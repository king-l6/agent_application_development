1. 文档更新了怎么办？

向量不会自动同步，需要手动处理。

方案做法适合场景
删旧重新编更新文档 → 删旧向量 → 重新 encode单篇更新，最常用
定时全量重建每天凌晨全部重新向量化文档稳定，实时性不高
事件驱动文档系统主动通知（webhook）有飞书/语雀/Confluence 平台

2. 怎么知道文档变没变？

两种检查方式，粒度不同：

方法粒度速度原理
mtime整个文件几毫秒读文件系统的修改时间戳，不改文件就不变
MD5每个内容块需要读内容算文本的哈希值，内容变了哈希就变
# mtime 检查（推荐，快） os.path.getmtime("退款政策.md") # → 秒级，一万个文件也就几秒 # MD5 检查（精准，需要读内容） hashlib.md5(文本.encode()).hexdigest()

3. 文档怎么"切块"存？

不是整个文档当一条向量，而是拆成块，每块有自己的向量和元数据：

原始文档（3000字） ├─ 块1（800字）→ 向量 + MD5 + section="退款政策" ├─ 块2（800字）→ 向量 + MD5 + section="退款政策" └─ 块3（800字）→ 向量 + MD5 + section="价格说明"

更新时，只重算真正变了的那几块，没变的跳过：

文档更新了 ├─ 块1（没变） → MD5 一样 → 跳过 ├─ 块2（改了） → MD5 不一样 → 重新向量化 └─ 块3（新加的） → 没有 MD5 → 新生成

4. 块之间怎么存层级关系？

靠元数据（metadata），和向量存在同一条记录里：

向量数据库里的一条记录： ┌───────────────────────────────────────┐ │ 向量 │ [0.12, 0.85, 0.33, ...] │ ← 算相似度用的 │ text │ "企业版退款政策是60天" │ ← 给LLM看的内容 │ doc_id │ help_center_001 │ ← 属于哪个文档 │ section │ 退款政策 │ ← 第几节 │ chunk_idx │ 3 │ ← 本节第几个块 │ md5 │ a1b2c3 │ ← 内容校验 │ version │ 2 │ ← 版本号 │ mtime │ 2026-06-18 14:30:00 │ ← 修改时间 └───────────────────────────────────────┘

搜索时可做过滤：只搜 section="退款政策" 的块

5. 文档中的表格

表格不是纯文字，不能直接向量化。需要先转成文字格式再存。

存法示例适合场景
Markdown 表格 | 套餐 | 价格 |\n|------|------|\n| 个人版 | 29元 | 原始文档就是 Markdown，最常用
JSON / CSV [{"套餐":"个人版","价格":"29元"}] 数据量大，LLM 需要做计算
HTML 表格 <table><tr><td>29元</td></tr></table> 原始文档就是 HTML
纯文字 个人版29元，企业版299元 最简单，但丢失了结构

关键：向量库存的是"存进去的原始文字"，返回也是原样返回。存 Markdown 格式，LLM 看到的就是表格，回答得更准确。

6. 文档中的图片

文本 embedding 模型不认识图片，不能直接向量化图片。两条路：

文字描述（你现在）直接存图片（进阶）
方法把图片转成文字描述再存用多模态模型（CLIP）直接把图片转向量
搜到返回文字描述，LLM 读文字后回答原图 URL 或 base64，LLM 直接看图
前提deepseek-v4-flash 够用需要多模态 LLM（GPT-4o、Claude Vision）
# 图片转文字描述再存（你现在走的路） 原始图片：[一张柱状图] 存入的文字： "2026年各季度营收对比图表：Q1 100万，Q2 150万，Q3 200万，Q4 300万" 搜索"营收趋势" → 匹配到这段文字 → LLM 回答

5. 核心结论：看场景

做法怎么做适用场景
直接覆盖 文档变了，删掉旧的向量，重新编码写入 只关心最新内容，旧的没用了
版本化 元数据加 version，新旧都保留，搜时只取最新 需要回溯旧回答、合规审计
# 版本化：元数据里加版本号 v1 → 向量 + text + version=1（原始） v2 → 向量 + text + version=2（更新） v3 → 向量 + text + version=3（再更新） # 搜索时过滤 version=3 就是当前版本

6. 一万个文档怎么检查更新？

最佳实践流程：

每天定时扫描
遍历所有文档，读 mtime（一万个几秒）
对比上次记录的 mtime
没变 → 跳过；变了 → 标记为待更新
对变了（没多少）的文档重新向量化
一般不多，真正变了可能只有几十个

不变的不处理，只处理真正变了的，效率最高。

7. PDF 文档怎么处理？

PDF 不是只有一种，先判断类型再选处理方法：

类型内部结构get_text() 能读到？应该用
电子版内嵌文字能，完整准确get_text() 直接提取
扫描件每页一张图片不能（空字符串）必须 OCR
混合型扫描图 + 上层文字层能，但可能乱序建议走 OCR

混合型最坑：表面能选中文字，实际提取出来顺序错乱、表格垮掉

判断逻辑：取前 3 页抽样，有任何一页文字少于 50 字就判为扫描件/混合型

def 判断pdf类型(pdf路径): doc = fitz.open(pdf路径) for i in range(min(3, len(doc))): if len(doc[i].get_text().strip()) < 50: return "扫描件/混合型" # 走 OCR return "电子版" # 走直接提取

完整处理流程：

① 打开 PDF
fitz.open() 读取文件
② 判断类型
前 3 页 get_text() 抽样 → 电子版 / 扫描件
③ 提取文字
电子版 → get_text() ；扫描件 → EasyOCR（dpi=200~300）
④ 输出 txt 校验
每页存为 page_N_ocr.txt，肉眼看有没有乱码
⑤ 按段落切块
按空行分段落，合并到接近 500 字上限，不切断语义
⑥ 向量化
SentenceTransformer → 768 维向量
⑦ 存入向量库
每条带 text + 向量 + 页码 + 块序号 + md5

切块策略对比：

策略做法结果
按字数硬切数到 500 字一刀切一句话被切成两半，搜不到
按段落合并段落（空行分隔）做最小单位，合并到接近上限语义完整，召回率高
# 段落优先切块 段落 = ["第一段...", "第二段...", "第三段..."] 缓存 = "" for 段 in 段落: if len(缓存) + len(段) < 500: 缓存 += 段 # 合并 else: 保存块(缓存) # 落盘 缓存 = 段 # 新块开始

参数建议值说明

9. 微调（Fine-Tuning）是什么？

用一个生活例子讲明白：

你有一个什么都会一点的助手（基础模型）。 他会说中文、会算数学、会写文章，但都是通用型的。 现在你让他专门做客服。 微调 = 给他做上岗培训 不是重新教他说话认字，而是让他熟悉你的业务流程。

10. LoRA（Low-Rank Adaptation）

LoRA 是微调的一种方式，核心思想是不训练整个模型，只训练一个"小插件"。

全量微调LoRA
类比让 100 项技能全部重学贴一张便签
训练参数100%（几亿 ~ 几千亿）0.1%~1%（几万 ~ 几十万）
显存需求（7B 模型）~56GB（需要 A100）~6GB（消费级显卡）
训练成本$30-40/次$2-5/次
效果基准接近 100%（通常差不到 1%）

类比：同一个浏览器，装不同插件做不同事。Chrome + AdBlock = 去广告，Chrome + 翻译 = 翻译。同一个模型 + 不同 LoRA adapter = 不同技能。

数学原理（不用怕，很简单）：

原始权重矩阵 W（4096×4096）= 1677 万参数 LoRA 不碰 W，在旁边插两个小矩阵： A（4096 × r）和 B（r × 4096），r 通常 = 8 或 16 训练参数量从 1677 万 → 3.2 万（减少 99%）
rank可训练参数占全模型比例适合场景
43,5840.89%简单分类
814,3363.47%单领域 Q&A
1628,6726.70%多领域指令跟随
3257,34412.56%复杂推理

几十 KB，就是你的"客服插件"

12. Loss（损失值）是什么？

Loss = 模型犯错的分数。越小越好。

Loss含义
8.69一开始，模型在乱猜
4.41学了几轮，方向对了但表述不准
1.8210 轮后，接近你的训练数据了
~0.5基本说对了，个别字不一样
0.0完美复刻你的数据

训练时看到 loss 不断下降，说明模型在学。如果 loss 不降了，说明学够了。

必须用 LoRA 的 3 种情况：

  • 输出格式固定（JSON / Markdown 表格 / 特定模板）
  • 风格要求极高（客服必须说"很抱歉"，Prompt 控制不稳定）
  • 蒸馏：大模型生成数据，小模型微调模仿，降成本 90%

15. 蒸馏（Distillation）

用大模型（教师）生成数据 → 小模型（学生）用 LoRA 模仿大模型的回答。

① 确定方向
客服 / 代码 / 翻译 / 文案 ... 只选一个方向
② 准备输入场景
列 50~200 个该方向常见的问题或需求
③ 大模型生成答案
deepseek-v4 或 GPT-4，用你的 API 批量跑
④ 得到训练数据
N 条"用户问 → 期望回答"的配对数据
⑤ LoRA 微调小模型
Qwen2.5-7B 或其他小模型，模仿大模型的风格
⑥ 部署使用
小模型 + adapter = 效果接近大模型，成本降低 90%+

数据格式（都一样）：

# 客服场景 {"prompt": "我的货丢了", "response": "很抱歉，请提供订单号"} # 代码场景 {"prompt": "写一个二分查找", "response": "def binary_search(...): ..."} # 翻译场景 {"prompt": "I love programming", "response": "我爱编程"}

关键认知：一个 LoRA adapter 只能管一件事。客服 adapter 只擅长客服，你问它数学它不懂。想要多能力就训练多个 adapter，切换着用。

16. 为什么不能完全复刻一个大模型？

一个简单的数学原因：

一个大模型 = 客服 + 翻译 + 写代码 + 写诗 + 数学 + 逻辑推理 + ... 蒸馏出来的小模型 = 只能学其中一样 想全部学 → 需要大模型全部的训练数据 → 几万亿 token → 几万张显卡 → 几千万成本 → 那还不如直接用大模型

所以蒸馏是针对性的：用到的能力全保留，用不到的丢掉。这就够了——你本来就不需要小模型会写诗，你只需要它会当客服。

14. 面试话术：RAG vs 微调 vs LoRA

什么时候 RAG，什么时候微调？ 知识/事实类问题 → RAG （文档里有的东西，检索出来给 LLM 看） 风格/格式/行为类问题 → 微调 （让模型以特定方式说话、输出特定格式） RAG 更便宜且可维护，微调是 RAG 解决不了时的选择。 先 prompt engineering → 再 RAG → 最后微调。 LoRA 的原理？ LoRA 是低秩适配（Low-Rank Adaptation），不更新原始权重 W， 而是在每层旁边插两个小矩阵 A 和 B，用 BA 的乘积来近似权重 更新量。训练参数量减少 99% 以上。训练完后可以把 BA 加回到 W 中合并，推理时零额外开销。 LoRA 的版本控制？ 原始模型永远不变。每个 adapter 就是一个版本， 切换 adapter 就是切换版本。不满意删掉 adapter 即可。

15. 面试怎么回答 PDF 处理？

我采用了分层策略：先检测 PDF 类型，电子版直接提取，扫描件才走 OCR。 因为 PDF 不是一个单一格式。90% 的文档是电子版，PyMuPDF 毫秒级 提取就够了。为了那 10% 的扫描件上全套版面分析模型，边际收益太低， 而且每次都要跑几秒。所以我先判断类型，不同情况走不同路径，这是 性价比最高的方案。 对于多栏排版、表格这类情况，目前的方案确实有局限。如果需要，可以 引入 Marker 或 Azure Document Intelligence 做版面感知提取，但那是 按需升级——不会一开始就上重武器，避免过度设计。 核心原则是：用 10% 的复杂度覆盖 90% 的场景，剩下的 10% 留到真正需要时再解决。

面试时这样说，体现的是架构思维，不是只会调 API。

总结

上午 · RAG 文档管理

  • 文档更新了，向量不会自动同步，需要手动触发重新编码
  • 用 mtime 快速检查哪些文档变了，比 MD5 快（不用读内容）
  • 文档切块存，每块带元数据（doc_id, section, version），搜索时可过滤
  • 表格要转成文字再存（Markdown / JSON / HTML 都行），图片要转成文字描述
  • 要不要版本，取决于你需不需要找回旧答案。不需要就直接覆盖

下午 · PDF 处理

  • PDF 分三种：电子版（直接提取）、扫描件（必须 OCR）、混合型（建议 OCR）
  • 先判断类型再走不同路径，用 10% 的复杂度覆盖 90% 的场景

下午 · LoRA 微调 + 蒸馏

  • 微调 = 上岗培训，LoRA = 只贴便签不重新学
  • LoRA 只训练 0.1%~1% 的参数，效果接近全量微调，成本低 95%
  • 训练出来的是 adapter（几十 KB），原始模型永远不变
  • 同一个模型可以加载不同 adapter 做不同任务，切换即回退
  • Loss 不断下降 = 模型在学；label masking = 只学回答部分不学提问
  • 0.5B 模型 + 5 条数据能理解原理，生产至少需要 7B + 100 条以上
  • RAG = 开卷考试翻书回答（知识类），LoRA = 改变能力本身（风格类）
  • 蒸馏 = 大模型生成数据 → 小模型微调模仿 → 降成本 90%+
  • 一个 LoRA adapter 只能管一件事，多能力需要多个 adapter 切换
