5. Embeddings Phase 11-04

  • Embedding = 把文字转成一串数字（向量）
  • 意思相近的文字 → 向量距离近 → 余弦相似度高
  • all-MiniLM-L6-v2 是英文模型，中文用 shibing624/text2vec-base-chinese
  • 维度越多表达越精细：384(轻量) / 768(中文) / 1536(OpenAI) / 3072(最强)
# 余弦相似度公式（只看方向是否一致） 相似度 = (向量A · 向量B) / (|向量A| × |向量B|)

6. 上下文工程 Phase 11-05

  • 上下文窗口是稀缺资源，不是越大越好
  • Lost-in-the-Middle — 模型最关注开头和结尾，中间容易忽略
  • 三明治原则：重要信息放开头(System)和结尾(最后一条消息)
  • 对话历史太长要截断或摘要
System（最重要） → 工具定义 → 检索文档 → 对话历史 → 当前问题 → 最后指令

7. RAG 检索增强生成 Phase 11-06

  • RAG 四步：文档向量化 → 问题转向量 → 搜索最相似文档 → 文档+问题一起给模型
  • 比微调便宜百倍，数据随时更新，可追溯来源
  • 基础 RAG 的问题："多少钱"搜不到"价格"文档（语义相似度不够）
# 搜索的核心代码 相似度 = (doc_vectors @ q_vec.T).flatten() / ( np.linalg.norm(doc_vectors, axis=1) * np.linalg.norm(q_vec) ) 排名 = np.argsort(相似度)[::-1] # 从高到低排序

9. Function Calling Phase 11-09

  • 模型输出结构化 JSON 说"我要调什么函数、参数是什么"
  • 你的代码执行工具，模型只做决策
  • 完整的6步循环：定义工具 → 传给模型 → 模型决定用哪个 → 代码执行 → 结果还回 → 模型最终回答
# 定义工具 工具 = [{ "name": "get_weather", "description": "获取某城市的天气", "input_schema": { "properties": {"city": {"type": "string"}}, "required": ["city"], } }] # 模型返回ToolUseBlock → 代码执行 → 结果返回

10. 缓存与成本优化 Phase 11-11

  • 40-60%的提问是同一意思换说法 → 语义缓存
  • 问题转向量，相似度超过阈值就返回缓存结果，不调API
  • 大量节省成本
class 语义缓存: def 获取(self, 问题): for 缓存向量, 回答 in self.缓存: if 相似度 > 阈值: return 回答 # 命中缓存，省一次API return None def 保存(self, 问题, 回答): self.缓存.append((向量, 回答))
