# 奖励模型设计器

在构建 RLHF 流水线以将语言模型与目标行为（有用性、编码能力、安全性、诚实性）对齐时，使用此框架来设计数据收集协议、训练奖励模型和配置 PPO。

## 输入要求

提供：
- **目标行为**（例如，"有用且无害的助手"、"Python 编码专家"、"带安全机制的医疗问答"）
- **基础模型**（例如，SFT 后的 Llama 3 8B、Mistral 7B Chat）
- **奖励模型大小**（通常与策略模型相同或更大）
- **标注预算**（可用的人工小时数或比较对数）
- **计算预算**（奖励模型训练 + PPO 的 GPU 小时数）

## 步骤 1：偏好数据收集

### 标注协议

1. **提示选择**：从 SFT 训练分布中采样，加上分布外的提示（10-20% 的新提示）
2. **回答生成**：使用 SFT 模型以不同温度（0.3、0.7、1.0）为每个提示生成 2-4 个回答
3. **比较格式**：向标注员展示恰好 2 个回答，询问"哪个回答更好？"
4. **标准模板**：为你的用例定义"更好"的含义

### 标准模板

| 标准 | 权重 | 描述 |
|------|------|------|
| 有用性 | 40% | 是否完整且正确地回答了问题？ |
| 无害性 | 25% | 是否避免了有害、偏见或误导性的内容？ |
| 诚实性 | 20% | 是否承认不确定性而非产生幻觉？ |
| 简洁性 | 15% | 回答的长度是否适合问题？ |

根据你的用例调整权重。编码助手可能将正确性权重设为 60%，简洁性设为 20%。

### 数据规模指南

| 规模 | 比较对 | 标注员小时数 | 预期奖励模型准确率 |
|------|--------|-------------|-------------------|
| 最小可行 | 5,000-10,000 | 400-800 | 60-65% |
| 生产 v1 | 20,000-50,000 | 1,600-4,000 | 65-72% |
| 生产 v2 | 100,000-500,000 | 8,000-40,000 | 72-78% |

InstructGPT 使用来自 40 名承包商的 33,000 条比较。Anthropic 的初始论文使用来自 20 名标注员的 22,000 条。标注员间一致性通常为 70-75%——奖励模型无法超过人类一致性水平。

### 质量控制

- **一致性过滤**：丢弃少于 70% 标注员同意的比较对
- **标注员校准**：在实际标注前，使用已知优差的样本进行校准轮次
- **偏差检测**：监控标注员是否一致偏好更长的回答、正式语言或特定模式
- **对抗性样本**：包含 5-10% 的样本，用于检测未仔细阅读的标注员

## 步骤 2：奖励模型架构

### 架构决策

| 决策 | 建议 | 理由 |
|------|------|------|
| 基础架构 | 与策略相同的 transformer | 从 SFT 检查点初始化权重，提供强大的初始特征 |
| 输出头 | 从最后隐藏状态做线性投影 | 从最完整的位置表示中获得标量奖励 |
| 模型大小 | >= 策略模型大小 | 较小的奖励模型会产生不可靠的信号，破坏 PPO 稳定性 |
| 初始化 | SFT 检查点 + 新输出头 | 预训练特征已捕捉语言质量 |

### 训练配置

| 参数 | 范围 | 说明 |
|------|------|------|
| 学习率 | 1e-5 到 5e-5 | 比 SFT 低，因为任务更简单 |
| Epochs | 1-3 | 在有限的比较数据下，过拟合是主要风险 |
| 批大小 | 64-256 | 每个"样本"是一个对，因此有效数据量是 2 倍 |
| 损失函数 | Bradley-Terry: -log(sigmoid(r_preferred - r_rejected)) | 成对比较的标准方法 |
| 验证集比例 | 10-20% | 监控在保留对上的准确率 |

### 评估指标

1. **成对准确率**：奖励模型正确排名的保留偏好对比例？目标：> 65%
2. **差值分布**：绘制 (r_preferred - r_rejected) 的分布。应集中在 0 以上，负值很少。
3. **校准度**：sigmoid(r_preferred - r_rejected) 是否接近实际的人类偏好概率？
4. **分布外泛化**：在与训练分布不同的提示上测试。准确率下降应 < 10%。

## 步骤 3：PPO 配置

### 超参数

| 参数 | 典型值 | 过高影响 | 过低影响 |
|------|--------|---------|---------|
| KL 系数 (beta) | 0.01-0.05 | 模型几乎不学习，太接近 SFT | 奖励破解，输出退化 |
| 学习率 | 5e-6 到 3e-5 | 训练不稳定，发散 | 收敛慢，浪费计算 |
| 裁剪比 (epsilon) | 0.1-0.3 | 更新幅度大，可能不稳定 | 非常保守，学习缓慢 |
| 每批次 PPO epochs | 1-4 | 对当前批次过拟合 | 未充分利用每个批次 |
| 生成批大小 | 128-512 | 内存问题 | 梯度估计有噪声 |
| 最大回答长度 | 256-1024 | 生成慢，内存问题 | 截断有用的回答 |

### 监控仪表板

PPO 训练期间跟踪以下指标：

1. **平均奖励**：应在训练期间上升。平稳是可接受的；下降意味着不稳定。
2. **KL 散度**：应保持在 10-20 nats 以下。飙升 = 奖励破解。
3. **回答长度**：应保持稳定。单调上升 = 冗长性奖励破解。
4. **熵**：token 分布熵应缓慢下降。快速下降 = 模式崩溃。
5. **奖励模型一致性**：用奖励模型给 PPO 回答打分，一致性应提高。

### PPO 期间的红旗

| 症状 | 可能原因 | 修复 |
|------|---------|------|
| 奖励上升但输出质量下降 | 奖励破解 | 增加 KL 系数，在对抗性样本上重新训练奖励模型 |
| KL 散度爆炸 | 学习率过高或 KL 系数过低 | 降低 lr，增加 beta |
| 回答长度单调增长 | 奖励模型奖励冗长 | 在奖励中添加长度惩罚，用长度控制的偏好对重新训练奖励模型 |
| 所有回答变得相同 | 模式崩溃 | 增加生成温度，减少 PPO epochs |
| 奖励剧烈波动 | PPO 不稳定 | 降低学习率，增加裁剪比 |

## 步骤 4：端到端验证

在部署 RLHF 训练的模型前：

1. **A/B 测试 vs SFT**：在 200+ 测试提示上运行 SFT 和 RLHF 模型。让 3+ 评估员比较回答。RLHF 模型应在 > 60% 的情况下获胜。
2. **安全评估**：在已知的对抗性提示上测试（越狱、有害请求）。RLHF 模型应适当拒绝。
3. **回归检查**：运行标准基准测试（MMLU、HumanEval、MT-Bench）以确认 RLHF 模型没有丢失核心能力。
4. **遗忘检查**：在通用文本语料上测量困惑度。相比 SFT 模型的增加应 < 10%。
5. **长度分析**：比较 SFT 和 RLHF 模型的平均回答长度。如果 RLHF 长 > 50%，奖励模型可能存在冗长偏差。
