# DualPipe 并行

> DeepSeek-V3 在 2,048 块 H800 GPU 上训练，MoE 专家分散在节点间。跨节点专家全对所有通信每消耗 1 GPU 小时的计算需要 1 GPU 小时的通信。GPU 有一半时间是空闲的。DualPipe（DeepSeek，2024 年 12 月）是一种双向流水线，它将前向和后向计算与它们触发的全对所有通信重叠。气泡减少，吞吐量提升，而保持两份模型参数副本（"Dual" 的名称由来）在专家并行已经将专家分散到各个 rank 时是廉价的。本节课是一个"学习"类型的演练，讲解 DualPipe 实际做什么，以及 Sea AI Lab 的 DualPipeV 改进如何以略微更紧的气泡为代价去掉了 2 倍参数成本。

**类型：** 学习
**语言：** Python（标准库，调度模拟器）
**前置知识：** 阶段 10 · 05（分布式训练，FSDP，DeepSpeed），阶段 10 · 14（开源模型架构和 MoE）
**时间：** ~60 分钟

## 学习目标

- 命名 DualPipe 前向-后向块的四个组成部分，并解释为什么每个部分都有自己重叠窗口。
- 解释大规模下的流水线气泡问题，以及"无气泡"在实践与营销中的区别。
- 为 8 个 PP rank 和 16 个微批次手动追踪 DualPipe 调度，并确认前向和反向流相互填充对方的空闲槽位。
- 陈述 DualPipeV（Sea AI Lab，2025）所做的权衡：在专家并行未激活时，以略微更大的气泡为代价去掉 2 倍参数复制。

## 问题

在 2k 块 H800 GPU 上训练 671B MoE 模型遇到三个复合瓶颈：

1. **内存压力。** 每个 GPU 持有模型的一部分。在 128 个头、61 层、序列长度 8k 下的激活内存是巨大的。
2. **流水线气泡。** 传统的流水线并行（GPipe、1F1B）在 GPU 等待其阶段的输入或梯度时使 GPU 处于空闲状态。在 8 个阶段下，即使使用 1F1B 调度，大约 12% 的 GPU 时间可能是气泡。
3. **跨节点全对所有。** 具有专家并行的 MoE 将专家分散到各个节点。每个前向传递触发一个全对所有来将 token 分派到它们的专家，以及另一个来合并。在 2k GPU 下，这很容易达到 1:1 的计算与通信比。

这些问题中的每一个都有单独的解决方案：用于内存的梯度检查点、用于流水线气泡的 Zero Bubble（Sea AI Lab，2023）、用于全对全的专家并行通信内核。DualPipe 所做的是让它们协同工作。该调度在单个前向-后向块内重叠计算和通信，同时从流水线的两端注入微批次，并利用生成的调度在计算窗口内隐藏全对所有。

报告结果：几乎消除了流水线气泡，在 DeepSeek-V3 的 14.8T token 训练运行中 GPU 利用率超过 95%。

## 概念

### 流水线并行复习

将 N 层模型分割到 P 个设备上。设备 `i` 持有层 `i * N/P .. (i+1) * N/P - 1`。一个微批次依次流过设备 0 到 P-1，然后从 P-1 到 0 反向。每个设备只能在前一个设备发送其输出后才能开始其前向阶段，并且只能在下游设备发送上游梯度后才能开始反向。

GPipe（Huang 等人，2019）一次调度一个微批次，浪费了大部分 GPU 时间。1F1B（Narayanan 等人，2021）为多个微批次交错前向和后向传递。Zero Bubble（Qi 等人，2023）将反向传递分为两部分——输入反向（B）和权重反向（W）——并调度它们以填充气泡。在 Zero Bubble 之后，流水线几乎是紧的。

DualPipe 是下一步。它在之上添加了两个想法：

### 想法 1：块分解

每个前向块被分割为四个组件：

- **注意力。** Q/K/V 投影、注意力、输出投影。
- **全对所有分派。** 将 token 发送到其专家的跨节点通信。
- **MLP。** MoE 专家计算。
- **全对所有合并。** 将专家输出带回的跨节点通信。

一个反向块添加每个组件的梯度版本。DualPipe 调度它们使得全对所有分派与下一个块的注意力计算并行发生，全对所有合并与后续块的 MLP 计算并行发生。

### 想法 2：双向调度

大多数流水线调度从阶段 0 注入微批次并流向阶段 P-1。DualPipe 从**两端**注入微批次。阶段 0 看到源自那里的前向微批次；阶段 P-1 也看到源自那里的前向微批次。两股流在中间相遇。

为了使其工作，设备 `i` 必须持有早期流水线层 `i` **和**后期流水线层 `P - 1 - i`。这就是 DualPipe 的"双"部分：每个设备保留两份它需要服务的模型层副本（每个方向一份）。在 DeepSeek-V3 的规模下，这是 2 倍的参数复制成本。这是可以承受的，因为专家并行已经将 MoE 专家分散得如此之薄，以至于复制两倍的非专家层是小问题。

关键的是，一个方向的前向流和另一个方向的反向流正好重叠在单向调度中气泡所在的位置。气泡消失了。

### 手动追踪的调度

考虑 P = 4 个 rank，8 个微批次，分为 4 前向 / 4 反向。时间从左到右移动；行是设备 rank。

```
           时间 →
rank 0:  F1 F2 F3 F4  F5R F6R F7R F8R  B1 B2 B3 B4  ...
rank 1:     F1 F2 F3  F4/F5R F6R F7R   B1 B2 ...
rank 2:        F1 F2  F3/F5R F4/F6R    B1 ...
rank 3:           F1  F2/F5R F3/F6R    ...
```

阅读"F4/F5R"符号：rank 1 在同一时间槽中运行微批次 4 的前向（在流水线中从左到右）**和**微批次 5 的前向（从右到左）。这就是"双向"在操作上的含义。

在 rank 2 处，交叉流更早重叠；在 rank 0 和 P-1 处，它们最晚重叠。在调度的稳定中间阶段，每个 rank 运行一个方向的前向与另一个方向的反向重叠。计算繁忙。前向传递的全对所有分派隐藏在反向计算内部。全对所有合并在前向计算内部隐藏。气泡被挤出。

### 气泡核算

标准 1F1B 流水线气泡（每个 rank 浪费的时间）：

```
bubble_1F1B = (P - 1) * forward_chunk_time
```

Zero Bubble 改进将其降低但未到零。DualPipe 在稳定阶段具有零气泡，如果微批次数量能被 2 倍流水线深度整除。在稳定阶段之外（预热和冷却），存在一些气泡，但它不会随微批次数量增长——论文强调的一个关键属性。

用营销术语来说："无气泡"。用技术术语来说：气泡不随微批次数量增长。Sea AI Lab 的后续分析（DualPipeV / Cut-in-half）表明，只有在专家并行不是瓶颈时才完全无气泡；在 EP 驱动的全对所有情况下，总是存在一些调度折中。

### DualPipeV——改进

Sea AI Lab（2025）观察到，当 EP 通信重叠不是关键点时，2 倍参数复制是浪费的。他们的 DualPipeV 调度将双向注入折叠为在单个参数副本上运行的"V 形"调度。气泡比 DualPipe 的稍大，但内存节省显著。DeepSeek 在其开源的 DualPipe 实现中采用了 DualPipeV 作为 EP 关闭模式。

权衡：

| 特性 | DualPipe | DualPipeV | 1F1B | Zero Bubble |
|---------|---------|-----------|------|------------|
| 每个设备的参数副本 | 2 | 1 | 1 | 1 |
| 气泡与微批次关系 | 恒定 | 小幅增长 | 增长 | 增长 |
| 计算-通信重叠 | 完全 | 部分 | 最小 | 部分 |
| 何时使用 | EP 密集型 MoE | 密集或 EP 轻量 | 基线 | 任何流水线 |

### 对 14.8T token 运行的意义

DeepSeek-V3 的预训练在约 2.8M GPU 小时内消耗了 14.8T token，使用 2,048 块 H800 GPU。使用天真的 1F1B，他们将损失 12-15% 的流水线气泡——340-420K GPU 小时，足够训练一个完整的 70B 模型。DualPipe 恢复了大部分。没有内部日志很难直接量化贡献，但论文中声称整个训练的平均 GPU 利用率超过 95%。

对于较小的运行（低于 1k GPU），DualPipe 是过度的——流水线气泡相对于总成本较小，密集模型训练很少遇到全对所有瓶颈。对于前沿 MoE 训练在数千 GPU 规模下，它实际上是必需的。

### 在栈中的位置

- 与 **FSDP**（阶段 10 · 05）互补。FSDP 在 rank 间分片模型参数；DualPipe 在 rank 间调度计算。它们结合使用。
- 与 **ZeRO-3** 梯度分片兼容。两份副本复制的簿记需要与 ZeRO 的分片梯度配合。

## 使用

`code/main.py` 是一个流水线调度模拟器。它接收 `(P, n_micro_batches, schedule)` 并为 1F1B、Zero Bubble、DualPipe 和 DualPipeV 打印稳定阶段利用率。它是一个教学工具——数字与论文中的定性声明一致，不是关于生产中测量到的加速比的声明。

该模拟器的价值：用不同的 P 和微批次数量运行它，观察 1F1B 的气泡分数如何增长而 DualPipe 不增长。

真实训练运行的集成考虑：

- 选择能干净地分割到你的微批次数量中的流水线并行深度。
- 确保你的专家并行网格支持双向全对所有。DeepSeek 的内核是参考。
- 预计第一次在调度本身上花费一周的调试时间。簿记很繁琐。
- 监控每个 rank 的 GPU 利用率，而不仅仅是聚合。DualPipe 的收益来自于收紧落后者的时间。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------------|------------------------|
| 流水线气泡 | "每个 rank 的空闲时间" | 因为流水线阶段在等待其输入或梯度而浪费的 GPU 周期 |
| 1F1B | "默认流水线调度" | 一个前向 / 一个反向交错调度；DualPipe 超越的基线 |
| Zero Bubble | "Sea AI Lab 2023" | 将反向拆分为 B（输入梯度）和 W（权重梯度）；几乎完全收紧流水线 |
| DualPipe | "DeepSeek-V3 调度" | 双向流水线 + 计算-通信重叠；气泡不随微批次数量增长 |
| DualPipeV | "Cut-in-half" | V 形改进，以略微更大的气泡为代价去掉 2 倍参数复制 |
| 块 | "流水线工作单元" | 一个微批次通过一个流水线阶段的前向或后向传递 |
| 全对所有分派 | "将 token 发送到专家" | 将 token 路由到其分配的 MoE 专家的跨节点通信 |
| 全对所有合并 | "将专家输出带回" | MLP 后收集专家输出的跨节点通信 |
| 专家并行（EP） | "跨 GPU 的专家" | 在 rank 间分片 MoE 专家，使得不同 GPU 持有不同的专家 |
| 流水线并行（PP） | "跨 GPU 的层" | 在 rank 间分片模型层；DualPipe 调度的维度 |
| 气泡分数 | "浪费的 GPU 时间" | (bubble_time / total_time)；DualPipe 驱动趋向于零的分数 |
