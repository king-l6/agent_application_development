# 原生稀疏注意力（DeepSeek NSA）

> 在 64k token 长度下，注意力机制消耗了 70-80% 的解码延迟。每个开源模型实验室都有自己的解决方案。DeepSeek 的 NSA（ACL 2025 最佳论文）是脱颖而出的那个：三个并行注意力分支——压缩粗粒度 token、选择性保留细粒度 token、以及用于局部上下文的滑动窗口——通过一个学习得到的门控机制组合在一起。它在硬件上对齐（内核友好），原生可训练（在预训练中生效，而非在推理时后装），并且在 64k 解码时比 FlashAttention 更快，同时匹配或超越全注意力的质量。本节课端到端地构建了这三个分支，并展示了为什么这种稀疏性是端到端可微的。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 7 · 12（KV 缓存，Flash Attention），阶段 7 · 15（注意力变体），阶段 10 · 16（微分注意力）
**时间：** ~60 分钟

## 学习目标

- 陈述 NSA 的三个注意力分支以及每个分支捕获的内容。
- 解释为什么 NSA 是"原生可训练的"，而先前的稀疏注意力方法仅限于推理时使用。
- 计算在 64k 上下文中，NSA 相对于全注意力的注意力计算节省量（作为压缩块大小和选择 top-k 的函数）。
- 在短合成序列上使用标准库 Python 实现三分支组合，并验证门控权重行为正常。

## 问题

序列长度 N 下的全注意力需要 `O(N^2)` 时间和每层 `O(N)` KV 缓存。在 64k token 下，计算和内存带宽的数字是灾难性的。根据 NSA 论文的测量理论估计：注意力占 64k 总解码延迟的 70-80%。所有下游指标——TTFT、token/秒、每百万 token 成本——都受注意力成本主导。

稀疏注意力是显而易见的答案。先前的尝试分为两类。固定模式稀疏性（滑动窗口、跨步、块局部）会丢弃信息，并在长距离召回任务上失败。推理时稀疏性（KV 缓存剪枝、H2O、StreamingLLM）应用于在密集注意力上预训练的模型，只能恢复潜在加速的一小部分，因为模型从未被要求通过稀疏模式路由信息。

原生稀疏注意力（Yuan 等人，DeepSeek + 北京大学 + 华盛顿大学，ACL 2025 最佳论文，arXiv:2502.11089）两者兼具：模型在预训练期间学习的稀疏模式，实现为内核对齐的算法，在推理时真正实现了计算节省。两年后，NSA 或其直接后代将成为每个前沿长上下文模型的默认注意力机制。

## 概念

### 三个并行分支

对于每个查询，NSA 运行三次注意力，针对 KV 缓存的三种不同视角：

1. **压缩分支。** Token 被分组为大小为 `l` 的块（通常为 32 或 64）。每个块通过一个小型学习到的 MLP 压缩为单个摘要 token。查询在这些压缩 token 上进行注意力计算，获得整个序列的粗粒度视图。

2. **选择分支。** 利用压缩分支的注意力分数，识别与当前查询最相关的 top-k 个块。读取这些块中的细粒度（未压缩）token，查询对所有 token 进行注意力计算。可以将压缩分支的注意力视为选择的路由信号。

3. **滑动窗口分支。** 查询对最近的 `W` 个 token（通常为 512）进行注意力计算以获取局部上下文。此分支捕获结构密集的短距离模式（语法、局部共指），这些模式是其他两个分支可能遗漏的。

三个分支的输出通过一个学习到的每个位置门控进行组合：

```
out = g_cmp * out_cmp + g_sel * out_sel + g_win * out_win
```

`g_cmp, g_sel, g_win` 是来自查询的小型 MLP 的门控权重。它们不必和为 1——它们可以独立地对分支进行加权。

### 为什么这是"原生可训练的"

选择步骤（top-k 块）是离散的。离散操作会破坏梯度流。先前的稀疏注意力工作要么跳过了通过选择的反向传播（限制了训练），要么使用了在推理时无法实现真正稀疏性的连续松弛方法。

NSA 绕过了这一点：压缩分支的注意力本身就是对整个序列的可微粗粒度注意力。top-k 操作仅复用压缩分支中的最高注意力分数来选取要加载的细粒度块。梯度流经压缩分支的分数（它影响压缩输出和选择逻辑），所选块对最终输出的贡献也是可微的。不可微的 `top_k` 操作在前向计算图中是一个无操作——它仅控制从内存中加载哪些块。

这就是为什么 NSA 可以端到端地用于预训练。模型学会了联合地通过三个分支路由信息，产生一个稀疏模式，在推理时真正实现承诺的加速。

### 硬件对齐的内核

NSA 的内核专为现代 GPU 内存层次结构设计。该内核按 GQA 组加载查询（外层循环），为每组获取相应的稀疏 KV 块（内层循环），并在 SRAM 上运行注意力。由于每个查询组看到相同的选定块（选择是按查询组进行的，而不是按查询头），KV 加载在组间分摊。算术强度保持在高水平。

该论文报告称，Triton 内核在 64k 解码上比 FlashAttention 快 9 倍，且加速比随序列长度增长。前向和反向内核都已提供。

### 计算预算

设 `N` 为序列长度，`l` 为压缩块大小，`k` 为 top-k 选择数量，`w` 为滑动窗口，`b` 为选定块大小（通常等于 `l`）。

- 压缩分支：每个查询 `O(N/l)` 个键，总计 `O(N * N / l)`。
- 选择分支：每个查询 `O(k * b)` 个键，总计 `O(N * k * b)`。
- 滑动分支：每个查询 `O(w)` 个键，总计 `O(N * w)`。

总计：`O(N * (N/l + k*b + w))`。

当 `N = 64k, l = 64, k = 16, b = 64, w = 512` 时：每个查询成本为 `1000 + 1024 + 512 = 2536` 个键。全注意力为 `64000` 个键。计算量减少 25 倍。

当 `N = 128k, l = 64, k = 16, b = 64, w = 512` 时：每个查询成本为 `2000 + 1024 + 512 = 3536` 个键。全注意力为 `128000` 个键。减少 36 倍。收益随序列长度增长，这正是关键所在。

### 对比表

| 方法 | 可微 | 真正推理加速 | 长距离召回 |
|--------|---------------|----------------------|-------------------|
| 仅滑动窗口 | 是 | 是 | 失败 |
| 跨步 / 块稀疏 | 是 | 是 | 部分 |
| KV 剪枝（H2O, StreamingLLM） | N/A（推理时） | 是 | 部分 |
| MoBA（Moonshot） | 部分 | 是 | 良好 |
| NSA | 是（原生） | 是（64k 下 9 倍） | 匹配全注意力 |

MoBA（Moonshot, arXiv:2502.13189）同期发表，采用类似的三比一更好的方法，将 MoE 原则应用于注意力块。NSA 和 MoBA 是 2026 年长上下文预训练中需要了解的两种架构。

## 构建

`code/main.py` 在短合成序列上实现了三个分支，并展示了：

- 压缩 MLP（为教学清晰起见，使用简单的均值池化基线；真实的 NSA 使用学习到的 MLP）。
- 由压缩分支分数驱动的 top-k 块选择。
- 对最后 `w` 个 token 的滑动窗口注意力。
- 门控组合。
- 与全注意力比较的计算量统计。

### 步骤 1：将 token 压缩为块

```python
def compress(K, l):
    n = len(K)
    n_blocks = (n + l - 1) // l
    out = []
    for b in range(n_blocks):
        start, end = b * l, min((b + 1) * l, n)
        block = K[start:end]
        summary = [sum(row[d] for row in block) / len(block) for d in range(len(K[0]))]
        out.append(summary)
    return out
```

### 步骤 2：压缩分支注意力

对查询与压缩键运行 softmax 注意力。压缩分支的分数兼作 top-k 选择的信号。

### 步骤 3：top-k 块选择

选取分数最高的 `k` 个压缩块的索引。加载这些块中原始的未压缩 token 并对它们运行注意力。

### 步骤 4：滑动窗口注意力

获取最后 `w` 个 token 并对它们运行标准注意力。

### 步骤 5：门控 + 组合

查询上的小型 MLP 产生三个门控权重。最终输出是三个分支输出的加权和。

### 步骤 6：计算量统计

打印每个分支每个查询关注的键数以及总数。与 `N`（全注意力）进行比较。在 `l = 32, k = 4, w = 128` 的 1024 token 合成数据上，NSA 每个查询关注 `32 + 128 + 128 = 288` 个键，而全注意力为 1024 个——减少了 3.5 倍。

## 使用

NSA 正在 DeepSeek 自己的长上下文预训练流程中交付。截至 2026 年 4 月在公共推理栈中的集成状态：

- **DeepSeek 内部**：原生支持，已发布权重使用 NSA 或其继任者 DSA（DeepSeek 稀疏注意力）。
- **vLLM**：正在为 DeepSeek-V3.x 权重开发实验性 NSA 支持。
- **SGLang**：NSA 基准测试已发布，生产路径跟随 vLLM。
- **llama.cpp / CPU**：不支持；内核分解的开销在 CPU 吞吐量下不值得。

何时使用 NSA：

- 预训练或继续训练运行，目标为 64k 以上上下文且有大量计算预算。
- 推理 DeepSeek 自己的长上下文检查点。权重是 NSA 原生的。

何时不使用：

- 服务于现有的密集注意力预训练模型。没有继续训练就无法改造 NSA。
- 上下文低于 16k。三分支的开销超过了节省。
- 批量 1 的交互式聊天。延迟敏感的解码虽有收益，但仅在长上下文中。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------------|------------------------|
| 压缩分支 | "粗粒度视图" | 对块平均键的注意力，以每个查询 O(N/l) 个键提供全局上下文 |
| 选择分支 | "Top-k 块" | 对压缩分支分数最高的 `k` 个块的细粒度注意力 |
| 滑动窗口 | "局部上下文" | 对最后 `W` 个 token 的注意力，用于短距离模式 |
| 原生可训练性 | "预训练时开启稀疏性" | 稀疏模式在预训练期间学习，而非推理时后装 |
| 压缩块大小 l | "粗粒度视图的组大小" | 多少个 token 合并为一个摘要；通常 32-64 |
| Top-k | "保留的块数" | 读取其未压缩 token 的压缩块数量；通常 16 |
| 滑动窗口 W | "局部注意力半径" | 通常为 512；更短会损害局部连贯性，更长会浪费计算 |
| 分支门控 | "如何混合三个分支" | 每个位置 MLP 输出的权重，对三个分支的贡献进行加权 |
| 硬件对齐 | "内核友好的稀疏性" | 选择稀疏模式使得实际 GPU 内核实现理论加速 |
| DSA | "NSA 的继任者" | DeepSeek 稀疏注意力，DeepSeek 系列中继 NSA 之后的架构 |
