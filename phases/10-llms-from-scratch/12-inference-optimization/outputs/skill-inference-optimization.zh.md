# LLM推理优化模式

两个阶段：预填充（计算密集型，并行）和解码（内存密集型，顺序）。
每个优化都针对其中一个或两个阶段。

```
请求 -> 预填充（处理提示词） -> 解码（生成token） -> 响应
             |                         |
         计算密集型               内存密集型
         优化：融合、              优化：批处理、
         前缀缓存                  量化、推测
```

## 决策框架

### 步骤1：确定你的瓶颈

测量工作负载的ops:byte比率：

| ops:byte | 瓶颈 | 优化什么 |
|----------|------|---------|
| < 50 | 内存 | 量化KV缓存，增加批次大小 |
| 50-200 | 过渡中 | 两者都重要，从批处理开始 |
| > 200 | 计算 | 内核融合，张量并行，FP8 |

### 步骤2：选择你的引擎

- **默认**：vLLM（最广泛的模型支持，PagedAttention，兼容OpenAI的API）
- **多轮/结构化输出**：SGLang（RadixAttention前缀缓存，约束解码）
- **最大NVIDIA吞吐量**：TensorRT-LLM（内核融合，H100上的FP8）

### 步骤3：按顺序应用优化

1. **KV缓存** -- 一直开启，无副作用
2. **连续批处理** -- 一直开启，无副作用（vLLM/SGLang默认启用）
3. **前缀缓存** -- 如果有共享系统提示词则启用（大多数聊天机器人有）
4. **量化** -- KV缓存INT8/FP8减少2-4倍内存，质量损失最小
5. **推测解码** -- 当延迟比吞吐量更重要时添加
6. **张量并行** -- 当模型不能装在一块GPU上时跨GPU分割

## KV缓存内存公式

```
每token = 2 * 层数 * KV头数 * 头维度 * 每参数字节数
总计 = 每token * 序列长度 * 并发用户数
```

常见模型的快速参考（BF16）：

| 模型 | 每token | 100用户 @ 4K |
|-------|---------|--------------|
| Llama 3 8B | 32 KB | 12.5 GB |
| Llama 3 70B | 320 KB | 125 GB |
| Llama 3 405B | 504 KB | 197 GB |

## 推测解码检查清单

- 草稿模型应比目标小5-10倍（例如，8B草稿用于70B）
- 接受率 > 70% 才能有意义地加速
- 最适用于可预测的文本（代码、结构化输出、自然语言）
- 最不适用于创造性/高采样任务（低温有帮助）
- EAGLE > 草稿-目标 > n-gram 适用于大多数工作负载

## 常见错误

- 以batch=1运行解码（内存密集型，GPU 95%计算空闲）
- 分配连续的KV缓存块（使用PagedAttention，获得接近零的浪费）
- 当80%的请求共享相同系统提示词时忽略前缀缓存
- 为模型权重过度配置GPU内存，不给KV缓存留空间
- 测量吞吐量而不测量延迟（10秒TTFT下的高吞吐量是无用的）
- 在高温度下使用推测解码（接受率降至50%以下）

## 监控检查清单

- 首个token时间（TTFT）：预填充延迟，交互式使用目标 < 500ms
- token间延迟（ITL）：解码速度，流式传输目标 < 50ms
- 吞吐量（token/秒）：所有并发用户的总计
- KV缓存利用率：已分配缓存的在用百分比
- 批处理利用率：每次迭代填充的批处理槽百分比
- 队列深度：等待批处理槽的请求数
