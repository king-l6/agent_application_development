## 问题

在70B模型上的自回归解码在H100上可能以每秒35个token的速度运行。GPU远未饱和。内存带宽是天花板：每个token从HBM加载70B的权重，执行一步算术，产生一个浮点数。计算单元大部分空闲。

推测解码将其转化为一个你可以实际解决的吞吐量问题。一个廉价的草稿在`N`次小型前向传播中提出`N`个token。验证者在前缀加上所有`N`个草稿上运行一次。如果验证者在位置`i`的分布与草稿一致（在我们将精确化的统计意义上），我们接受；如果不一致，我们拒绝并从残差分布中采样一个修正。一次大型模型前向传播产生多达`N+1`个接受token，而不是一个。

重要的定理是Leviathan、Kalman、Matias（ICML 2023）：输出分布与直接从验证者采样产生的分布相同。不是近似。完全相同。这就是推测解码在生产中可接受的整个原因——它是一种纯粹的延迟优化，没有质量权衡。

阶段7 · 课程16给你的是数学。本课程给你的是训练栈。一个好的草稿比一个廉价的草稿多提供2倍的加速。EAGLE、EAGLE-2和EAGLE-3（Li等，2024-2025）将"草稿 = 同一模型的更小版本"转变为精确的工程学科。2026年的生产推理服务器默认使用EAGLE-3。

## 概念

### 不变项：Leviathan拒绝采样

设`p(t)`为草稿模型在给定前缀下对下一个token的分布，`q(t)`为验证者的分布。采样一个草稿token `d ~ p`。以概率`min(1, q(d) / p(d))`接受。拒绝时，从残差分布`(q - p)_+ / ||(q - p)_+||_1`采样。得到的样本根据`q`分布。无论`p`有多差，这都是成立的——越差，你拒绝的次数越多，但输出仍然是精确的。

将`N`次这样的调用背靠背堆叠，使用一次验证者前向传播对`prefix + d_1 + ... + d_N`进行处理。验证者同时返回`q_1, q_2, ..., q_{N+1}`。从左到右遍历。在位置`j`的第一次拒绝时，从`residual(q_j, p_j)`采样并停止。完全接受时，从`q_{N+1}`采样一个奖励token。

### 什么决定加速

设`α`为每个草稿token的预期接受率。设`c = cost(draft) / cost(verifier)`为成本比。每次验证者前向传播的预期接受token数为：

```
E[accepted] = (1 - α^(N+1)) / (1 - α)
```

每个接受token的预期总挂钟时间为`(N * c + 1) / E[accepted]`。关于`N`最小化这个值，你就得到了最佳点。对于`α = 0.8, c = 0.05`：最优`N`约为5-7，加速比为3.2倍。对于`α = 0.95, c = 0.02`：最优`N`约为8-10，加速比可推至5倍。

单一最大的杠杆是`α`。在固定`N = 5`下从`α = 0.6`（普通草稿）到`α = 0.9`（EAGLE-3），每次验证者前向传播的预期接受token数从2.2增加到4.1。来自相同验证者的吞吐量提高了近2倍。

### 两年演进

**普通推测（Leviathan，2023）。** 草稿模型是一个独立训练的更小LLM，来自同一家族。容易搭接，`α ≈ 0.6`，加速比最好约2倍。

**EAGLE-1（Li等，2024）。** 草稿是一个小型Transformer——通常是一两层——以验证者的最后一层隐藏状态作为输入，直接预测下一个token。因为草稿看到了验证者的特征表示，其分布更接近验证者的分布。`α`攀升至0.7-0.8。

**EAGLE-2（Li等，2024）。** 添加了动态草稿树：不是提出单个`N`个token的序列，而是提出一个小型候选树，在一次前向传播中（树注意力）用验证者为每个候选打分，然后走最高概率路径。草稿长度变得每个步骤自适应。每个接受路径token的`α`攀升至0.85以上。

**EAGLE-3（Li等，2025，NeurIPS）。** 两个更多变化。首先，完全放弃特征预测损失——EAGLE-1/2训练草稿以匹配验证者的隐藏状态，这限制了数据能帮助的程度。EAGLE-3直接训练token预测。其次，训练时测试（TTT）：在草稿训练期间，将草稿自身的先前预测作为输入回馈多个步骤，与推理时操作的方式相同。这对齐了训练和测试分布，并阻止了错误累积。测量的加速比：聊天场景最高6.5倍，在SGLang上H100在batch 64下吞吐量提升38%。

### KV缓存回滚

验证在一次前向传播中扩展验证者的KV缓存`N`个条目。如果在位置`j`发生拒绝，位置`j-1`之后的缓存内容现在是错误的。两种常见实现：写入暂存缓冲区并在接受时提交（vLLM、TensorRT-LLM），或保留物理KV缓存加逻辑长度并在拒绝时截断。无论哪种方式，回滚成本是每层每头的字节数，与前向传播成本相比可以忽略不计。

对于EAGLE-2树搜索，验证者使用尊重树拓扑的非因果掩码运行注意力。工程实现比较繁琐，但计算是一个标准的使用自定义掩码的flash-attention调用。

### 2026年的草稿架构

| 策略 | 草稿类型 | `α` | 加速比 | 训练成本 |
|------|---------|-----|--------|----------|
| 普通 | 独立小LLM | 0.55-0.70 | 1.8-2.3倍 | 无（重用现有小模型） |
| Medusa | 验证者上的额外LM头 | 0.65-0.75 | 2-3倍 | 约1B SFT token |
| EAGLE-1 | 隐藏状态上的1层Transformer | 0.70-0.80 | 2.5-3倍 | 约60B token |
| EAGLE-2 | EAGLE-1 + 动态草稿树 | 0.80-0.88 | 3-4倍 | 约60B token |
| EAGLE-3 | 多层特征融合 + TTT | 0.88-0.92 | 3.5-6.5倍 | 约60-200B token |
| 前瞻 | 无草稿（Jacobi迭代） | 不适用 | 1.3-1.6倍 | 无 |

2026年生产中：vLLM和SGLang默认使用EAGLE-3（可用时），否则使用EAGLE-2。TensorRT-LLM为Meta和NVIDIA公开模型提供最快的Medusa路径。llama.cpp为CPU部署提供普通草稿。

## 构建它

见`code/main.py`。这是完整的Leviathan推测循环，包含所有组件：N个草稿、验证者并行前向传播、逐位置拒绝、残差采样、奖励token、KV回滚、以及输出分布与直接从`q`采样相匹配的经验验证。

### 步骤1：拒绝规则

```python
def accept(q_prob, p_prob, u):
    if p_prob <= 0:
        return True
    return u < min(1.0, q_prob / p_prob)
```

### 步骤2：残差分布

```python
def residual(q, p):
    raw = [max(0.0, qi - pi) for qi, pi in zip(q, p)]
    s = sum(raw)
    if s == 0:
        return list(q)
    return [r / s for r in raw]
```

### 步骤3：完整的推测步骤

`spec_step`函数从`p`草稿`N`个token，然后在一个并行`q`评估中验证所有token。对每个草稿token应用拒绝规则，并在第一次拒绝时从残差采样修正。如果一切都被接受，从`q_{N+1}`发出一个奖励token。

### 步骤4：KV回滚簿记

模拟器为每个工作者跟踪一个逻辑`kv_length`。接受`k`个草稿时，`kv_length += k`。在位置`j`拒绝时，缓存已经写过了`j`，但逻辑长度设置为`prefix_length + j + 1`——比修正token多一个。后续的读取截断到逻辑长度。

### 步骤5：Leviathan检查

运行50,000个推测步骤。计数接受token的经验分布。与50,000个来自`q`的直接样本比较。卡方统计量应明显低于临界值。该定理在实践中成立。

### 步骤6：加速 vs α

通过在不同幅度下扰动`p`远离`q`来扫描草稿质量。测量`α`，然后绘制预期token数随`α`和`N`变化的函数。代码打印一个表格，展示EAGLE-3级别的草稿质量（`α ≈ 0.9`）如何解锁每次验证者调用4-5个token。

## 使用它

使用EAGLE-3的生产级`vllm serve`：

```bash
vllm serve meta-llama/Llama-3.3-70B-Instruct \
  --speculative-config '{
    "model": "yuhuili/EAGLE3-LLaMA3.3-Instruct-70B",
    "num_speculative_tokens": 5,
    "method": "eagle3"
  }'
```

在H100上使用EAGLE-3的SGLang在batch 64下：根据EAGLE-3论文，吞吐量大约是batch-64普通解码的1.38倍。

何时使用推测解码：

- 任何交互式聊天工作负载，其中p50延迟比峰值吞吐量更重要。
- 代码生成和结构化输出（JSON、SQL）。`α`高于0.9，因为目标分布高度可预测。
- 长文本生成（数千个token）。摊销的加速持续发挥作用。

何时不使用：

- 非常小的模型（< 3B）。草稿并不比验证者便宜多少。
- 极小的batch-1 CPU部署。草稿模型的内存开销可能不值得。
- 非常高温度的创造性采样，其中`α`崩溃。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| Leviathan规则 | "min(1, q over p)" | 伯努利接受/拒绝，概率为`min(1, q(d)/p(d))`，当从拒绝时的残差采样时精确保留验证者分布 |
| 残差分布 | "(q minus p) plus, normalized" | `(q - p)_+` 在零处截断并重新归一化——拒绝时采样的正确分布 |
| 接受率α | "草稿正确率" | 在拒绝规则下期望的每token伯努利成功概率；支配所有加速数学 |
| EAGLE-1 | "隐藏状态草稿" | 以验证者最后一层隐藏状态为条件的小型Transformer草稿（Li等，2024） |
| EAGLE-2 | "动态草稿树" | EAGLE-1加上一个候选延续树，在一次验证者前向传播中用树注意力评分 |
| EAGLE-3 | "训练时测试" | 放弃特征预测损失，在训练期间以草稿自身输出作为输入，直接训练token预测 |
| 训练时测试（TTT） | "暴露偏差修复" | 在训练期间自回归运行草稿，使训练和测试输入分布匹配——计划采样的直接类比 |
| KV回滚 | "撤销拒绝的草稿" | 在拒绝后将验证者的KV缓存重置为接受前缀长度的簿记 |
| 奖励token | "免费的那个" | 当所有`N`个草稿都被接受时，从`q_{N+1}`额外采样一个，无需额外验证者成本 |
| 树注意力 | "一次验证多个候选" | 使用尊重草稿树拓扑的非因果掩码的注意力；在一次前向传播中计算树中每个节点的`q_i` |
