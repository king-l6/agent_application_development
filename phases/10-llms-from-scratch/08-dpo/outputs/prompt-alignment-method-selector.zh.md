# 对齐方法选择器

在为语言模型选择对齐方法时，使用此框架来评估你的数据、计算和质量要求，然后选择最适合你约束条件的方法。

## 输入要求

提供：
- **基础模型**（例如，Llama 3 8B、Mistral 7B、Qwen 2.5 72B）
- **起点**（基础模型，或已经 SFT 了？）
- **可用数据**（指令对、偏好对、非配对评分，或无）
- **计算预算**（GPU 小时数、GPU 数量）
- **质量目标**（原型可用、与开源竞争、最先进水平）
- **时间线**（天、周、月）

## 决策矩阵

### 快速选择

| 你的情况 | 推荐方法 | 原因 |
|---------|---------|------|
| 没有偏好数据，只有指令对 | 仅 SFT | 没有偏好信号就无法对齐 |
| < 5,000 个偏好对，计算有限 | DPO | 更简单的流水线，小数据也能工作良好 |
| 非配对反馈（仅点赞/点踩） | KTO | 唯一不需要成对比较的方法 |
| 希望单次训练完成对齐 | ORPO | 合并 SFT + 对齐，无需参考模型 |
| 内存受限（无法容纳参考模型） | SimPO | 不需要参考模型 |
| 大规模、多目标对齐 | RLHF (PPO) | 独立的奖励模型捕捉复杂偏好 |
| 使用在线数据的迭代对齐 | RLHF (PPO) | 可以在循环中生成、评分和重新训练 |
| RLHF 后的精炼 | DPO | 在有针对性的偏好上微调 RLHF 模型 |

### 详细对比

| 方法 | 数据要求 | 内存中的模型 | 训练循环 | 稳定性 | 最佳规模 |
|------|---------|-------------|---------|-------|---------|
| SFT | 指令对 (10K+) | 1 | 1 | 高 | 任意 |
| RLHF | 偏好对 (20K+) | 3-4 | 3 | 低 | 大型 (70B+) |
| DPO | 偏好对 (5K+) | 2 | 2 (SFT + DPO) | 高 | 中小型 (7B-70B) |
| KTO | 非配对评分 (5K+) | 2 | 2 (SFT + KTO) | 高 | 任意 |
| ORPO | 偏好对 (10K+) | 1 | 1 | 高 | 中小型 |
| SimPO | 偏好对 (5K+) | 1 | 2 (SFT + SimPO) | 高 | 中小型 |

## 方法特定配置

### SFT

- **何时停止**：1-3 个 epoch 后或验证损失停止下降时
- **关键超参数**：学习率 (1e-5 到 5e-5，更大的模型用更低的值)
- **关键细节**：在损失中屏蔽指令 token
- **陷阱**：超过 3 个 epoch 会导致记忆化；混入 2-5% 的预训练数据

### RLHF (PPO)

- **何时使用**：有 20K+ 比较对、需要多目标对齐或想要迭代在线学习
- **关键超参数**：KL 系数 (0.01-0.05)、PPO 裁剪比 (0.1-0.3)、学习率 (5e-6 到 3e-5)
- **关键细节**：奖励模型应 >= 策略模型大小
- **陷阱**：PPO 不稳定；持续监控 KL 散度和奖励曲线

### DPO

- **何时使用**：有偏好对，想要比 RLHF 更简单的流水线
- **关键超参数**：Beta (0.1-0.5；更低 = 允许更多偏离参考)
- **关键细节**：参考模型必须是 SFT 检查点的冻结副本
- **陷阱**：对 beta 非常敏感；在 [0.05, 0.1, 0.2, 0.5] 上扫描

### KTO

- **何时使用**：只有"好"或"坏"标签，没有成对比较
- **关键超参数**：Beta（与 DPO 相同），损失厌恶乘数（坏回答 1.5x）
- **关键细节**：需要大致平衡的好/坏示例（40-60% 比例）
- **陷阱**：没有配对，梯度信号较弱；可能需要比 DPO 更多的数据

### ORPO

- **何时使用**：想跳过 SFT，直接从基础模型到对齐模型
- **关键超参数**：Lambda（偏好项与 SFT 项的权重）
- **关键细节**：需要一个数据集中同时包含指令标签和偏好对
- **陷阱**：组合目标可能难以平衡；如果 SFT 损失占主导，对齐会较弱

### SimPO

- **何时使用**：内存受限，无法容纳参考模型
- **关键超参数**：Beta、gamma（长度归一化指数）
- **关键细节**：长度归一化防止模型偏向短回答
- **陷阱**：没有参考模型锚点，模型可能漂移更多；仔细监控

## 流水线模板

### 模板 1：快速原型（1-2 天）

```
基础模型 -> SFT (1 epoch, 10K 示例) -> DPO (3 epochs, 5K 对)
```

计算：7B 模型在 A100 上约需 4 GPU 小时
质量：扎实的指令遵循，基本的偏好对齐

### 模板 2：生产质量（1-2 周）

```
基础模型 -> SFT (2 epochs, 50K 示例) -> DPO (5 epochs, 20K 对) -> 评估 -> 迭代
```

计算：7B 约需 40 GPU 小时，70B 约需 200 GPU 小时
质量：与开源 RLHF 模型竞争

### 模板 3：最先进水平（1-3 个月）

```
基础模型 -> SFT (2 epochs, 100K+ 示例) -> RLHF (PPO, 50K+ 对) -> DPO (有针对性的精炼) -> 评估 -> 迭代
```

计算：70B 约需 500+ GPU 小时
质量：接近前沿模型对齐

### 模板 4：最小数据（1-2 天）

```
基础模型 -> SFT (1 epoch, 5K 示例) -> KTO (来自用户的非配对点赞/点踩)
```

计算：7B 约需 2 GPU 小时
质量：在数据收集开销最小的情况下优于仅 SFT

## 评估协议

对齐后，在以下维度进行评估：

1. **偏好胜率**：在 200+ 测试提示上比较对齐模型与 SFT 模型，使用人类评审。目标：> 60% 胜率。
2. **基准保留**：MMLU、HumanEval 或领域特定基准。不应比 SFT 基线下降 > 5%。
3. **MT-Bench 或 AlpacaEval**：标准对齐质量基准。与已发布的基线比较。
4. **安全评估**：测试对抗性提示、越狱和有害请求类别。
5. **回答多样性**：测量 100 个提示的回答熵。低熵 = 模式崩溃。

## 常见失败模式

| 症状 | 原因 | 针对方法的修复 |
|------|------|--------------|
| 冗长、填充的回答 | 奖励模型/隐式奖励偏好长度 | DPO：增加 beta。RLHF：添加长度惩罚。SimPO：调整 gamma。 |
| 模型同意一切 | 偏好数据偏差导致的谄媚 | 添加正确回答与用户不一致的偏好对 |
| 拒绝良性请求 | 安全数据上的过度对齐 | 减少安全示例比例，添加更多良性-拒绝对 |
| 输出与 SFT 几乎相同 | beta 过高 (DPO/KTO) 或 KL 系数过高 (PPO) | 降低 beta / KL 系数；模型没有在学习 |
| 训练损失振荡 | 学习率过高或数据不足 | 降低 lr 2-3 倍；增加偏好数据 |
