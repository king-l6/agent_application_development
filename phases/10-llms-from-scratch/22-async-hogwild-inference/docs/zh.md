# 异步和 Hogwild! 推理

> 投机解码（阶段 10 · 15）在单个序列内并行化 token。多智能体框架跨整个序列并行化但需要显式协调（投票、子任务拆分）。Hogwild! 推理（Rodionov 等人，arXiv:2504.06261）做了别的事情：运行同一 LLM 的 N 个实例，共享一个 KV 缓存。每个工作线程立即看到其他每个工作线程生成的 token。现代推理模型——QwQ、DeepSeek-R1——可以通过该共享缓存自我协调，无需任何微调。该方法是实验性的，但它开辟了一个全新的推理并行维度，与投机解码正交。本节课在标准库 Python 中实现了一个两工作线程的 Hogwild! 模拟器，并解释了为什么共享缓存协作从现有模型的推理能力中涌现。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 10 · 12（推理优化），阶段 10 · 15（投机解码）
**时间：** ~60 分钟

## 学习目标

- 描述三种常见的并行 LLM 拓扑（投票、子任务、Hogwild!），并命名每种拓扑针对的问题。
- 陈述 Hogwild! 的核心设置：多个工作线程、一个共享 KV 缓存、通过自我提示实现的涌现协调。
- 计算 Hogwild! 的墙钟加速作为工作线程数 `N`、任务级并行度 `p` 和协调开销 `c` 的函数。
- 在玩具问题上实现一个两工作线程的 Hogwild! 模拟器，并观察涌现的任务划分。

## 问题

现代 LLM 通过产生长的推理链来解决难题——5000 个 token 的逐步逻辑很常见，数万个 token 发生在深度数学问题上。在 70B 模型上以 35 token/秒解码时，50k 个 token 需要 24 分钟。谈不上交互性。

投机解码（阶段 10 · 15）通过在一个序列内并行化带来了 3-5 倍的加速。除此之外，自回归解码的顺序依赖是硬性天花板。每个新 token 依赖于每个先前的 token。

显而易见的问题：我们能否跨序列并行化？在同一问题上运行同一模型的多个副本，让它们合作，让它们分工？

先前工作：投票集成（运行 N 个模型，选择多数答案）、思维树（分支推理路径并重组）和多智能体框架（给每个智能体分配一个子任务，使用协调器）。这些都在特定任务领域有帮助。它们都引入了显式的协调机制——投票规则、分支剪枝逻辑、智能体间消息传递协议。

Hogwild! 推理采用了不同的方法。N 个工作线程共享一个 KV 缓存。每个工作线程立即看到其他每个工作线程生成的 token，就像它们是自己上下文的一部分一样。这些工作线程——无需任何训练或微调——就能弄清楚如何分工。现代推理模型（QwQ、DeepSeek-R1、Claude 家族推理模式）可以读取共享缓存并说出诸如"我看到工作线程 2 已经处理了基本情况，所以我将处理归纳步骤"之类的内容。

加速效果取决于工作负载，截至 2026 年 4 月仍是实验性的。但这个想法值得了解，因为它开辟了一个新的推理并行维度。

## 概念

### 设置

初始化 N 个工作进程，都运行相同的 LLM。不使用每个工作线程各自的 KV 缓存，而是维护**一个**共享缓存。当工作线程 `i` 生成 token `t_j` 时，该 token 被写入共享缓存的下一个位置。当工作线程 `k` 进行下一步时，它读取缓存的当前状态（包括所有 N 个工作线程到目前为止生成的所有内容）。

在每一步，工作线程竞相写入 token。没有每个工作线程的位置索引——缓存是一个增长的单一序列。顺序由写入到达时间决定。

### 为什么协调会涌现

工作线程共享一个提示。通常是类似"你是共同解决此问题的 N 个实例之一。每个实例读取共享内存并可以看到其他实例编写的内容。避免重复工作。"这样的内容。提示加上共享缓存就足够了。推理模型读取缓存，注意到问题的哪些部分已经被尝试过，并且（通常但不总是）转向未探索的部分。

Hogwild! 论文（Rodionov 等人，2025）报告了如下观察：

- 工作线程制定计划并通过缓存将这些计划传达给其他工作线程。
- 工作线程注意到其他工作线程推理中的错误并指出来。
- 工作线程在计划失败时进行调整并提出替代方案。
- 当被提示检查冗余时，工作线程会检测到冗余并转向。

这些都不需要微调。涌现的行为来自模型已有的推理能力。

### 命名由来

该论文的名称灵感来自 Hogwild! SGD（Recht 等人，2011），一种异步更新优化器。类比：SGD 的异步工作线程都写入一个共享参数向量；Hogwild! 推理的工作线程都写入一个共享 KV 缓存。两者都依赖经验收敛而不是同步保证。

### RoPE 使其可行

旋转位置嵌入（RoPE，Su 等人，2021）通过 Q 和 K 向量中的旋转编码位置信息。因为位置是旋转而不是硬编码的偏移，token 的位置可以在不重新计算 KV 缓存条目的情况下移动。当工作线程 `i` 在位置 `p` 写入共享缓存时，其他工作线程读取该位置可以直接使用缓存的条目——无需重新旋转。

在学习位置或绝对位置模型中，Hogwild! 需要在每次并发写入时进行缓存失效。RoPE 让缓存保持稳定。

### 墙钟时间数学

令 `T_serial` 为一个工作线程单独解决问题的时间。令 `p` 为任务级别可并行化部分。令 `c` 为每步协调开销（读取扩展的缓存，决定写什么）。

单工作线程时间：`T_serial`。
N 工作线程 Hogwild! 时间，如果协调免费：`T_serial * ((1 - p) + p / N)`。经典的 Amdahl 定律。
带协调开销：`T_serial * ((1 - p) + p / N) + c * steps_per_worker`。

要使工作线程高效，`c` 必须相对于每步解码时间很小。在产生 5k+ token 的推理模型上，工作线程可以负担数百个 token 的协调开销并且仍然胜出。在短聊天任务上，协调占主导，Hogwild! 比串行更差。

### 具体示例

推理问题：10k 个 token 的思维链。假设问题有 `p = 0.7` 的可并行化内容（不同的证明策略、不同的情况分析）和每工作线程 `c = 200` 个 token 的协调开销。使用 `N = 4` 个工作线程：

- 串行时间：10000 个解码步骤。
- Hogwild! 时间：10000 * (0.3 + 0.7 / 4) + 200 * 4 = 10000 * 0.475 + 800 = 5550 个解码步骤。
- 加速：10000 / 5550 = 1.8 倍。

这还算适中。但在更长的推理问题（50k 个 token）上，协调开销被摊薄，加速比推到 2.5-3 倍。Hogwild! 是推理等价于允许你自然地编写多线程代码的语言中的线程级并行。

### 何时使用 Hogwild!

- 长推理问题（数千个 token），其中任务可以在独立子目标间并行化。
- 已经训练好逐步思考的推理模型。非推理模型不能很好地自我协调。
- 单节点部署，有足够的 VRAM 容纳共享缓存加上 N 个工作进程。缓存是共享的，但每个工作线程有自己的激活内存。

### 何时不使用

- 短交互式聊天。协调开销占主导。
- 不能并行化的任务（单一线性证明、单一编译）。N=1 是最大值。
- 非推理模型。没有协调涌现。
- 多节点部署。共享缓存需要非常快速的跨工作线程同步。节点内可以；跨节点是延迟灾难。

### 实验状态

截至 2026 年 4 月，Hogwild! 是一种研究方法，有开源的 PyTorch 实现。尚未在生产中得到采用。三个障碍：

1. 跨并发进程的共享 KV 缓存管理是非平凡的工程问题。
2. 涌现协调依赖于任务；基准测试仍在构建中。
3. 加速比与投机解码已经提供的相比还算适中，两者可以结合，但结合的工程又是一层。

值得了解。值得实验。还不足以在产品上押注。

## 构建

`code/main.py` 实现了一个玩具 Hogwild! 模拟器：

- 两个工作进程，每个是一个确定性的"LLM"，以已知概率产生几种 token 类别之一（工作 token、观察 token、协调 token）。
- 一个共享缓存（只是一个 token 列表），两个工作线程都可以读写。
- 一个简单的协调逻辑：当一个工作线程看到另一个已经在一个类别中产生了足够多的工作 token 时，它选择一个不同的类别。

模拟器在固定的步骤预算下运行并报告：

- 总产生的工作 token 数。
- 总墙钟时间（工作线程步数）。
- 相对于单个工作线程的有效加速。
- 哪个工作线程写了哪个 token 的追踪。

### 步骤 1：共享缓存

两个工作线程都追加的列表。在实际实现中使用简单锁定（Python `threading.Lock`）；我们使用计数器模拟。

### 步骤 2：工作线程循环

每个工作线程在每一步：

- 读取当前的共享缓存。
- 根据已有的内容决定写哪个类别的 token。
- 写一个 token。

### 步骤 3：协调启发式

如果类别 X 在缓存中已经有 K 个 token 且工作线程的意图类别是 X，工作线程切换到类别 Y。这是一个玩具替代品，模拟推理模型"注意到这已经被覆盖了，转而做其他事情"的行为。

### 步骤 4：测量加速

使用 N=1 工作线程和 N=2 工作线程运行模拟器，相同的总步骤预算。计算产生的工作 token。N=2 应该产生大约 1.5-1.8 倍更多的工作 token，因为协调驱动的任务划分。

### 步骤 5：给协调施压

降低协调启发式的敏感性。再次运行。观察到没有良好的协调，N=2 会冗余产生相同的 token，加速下降到 1 以下。这与论文的观察一致：只有在工作线程具有自我协调的推理能力时，这个技巧才有效。

## 使用

截至 2026 年 4 月，Hogwild! 在生产中的集成属于研究级别。来自 Yandex/HSE/IST 的参考实现基于 PyTorch，针对 DeepSeek-R1 和 QwQ 模型上的单节点多进程设置。

务实的采用路径：

1. 分析你的推理任务工作负载。测量探索性（多种策略、情况分析、搜索）与线性的 token 比例。
2. 如果探索占主导，运行两工作线程 Hogwild! 实验。测量墙钟时间改进。
3. 如果改进低于 1.3 倍，你处于协调主导的 regime。回退到单工作线程。
4. 如果改进超过 1.5 倍，推至 N=4 并再次测量。收益递减通常在 N=4-8 时出现。

与投机解码结合：每个 Hogwild! 工作线程可以独立使用投机解码。两种加速大约相乘（粗略地），将 3 倍投机解码和 1.8 倍 Hogwild! 组合成相对于天真的单工作线程解码有效的 5.4 倍。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------------|------------------------|
| Hogwild! | "并行工作线程，共享缓存" | 同一 LLM 的 N 个实例并发运行，共享一个 KV 缓存；通过自我提示涌现协调 |
| 共享 KV 缓存 | "协调媒介" | 所有工作线程读写的一个增长的 KV 缓冲区；实现跨工作线程即时 token 可见性 |
| 涌现协调 | "无需训练" | 具备推理能力的 LLM 可以读取共享缓存并分配工作，无需任何微调或显式协议 |
| 协调开销 (c) | "用于定位的 token" | 每个工作线程读取扩展缓存并决定做什么的成本；必须相对于总解码时间保持较小 |
| 可并行化部分 (p) | "可以并行运行的内容" | 任务级并行度：总工作中本质非顺序的部分的比例 |
| RoPE 使 Hogwild! 可行 | "旋转位置是移位不变的" | 因为位置是旋转，写入共享缓存不需要重新计算先前的 token |
| 投票集成 | "运行 N 个，选多数" | 最简单的并行推理拓扑；对分类有用，对长形式推理帮助不大 |
| 思维树 | "分支和剪枝" | 探索多个分支并剪枝的推理策略；显式协调逻辑 |
| 多智能体框架 | "分配子任务" | 每个智能体获得一个角色；一个协调器进行编排；协议开销大 |
