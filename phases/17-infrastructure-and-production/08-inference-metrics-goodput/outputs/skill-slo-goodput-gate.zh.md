给定工作负载（模型、硬件、目标并发度、面向用户的交互类型——流式聊天 / 单次 / 语音 / 智能体），生成基于 goodput 的 SLO 门禁用于 CI/CD。

输出内容：

1. **SLO 规范**。三个阈值：TTFT P99 上限、TPOT P99 上限、E2E P99 上限。根据交互类型选择合理的值（流式聊天：TTFT 500 ms，TPOT 25 ms，E2E 3 s；语音：TTFT 300 ms 更严格；智能体：E2E 5 s 更宽松）。
2. **基准测试方案**。工具选择（LLMPerf 或 GenAI-Perf —— 说明选择及其原因）。提示分布（输入和输出 token 的平均值 + 标准差）。并发度扫描（目标的 25%、50%、100%、150%）。
3. **Goodput 计算**。公式：同时满足所有三个约束的请求比例。目标：生产环境 >= 99%，金丝雀 >= 95%。
4. **百分位报告**。对每个指标，报告 P50、P90、P99（绝不单独报告平均值）。仅在合理性检查时标注平均值。
5. **工具陷阱说明**。说明工具是否在 ITL 中包括或排除 TTFT。在跨团队比较之前固定定义。
6. **门禁逻辑**。CI 通过条件：在目标并发度下 goodput >= 目标值。如果 goodput 在 100% 到 150% 并发度之间下降超过 5 个百分点，标记告警——表明缺少负载测试余量。

硬性拒绝：
- 仅以吞吐量为门禁条件。拒绝并要求使用 goodput。
- 报告平均值而不报告 P99。拒绝。
- 省略工具名称和工具版本。拒绝。
- 仅在目标并发度下进行基准测试；始终做扫描。

拒绝规则：
- 如果用户没有写下的 SLO，拒绝并首先基于交互类型编写一个。
- 如果提示分布是"循环中使用相同的提示"，拒绝——这是提示一致性陷阱。要求使用逼真的合成数据。
- 如果基准测试少于 30 次运行或每次运行少于 100 个请求，拒绝，认为统计上不充分。

输出：一页 SLO 门禁规范，列出阈值、基准测试方案、工具选择、百分位报告模板和 CI 通过/失败规则。结尾附上"下一步测量什么"段落，根据已知弱点推荐 goodput vs 并发度曲线、提示分布敏感性或分块预填充开启/关闭的尾部比较。
