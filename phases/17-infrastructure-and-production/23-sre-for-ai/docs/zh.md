# AI 的 SRE — 多智能体事件响应、运行手册、预测性检测

> AI SRE 使用基于基础设施数据（日志、运行手册、服务拓扑）通过 RAG 进行推理的 LLM，来自动化调查、文档记录和协调阶段。2026 年的架构模式是多智能体编排 — 由监督者协调专门化智能体（日志、指标、运行手册）；AI 提出假设和查询，人类批准判断决策。Datadog Bits AI 和 Azure SRE Agent 以托管产品形式提供此功能。运行手册正在进化：NeuBird Hawkeye 使用对抗性评估（两个模型分析同一事件；一致 = 置信，分歧 = 不确定）；运营记忆在团队变更中持续存在。自动修复保持谨慎：AI 建议，人类批准。完全自主的行动范围狭窄（重启 pod、回滚特定部署）并具有严格护栏 — 任何兜售"一劳永逸"的人都是在过度推销。新兴前沿：事件前预测。MIT 研究报告，一个在历史日志 + GPU 温度 + API 错误模式上训练的 LLM 预测了 89% 的故障，提前 10-15 分钟。预测：到 2026 年底，95% 的企业 LLM 将具有自动故障转移。

**类型：** 学习
**语言：** Python（标准库，玩具多智能体事件分类模拟器）
**前置知识：** 阶段 17 · 13（可观测性），阶段 17 · 24（混沌工程）
**时间：** ~60 分钟

## 学习目标

- 绘制多智能体 AI SRE 架构图：监督者 + 专门化智能体（日志、指标、运行手册）+ 人类批准门控。
- 解释为什么自动修复是狭窄的（重启 pod、回滚部署）而不是广泛的（重新架构服务）。
- 说出对抗性评估模式（NeuBird Hawkeye）：两个模型一致 = 置信；不一致 = 升级。
- 引用 MIT 89% 早期检测结果和运营约束：没有执行的预测只是仪表盘。

## 问题

一个值班工程师在凌晨 3 点收到告警。"结账错误率过高。"他们检查 Datadog、Loki、三个运行手册、部署日志。30 分钟后他们意识到根本原因是 vLLM 因 KV 缓存尖峰导致的 OOM。他们重启 pod；错误消失。

在 2026 年，前 20 分钟的调查工作是可自动化的。按服务分组日志、与最近部署关联、匹配运行手册 — 这些都是 RAG + 工具使用。一个有监督的智能体可以在人类打开 Datadog 之前进行首次分类并呈现假设。

完全自主的修复是另一个不同的问题。重启 pod：安全。扩展 GPU 池：如果策略允许则安全。重新架构服务：绝对不能。这门学科就是划清这条窄线。

## 概念

### 多智能体架构

```
          事件
             │
             ▼
        监督者
        /    |    \
       ▼     ▼     ▼
  日志智能体  指标智能体  运行手册智能体
       │     │     │
       └─────┴─────┘
             │
             ▼
        假设 + 证据
             │
             ▼
        人类批准
             │
             ▼
        行动（狭窄集合）
```

监督者将事件分解为子查询。专门化智能体拥有工具访问权限（日志搜索、PromQL、文档检索）。监督者综合信息，向人类呈现假设 + 证据。人类批准或重新定向。

### 自动修复范围

**安全（狭窄）**：重启 pod、回滚特定部署、在预先批准的范围内扩展池、启用预先批准的功能标志。

**不安全（广泛）**：更改服务拓扑、修改资源限制、部署新代码、更改 IAM、修改数据库。

任何兜售"一劳永逸"的人都是在过度推销。随着 AI SRE 的成熟，安全集不断扩大，但边界是真实存在的。

### 对抗性评估（NeuBird Hawkeye）

两个模型独立分析同一事件。如果它们在根本原因上达成一致，置信度很高。如果它们不一致，将两种假设都展示给人类进行升级。简单的模式，有效的针对幻觉原因筛选。

### 运营记忆

团队人员流动是传统 SRE 的无声杀手 — 团队经验知识会流失。AI SRE 将运行手册 + 事后分析存储在向量数据库中；智能体在每次新事件时检索。当新工程师加入时，AI 拥有全部历史。

### 事件前预测

MIT 2025 年研究：在历史日志、GPU 温度、API 错误模式上训练的 LLM 在测试集上预测了 89% 的故障，提前 10-15 分钟。

现实检查：没有执行的预测只是仪表盘。运营问题是"当我们预测到时，我们做什么？" 预防性排空？告警？自动扩缩？答案是策略特定的。

### 2026 年的产品

- **Datadog Bits AI** — Datadog 内部的托管 SRE 协同助手。
- **Azure SRE Agent** — Azure 原生。
- **NeuBird Hawkeye** — 对抗性评估 + 运营记忆。
- **PagerDuty AIOps** — 分类 + 去重。
- **Incident.io Autopilot** — 事件指挥官 + 协调。

### 运行手册即代码

运行手册从 Confluence 页面演变为带有结构化章节（症状、假设、验证、行动）的版本化 markdown。结构化运行手册有助于更好的 RAG 检索。任何 AI-SRE 部署的第一步都是将非结构化运行手册转变为结构化。

### 你应该记住的数字

- MIT 早期检测：89% 的故障，提前 10-15 分钟。
- 多智能体分类：监督者 +（日志、指标、运行手册）+ 人类。
- 安全的自动修复集：重启 pod、回滚部署、在范围内扩展。
- 对抗性评估：两个模型独立；一致 = 置信。

## 使用它

`code/main.py` 模拟了多智能体分类：日志智能体发现错误，指标智能体发现 CPU 尖峰，运行手册智能体匹配到已知问题。监督者对假设进行排序。

## 练习

1. 运行 `code/main.py`。如果日志和指标智能体意见不一致怎么办？监督者如何解决？
2. 为你的服务定义三个"安全"的自动修复操作。为每个操作提供论证。
3. 编写结构化的运行手册模板：章节、必填字段、验证命令。
4. 预测性检测在提前 12 分钟触发。你的策略是什么 — 告警、预防性排空，还是两者都做？
5. 论证一个 3 人团队是否应该在 2026 年采用 AI SRE 还是等待。考虑成熟度、量级和风险。
