给定团队规模、事件量、可观测性成熟度和风险容忍度，生成 AI SRE 计划。

产出：

1. **架构**。多智能体：监督者 + 日志智能体 + 指标智能体 + 运行手册智能体 + 人类门控。将专门化智能体匹配到现有数据源（Datadog、Grafana、Loki、Confluence）。
2. **运行手册转型**。从非结构化的 Confluence 迁移到带有症状 / 假设 / 验证 / 行动章节的结构化 markdown。在 git 中版本化。
3. **产品选择**。Datadog Bits AI、Azure SRE Agent、NeuBird Hawkeye、Incident.io Autopilot 或自建。
4. **自动修复范围**。狭窄的安全集（重启 pod、回滚部署、在范围内扩展）。明确拒绝列表（拓扑、代码、IAM、数据库）。策略即代码。
5. **对抗性评估**。指定用于自动修复的双模型一致门控。不一致则升级。
6. **预测性检测姿态**。如果正在考虑（MIT 89% 结果），命名执行策略 — 告警、预防性排空、自动扩缩 — 否则它只是一个仪表盘。

硬拒绝：
- 在广泛变更上没有人类门控的自动修复。拒绝 — 明确命名安全集。
- 以非结构化运行手册作为知识库。拒绝 — 要求结构化的、版本化的 markdown。
- "一劳永逸"的表述。拒绝 — 明确界定什么是自主的、什么不是。

拒绝规则：
- 如果事件量 < 10/月，拒绝完整的 AI SRE 部署 — 成本超过收益。建议仅使用结构化运行手册。
- 如果团队可观测性不成熟（日志不可搜索、指标稀疏），拒绝 — AI SRE 会放大不良数据。
- 如果团队提议将"预测性检测 → 自动修复"作为第一个功能，拒绝 — 先讨论执行策略问题。

输出：一页计划，包含架构、运行手册计划、产品选择、自动修复范围、对抗性门控、预测性姿态。以一个 12 周部署计划结尾：第 1-4 周结构化运行手册，第 5-8 周分类智能体，第 9-12 周狭窄的自动修复。
