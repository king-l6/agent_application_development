# LLM 的 FinOps — 单位经济学与多租户归因

> 传统 FinOps 在 LLM 花费上失效。成本是 token 交易，而非资源运行时间。标签无法映射 — API 调用是交易，而非资产。工程决策（提示设计、上下文窗口、输出长度）是财务决策。2026 年的操作手册有三个归因维度需要在第一天就进行 instrumentation：每个用户（`user_id`）用于席位定价和扩展、每个任务（`task_id` + `route`）用于产品面成本确定和优先级排序、每个租户（`tenant_id`）用于单位经济学和续约。四个 token 层 — 提示、工具、记忆、响应 — 一个桶会隐藏花费。多租户产品的执行阶梯：每个租户的速率限制（2-3 倍预期峰值，清晰的 429 + Retry-After）；每日花费上限（1.5-3 倍合同上限；触发速率收紧 + 告警）；花费 z-score > 4 时的终止开关（自动暂停 + 呼叫值班人员）。归因模式：标签-聚合、遥测连接器（跟踪 ID → 计费；最高准确度）、采样-外推、基于模型的分配、事件溯源、实时流式处理。单位指标：每个已解决查询的成本、每个已生成工件的成本 — 而非 $/M tokens。追溯性标记总会遗漏；在请求创建时进行 instrumentation。

**类型：** 学习
**语言：** Python（标准库，带终止开关的玩具成本归因模拟器）
**前置要求：** 第 17 阶段 · 13（可观测性），第 17 阶段 · 14（缓存）
**时间：** 约 60 分钟

## 学习目标

- 解释为什么传统 FinOps（标签 + 层级）在 LLM 花费上失效，并说出三个新的归因维度。
- 列举四个 token 层（提示、工具、记忆、响应）以及为什么单桶计费隐藏了成本。
- 为多租户产品设计执行阶梯（速率限制 → 花费上限 → 终止开关）。
- 选择单位指标（每个已解决查询/工件的成本）而不是 $/M tokens。

## 问题

你的账单显示 40,000 美元。你不知道：
- 哪个租户花费了它。
- 哪个产品功能导致了它。
- 是否有任何个别用户有滥用行为。
- 是提示膨胀、工具调用还是记忆放大是罪魁祸首。

基于提供商的标签-聚合适用于云资源（EC2、S3），其中标签传播到行项目。LLM API 调用不会自动标记 — 你必须在调用点标记 user/task/tenant 并贯穿始终。追溯性归因总会遗漏边缘情况。

## 概念

### 三个归因维度

**每个用户**（`user_id`）：谁花费了什么。驱动席位定价、扩展对话、识别高价值用户。

**每个任务**（`task_id` + `route`）：哪个产品面花费了什么。驱动功能优先级排序、终止昂贵功能的决策。

**每个租户**（`tenant_id`）：哪个客户是盈利的。驱动单位经济学、续约定价、层级阈值。

在调用点的第一天对所有三个进行 instrumentation。追溯性总是更差。

### 四个 token 层

| 层 | 示例 | 占总量的典型百分比 |
|-------|---------|---------------------|
| 提示 | 系统 + 用户输入 | 40-60% |
| 工具 | 工具调用结果反馈 | 20-40%（智能体工作负载） |
| 记忆 | 先前对话 / 检索的文档 | 10-30% |
| 响应 | 模型输出 | 10-30% |

将四者放在同一个桶中会使优化变得盲目。在你的归因模式中将它们分开。

### 执行阶梯

1. **速率限制** 每个租户。2-3 倍预期峰值。返回 429 并附带 `Retry-After`。租户看到摩擦；没有意外账单。

2. **每日花费上限** 每个租户。1.5-3 倍合同上限。触发：收紧速率限制 + 告警客户成功团队。

3. **终止开关** 花费 z-score > 4（相对于租户基线）。自动暂停租户；呼叫值班人员；升级到运维 + CS。

### 归因模式

- **标签-聚合**：标记元数据头部；稍后聚合。简单；粗略。
- **遥测连接器**：通过跟踪 ID 将跟踪连接到计费。最高准确度。成熟团队的做法。
- **采样 + 外推**：采样 5-10%，乘以倍数。对于粗略花费具有成本效益；遗漏尾部。
- **基于模型的分配**：回归推断成本驱动因素。适用于没有标签的遗留数据。
- **事件溯源**：成本作为流中的事件（Kafka / Kinesis）。实时。
- **实时流式处理**：仪表板亚秒级更新。

### 每个 X 的成本是单位指标

$/M tokens 是供应商用语。产品指标：

- 每个已解决支持工单的成本。
- 每篇已生成文章的成本。
- 每个成功智能体任务的成本。
- 每个用户会话分钟的成本。

将成本与产品成果挂钩。否则优化没有锚点。

### 成本归因跟踪形状

```
trace_id: abc123
  user_id: u_42
  tenant_id: t_7
  task_id: task_classify_doc
  route: model_haiku
  layers:
    prompt_tokens: 1800
    tool_tokens: 600
    memory_tokens: 400
    response_tokens: 150
  cost_usd: 0.0135
  cached_input: true
  batch: false
```

每次调用时发出。存储在数据湖中。按每个维度聚合。第 17 阶段 · 13 的可观测性栈是它的存放位置。

### 复合节省栈

栈：缓存 + 批处理 + 路由 + 网关。使用所有四个：
- 缓存 L2（第 17 阶段 · 14）：输入成本约降低 10 倍。
- 批处理（第 17 阶段 · 15）：5 折优惠。
- 路由到廉价模型（第 17 阶段 · 16）：成本降低 60%。
- 网关效率（第 17 阶段 · 19）：冗余 + 重试。

最佳情况叠加：约为基础线的 5-10%。大多数团队使用了 2-3 个杠杆；很少有团队全部四个都使用。

### 你应该记住的数字

- 归因维度：每个用户、每个任务、每个租户。
- 四个 token 层：提示、工具、记忆、响应。
- 终止开关：花费 z-score > 4。
- 单位指标：每个已解决查询的成本，而非 $/M tokens。
- 叠加优化：可达到基础线的约 5-10%。

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|------|----------------|------------------------|
| 每个用户归因 | "用户级成本" | 每次调用上标记 `user_id` |
| 每个任务归因 | "功能成本" | `task_id` + `route` 标识产品面 |
| 每个租户归因 | "客户成本" | `tenant_id`；驱动单位经济学 |
| 四个 token 层 | "成本层" | 提示 + 工具 + 记忆 + 响应 |
| 速率限制 | "429 防护" | 在网关上执行的每个租户上限 |
| 每日花费上限 | "每日上限" | 租户范围的预算，带告警 |
| 终止开关 | "自动暂停" | 花费 z-score > 4 触发自动暂停 |
| 每个已解决成本 | "产品单位指标" | 成本与产品成果挂钩，而非 token |
