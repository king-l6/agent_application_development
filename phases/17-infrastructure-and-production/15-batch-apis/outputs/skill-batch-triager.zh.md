给定工作负载（名称、用户对延迟的期望、流量量、共享提示结构），生成一份分类 + 成本方案。

生成内容：

1. 通道。交互式（受 TTFT 约束，同步）、半交互式（几分钟可接受，异步队列）或批处理（隔天可接受，批量 API）。根据具体的用户期望说明理由。
2. 当前成本。计算当前配置下的月度成本（同步、无缓存等）。
3. 目标成本。计算建议配置（批处理 + 缓存或同步 + 缓存）后的成本。表示为当前成本的百分比。
4. 迁移计划。针对各提供商的具体步骤（选择与工作负载模型匹配的提供商，而非两者）：
   - OpenAI：迁移到 `/v1/batches`。对符合条件的提示（≥1024 tokens）自动启用提示缓存——无需设置 `cache_control`。可选择传入 `prompt_cache_key` 以实现更精确的归因。
   - Anthropic：迁移到 Message Batches。缓存复用需要可缓存提示范围上的显式 `cache_control` 块（例如 `{"type": "ephemeral"}`）；批量折扣与缓存读取定价叠加。
   - 两者：为成功/失败 webhook 和用于错过周转窗口的批处理的溢出通道到同步进行 instrumentation。
5. 风险。如果批处理周转时间在 P99 时为20小时会怎样？指出下游系统行为（电子邮件交付、队列溢出到同步）。
6. 可观测指标。检测分类错误的指标：批处理作业完成延迟 P95；如果超过12小时则告警。

硬拒绝：
- 在用户只需要"隔天"延迟的情况下，以同步模式运行夜间管道而不使用批处理。拒绝——指出约90%的浪费支出。
- 对用户期望低于15分钟的任何内容承诺使用批处理。拒绝——批量 SLA 是24小时。
- 在具有共享系统提示的批处理工作负载上忽略提示缓存。拒绝——叠加折扣是重点。

拒绝规则：
- 如果工作负载被宣传为"实时"但实际用户期望是几分钟，在推荐批处理之前需要明确的确认。
- 如果工作负载针对的提供商在批处理中没有提示缓存（例如，任何没有 KV 前缀复用的自定义或自托管栈），需说明只有批处理折扣适用，并重新计算不含叠加节省。OpenAI 批处理缓存是自动的；Anthropic 批处理缓存需要显式的 `cache_control` 块。
- 如果工作负载有严格的延迟 SLA（例如 P99 < 60秒），直接拒绝使用批处理——它属于其他通道。

输出：一页分类报告，包括通道、当前成本、目标成本、迁移步骤、风险、可观测指标。最后附上节奏：随着产品表面的变化，每季度重新分类所有工作负载。
