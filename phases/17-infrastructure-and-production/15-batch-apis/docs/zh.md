# 批量 API —— 作为行业标准的50%折扣

> 每个主要提供商都提供异步批量 API，具有50%的折扣和约24小时的周转时间。OpenAI、Anthropic、Google 以及大多数推理平台（Fireworks batch tier、Together batch）都实现了相同的模式。将批处理与提示缓存叠加，夜间管道可降至同步未缓存成本的约10%。规则极其简单：如果不是交互式的，就应该使用批处理。内容生成管道、文档分类、数据提取、报告生成、批量标注、目录标签 —— 任何能容忍24小时延迟的工作负载，在迁移到批处理之前都是在浪费钱。2026年的生产模式是将每个新的 LLM 工作负载分为三类：交互式（带缓存的同步）、半交互式（带回退的异步队列）、批处理（夜间，叠加缓存输入）。那些假装是交互式但实际能容忍几分钟延迟的工作负载浪费最大。

**类型：** 学习
**语言：** Python（标准库，玩具级批量 vs 同步成本模拟器）
**前置知识：** 第17阶段·14（提示与语义缓存）
**时间：** 约45分钟

## 学习目标

- 指出三种提供商批量 API（OpenAI、Anthropic、Google）以及通用的50%折扣 + 24小时周转保证。
- 计算在夜间分类工作负载上叠加批处理 + 缓存输入的成本，并与同步未缓存基线进行比较。
- 将工作负载分为交互式/半交互式/批处理三类，并说明分类理由。
- 指出两个陷阱：部分交互性（用户期望快于24小时）和输出模式漂移（批量文件格式因提供商而异）。

## 问题

你的团队运行一个夜间报告生成管道。5万份文档，对每份进行摘要，对摘要进行聚类，起草一份执行简报。同步运行需要4小时，每晚花费2000美元。你听说了批量 API。

批处理给你50%的折扣。你还在系统提示上启用了提示缓存（在所有5万次调用中共享）。叠加后，账单降至每晚180美元——约为基线的9%。同一个管道，三次配置变更。

批量 API 是 LLM 成本工具包中最便宜的杠杆，但几乎没人使用。原因主要是组织方面的：团队把 SLA 实际上是"到早上之前"的任务当成了"实时"。本课程就是要让你不要把90%的账单留在桌上。

## 概念

### 三种批量 API

**OpenAI 批量 API**：上传一个包含请求列表的 JSONL 文件。承诺24小时周转（实践中通常约2-8小时）。输入和输出 token 均享受50%折扣。`/v1/batches` 端点。符合条件的缓存输入还可叠加缓存输入定价。

**Anthropic Message Batches**：JSONL 上传。24小时周转。50%折扣。支持 `cache_control`——缓存写入是显式的，读取在批处理内部自动进行。

**Google Vertex AI 批量预测**：BigQuery 或 GCS 输入。Gemini 类似50%折扣。与 Vertex 管道集成。

### 语义：异步，而非缓慢

批量 API 的意思是"我承诺在24小时内返回"——而不是"这需要24小时"。典型的 P50 是2-6小时。提供商会将你的批处理安排在 GPU 库存利用率较低的的非高峰窗口。

### 与缓存叠加

一个5万文档的摘要任务，使用相同的4K token 系统提示：

- 同步未缓存：50000 ×（输入价格 × 4000 + 输出价格 × 200），按全价计算。
- 同步缓存：系统提示在第一次写入后缓存；剩余49999次获得10倍更低的输入价格。
- 批处理 + 缓存：以上所有叠加，读写均享受50%折扣。

叠加效果：批处理 + 缓存 = 同步未缓存账单的约10%。任何在夜间运行且有共享系统提示的工作负载都应使用这种方式。

### 工作负载分类

**交互式** —— 用户等待响应。TTFT 至关重要。使用提示缓存的同步调用。不能使用批处理。

**半交互式** —— 用户提交任务，几分钟后回来看结果。异步队列，如果批处理不可用则回退到同步。考虑中等规模的 RAG 索引。

**批处理** —— 用户期望"到早上"或"下一小时"拿到结果。内容管道、大规模分类、离线分析。始终使用批处理，始终叠加缓存。

常见错误：因为管道是生产环境就将所有内容归类为交互式。生产环境不是延迟规格——SLA 才是。

### 部分交互性陷阱

某些功能看起来是交互式的，但实际上能容忍5-10分钟。例如：带有"刷新"按钮的夜间客户健康报告。用户点击刷新；等待10分钟完全没问题。团队却将其作为同步实现。50个并发刷新消耗的成本是批处理并通过电子邮件发送的10倍。

要问的问题是："24小时对这个用户意味着什么？"如果答案是"他们不会注意到"，就用批处理。

### 输出模式陷阱

批量文件格式因提供商而异：

- OpenAI：JSONL，每行一个请求。
- Anthropic：JSONL，每行一条消息；响应格式嵌入其中。
- Vertex：BigQuery 表或带有 TFRecord 的 GCS 前缀。

编写"一个批量客户端"跨提供商意味着每个提供商都需要适配器代码。声称支持多提供商批处理的网关（Portkey、LiteLLM 部分层级）仍然只是对原始格式的薄封装。

### 你应该记住的数字

- 各提供商的批量折扣：输入 + 输出统一50%。
- 周转 SLA：保证24小时，典型 P50 为2-6小时。
- 叠加批处理 + 缓存输入：约同步未缓存成本的10%。
- 工作负载分类规则：如果24小时延迟可接受，始终使用批处理。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 批量 API | "异步折扣" | 50%折扣，24小时周转 |
| JSONL | "批量格式" | 每行一个 JSON 请求；OpenAI/Anthropic 标准 |
| Message Batches | "Anthropic 批处理" | Anthropic 的批量 API 产品名称 |
| 批量预测 | "Vertex 批处理" | Vertex AI 的批量 API 产品 |
| 周转 SLA | "24小时承诺" | 保证值，非典型值；典型值为2-6小时 |
| 工作负载分类 | "交互性决策" | 交互式/半交互式/批处理路由决策 |
| 输出模式 | "响应格式" | 每个提供商的 JSONL 布局；不可移植 |
| 叠加折扣 | "批处理 + 缓存" | 两者同时应用时约为同步未缓存账单的10% |
