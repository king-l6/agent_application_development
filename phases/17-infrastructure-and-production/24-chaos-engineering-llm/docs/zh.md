# LLM 生产环境的混沌工程

> 2026 年，LLM 的混沌工程已成为一门独立学科。在生产环境中运行实验之前的前置条件：已定义的 SLI/SLO、跟踪+指标+日志可观测性、自动化回滚、应急预案、值班人员。架构有四个平面：控制（实验调度器）、目标（服务、基础设施、数据存储）、安全（防护 + 中止 + 流量过滤）、可观测性（指标 + 跟踪 + 日志）、反馈（反馈到 SLO 调整）。防护栏是强制性的：燃烧率告警在每日错误预算消耗超过预期的 2 倍时暂停实验；抑制窗口 + 跟踪 ID 相关性去重告警噪声。节奏：每周小型金丝雀 + SLO 评审；每月演练日 + 事后复盘；每季度跨团队韧性审计 + 依赖映射。LLM 特定实验：内存过载、网络故障、提供商中断、格式异常的提示、KV 缓存逐出风暴。工具：Harness Chaos Engineering（LLM 驱动的建议、爆炸半径降级、MCP 工具集成）；LitmusChaos（CNCF）；Chaos Mesh（CNCF Kubernetes 原生）。

**类型：** 学习
**语言：** Python（标准库，玩具混沌实验运行器）
**前置要求：** 第 17 阶段 · 23（AI 的 SRE），第 17 阶段 · 13（可观测性）
**时间：** 约 60 分钟

## 学习目标

- 说出五个混沌工程前置条件（SLI/SLO、可观测性、回滚、应急预案、值班人员）并解释为什么跳过任何一个都会破坏实践。
- 图示四个平面（控制、目标、安全、可观测性）以及反馈循环到 SLO。
- 列举五个 LLM 特定实验（内存过载、网络故障、提供商中断、格式异常的提示、KV 逐出风暴）。
- 根据技术栈选择工具 — Harness、LitmusChaos、Chaos Mesh。

## 问题

传统技术栈的混沌测试已经成熟。LLM 技术栈增加了新的故障模式。一个带有恶意字符的 4K token 提示可使分词器停顿 12 秒。上游提供商返回 429；你的网关重试；你的服务在重试放大的并发下 OOM。在突发负载下，KV 缓存逐出风暴导致重新填充级联，使计算资源饱和。

这些都不会在单元测试中出现。混沌工程是你如何在用户发现之前发现它们的方法。

## 概念

### 前置条件

不要在没有以下条件的情况下在生产环境中运行混沌：

1. **SLI/SLO** — 定义的服务水平指标和目标。
2. **可观测性** — 跟踪、指标、日志，接入仪表板。
3. **自动化回滚** — 第 17 阶段 · 20 策略标志回滚。
4. **应急预案** — 结构化的，第 17 阶段 · 23。
5. **值班人员** — 有人响应。

缺少任何一项都意味着混沌会变成真实事故。

### 四个平面 + 反馈

**控制平面** — 实验调度器（Litmus 工作流、Chaos Mesh 调度、Harness UI）。

**目标平面** — 服务、Pod、节点、负载均衡器、数据存储。

**安全平面** — 终止开关、抑制窗口、爆炸半径限制、错误预算门控。

**可观测性平面** — 正常指标 + 跟踪 ID 相关性，以区分混沌引起的故障和自然故障。

**反馈循环** — 发现反馈到 SLO 调整、应急预案更新、代码修复。

### 防护栏是强制性的

- **燃烧率告警**：如果每日错误预算消耗超过预期的 2 倍，暂停实验。
- **抑制窗口**：在实验期间，在爆炸半径内静默非实验告警。
- **跟踪 ID 相关性**：所有实验引起的错误都携带标签，以便值班人员去重。

### 五个 LLM 特定实验

1. **内存过载** — 通过发送高并发的长上下文请求来强制 KV 缓存抢占风暴。观察：服务是优雅地降级还是崩溃？

2. **网络故障** — 切断推理网关和提供商之间的连接。观察：回滚是否在 SLA 内启动？（第 17 阶段 · 19）

3. **提供商中断模拟** — OpenAI 返回 100% 429。观察：路由是否故障转移到 Anthropic？（第 17 阶段 · 16、19）

4. **格式异常的提示** — 注入使分词器停顿的有效载荷（例如，深度嵌套的 unicode、巨大的 UTF-8 码点）。观察：单个请求是否会锁住一个工作器？

5. **KV 逐出风暴** — 通过饱和 vLLM 块预算来强制逐出。观察：LMCache 是恢复还是服务降级？

### 节奏

- **每周** — 在预演环境中进行小型金丝雀实验，也许 5% 的生产环境。
- **每月** — 针对特定场景的预定演练日；跨团队参与；事后复盘。
- **每季度** — 跨团队韧性审计；依赖映射更新。

### 工具

- **Harness Chaos Engineering** — 商业软件；AI 驱动的实验建议；爆炸半径降级；MCP 工具集成。
- **LitmusChaos** — CNCF 毕业项目；基于 Kubernetes 工作流。
- **Chaos Mesh** — CNCF 沙箱；Kubernetes 原生 CRD 风格。
- **Gremlin** — 商业软件；广泛支持。
- **AWS FIS** / **Azure Chaos Studio** — 托管云服务。

### 从小规模开始

第一个实验：在稳定流量下杀死一个解码副本。观察重新路由和恢复。如果这有效且看起来安全，升级到网络混沌。

第一个 LLM 特定实验：注入一个提供商 429 持续 5 分钟。观察回滚。大多数团队发现他们的回滚没有经过充分测试。

### 你应该记住的数字

- 四个平面：控制、目标、安全、可观测性。
- 燃烧率暂停：每日预算消耗超过预期的 2 倍。
- 节奏：每周金丝雀、每月演练日、每季度审计。
- 五个 LLM 实验：内存、网络、提供商、格式异常的提示、KV 风暴。
