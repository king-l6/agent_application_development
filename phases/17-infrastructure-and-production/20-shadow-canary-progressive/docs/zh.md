# 影子流量、金丝雀发布和 LLM 的渐进式部署

> LLM 发布结合了软件部署中最困难的部分：没有单元测试、故障模式分散、信号延迟。流程是 (1) 影子模式 — 将生产请求复制到候选模型，记录日志，进行比较，零用户影响；捕获明显的分布问题，但不是质量保证；(2) 金丝雀发布 — 分阶段流量迁移 10% → 25% → 50% → 75% → 100%，每个阶段有门控；跟踪延迟百分位、每次请求成本、错误/拒绝率、输出长度分布、用户反馈率；(3) 对不同的替代方案在稳定性确认后进行 A/B 测试。非确定性是不可约的 — 由于 GPU FP 非结合性加上批次大小差异，相同输入在不同运行间的准确度变化可达 15%。成本是一个变量，不是常量 — 一个好 20% 的模型每次调用可能贵 3 倍。回滚速度是决定性的：如果回滚需要重新部署，你就太慢了。策略存在于配置/标志中；模型存在于带有固定摘要的注册表中；回滚 = 切换策略 + 恢复阈值 + 在几秒内固定旧模型。

**类型：** 学习
**语言：** Python（标准库，玩具金丝雀进度模拟器）
**前置知识：** 阶段 17 · 13（可观测性），阶段 17 · 21（A/B 测试）
**时间：** ~60 分钟

## 学习目标

- 区分影子模式（零影响对比）、金丝雀（实时流量渐进）和 A/B（稳定性确认后的比较）。
- 列举五个 LLM 特定的金丝雀指标（延迟、成本/请求、错误/拒绝、输出长度分布、用户反馈）。
- 解释为什么 LLM 非确定性（高达 15%）改变了发布中"稳定"的含义。
- 设计一个以秒计（策略切换）而非以小时计（重新部署）的回滚路径。

## 问题

你发布了一个新模型。离线评估显示准确率提升 3%。你将其在生产环境启用。24 小时内，成本上涨 40%，用户点踩率上升 8%，三个客户工单报告"奇怪的答案"。你回滚。重新部署需要 3 小时。你的周末毁了。

其中的每一个问题都是可以避免的。影子模式会在任何用户看到之前就捕获 40% 的成本飙升。金丝雀会在点踩率变化时就在 10% 阶段停止。策略标志回滚只需 30 秒。这种纪律填补了"离线评估看起来不错"和"真实用户满意"之间的空白。

## 概念

### 影子模式

候选模型接收与生产环境相同的请求；输出被记录，不返回给用户。零用户影响。记录：

- 输出内容（与生产环境的差异）。
- 令牌数（成本差异）。
- 延迟。
- 拒绝和错误。

捕获的问题：成本激增、长度退化、明显的拒绝变化、硬错误。不捕获的问题：用户会感知到的质量差异。影子是一种冒烟测试，不是质量测试。

### 金丝雀发布

分阶段流量迁移，每个阶段有门控。典型进程：1% → 10% → 25% → 50% → 75% → 100%。每个步骤在 5 个指标上设置门控：

1. **延迟百分位** — P50、P95、P99。违规条件：金丝雀的 P99 > 基线的 1.5 倍。
2. **每次请求成本** — 混合平均。违规条件：超过基线 20%。
3. **错误 / 拒绝率** — 5xx 加明确拒绝。违规条件：基线的 2 倍。
4. **输出长度分布** — 均值 + P99。违规条件：分布偏移。
5. **用户反馈率** — 点踩 / 工单提交。违规条件：基线的 1.5 倍。

### 非确定性是新方差

相同的输入产生不同的输出。原因：

- GPU FP 非结合性（浮点归约顺序因批次而异）。
- 批次大小差异（相同提示在 128 批次与 16 批次中）。
- 采样（温度 > 0）。

测量结果：相同评估集上运行间准确度变化高达 15%。发布中的"稳定"意味着指标在预期方差范围内，而不是与基线相同。将门控设置在噪声基底以上。

### 成本是一个变量

一个好 20% 的模型每次调用可能贵 3 倍。每次请求成本是五个门控之一。发布一个"更好"但破坏单位经济模型的模型是需要回滚的情况。

### 回滚是你的武器

- **策略标志**（功能标志系统）：在配置中切换百分比；只需几秒。
- **模型固定**（注册表摘要）：固定的模型不会自动升级。
- **回滚** = 恢复标志 + 设置固定摘要为之前的版本。秒级，不是小时级。

如果你的堆栈需要重新部署来回滚，在发布之前解决这个问题。

### 工具

**Argo Rollouts** / **Flagger** — Kubernetes 渐进式交付控制器。与 Istio/Linkerd 加权路由集成。

**Istio 加权路由** — 服务网格级别的流量拆分。

**KServe / Seldon Core** — 内置金丝雀功能的模型服务。

**功能标志** — LaunchDarkly、Flagsmith、Unleash。策略级切换，无需重新部署。

### 指标节奏

根据流量大小，金丝雀门控每 5-15 分钟检查一次。1% 流量，10 req/min，每个窗口给出 50-150 个数据点 — 对延迟来说足够，但对用户反馈来说噪声较大。10% 流量给出约 10 倍的数据。每次推进应暂停足够长的时间以在每个步骤积累足够的样本。

### A/B 步骤是可选的

如果新模型有明显不同（不同的行为、不同的成本曲线、不同的语气），在金丝雀通过后在 50% 进行 A/B 测试。如果只是改进版本，金丝雀门控通过后直接跳到 100%。

### 你应该记住的数字

- 金丝雀进程：1% → 10% → 25% → 50% → 75% → 100%。
- 非确定性上限：相同输入运行间变化高达 15%。
- 五个金丝雀指标：延迟、成本、错误/拒绝、输出长度、用户反馈。
- 成本门控：超过基线 20% 是违规。
- 回滚：秒级，不是小时级。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 影子模式 | "复制到新模型" | 零影响发送到候选模型进行日志记录 |
| 金丝雀 | "渐进流量" | 逐步向用户开放、带有门控的发布 |
| 门控 | "发布检查" | 阻止进程推进的指标阈值 |
| 非确定性 | "LLM 方差" | 不可约的运行间差异 |
| 策略标志 | "标志切换回滚" | 配置级回滚，秒级不是小时级 |
| 模型固定 | "注册表摘要" | 模型版本的不可变引用 |
| Argo Rollouts | "K8s 渐进" | Kubernetes 原生的金丝雀/回滚控制器 |
| KServe | "推理 K8s" | 带有金丝雀原语的模型服务 |
| Istio 加权 | "网格拆分" | 服务网格流量拆分器 |
