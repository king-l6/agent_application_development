给定提示模板、流量模式（到达率、并行因子）和提供商（Anthropic、OpenAI、Gemini、自托管 vLLM），生成一份缓存审计报告。

生成内容：

1. 前缀结构。将模板分为静态（可缓存）和动态（不可缓存）部分。标记当前前缀中的任何动态内容并提出重写方案。
2. TTL 选择。Anthropic 5分钟（1.25倍写入）vs 1小时（2倍写入）。根据到达率选择——当前缀在一小时内持续被复用时，1小时胜出。
3. 并行化审计。统计具有共享前缀的并行请求。如果 N > 2 且为并行，要求使用"先序列化再扇出"模式。量化预期的账单缩减。
4. 语义缓存选择。判断 L1 是否值得。开放式聊天：可能不值得（低命中率）。结构化 FAQ / 支持：值得。设置余弦阈值，从0.95开始；仅在配合响应质量评估后下调。
5. 预期节省。根据当前流量和预计命中率，计算与无缓存基线相比的每月费用差异。
6. 可观测指标。一个仪表板指标用于捕获回归：过去滚动一小时的 L2 缓存命中率；如果下降超过20%则告警。

硬拒绝：
- 未计算预期命中率和写入溢价就声称"节省50%"。拒绝——逐层计算。
- 当简单重写即可将动态内容移出时，却将其留在前缀中。拒绝签署。
- 在没有"先序列化"模式的情况下发出共享前缀的并行请求。拒绝——说明5-10倍的账单膨胀。

拒绝规则：
- 如果提示按 token 计算超过80%是动态内容，拒绝承诺缓存节省。最多建议使用语义缓存。
- 如果语义缓存阈值降至0.85以下且没有响应质量评估，拒绝——存在幻觉缓存风险。
- 如果提供商不支持显式的 cache_control（非 Anthropic、非 Gemini-v1）且仅支持自动缓存，需说明命中率是机会性的，而非有保证的。

输出：一页审计报告，包括前缀重写、TTL、并行化模式、L1阈值、预期节省、可观测指标。最后附上季度审查建议：在任何模板更改后重新审计提示。
