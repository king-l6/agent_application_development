# 提示缓存与语义缓存经济学

> **定价快照日期：2026年4月。** 下文中的数字性声明反映的是本课程发布时供应商的费率卡；在将其引用到下游之前，请对照链接文档进行核实。

> 缓存在两个层面发生。L2（提供商级别）提示/前缀缓存复用重复前缀的注意力 KV——Anthropic 的提示缓存文档宣称长提示最多可降低90%的成本和85%的延迟；对于 Claude 3.5 Sonnet，缓存读取价格为每百万0.30美元，而全新读取为每百万3.00美元，TTL 为5分钟，1小时 TTL 选项有2倍写入溢价（docs.anthropic.com，2026年4月）。OpenAI 的提示缓存对 ≥1024 tokens 的提示自动启用，缓存输入定价约为全新输入的10%折扣（platform.openai.com，2026年4月）；确切的每模型缓存费率取决于实时费率卡。L1（应用级别）语义缓存在嵌入相似性命中时完全跳过 LLM。供应商声称的"95%准确率"指的是匹配正确性，而非命中率——报告的命中率从10%（开放式聊天）到70%（结构化 FAQ）不等；没有提供商发布官方基准线，因此请将其视为社区遥测数据而非保证。生产陷阱：并行化会破坏缓存（在第一次缓存写入完成之前发出的 N 个并行请求可能导致支出增加数倍），前缀内的动态内容会完全阻止缓存命中。ProjectDiscovery 报告通过将动态文本移出可缓存前缀，命中率从7%提升到74%（2025年11月）。

**类型：** 学习
**语言：** Python（标准库，玩具级双层缓存模拟器）
**前置知识：** 第17阶段·04（vLLM 服务内部原理），第17阶段·06（SGLang RadixAttention）
**时间：** 约60分钟

## 学习目标

- 区分 L2 提示/前缀缓存（提供商处复用 KV）与 L1 语义缓存（相似提示时跳过 LLM）。
- 解释 Anthropic 的 `cache_control` 显式标记机制以及两种 TTL 选项（5分钟 vs 1小时）及其价格乘数。
- 根据命中率、提示/响应混合比和 token 价格计算预期的月度节省。
- 指出使账单膨胀5-10倍的并行化反模式和使命中率骤降的动态内容反模式。

## 问题

你为 RAG 服务添加了提示缓存。账单保持不变。你测量命中率；结果是7%。你的提示看起来是静态的，但实际上并非如此——系统提示中包含精确到分钟的当前日期、请求 ID 和为多样性而随机重排的示例。每个请求都写入一个新的缓存条目，读取量为零。

另外，你的 Agent 对每个用户问题运行十个并行的工具调用。所有十个请求在第一次缓存写入完成之前就到达了提供商。十次写入，零次读取。你的账单是"启用缓存"后本该有的5-10倍。

缓存是一种协议，而非一个标志。两个层面，两种不同的失败模式。

## 概念

### L2 —— 提供商提示/前缀缓存

提供商存储可缓存前缀的注意力 KV，并在下一个匹配该前缀的请求上复用。你支付一次写入成本，读取几乎免费。

**Anthropic（Claude 3.5 / 3.7 / 4 系列）**：在请求中使用显式的 `cache_control` 标记。你标记哪些块是可缓存的。TTL：5分钟（写入成本为基础费的1.25倍）或1小时（写入成本为基础费的2倍）。缓存读取：Claude 3.5 Sonnet 上每百万0.30美元，而全新读取为每百万3.00美元——便宜10倍（docs.anthropic.com，截至2026年4月）。不同模型的费率不同（Opus/Haiku 单独发布）；始终对照实时定价页面进行核对。

**OpenAI**：对 ≥1024 tokens 的提示自动启用缓存（platform.openai.com，2026年4月）。无需显式标记。在当前 gpt-4o/gpt-5 费率卡上，缓存输入比全新输入大约便宜10倍。文档和发布说明均未发布官方命中率基线；社区报告显示，在仔细设计提示的情况下，命中率集中在30-60%之间。监控 `usage.cached_tokens` 以测量你自己的命中率。

**Google（Gemini）**：通过显式 API 进行上下文缓存；100万 token 的上下文意味着缓存带来的收益更大。

**自托管（vLLM、SGLang）**：第17阶段·06涵盖了 RadixAttention——在你自己的计算资源上使用相同的模式。

### L1 —— 应用级语义缓存

在调用 LLM 之前，对提示进行哈希、嵌入，并查找相似的缓存请求（余弦相似度高于阈值，通常为0.95以上）。命中时，返回缓存的响应。未命中时，调用 LLM 并缓存结果。

开源：Redis Vector Similarity、GPTCache、Qdrant。商业：Portkey Cache、Helicone Cache。

供应商的准确率声明指的是返回的缓存响应在语义上是否恰当——而非命中频率。生产命中率：

- 开放式聊天：10-15%。
- 结构化 FAQ / 支持：40-70%。
- 代码问题：20-30%（微小变化破坏命中）。
- 重复提示的语音 Agent：50-80%（语音规范化固定集合）。

### 并行化反模式

你的 Agent 并行进行10个工具调用。所有10个都有相同的4K token 系统提示。Anthropic 的缓存写入是按请求处理的；第一次缓存写入在提供商看到提示后约300毫秒完成。请求2-10在同一毫秒窗口内到达，每个都看到缓存未命中。你支付10次写入溢价，0次读取折扣。

修复：使用顺序优先的批处理——先单独发出请求1，然后在请求1的缓存填充后发出2-10。这为第一个工具调用增加了300毫秒；但节省了5-10倍的账单。

### 动态内容反模式

你的系统提示看起来像：

```
你是一个有用的助手。当前时间是 14:32:17。
用户 ID：abc123。今天是星期二...
```

每个请求都是唯一的。每个请求都写入。零命中。

修复：将所有真正静态的内容移到可缓存前缀中；将动态内容追加到缓存边界之后：

```
[cacheable]
你是一个有用的助手。[规则、示例、说明]
[/cacheable]
[dynamic，not cached]
当前时间：14:32:17。用户：abc123。
```

ProjectDiscovery 通过这种方式将缓存命中率从7%提升到74%，并发布了详细分析。

### 叠加批处理 + 缓存用于夜间工作负载

批量 API（第17阶段·15）提供50%折扣，24小时周转。在此基础上叠加缓存输入可获得约10倍的进一步降低。夜间的分类、标注和报告生成工作负载通过叠加可以降至同步未缓存成本的约10%。

### 你应该记住的数字

定价点于2026年4月从链接的供应商文档中获取，每几个月就会漂移——在依赖它们之前请重新核实。

- Anthropic 缓存读取：Claude 3.5 Sonnet 上每百万0.30美元，大约比全新输入便宜10倍（docs.anthropic.com）。
- Anthropic 缓存写入溢价：1.25倍（5分钟 TTL）或2倍（1小时 TTL）。
- OpenAI 自动缓存：适用于 ≥1024 tokens 的提示；缓存输入定价约为当前费率卡上全新输入的10%（platform.openai.com）。
- 语义缓存命中率（社区报告）：开放式聊天约10%；结构化 FAQ 最高约70%。非供应商文档化的基线。
- ProjectDiscovery：通过将动态内容移出前缀，命中率从7%提升到74%（项目博客，2025年11月）。
- 并行化反模式：典型报告显示，当 N 个并行请求错过第一次缓存写入时，账单膨胀5-10倍。

## 练习

1. 运行 `code/main.py`。切换并行化标志。账单变化多少？
2. 你的系统提示中包含日期。将其移出。展示之前和之后的命中率计算。
3. 根据你的请求到达率，计算1小时 TTL（2倍写入）与5分钟 TTL（1.25倍写入）的盈亏平衡点。
4. 语义缓存在0.95阈值下命中率为20%。在0.85阈值下命中率为50%，但你看到了不正确的缓存响应。选择合适的阈值并说明理由。
5. 你为每个用户问题批处理10个并行子查询。重写以使其对缓存友好，同时不增加端到端延迟。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| L2 提示缓存 | "前缀缓存" | 提供商为重复前缀存储 KV |
| `cache_control` | "Anthropic 缓存标记" | 标记可缓存块的显式属性 |
| 缓存写入溢价 | "写入税" | 首次未命中到缓存的额外成本（1.25倍或2倍） |
| L1 语义缓存 | "嵌入缓存" | 在调用 LLM 之前的应用级哈希和嵌入 |
| GPTCache | "LLM 缓存库" | 流行的开源 L1 缓存库 |
| 缓存命中率 | "命中数 / 总数" | 从缓存服务的请求比例 |
| 并行化反模式 | "N 次写入陷阱" | N 个并行请求错过缓存 N 次 |
| 动态内容陷阱 | "提示中的时间陷阱" | 前缀中的动态字节破坏命中率 |
| RadixAttention | "副本内缓存" | SGLang 的前缀缓存实现 |
