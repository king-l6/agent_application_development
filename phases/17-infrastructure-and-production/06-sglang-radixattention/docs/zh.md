# SGLang 与 RadixAttention：前缀繁重工作负载的利器

> SGLang 将 KV 缓存视为一级可重用资源，存储在基数树中。vLLM 按 FCFS（先来先服务）调度请求，而 SGLang 的缓存感知调度器优先考虑共享前缀更长的请求 — 实际上是深度优先的基数遍历，使热分支保持在 HBM 中。在 Llama 3.1 8B 上使用 ShareGPT 风格的 1K 提示，SGLang 达到约 16,200 tok/s，vLLM 约 12,500，优势约 29%。在前缀繁重的 RAG 工作负载上，优势可达 6.4 倍。在语音克隆形状的工作负载上，缓存命中率超过 86%。2026 年已部署在 400,000+ GPU 上，涵盖 xAI、LinkedIn、Cursor、Oracle、GCP、Azure、AWS。陷阱在于，当前缀排序不一致时，6.4 倍的数字会消失 — 排序是工程师的杠杆。

## 学习目标

- 绘制 RadixAttention：前缀如何存储在基数树中，以及 KV 块如何在同一分支上不同序列之间共享。
- 解释缓存感知调度，以及为什么 FCFS 对于前缀繁重的流量是错误的。
- 给定前缀缓存命中率和提示长度分布，计算工作负载的预期加速比。
- 指出使 6.4 倍数字变为现实的提示排序纪律 vs 失去的收益。

## 问题

经典服务将每个请求的提示视为不透明。即使 5,000 个 RAG 请求都以相同的 2,000 token 系统提示加上相同的检索前导开始，vLLM 也会预填充那 2,000 token 前缀 5,000 次。GPU 一遍又一遍地做同样的工作。

观察：在 agentic 和 RAG 工作负载中，提示几乎总是共享长前缀。系统提示、工具模式、few-shot 示例、检索标头、对话历史 — 这些跨请求重复。如果你存储了该前缀的 KV 缓存一次并重用它，你就不需要再次预填充它。

RadixAttention 正是这样做的。Token 在基数树中建立索引；每个节点拥有其从根路径上的 token 序列的 KV 块。新请求遍历树：任何 token 匹配的节点重用该节点的 KV 块。预填充成本变得与"新"后缀成正比，而不是完整提示。

挑战在于调度。如果两个请求共享一个 2,000 token 前缀，而第三个只共享同一前缀的 200 个 token，你希望将两个长共享请求一起服务，以便长前缀保留在 HBM 中。FCFS 反其道而行之 — 它按先到先服务提供服务，可能在下一次长前缀请求到来之前驱逐热分支。

## 概念

### 基数树作为 KV 索引

基数树（紧凑字典树）存储 token 序列。每个节点拥有一个 token 范围以及为该范围计算的 KV 块。子节点扩展该序列一个或多个 token。

```
root
 |- "You are a helpful assistant..."  (2,000 tokens, 124 KV blocks)
      |- "Context: <doc A>..."        (500 tokens, 31 blocks)
           |- "Question: Alice..."    (80 tokens, 5 blocks)
           |- "Question: Bob..."      (95 tokens, 6 blocks)
      |- "Context: <doc B>..."        (520 tokens, 33 blocks)
```

一个新请求进来，带有系统提示 + "Context: <doc A>" + "Question: Carol"。调度器遍历：系统前缀匹配（重用 124 个块），doc-A 分支匹配（重用 31 个块），然后只为 "Question: Carol" 分配新块（4 个块）。预填充成本：4 个新 token 的块。没有树：160 个块。预填充节省约 40 倍。

### 缓存感知调度

如果缓存不断流失，基数树支持的重用就毫无意义。两个关键策略：

1. **深度优先派发。** 从队列中选择下一个请求时，优先选择与当前运行集在同一分支上根植的请求。这使热分支保持 pinned。
2. **分支级 LRU，而非块级 LRU。** 驱逐整个分支（从最短使用的叶子开始）而非单个块，这样缓存的形状与基数形状匹配。

FCFS 违反了两者。一个共享 2,000 token 的请求排在一个共享 50 token 的请求后面，然后 2,000 token 分支被驱逐以接纳 50 token 的分支。

### 你应该记住的基准测试数字

- Llama 3.1 8B，H100，ShareGPT 1K 提示：SGLang 约 16,200 tok/s vs vLLM 约 12,500（约 29% 优势）。
- 前缀繁重 RAG（相同系统 + 相同文档，变化的问题）：SGLang 上高达 6.4 倍。
- 语音克隆工作负载：86.4% 前缀缓存命中率。
- SGLang 客户的生产命中率：50-99%，取决于提示纪律。
- 2026 年已部署在 400,000+ GPU 上。

### 排序陷阱

6.4 倍的数字依赖于一致的提示模板排序。如果你的客户端在某些请求中将提示构造为 `[system, tools, context, history, question]`，而在另一些请求中构造为 `[system, context, tools, history, question]`，树无法找到共享前缀。对人类来说看似共享的前缀，对基数树来说是两个不同的序列。

工程师的杠杆：你的提示模板是一个缓存键。固定顺序。将所有不可变内容（系统、工具、模式）放在前面。将检索上下文放在中间。将用户问题放在最后。不要将动态内容插入到前缀中。

研究中的真实案例：将动态内容移出可缓存前缀，一次部署就将缓存命中率从 7% 提高到 74%。

### RadixAttention 胜出和失利的场景

胜出：
- RAG（相同的检索前导，变化的问题）。
- Agent（相同的工具模式，变化的查询）。
- 带有长系统提示的聊天。
- 具有重复前导的语音/视觉工作负载。

失利（回到 vLLM 级别的吞吐量）：
- 具有唯一提示的单次生成（代码补全、无系统提示的开放式聊天）。
- 动态提示，每个请求将唯一内容插入到前缀中。

### 为什么这是调度器问题，而不仅仅是核函数问题

你可以将 KV 重用实现为核函数技巧。SGLang 的洞察是，只有调度器保持热分支驻留时，重用才有效。一个朴素的"可用即重用"策略会在混合负载下使缓存不断流失。基数树索引的调度器才是将核函数技巧转化为 29% 生产优势的关键。

### 与 vLLM 的相互关系

这两个系统并非严格意义上的竞争对手。2026 年 vLLM 添加了前缀缓存（`--enable-prefix-caching`）和一个缓存感知路由器（Rust 编写的 vLLM Router）。差距缩小了但并未完全消失 — SGLang 的整个栈是以基数为先的；vLLM 是嫁接的。对于以前缀重用为主的工作负载，SGLang 仍然是默认选择。对于没有强前缀模式的通用服务，vLLM 仍然相等或更好。

## 关键术语

| 术语 | 人们说的是 | 实际含义 |
|------|-----------|---------|
| RadixAttention | "SGLang 的东西" | KV 缓存以基数树索引，使共享前缀重用块 |
| 基数树 | "紧凑字典树" | 每个节点拥有一个 token 范围及其 KV 块的树 |
| 缓存感知调度器 | "热分支优先" | 优先选择与驻留分支共享的请求的调度器 |
| 前缀缓存命中率 | "你的提示有多少是免费的" | 从重用 KV 块服务的提示 token 比例 |
| FCFS | "先来先服务" | 破坏前缀局部性的默认调度 |
| 分支级 LRU | "驱逐叶子" | 与基数形状匹配的驱逐策略 |
| 提示模板排序 | "缓存键" | 提示的组件顺序决定树可以共享什么 |
| 系统提示固定 | "驻留前缀" | 保持不可变系统部分固定，避免驱逐抖动 |
