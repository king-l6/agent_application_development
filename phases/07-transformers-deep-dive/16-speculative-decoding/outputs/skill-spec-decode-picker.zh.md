# 推测解码选择器

帮助工程师在普通推测、Medusa、EAGLE 或 lookahead 解码之间做出选择，并为特定工作负载调优 `N`（草稿长度）。

## 收集的输入

1. **验证器模型**——哪个 LLM 产生最终输出。大小很重要（草稿成本必须小于验证器成本才能获得加速）。
2. **工作负载类型**——代码、聊天、结构化输出、摘要。决定接受率。
3. **采样策略**——贪心、低温、高温、束搜索。高温采样会降低接受率。
4. **硬件目标**——内存预算决定了你是否能容纳一个独立的草稿模型。
5. **工程预算**——Medusa 和 EAGLE 需要微调；普通推测和 lookahead 不需要。
6. **延迟目标**——交互式聊天（<500ms TTFT，<50ms 每 token）与批处理（吞吐量优先）。

## 决策规则

- **快速启动，无需训练**：使用同家族的 1B-3B 草稿模型的普通推测。典型 2 倍加速。
- **你可以微调**：EAGLE-2 或 EAGLE-3，使用验证器的隐藏状态。典型 3-4 倍。
- **你可以微调但不能运行两个模型**：Medusa（验证器上的额外头）。2-3 倍。
- **没有训练预算，没有可用的草稿模型**：lookahead 解码。1.3-1.6 倍。
- **批处理为主的服务**：连续批处理更重要；随着批次增大，推测收益递减，因为验证器已经饱和。
- **高温或随机采样**：接受率急剧下降。考虑降低 N（2-3）或禁用。
- **结构化输出（JSON、代码）**：接受率高。将 N 推到 7+ 以获得最大加速。

## 调优

- **N（草稿长度）**：从 5 开始。测量接受率。如果 α > 0.9，推到 7。如果 α < 0.6，降到 3。
- **草稿温度**：匹配验证器的温度。不匹配的草稿采样会损失 α。
- **树深度（EAGLE-2 / Medusa）**：3-5 个分支；更宽的树仅在 α > 0.8 时有帮助。
- **草稿模型大小**：能实现 α > 0.7 的最小模型。对于 70B 验证器，典型的草稿是 1B；不要低于验证器的 tokenizer / 嵌入兼容性。

## 始终标记

- 检查草稿和验证器是否共享 tokenizer。不同的 BPE 分割会破坏推测保证。
- 推测解码在 vLLM 中与连续批处理交互：当批次已经饱和时，每请求的加速下降。
- EAGLE 的隐藏状态输入需要验证器内部结构；并不总是通过 HF API 暴露。优先使用 vLLM 或 SGLang 运行时。
- Medusa 头需要在验证器自己的输出上进行监督微调。数据收集步骤通常是主要成本。

## 输出格式

返回：

1. **推荐**——一个策略名称和调优参数（例如"EAGLE-2，N=5，tree_depth=4"）。
2. **预期加速**——附明确的 α 假设。
3. **兼容性检查**——tokenizer 匹配、运行时支持、KV cache 回滚支持。
4. **回退方案**——如果主要策略表现不佳，接下来尝试什么。
5. **测量计划**——如何在代表性样本上验证接受率和加速比。
