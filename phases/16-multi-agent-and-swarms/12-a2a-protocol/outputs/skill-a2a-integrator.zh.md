给定两个需要互操作的智能体系统，生成 A2A 集成方案：智能体卡片内容、任务模式、认证、传输模式。

生成内容：

1. **智能体卡片。** 名称、版本、技能、端点、支持的模态（文本、结构化、图片、音频、视频）、protocol_version、认证声明。
2. **每个技能的任务模式。** 输入 JSON 模式 + 产物 JSON 模式。要明确——客户端将进行验证。
3. **认证选择。** Bearer 令牌（OAuth2 或不透明）、mTLS 或签名请求。根据威胁模型（公共互联网、VPC、混合）说明理由。
4. **传输模式。** 轮询 vs SSE 流式传输 vs webhook 回调。长时间运行或进度密集型任务用流式传输；短任务用轮询。
5. **速率限制。** 每个客户端和每个任务的限制。防止滥用。
6. **幂等性。** 重复 `POST /tasks` 请求的策略（客户端侧任务键、服务器侧去重）。
7. **失败处理。** `failed` 之外的任务状态（可重试 vs 致命）、死信策略、错误产物模式。
8. **MCP vs A2A 分工。** 如果远程智能体内部使用 MCP，指出哪些工具是暴露的，哪些是内部保留的。

硬拒绝：

- 没有声明协议版本的智能体卡片。
- 用例需要结构化但任务模式却是自由形式文本。
- 在公共互联网部署中 auth=none。

拒绝规则：

- 如果两个智能体在同一进程中运行，拒绝 A2A 并推荐直接的 Python/JS 调用。A2A 用于跨系统边界。
- 如果延迟要求是亚 100ms 往返，拒绝 A2A 并推荐带有共享模式的直接 RPC。
- 如果远程智能体没有声明智能体卡片，拒绝集成并推荐先发布一个。

输出：一页的集成简报。以内联粘贴的智能体卡片 JSON 结束，以便工程团队可以将其放入 `/.well-known/agent.json`。
