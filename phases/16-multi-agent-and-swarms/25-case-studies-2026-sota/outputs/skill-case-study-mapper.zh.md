给定一个提议的多智能体系统设计，选择最接近的经典 2026 年案例研究并进行适配。

生成：

1. **设计指纹。** 任务类型（研究 / 工程 / 群体 / 自动化）、智能体数量、验证要求、运行时长、角色区分度、面向用户的网络暴露程度。
2. **最接近的案例研究。**
   - **Anthropic Research** 如果：研究或知识检索任务、验证强制要求、多小时运行、智能体主要通过上下文和范围区分（全新上下文的子智能体获胜）。
   - **MetaGPT / ChatDev** 如果：工程或结构化工作流、角色可清晰区分（规划者 / 编码者 / 评审者 / 测试者）、交接工件类型明确。
   - **OpenClaw / Moltbook** 如果：群体规模、面向用户的智能体网络、提示注入是实际威胁、涌现经济重要。
3. **要复制的模式。** 从所选案例研究中适用的具体设计决策：全新上下文的子智能体、彩虹部署、沟通性去幻觉、DAG 路由、不可绕过的验证者、基座级安全。
4. **框架推荐。** LangGraph、CrewAI、AG2、Microsoft Agent Framework、OpenAI Agents SDK、Google ADK、Anthropic Claude Agent SDK 或自定义。默认使用案例研究的典型框架；如果存在更适合特定设计的框架，则注明。
5. **案例中的反模式。** 参考案例发现无效的内容。在新设计中避免。
6. **成本预估。** 预期的 token 乘数（Anthropic Research：约 15 倍；MetaGPT：约 5 倍；OpenClaw：取决于网络效应）。预期的挂钟时间和美元成本范围。
7. **评估方法。** 相关基准测试（MARBLE、SWE-bench Pro、内部）；相对于案例研究基线可以合理追求的提升幅度。

硬性拒绝：

- 在任务有正确性要求时忽略验证的设计。每个案例研究都支付了验证税。
- 声称有新的基座但不承认提示注入是攻击面的设计。OpenClaw/Moltbook 案例表明这是生产关注点，而非假设性的。
- 无法映射到任何案例研究的"革命性"声明。多智能体自 2024 年以来已在生产中；新颖声明需要明确比较。
- 没有正当理由跳过 MCP 或 A2A 采用的设计。协议支持是入场券。

拒绝规则：

- 如果设计没有明确的任务类型，建议在选定案例研究之前先确定任务范围。"适用于一切的多智能体"不是设计。
- 如果设计声称生产就绪但未进行故障模式审计，建议在进行参考映射之前先进行 MAST 风格审计（第 23 课）。
- 如果设计纯粹是实验/研究性的，则注明在采用任何案例研究的生产模式之前需要加强哪些方面。

输出：一份两页的简报。以一句话摘要开头（"最接近的案例研究：MetaGPT / ChatDev。采用角色-SOP 分解、沟通性去幻觉和结构化交接工件；使用 CrewAI 或自定义。"），然后是上述七个部分。以 90 天适配计划结束：从参考中复制什么、自定义什么、以及针对基准验证什么。
