给定一个多智能体系统和采样的执行轨迹，运行失败模式审计。

生成：

1. **样本构建。** 至少 200 条来自生产的轨迹，跨任务类型和时间窗口均匀采样。记录采样方法和偏差风险。
2. **分类遍历。** 对每条轨迹，标记 `成功 | 失败`。对于失败，分配一个 MAST 类别（规约/协调/验证），并在适用时分配一个或多个群体思维家族标签（monoculture / 从众 / ToM / 混合动机 / 级联）。
3. **分布表。** 按 MAST 类别和群体思维标签的计数和百分比。与 Cemri 2025 的参考分布（41.77 / 36.94 / 21.30）比较。严重偏离参考的系统通常有一个特定的薄弱层。
4. **主要失败模式。** 识别 3 个最常见的具体模式（例如"两个智能体都进行评审"）。记录复现步骤。
5. **缓解措施排序。** 针对每个主要模式，从标准库中提出缓解措施：显式角色合约、版本化共享状态、独立验证者、断路器、检测-诊断-验证（STRATUS）三人组。根据模式频率按预期失败减少量排序。
6. **无声失败的风险。** 多少失败产生看似合理但错误的输出 vs 响亮错误？无声率驱动验证层投资。
7. **缓慢失败代理。** 推荐 2-3 个能够在其变成响亮错误之前发现漂移的实时指标：一致率、重试率、输出长度分布、智能体间编辑距离。

硬性拒绝：

- 没有随机或分层样本的审计。手工挑选的失败会过度代表戏剧性案例并错过缓慢失败漂移。
- 没有基线测量的缓解措施建议。"添加验证者"如果当前失败率未知则毫无意义。
- 忽略 MAST 未知事件。如果轨迹不适合某个类别，分类法不完整；提议扩展而非强制分类。
- 声称在没有操作级缓慢失败监控的情况下季度审计就足够了。季度审计无法捕捉审计间隔间的漂移。

拒绝规则：

- 如果轨迹缺乏每个智能体的归属（谁写了什么、谁读了什么），审计无法区分协调失败和角色冲突。建议在重新审计前添加结构化的每个智能体日志。
- 如果系统总共有少于 50 条失败轨迹，样本太小无法产生分布估计。建议更长的观察窗口。
- 如果轨迹包含 PII，在分析前进行脱敏。

输出：三页报告。以一句话总结开头（"41% 规约失败、12% 协调、39% 验证缺口、8% 未知；主要模式是双重评审者冲突；最高 ROI 的缓解措施是显式角色合约。"），然后是以上七个部分。以优先行动列表结束：三个缓解措施，附估计的实施成本和预期的失败率降低。
