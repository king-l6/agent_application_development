# 失败模式 —— MAST、群体思维、 monoculture、级联错误

> 2026 年的参考分类法是 **MAST**（Cemri 等人，NeurIPS 2025，arXiv:2503.13657），基于 7 个最先进的开源 MAS 的 1642 条执行轨迹推导得出，显示 **41–86.7% 的失败率**。三个根类别：**规约问题**（Specification Problems，41.77%）—— 角色模糊、任务定义不明确；**协调失败**（Coordination Failures，36.94%）—— 通信中断、状态不同步；**验证缺口**（Verification Gaps，21.30%）—— 缺少验证、缺乏质量检查。**群体思维**家族（arXiv:2508.05687）补充了：monoculture 崩溃（相同基座模型 → 关联性失败）、从众偏差（智能体相互强化错误）、缺陷心智理论、混合动机动态、级联可靠性失败。级联示例：重试风暴 —— 支付失败触发订单重试，订单重试触发库存重试，库存重试压垮库存服务（数秒内 10 倍负载 —— 需要断路器）。内存中毒：一个智能体的幻觉进入共享内存，下游智能体将其当作事实；准确率逐渐衰减，使根因诊断变得痛苦。**STRATUS**（NeurIPS 2025）报告通过专门的检测/诊断/验证智能体实现了 1.5 倍的缓解成功提升。本课将失败模式作为头等工程目标来对待。

**类型：** 学习
**语言：** Python（标准库）
**前置条件：** Phase 16 · 13（共享内存），Phase 16 · 14（共识与 BFT），Phase 16 · 15（投票与辩论拓扑）
**时间：** ~75 分钟

## 问题

多智能体系统在真实任务上的失败率为 41-86.7%（Cemri 等人 2025 年跨越 7 个开源 MAS 测量得到）。这不是通过"多加几个智能体"就能调试的。失败有结构性原因。MAST 分类法为你提供了类别。本课将每个类别映射到具体的检测、诊断和缓解模式，使这些数字不再显得随意。

2026 年的生产实践是将失败模式视为设计输入。你的架构只有在你能够针对每个 MAST 类别指出你所部署的缓解措施时，才算是"足够好"。

## 概念

### MAST 类别

**规约问题（占失败的 41.77%）。** 智能体的任务定义不够精确。示例：

- 角色模糊：两个智能体都认为自己是评审者。
- 任务说明不足：用户想要特定角度时只说"总结一下这个"。
- 成功标准隐式：智能体无法判断自己是否成功。

缓解措施：
- 编写明确的角色合约。每个智能体的提示说明它做什么*以及不做什么*。
- 每个任务的验收测试。在智能体开始前，定义"完成看起来像 X"。
- 前置规约检查：一个单独的智能体在分派前审查任务定义。

**协调失败（占 36.94%）。** 通信或状态中断。

示例：
- 两个智能体不同步地更新共享状态。
- 智能体间消息丢失（队列故障、超时）。
- 状态漂移：智能体 A 认为任务已完成；智能体 B 仍在执行。

缓解措施：
- 带乐观并发的版本化共享状态。
- 关键消息的显式确认（重试直到确认）。
- 周期性状态同步检查点；及早检测漂移。

**验证缺口（占 21.30%）。** 对输出没有独立检查。

示例：
- 一个智能体声称成功；无人验证。
- 智能体链中的每个智能体信任前一个的输出。
- 对涌现的组合行为缺少测试覆盖。

缓解措施：
- 独立验证者智能体（第 13 课）。只读、独立源访问。
- 显式交接合约："A 的输出必须通过检查器 C 后 B 才能开始。"
- 用于事后分析的结果日志记录。

### 群体思维家族（arXiv:2508.05687）

当智能体同质化或相互模仿时的五种相关失败：

**Monoculture 崩溃。** 相同的基座模型或训练数据 → 关联性错误。当三个智能体共享同一个 LLM 时，它们共享同一个 LLM 的幻觉。

**从众偏差。** 智能体朝着声音最大或最自信的同伴调整，即使对方是错误的。

**缺陷心智理论。** 智能体无法模拟彼此的信念；协调崩溃（第 18 课）。

**混合动机动态。** 激励部分对齐的智能体漂向妥协的中点，结果无人满意。

**级联可靠性失败。** 一个组件的错误模式触发依赖组件中的错误模式。

### 级联示例 —— 重试风暴

一个经典的 2026 年事故模式：

```
支付服务 10% 的请求失败
   ↓
订单智能体重试支付（指数退避但 naive）
   ↓
每次重试都是一次新的订单库存检查
   ↓
库存服务看到 2 倍正常负载
   ↓
库存服务开始超时
   ↓
每个订单重试库存检查
   ↓
库存服务看到 10 倍正常负载
   ↓
集群宕机
```

修复方法是经典的：**断路器**。当下游错误率超过阈值时，用缓存或默认结果短路。再加上每个请求的重试预算上限。

断路器是少数几个你直接从分布式系统借用而无需修改的多智能体失败缓解措施之一。

### 内存中毒（重访）

来自第 13 课：一个智能体的幻觉成为共享内存中的事实；下游智能体在中毒的事实上推理。按 MAST 术语，这是共享内存层的一个验证缺口。

逐渐的准确率下降是症状。你不会得到一个崩溃；你得到的是难以根因诊断的缓慢漂移。

缓解措施：仅追加日志、来源追踪、不可写验证者。第 13 课已涵盖。

### STRATUS —— 用于失败检测的专门智能体

STRATUS（NeurIPS 2025）报告，当你部署以下角色时，缓解成功提升 1.5 倍：

- **检测智能体。** 监控症状模式（高度分歧、重试尖峰、准确率漂移）。
- **诊断智能体。** 给定症状，从 MAST 分类法推断可能的根因。
- **验证智能体。** 应用缓解措施后，检查症状是否消失。

这是 SRE 风格的事故响应，应用于智能体系统。这三个角色都可以是通过专门提示实现的 LLM 智能体。

### 失败模式审计

2026 年的一个最佳实践是年度（或每次重大发布）失败模式审计：

1. **轨迹采样。** 收集约 1000 条真实执行轨迹。
2. **分类。** 对于每条轨迹的失败，映射到 MAST + 群体思维类别。
3. **按类别计算失败率。** 哪些类别在你的系统中占主导？
4. **排序缓解措施。** 哪种修复能消除最多的失败？
5. **选择 2-3 个缓解措施。** 实施；下个季度重新审计。

这种纪律比具体选择更重要。没有审计，失败会混入噪声中，永远得不到系统性解决。

### 当系统无声失败时

最危险的失败类别是无声的正确性失败。一个大声失败的系统（崩溃、异常、告警）可以被监控。一个产生看似合理但错误输出的系统无法通过异常日志检测。这就是为什么验证缺口按每次失败的成本计算是最昂贵的类别，尽管按数量只占 21.30%。

投资于：
- 基于样本的人工审查。
- 金数据集回归测试。
- 重要输出上的跨智能体交叉检查。

### 快速失败 vs 缓慢失败

有些失败是即时的；有些是缓慢的。即时失败（超时、模式不匹配、认证错误）检测成本低。缓慢失败（内存中毒、monoculture 漂移、角色模糊）检测和预防成本高。

2026 年的工程手段：为缓慢失败设置代理指标，以便你能够在漂移变成可见错误之前捕获它。一致率、重试率、输出长度分布和连续智能体版本之间的编辑距离都是有用的代理指标。

## 构建它

`code/main.py` 实现：

- `FailureTaxonomy` —— 将模拟事件分类为 MAST + 群体思维类别。
- `CircuitBreaker` —— 经典模式；错误率超过阈值时断开。
- `RetryStormSimulator` —— 展示级联失败；切换断路器开/关。
- `DetectionAgent` —— 脚本化的 STRATUS 风格症状匹配器。

运行：

```
python3 code/main.py
```

预期输出：
- 无断路器的重试风暴：库存错误爆炸（模拟）。
- 有断路器：在阈值处限制；提供降级模式响应。
- 检测智能体标记该模式并命名 MAST 类别。

## 使用它

`outputs/skill-mast-auditor.md` 在多智能体系统上运行 MAST 风格的失败模式审计。轨迹 → 分类 → 缓解措施排序。

## 交付它

生产中的失败模式纪律：

- **每季度 MAST 审计。** 不是每年。类别会随系统增长而变化。
- **处处使用断路器。** 每个向任何依赖服务的出站调用。默认在 5-10% 错误率处断开。
- **金数据集。** 小型、高质量、人工审计。每周进行回归测试。
- **STRATUS 三人组。** 检测 + 诊断 + 验证智能体监控生产。先从检测智能体开始；当症状嘈杂时再添加诊断。
- **失败预算。** 按类别的失败率显式 SLO。超出预算触发停止发布的讨论。

## 练习

1. 运行 `code/main.py`。确认断路器限制重试风暴。改变失败阈值并观察权衡。
2. 实现一个**缓慢失败代理**：3 个并行智能体间的一致率。当它急剧下降时，触发告警。通过逐渐关联智能体输出来模拟 monoculture 漂移。
3. 阅读 Cemri 等人（arXiv:2503.13657）。选择其中的一个 MAS 系统，映射其前 3 个失败类别。它们与 MAST 预测的相比如何？
4. 阅读群体思维论文（arXiv:2508.05687）。识别五种模式中在生产中最难检测的一种。提出一个代理指标。
5. 为你所知的特定多智能体系统设计一个 STRATUS 风格的检测-诊断-验证三人组。检测关注哪些症状？诊断推荐哪些缓解措施？验证如何确认它们有效？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|---|---|---|
| MAST | "2026 年分类法" | Cemri 2025；3 个根类别 + 14 个子类型的失败。 |
| Specification Problem | "角色模糊" | 任务或角色定义不足；智能体不知道要做什么。 |
| Coordination Failure | "状态漂移" | 智能体间的通信或同步中断。 |
| Verification Gap | "无人检查" | 输出未经独立验证即被接受。 |
| Groupthink family | "同质化失败" | Monoculture、从众、缺陷 ToM、混合动机、级联。 |
| Monoculture collapse | "相同模型，相同幻觉" | 来自共享基座模型或训练数据的关联性错误。 |
| Retry storm | "级联错误放大" | 一个失败触发重试，放大下游负载。 |
| Circuit breaker | "错误率时快速失败" | 错误率超过阈值时断开；用默认值短路。 |
| STRATUS | "事故响应三人组" | 检测 + 诊断 + 验证智能体。1.5 倍缓解成功。 |
| Memory poisoning | "幻觉传播" | 共享内存事实被污染；下游智能体在毒化信息上推理。 |
