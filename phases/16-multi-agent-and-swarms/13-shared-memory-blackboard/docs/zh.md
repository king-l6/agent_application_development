# 共享记忆与黑板模式

> 2026 年多智能体系统中并存两种方法：**消息池**（每个人看到所有人的消息，如 AutoGen GroupChat 或 MetaGPT）和**带订阅的黑板**（智能体订阅相关事件，如 Context-Aware MCP 或 Matrix 框架）。两者都是多智能体系统中唯一有状态的部分——这意味着它们也是有趣错误的主要来源。参考失败模式是**记忆中毒**：一个智能体幻觉出一个"事实"，其他智能体将其视为已验证，准确度逐渐下降，其调试难度远高于即时崩溃。本课从标准库构建两种结构，注入一次中毒攻击，并展示三种在生产环境中真正有效的缓解措施。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置知识：** 第 16 阶段 · 04（基础模型），第 16 阶段 · 09（并行群网络）
**时间：** 约 75 分钟

## 问题

多智能体系统需要一个让智能体共享事实的地方。一个直接的选择是"在消息中传递一切"——但这用额外的复制重新发明了共享状态。另一个是"给每个人一个全局日志"——但全局日志无界增长且容易中毒。第三个是"为每个智能体投影一个视图"——可扩展但模式繁重。

当一个智能体产生幻觉并将幻觉写入共享状态时，每个读取该状态的下游智能体都会将该幻觉采纳为事实。等到人类注意到时，推理链已经深入五步，根本原因是第三条写入的消息。调试多智能体准确度衰减比调试崩溃更难。

这就是记忆中毒。它是 MAST 分类法（Cemri 等人，arXiv:2503.13657）中记录第二多的失败族，并且是结构性的：任何没有来源追踪和不可写入验证者的共享记忆设计最终都会出现这个问题。

## 概念

### 两种主要拓扑结构

**全消息池。** 每个智能体读取每条消息。AutoGen GroupChat 和 MetaGPT 使用此方式。简单、透明、可检查，但无法扩展到约 10 个智能体以上，因为每个智能体的上下文会被其他智能体的工作填满。

```
智能体-A ──写入──▶ ┌────────────────┐ ◀──读取── 智能体-D
                    │ 消息池         │
智能体-B ──写入──▶ │                │ ◀──读取── 智能体-E
                    │ （全局日志）   │
智能体-C ──写入──▶ └────────────────┘ ◀──读取── 智能体-F
```

**带订阅的黑板。** 智能体声明对主题的兴趣；底层基础设施只路由相关的消息。CA-MCP（arXiv:2601.11595）和 Matrix 去中心化框架（arXiv:2511.21686）使用此方式。可扩展性更好，但需要预先设计模式以使订阅有意义。

```
                   ┌─ 主题：价格 ──┐
智能体-A ──发布──▶ │              │ ──▶ 智能体-D（已订阅）
                   ├─ 主题：订单 ──┤
智能体-B ──发布──▶ │              │ ──▶ 智能体-E（已订阅）
                   ├─ 主题：告警 ──┤
智能体-C ──发布──▶ │              │ ──▶ 智能体-F（已订阅）
                   └──────────────┘
```

### 各自的优势场景

- **全池**在智能体少（< 10）、异构且对话时间短时胜出。当每个人都能看到一切时，推理谁说了什么很简单。
- **黑板**在智能体多、角色同质但实例众多（群）且对话时间长时胜出。路由节省了令牌成本和上下文污染。

生产系统通常混合使用：顶层（规划层）用小型的全池，下层（工作层）用黑板。

### 记忆中毒，一个场景

三个智能体处理一个研究任务。智能体 A 是检索智能体。智能体 B 是摘要生成器。智能体 C 是分析师。

1. A 获取了一个页面并向共享状态写入一条消息："研究报告了 42% 的准确率提升。"
2. 获取的页面实际上说的是"4.2% 的提升。"A 幻觉了一个小数点。
3. B 读取共享状态，写入："报告了大幅 42% 的准确率提升（来源：A）。"
4. C 读取共享状态，写入："建议采用——42% 的提升是变革性的。"
5. 最终报告引用了一个从未存在的 42% 的数字。

没有智能体崩溃。没有测试失败。系统"正常运行"。幻觉通过共享状态从一个智能体的上下文传播到了每个下游智能体的推理中。

### 为什么这是结构性的

没有共享状态，智能体 A 的幻觉停留在 A 的上下文中。下游智能体会重新获取或重新推导，可能发现错误。有了天真的共享状态，A 的上下文变成了每个人的上下文，幻觉被洗白为事实。

问题不在于共享状态本身——而在于共享状态**没有来源追踪且没有独立的验证者**。三种缓解措施解决了这个问题：

1. **在每次写入上标注来源。** 共享状态中的每个条目记录谁写的、何时、在什么提示下以及（如果适用）智能体引用了什么来源。下游智能体根据来源信息带着怀疑态度阅读。
2. **版本化写入；仅追加。** 更正是覆盖旧条目的新条目，而不是原地更新。审计追踪得以保留。
3. **至少保持一个不能写入共享状态的智能体。** 一个只读的验证者智能体采样条目、重新获取来源并标记不一致。因为它不能写入池，所以不会被池污染。

### 黑板先例（Hayes-Roth，1985）

黑板模式比 LLM 智能体早四十年。Hayes-Roth（1985，"A Blackboard Architecture for Control"）描述了观察全局黑板、贡献部分解决方案并触发其他来源的专家知识源。2026 年的黑板（CA-MCP、Matrix）是相同的模式，只是将 LLM 智能体作为知识源，JSON 块作为部分解决方案。旧文献已经记录了对写竞争、机会性控制和一致性的解决方案，现代系统正在重新发现这些方案。

### 投影与全视图

纯黑板给每个订阅者相同的投影（按主题范围）。一个更具侵略性的设计是**按智能体投影**：每个智能体获得一个根据其角色定制的视图。LangGraph 的状态归约器是 2026 年的规范实现——归约函数将全局状态折叠成一个角色特定的切片。

按智能体投影可扩展性更好，但需要模式说明。没有模式，你会在每个智能体的提示中重新构建临时的投影。

### 写竞争模式

多个智能体同时写入是一个并发问题，而不仅仅是 LLM 问题。三种模式有效：

- **顺序写入器（单一生产者）。** 所有写入通过一个协调智能体串行化。简单，但成为瓶颈。
- **带版本化的乐观并发。** 每个条目有一个版本；写入者在版本不匹配时失败并重试。经典的数据库技术。
- **主题分区。** 不同智能体拥有不同的主题。没有跨主题竞争。需要设计分区边界。

大多数 2026 框架默认使用顺序写入器，因为 LLM 调用足够慢，竞争很少发生，瓶颈影响不大。

### 不可写入的验证者

最重要的缓解措施是只读验证者。实现规则：

- 验证者与团队共享状态（读取黑板或池）。
- 验证者没有写入共享状态的权限——只能写入一个独立的验证通道。
- 验证者独立获取写入中引用的来源。标记不一致。
- 验证者自己的输出被路由到人类或独立的决策智能体，永远不会送回到池中。

没有这种分离，验证者的输出会成为池中的新条目，这意味着中毒的池会毒害验证者，进而毒害其验证结果。

## 构建它

`code/main.py` 在标准库 Python 中实现两种拓扑结构，加上一个小型的中毒攻击演示和三种缓解措施。

- `MessagePool` —— 线程安全的仅追加日志，支持完全读取。
- `Blackboard` —— 按主题键的发布/订阅，支持按智能体订阅。
- `ProvenanceEntry` —— 每次写入记录（写入者、时间戳、prompt_hash、source_uri）。
- `PoisoningScenario` —— 运行一个三智能体研究任务，其中智能体 A 幻觉了一个小数点。打印最终报告。
- `Verifier` —— 一个只读智能体，重新获取来源并标记不一致。在有验证者的情况下运行相同的场景。

运行：

```
python3 code/main.py
```

预期输出：
- 运行 1（无验证者）：幻觉的 42% 传播到最终报告。
- 运行 2（有验证者）：验证者标记不一致，池被标记为"flagged"，最终报告包含撤回声明。

## 使用它

`outputs/skill-memory-auditor.md` 是一个技能，用于审计任何多智能体系统的共享记忆设计，检查来源追踪、版本化和验证者分离。在生产环境之前对新多智能体架构运行此技能。

## 交付清单

对于任何共享记忆设计：

- 在每次写入上记录来源：`(writer, timestamp, prompt_hash, tool_calls_cited, source_uri)`。
- 使日志仅追加。更正是引用被覆盖条目的新条目。
- 部署至少一个具有独立来源访问权限的只读验证者智能体。
- 将验证者输出路由到独立通道，而不是返回共享池。
- 记录覆盖条目的写入比例——上升比例是幻觉模式的早期证据。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 消息池 (Message pool) | "共享聊天历史" | 每个智能体读取的仅追加日志。完全透明，可扩展性差。 |
| 黑板 (Blackboard) | "共享工作区" | 按主题键的发布/订阅。智能体订阅相关主题。可扩展性更好。 |
| 来源追踪 (Provenance) | "谁写了什么" | 每次写入的元数据：写入者、时间戳、提示、来源。 |
| 记忆中毒 (Memory poisoning) | "幻觉传播" | 一个智能体的错误进入共享状态，下游智能体将其采纳为事实。 |
| 仅追加 (Append-only) | "没有原地更新" | 更正是覆盖旧条目的新条目。保留审计追踪。 |
| 不可写入验证者 (Unwritable verifier) | "独立审计者" | 重新获取来源并标记不一致的只读智能体。 |
| 投影 (Projection) | "有范围的视图" | 从全局状态计算出的按智能体视图。LangGraph 归约器是规范案例。 |
| 知识源 (Knowledge Source) | "专家智能体" | Hayes-Roth 1985 年对黑板参与者的术语。 |
