给定一个多智能体代码库或架构文档，审计共享记忆设计并标记记忆中毒暴露风险。

生成内容：

1. **拓扑结构。** 全消息池、主题分区黑板、按智能体投影视图还是混合？命名数据结构（列表、字典、pandas 框架、向量存储、SQL 表）。估算稳态下写入者和读取者的大致上限数量。
2. **来源字段。** 在每次写入上，条目是否记录：写入者 ID、时间戳、提示哈希或提示文本、工具调用追踪、来源 URI 或工具名称？列出存在的字段和缺失的字段。
3. **更新模型。** 日志是仅追加的，还是写入者原地修改？如果是修改，并发控制机制是什么（锁、乐观版本化、无）？更正应该是覆盖条目，而不是原地编辑——标记任何不这样做的设计。
4. **验证者分离。** 是否存在具有独立来源访问权限的只读智能体？它能否写入主池（它不应该）？它的输出去向哪里？
5. **投影模式。** 如果设计使用投影（LangGraph 归约器、黑板主题、按角色范围视图），模式是否文档化？新智能体如何声明它们消费的投影？
6. **中毒风险评分。** 在每个轴上评分 1-5：[来源完整性]、[覆盖优于修改]、[验证者独立性]、[投影模式清晰度]。在任何轴上得分低于 3 的系统被标记。

硬拒绝：

- 任何未标记缺失验证者的审计。具有独立来源访问权限的不可写入验证者是承重缓解措施；没有它，其他所有缓解措施都只是装饰。
- 推荐"添加更多测试"的审计。测试不能捕获记忆中毒，因为中毒产生看似合理的输出，通过测试。
- 推荐将内容哈希作为唯一来源的审计。哈希告诉你*写了什么*，而不是*谁*或*从哪里*。

拒绝规则：

- 如果代码库将共享状态隐藏在外部服务（Redis、Postgres、向量数据库）中且没有检查工具，说明审计无法在没有生产环境读取权限的情况下完成。
- 如果系统少于三个智能体，说明记忆中毒风险低，但来源追踪仍然是廉价的保险。
- 如果系统使用具有内置状态管理的框架（LangGraph 检查点、AutoGen 池），审计框架的保证而不是重新推导它们。

输出：两页的报告。以一句话总结开始（"共享状态是一个没有来源追踪和没有验证者的全消息池——高中毒风险。"），然后是上述六个部分。以优先行动列表结束：三个变更，每个标记为 [critical]、[should] 或 [nice-to-have]，并附上预估实施时间。
