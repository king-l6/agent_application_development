# 层级架构及其故障模式

> 层级结构就是嵌套的监督者。管理者智能体管理子管理者，子管理者管理工作者。CrewAI `Process.hierarchical` 是教科书式的版本：一个 `manager_llm` 动态委派任务并验证输出。LangGraph 的等价物是 `create_supervisor(create_supervisor(...))`。当任务是一个真实的组织架构时，这是自然的模式。但它也是最容易崩溃为管理层循环的模式——管理者智能体分配工作不佳，误解子输出，或无法达成共识。顺序模式常常胜过它。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 第16阶段 · 05（监督者模式）
**时间：** ~60分钟

## 问题

一旦监督者模式被掌握，自然的下一步就是"如果工作者自己也是监督者呢？"团队有子团队；公司有部门的部门。层级架构反映了这一点。

问题是：LLM 管理者与人类管理者不同。人类管理者对其下属知道什么有稳定的先验知识。LLM 管理者每轮都会根据其上下文中的内容重新推理整个组织。该上下文中微小的漂移就会导致整个树错误分配工作。

## 概念

### 形状

```
                 管理者
                 ┌─────┐
                 └──┬──┘
           ┌────────┴────────┐
           ▼                 ▼
       子管理者 A         子管理者 B
       ┌─────┐           ┌─────┐
       └──┬──┘           └──┬──┘
         ┌┴──┬──┐          ┌┴──┐
         ▼   ▼  ▼          ▼   ▼
       W1  W2  W3         W4  W5
```

每个内部节点进行规划、委派和综合。只有叶子节点做实际工作。

### 它在哪里闪耀

- **清晰的组织映射。** 如果真正的任务是部门级的（"法务审查文档、财务审查文档、工程审查文档，然后为高管总结"），层级结构是显式的。
- **本地总结。** 每个子管理者在顶层管理者看到之前综合其团队的输出。顶层管理者看到三个子管理者摘要，而不是十五个工作者的输出。

### 它在哪里出问题

2026年的事后分析不断发现的三种故障模式：

1. **任务分配错误。** 管理者阅读目标，幻觉出一个分解，并委派给错误的子管理者。因为子管理者顺从地处理分配给它的任务，错误只在顶层综合时才显现——比人类能够捕捉它的位置远离了一层。
2. **输出误解。** 子管理者返回"无法验证声称 X"。顶层管理者总结为"声称 X 未确认"。意义在每一层漂移。
3. **共识循环。** 两个子管理者意见不一致；顶层管理者要求它们协调；它们向下重新委派；工作者重新运行；子管理者返回略有不同的答案；循环。CrewAI 的 `Process.hierarchical` 通过步骤限制防止这一点，但这个限制本身现在是一个超参数。

### 决定性问题

顺序（线性流水线）vs 层级：你的任务是否实际有独立的子团队，还是它只是一个假装成树结构的线性流程？如果是后者，使用顺序。如果是前者，使用层级但要预算显式的协调规则。

### CrewAI 的实现

`Process.hierarchical` 将一个管理者 LLM 连接在专家 crew 之上。管理者：

- 接收顶层任务，
- 将子任务分配给 crew，
- 评估 crew 输出，
- 决定是接受、重新委派还是迭代。

文档：https://docs.crewai.com/en/introduction（在核心概念下查找"Hierarchical Process"）。

### LangGraph 的实现

LangGraph 使用嵌套的 `create_supervisor` 调用。内部监督者有它自己的图；外部监督者将内部图视为一个不透明节点。这在调试方面比 CrewAI 更清晰（你可以逐步通过每个图），但更难表达树的动态重塑。

参考：https://reference.langchain.com/python/langgraph-supervisor。

## 动手构建

`code/main.py` 运行一个 3 层层级：

- 顶层管理者：将任务拆分为"工程"和"法务"分支，
- 工程子管理者：拆分为"前端"和"后端"工作者，
- 法务子管理者：一个工作者。

演示对比了顺利路径（每个人都同意）与一个**扰动路径**，其中顶层管理者的分解将"法务"误标为"财务"，并观察错误的级联——子管理者顺从地做财务工作，顶层综合器报告财务发现，原始法务问题未得到回答。

运行：

```
python3 code/main.py
```

输出显示两条路径，并清楚地并排对比"被问的是什么"与"实际交付的是什么"。

## 使用它

`outputs/skill-hierarchy-fitness.md` 评估给定任务应该使用层级、顺序还是扁平监督者。输入：任务描述、组织架构、协调预算。输出：模式推荐，并附有待防范的具体故障模式。

## 投入生产

如果你部署层级：

- **将树深度限制为 2。** 三个层级就已经将大多数错误隐藏在可观测性之外。
- **显式协调预算。** 设置顶层管理者必须提交前的最大轮数。通常为 2。
- **每次综合的来源追溯。** 每个节点的摘要必须引用产生它的叶子输出。
- **分解漂移告警。** 记录管理者每步的分解；与用户查询进行对比。如果分解不再覆盖查询，发出告警。

## 练习

1. 运行 `code/main.py` 并比较顺利路径 vs 扰动路径。在顶层输出完全偏离用户问题之前，需要多少层管理者交接？
2. 添加第三层（顶层 → 子层 → 子子层 → 工作者）。随着深度增加，测量扰动路径自我纠正 vs 完全偏离的频率。
3. 在每个子管理者处实现一个"金丝雀"工作者，它始终被问及原始的未更改的用户问题。使用金丝雀答案检测分解漂移。当金丝雀与综合答案不一致时，管理者应该如何反应？
4. 阅读 CrewAI 的 `Process.hierarchical` 文档。确定 CrewAI 应用的一个具体防护措施（步骤限制、manager_llm 约束）并描述它针对哪种故障模式。
5. 比较嵌套的 LangGraph 监督者与 CrewAI 层级。哪个使协调循环更便宜检测？

## 关键术语

| 术语 | 人们所说的 | 实际含义 |
|------|-----------|---------|
| 层级（Hierarchical） | "组织架构模式" | 监督者之上还有监督者；只有叶子做工作。 |
| 管理者 LLM（Manager LLM） | "老板" | 在内部节点进行分解、分配和验证的 LLM。 |
| 分解漂移（Decomposition drift） | "老板迷失方向" | 顶层管理者的拆分不再覆盖原始问题。 |
| 协调循环（Reconciliation loop） | "无尽的会议" | 子管理者意见不合；顶层重新委派；工作者重新运行；循环直到预算耗尽。 |
| 深度-2 上限（Depth-2 ceiling） | "不要超过 2 层" | 经验防护措施：3 层以上会崩溃可观测性。 |
| 金丝雀问题（Canary question） | "每一层的真实基准" | 一个始终被问及原始未更改查询的工作者，用于检测漂移。 |
| 来源链（Provenance chain） | "谁说了什么" | 从每个综合追溯到产生它的叶子输出。 |
