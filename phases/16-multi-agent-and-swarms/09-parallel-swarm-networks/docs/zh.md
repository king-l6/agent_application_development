# 并行 / 群 / 网络化架构

> 与监督者模式对比：没有中央决策者。智能体读取共享事件总线，异步领取工作，将结果写回。LangGraph 明确支持适用于去中心化、动态环境的"群架构"。Matrix（arXiv:2511.21686）将控制流和数据流都表示为通过分布式队列传递的序列化消息，以消除协调者瓶颈。其权衡是明确的：用确定性和可追溯性换取可扩展性。群架构适合具有许多独立子问题的任务；不适合需要单一连贯计划的任务。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`，`queue`）
**前置知识：** 第 16 阶段 · 05（监督者模式），第 16 阶段 · 04（基础模型）
**时间：** 约 75 分钟

## 问题

监督者模式可以扩展到几个工作者。但几百个呢？监督者本身成了瓶颈：每一个关于谁做什么的决策都通过一个智能体漏斗传输。一个缓慢的计划步骤就会使整个系统停滞。

群架构翻转了设计。不是中央规划者分派工作，而是工作者从共享队列中领取工作。"协调"内置在事件总线的语义中。没有协调者；系统一直扩展到队列无法承受为止。

## 概念

### 结构

```
                ┌──── 共享队列 ────┐
                │                  │
       ┌────────┼────────┐  ◄──────┴───┐
       ▼        ▼        ▼             │
     工作者   工作者   工作者      工作者
       A       B       C        D
       │        │        │         │
       └────────┴────────┴─────────┘
                 │
                 ▼
             结果池
```

没有协调者。每个工作者重复：拉取任务、处理、写入结果（并可选地入队后续任务）。

### 群架构适合的场景

- **许多独立的任务。** 爬取、转换、分类。任务之间互不依赖。
- **可变时长的工作。** 如果一些任务耗时 100ms 而另一些耗时 10s，群架构会自动平衡负载——快速工作者领取下一个任务。监督者必须预先估计持续时间。
- **吞吐量优先于确定性。** 你关心的是总完成时间，而不是严格的顺序。

### 群架构不适用的场景

- **有序的工作流。** 如果步骤 3 需要步骤 2 的输出，群架构可能让步骤 3 在步骤 2 完成之前就开始执行。
- **需要全局计划的任务。** 复杂的研究问题需要规划者。一群研究者产生独立的事实，而不是一份连贯的报告。
- **调试。** 没有中央日志且工作是异步的，重现一个错误成本高昂。

### Matrix（arXiv:2511.21686）

Matrix 是 2025 年的论文，它将群架构推向了其自然结论：控制流和数据流都是分布式队列上的序列化消息。没有中央协调者。容错性来自消息的持久性。可扩展性成为消息代理的问题，而不是系统的问题。

贡献：一种编程模型，其中多智能体协调是"这个智能体订阅哪个消息主题？"而不是"监督者下一步选择哪个智能体？"这使得系统看起来像一个发布/订阅事件网格。

### LangGraph 的群架构

LangGraph 2025 文档明确将"群架构"描述为多智能体模式之一：智能体是节点，但边形成一个带有循环的有向图，任何节点都可以从池中被激活。工作者通过条件从可用工作中选择，而不是通过监督者分配。

### 失败模式：饥饿和热点

如果所有工作者都拉取最快可用的任务，长时间运行的任务永远得不到处理，直到它们是唯一剩余的任务。经典的队列饥饿。

缓解措施：
- 带有显式老化机制的优先级队列（随着等待时间增加优先级）。
- 工作者专业化：一些工作者只处理"长"任务。
- 背压：限制进入队列的快速任务数量。

### 与基于内容的路由的关联

群架构自然地与基于内容的路由（第 22 课）配合。不使用通用队列，而是每种消息类型一个队列。专业工作者只订阅它们自己的类型。这是可以扩展到数千个智能体的消息总线架构的基础。

## 构建它

`code/main.py` 实现了一个由 4 个工作线程组成的群，从共享的 `queue.Queue` 中拉取任务。任务具有不同的持续时间（一些快，一些慢）。演示对比了：

- **顺序基线：** 一个工作者串行处理所有任务。
- **固定分配：** 每个任务预先分配给特定工作者（监督者风格）。
- **群架构：** 工作者从共享队列中拉取。

群架构自动平衡负载；固定分配在任务较慢时使快速工作者闲置。

运行：

```
python3 code/main.py
```

输出显示每个工作者的任务计数（群架构分配不均衡但最优）以及墙上时钟时间。

## 使用它

`outputs/skill-swarm-fit.md` 评估一个任务应该使用群架构还是监督者模式。输入：任务独立性、持续时间变化、排序需求、可调试性需求。

## 交付清单

- **带老化机制的优先级队列。** 防止长任务饥饿。
- **工作者幂等性。** 如果工作者在中途崩溃，一个任务可能被拉取多次。工作者必须具有幂等性。
- **持久化队列。** 生产环境使用 Kafka、Redis Streams 或基于数据库的队列。`queue.Queue` 仅存在于内存中。
- **每个任务的可观测性。** 每个任务有一个跟踪 ID；每个工作者记录开始/结束时间。
- **背压。** 如果队列增长速度快于工作者处理速度，减慢生产者。

## 练习

1. 运行 `code/main.py`。在可变时长工作负载下，群架构比顺序快多少？比固定分配快多少？
2. 添加一个优先级队列变体（使用 `queue.PriorityQueue`）。按任务的"重要性"字段分配优先级。观察低优先级任务是否会持续负载下饿死。
3. 实现一个热点检测器：当某个工作者处理的任务数量是最慢工作者的 3 倍以上时记录日志。这说明了任务时长分布的什么信息？
4. 阅读 Matrix 论文（arXiv:2511.21686）的摘要和第 3 节。指出 Matrix 接受的一个具体权衡（可扩展性提升）和放弃的一个权衡（可追溯性、确定性）。
5. 将群演示转换为使用 `(task_type, payload)` 元组的 `queue.Queue`，工作者只订阅特定类型。当任务异构时，哪些路由规则是有意义的？

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 群架构 (Swarm architecture) | "去中心化智能体" | 工作者从共享队列中拉取；没有中央协调者。 |
| 事件总线 (Event bus) | "智能体订阅主题" | 按类型或内容将任务路由到工作者的消息代理。 |
| 饥饿 (Starvation) | "任务从不运行" | 低优先级任务因更高优先级的工作持续到达而永远得不到处理。 |
| 热点 (Hot-spotting) | "一个工作者不堪重负" | 负载不均衡，一个工作者处理大部分任务。 |
| 背压 (Back-pressure) | "减慢生产者" | 在队列填满时向上游发出停止生产信号的机制。 |
| 幂等工作 (Idempotent worker) | "安全地重跑" | 一个任务处理两次产生相同的结果。因工作者可能中途崩溃而必需。 |
| 持久化队列 (Durable queue) | "崩溃后幸存" | 由磁盘或复制存储支持的队列；工作者崩溃时任务不会丢失。 |
| Matrix 框架 | "完全消息传递群" | 数据流和控制流都是分布式队列上的序列化消息。 |
