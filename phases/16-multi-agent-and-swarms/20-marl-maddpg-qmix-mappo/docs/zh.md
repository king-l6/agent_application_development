# MARL —— MADDPG、QMIX、MAPPO

> 多智能体协调的强化学习遗产，至今仍影响着 2026 年的 LLM 智能体系统。**MADDPG**（Lowe 等人，NeurIPS 2017，arXiv:1706.02275）引入了集中式训练与分散式执行（CTDE）：训练时每个评论家看到所有智能体的状态和动作；测试时只运行局部策略。适用于合作、竞争和混合场景。**QMIX**（Rashid 等人，ICML 2018，arXiv:1803.11485）是基于单调混合网络的价值分解；各智能体的 Q 值组合为联合 Q 值，使得 `argmax` 可以干净地分布 —— 在 StarCraft Multi-Agent Challenge（SMAC）上占据主导地位。**MAPPO**（Yu 等人，NeurIPS 2022，arXiv:2103.01955）是带集中式价值函数的 PPO；"出乎意料地有效"，在粒子世界、SMAC、Google Research Football、Hanabi 上仅需极少的调参即可表现良好。这些算法支撑了必须分散执行的智能体团队训练策略。MAPPO 是 **2026 年合作式 MARL 的默认基线**。本课通过一个小型网格世界玩具构建每种算法，在触及 LLM 智能体训练之前将这些思想深深植入肌肉记忆。

**类型：** 学习
**语言：** Python（标准库，小型无 NumPy 实现）
**前置条件：** Phase 09（强化学习），Phase 16 · 09（并行群体网络）
**时间：** ~90 分钟

## 问题

LLM 智能体系统越来越多地训练智能体间的协调策略：何时退让、何时行动、调用哪个同伴。教你如何训练此类策略的文献是多智能体强化学习（MARL），它早于 LLM 浪潮，并有一小套主导算法。

没有模式词汇表阅读 MARL 论文是痛苦的。集中式训练与分散式执行（CTDE）、价值分解和集中式评论家不是流行语 —— 它们是针对特定问题的特定答案：

- 独立 RL（每个智能体单独学习）从每个智能体的视角来看是非稳定的。糟糕。
- 集中式 RL（一个智能体控制所有）不可扩展且违反执行约束。
- CTDE 兼得两者之优：训练时使用全局信息，部署时使用局部策略。

## 概念

### 论文使用的三种环境

- **粒子世界（multi-agent particle env）**。简单的 2D 物理，包含合作/竞争任务。MADDPG 的原始测试平台。
- **StarCraft Multi-Agent Challenge（SMAC）**。合作式微操，部分观测。QMIX 的测试平台。离散动作，连续状态。
- **Google Research Football、Hanabi、MPE**。MAPPO 基线。

不同的环境有不同的动作/观测类型。算法相应选择。

### MADDPG（2017）—— CTDE 模式

每个智能体 `i` 有一个策略 `mu_i(o_i)`，将自身观测映射为动作。每个智能体还有一个评论家 `Q_i(x, a_1, ..., a_n)`，训练时看到所有观测和所有动作。策略通过策略梯度根据评论家的评估进行更新。

```
策略更新：    grad_theta_i J = E[grad_theta mu_i(o_i) * grad_a_i Q_i(x, a_1..n) at a_i=mu_i(o_i)]
评论家更新：   TD on Q_i(x, a_1..n) given next-state joint estimate
```

为什么用 CTDE：训练时我们知道所有人的动作；我们利用这一点来降低每个评论家的方差。部署时每个智能体只看到 `o_i` 并调用 `mu_i(o_i)`。

失败模式：评论家随 N 个智能体增长（输入包含所有动作）。在无近似的情况下无法扩展到约 10 个以上的智能体。

### QMIX（2018）—— 价值分解

仅限合作。全局奖励是各智能体 Q 值的单调函数的和：

```
Q_tot(tau, a) = f(Q_1(tau_1, a_1), ..., Q_n(tau_n, a_n)),   df/dQ_i >= 0
```

单调性保证了 `argmax_a Q_tot` 可以通过每个智能体独立选择 `argmax_{a_i} Q_i` 来计算。这正是你需要的**分散式执行属性**。训练时，混合网络从各智能体的 Q 值产生 `Q_tot`。

为什么 QMIX 在 SMAC 上获胜：合作式 StarCraft 微操具有同构智能体、局部观测、全局奖励 —— 完美匹配价值分解。

失败模式：单调性约束具有限制性；某些任务的奖励结构不可单调分解（一个智能体为团队牺牲）。扩展方法（QTRAN、QPLEX）放宽了这一限制。

### MAPPO（2022）—— 被忽视的默认选项

多智能体 PPO：带集中式价值函数的 PPO。每个智能体有自己的策略；所有智能体共享（或各自拥有）看到完整状态的价值函数。Yu 等人 2022 年在五个基准上将 MAPPO 与 MADDPG、QMIX 及其扩展进行了比较，发现：

- MAPPO 在粒子世界、SMAC、Google Research Football、Hanabi、MPE 上匹敌或超越 off-policy MARL 方法。
- 需要极少超参数调优。
- 训练稳定；跨种子可复现。

社区在这篇论文之前低估了 on-policy MARL。到 2026 年，MAPPO 是合作式 MARL 的默认基线；任何新方法都必须超越它。

### 为什么 LLM 智能体工程师应该关注

三个直接用途：

1. **路由训练。** 一个元智能体选择哪个子智能体处理任务。这是一个具有 N 个分散式子智能体和一个集中式路由器的 MARL 问题。MAPPO 适合。
2. **角色涌现。** 在生成式智能体模拟中，训练智能体随时间采用互补角色是一个伪装下的 MARL 问题。QMIX 风格的价值分解通过构造强制互补性。
3. **多智能体工具使用。** 当智能体共享工具并竞争预算时，通过 CTDE 训练它们会产生尊重资源约束的可部署局部策略。

实际警告：到 2026 年，大多数生产型 LLM 智能体系统通过提示来制定策略而非训练它们。当你具备（a）大量交互数据，（b）清晰的奖励信号，以及（c）投入训练基础设施的意愿时，MARL 才进入场景。

### CTDE 作为超越 RL 的设计模式

即使没有训练，CTDE 也是一个有用的架构模式：

- 在**设计**期间，假设完全的团队可见性。
- 在**运行时**，强制分散式执行：每个智能体只看到 `o_i`。

该模式迫使你在前期就保持各智能体状态的明确性并考虑部分可观测性。许多生产型多智能体系统默认假设处处共享状态 —— CTDE 原则可以防止这一点。

### 非平稳性问题

当多个智能体同时学习时，每个智能体的环境（包含其他智能体的策略）是非平稳的。经典的单智能体 RL 证明失效。本课中的 MARL 算法都解决了这一问题：

- MADDPG：全局评论家看到所有动作，因此其价值估计是平稳的。
- MAPPO：集中式价值函数抑制了来自其他策略变化的方差。

在 LLM 智能体系统中，非平稳性表现为"我的智能体上个月工作正常，现在上游那个其他智能体变了，我的智能体开始行为异常。"使用 CTDE 训练 MARL 是原则性的修复方法；提示级别的修复更快但不够持久。

### 本课不涵盖的内容

训练实际网络是 Phase 09 的主题。本课构建脚本化策略版本，在不使用梯度更新的情况下演示 CTDE、价值分解和集中式价值模式。目标是在你接触完整的 MARL 库（PyMARL、MARLlib、RLlib multi-agent）之前内化这些模式。

## 构建它

`code/main.py` 实现了三种模式演示，全部在一个小型 2 智能体合作式网格世界上运行：

- 环境：2 个智能体在 4x4 网格上，一个奖励点。奖励 = 1，如果任一智能体到达奖励点；任务结束。
- `IndependentAgents` —— 每个智能体将其他智能体视为环境。基线。
- `MADDPGStyle` —— 集中式评论家计算联合价值；策略从其中更新。脚本化策略改进。
- `QMIXStyle` —— 使用单调混合器的价值分解。
- `MAPPOStyle` —— 集中式价值函数；策略针对共享基线更新。

所有四个运行相同的回合数，并报告平均步数到目标。CTDE 变体收敛到比独立基线更短的路径。

运行：

```
python3 code/main.py
```

预期输出：独立智能体平均约 6 步；CTDE 变体收敛到约 3.5 步（4x4 网格最优为 3 步）。尽管使用脚本化策略，模式差异依然显现。

## 使用它

`outputs/skill-marl-picker.md` 是一个为给定多智能体任务选择 MARL 算法的能力文件：合作 vs 竞争、同构 vs 异构、动作空间类型、规模、奖励信号。

## 交付它

MARL 在生产中很少见。当你使用时：

- **从 MAPPO 开始。**2022 年的论文将其确立为基线；先复现它可以节省数周追逐更花哨方法的时间。
- **记录每个智能体的观测和动作流。**没有每个智能体轨迹的调试 MARL 是无望的。
- **将训练代码与执行代码分离。**CTDE 是一种原则；让执行路径真的只看到 `o_i`。
- **奖励塑形警告。**MARL 对奖励设计极为敏感。塑形中一个协调错误就会让智能体学会利用它。进行对抗性测试。
- **对于 LLM 智能体**，首先考虑提示级别的策略。只有当交互数据 + 奖励信号 + 基础设施都就绪时，才投资于 MARL 训练。

## 练习

1. 运行 `code/main.py`。测量独立智能体与 MAPPO 风格智能体之间的步数到目标差距。在 6x6 网格上差距是增大还是缩小？
2. 实现一个竞争变体：两个智能体，一个奖励点，只有先到达的获得奖励。哪种模式能干净地处理竞争？历史上是 MADDPG。
3. 阅读 MADDPG（arXiv:1706.02275）第 3 节。用你自己的话以伪代码符号化地实现确切的评论家更新规则。
4. 阅读 MAPPO（arXiv:2103.01955）。作者为什么认为集中式价值 + PPO 在其基准测试中优于 off-policy MARL？列出三个最有力的主张。
5. 将 CTDE 作为设计模式应用于一个假设的 LLM 智能体系统（例如，研究智能体 + 总结器 + 编码器）。设计时可用的联合信息中哪些在运行时不可用？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|---|---|---|
| MARL | "多智能体 RL" | 针对多智能体系统的强化学习。 |
| CTDE | "集中式训练，分散式执行" | 训练时使用全局信息；部署时使用局部策略。 |
| MADDPG | "多智能体 DDPG" | CTDE，每个智能体的评论家看到所有观测 + 动作。 |
| QMIX | "价值分解" | 各智能体 Q 值的单调混合。合作式。 |
| MAPPO | "多智能体 PPO" | 带集中式价值函数的 PPO。2026 年默认基线。 |
| Value decomposition | "个体 Q 值之和" | 联合 Q 值表示为各智能体 Q 值的单调函数。 |
| Non-stationarity | "移动目标" | 每个智能体的环境随其他智能体学习而变化。MARL 的核心问题。 |
| On-policy / off-policy | "从当前/回放中学习" | PPO 是 on-policy；DDPG 和 Q-learning 是 off-policy。 |
| SMAC | "StarCraft 多智能体挑战" | 合作式微操基准；QMIX 的主场。 |
