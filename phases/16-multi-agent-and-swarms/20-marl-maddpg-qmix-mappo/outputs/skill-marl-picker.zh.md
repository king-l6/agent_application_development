给定一个多智能体任务描述，选择 MARL 算法。

生成：

1. **任务分类。** 完全合作（共享奖励）、完全竞争（零和）、混合、一般和。智能体数量。同构 vs 异构。
2. **可观测性。** 完全（每个智能体看到全局状态）、部分（每个只看到自身观测）或启用通信。
3. **动作空间。** 离散（Atari 风格、SMAC）或连续（粒子世界、MuJoCo）。影响算法选择。
4. **奖励结构。** 密集（每步塑形）vs 稀疏（仅终止）。密集使 MAPPO 实用；稀疏需要信用分配帮助（QMIX 的价值分解）。
5. **算法建议。** 从 MAPPO 开始作为基线（Yu 等人 2022）。切换到：
   - QMIX：当合作 + 同构 + 需要强稀疏奖励信用分配时
   - MADDPG：当混合（合作 + 竞争）+ 连续动作时
   - 扩展（QTRAN、QPLEX、FACMAC）：当单调性约束过于严格时
6. **训练基础设施。** 你是否有：足够的交互数据、计算预算、奖励塑形专业知识、稳定性预算（每次实验 5-10 个种子）？如果没有，推荐 LLM 智能体的提示级别策略。
7. **部署契约。** CTDE：部署时每个智能体只看到局部观测。显式编写契约，使运行时代码尊重它。

硬性拒绝：

- 首次运行选择非 MAPPO 基线。MAPPO 是 2026 年的基线；从那里开始。
- 对混合合作-竞争任务使用 QMIX。价值分解假设单调聚合。
- 为缺乏交互数据或奖励信号的 LLM 智能体系统推荐 MARL 训练。在数据就绪之前，提示级别策略将表现更好。
- 训练时不记录每个智能体的观测和动作。调试是不可能的。

拒绝规则：

- 如果任务少于约 1000 回合的交互数据，推荐提示级别策略或监督微调。
- 如果任务是非马尔可夫的（需要记忆）但推荐不包括循环评论家，标记该缺口。
- 如果任务是一般和竞争（多个均衡），仅 MARL 无法选择一个；推荐机制设计或均衡选择。

输出：一页简报。以一句话建议开头（"MAPPO 基线，使用集中式价值函数；每个智能体离散策略；部署时 CTDE；每次实验 5 个种子。"），然后是以上七个部分。以训练到部署的管线结束：数据收集、训练、评估、发布。
