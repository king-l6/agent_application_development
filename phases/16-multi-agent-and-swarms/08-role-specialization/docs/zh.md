# 角色专业化——规划者、评审者、执行者、验证者

> 2026 年最常见的多智能体分解方式：一个智能体规划，一个执行，一个评审或验证。MetaGPT（arXiv:2308.00352）将其形式化为编码到角色提示中的标准操作程序（SOP）——产品经理、架构师、项目经理、工程师、QA 工程师，遵循 `Code = SOP(Team)` 的模式。ChatDev（arXiv:2307.07924）通过"聊天链"和"沟通去幻觉化"（智能体明确请求缺失的细节）将设计师、程序员、评审者、测试者串联起来。验证者是承重角色：Cemri 等人（MAST，arXiv:2503.13657）表明，每一个多智能体故障都可以追溯到缺失或失效的验证。普华永道报告称，在 CrewAI 中引入结构化验证循环后，准确率提升了 7 倍（从 10% 提升到 70%）。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 第 16 阶段 · 04（基础模型），第 16 阶段 · 05（监督者）
**时间：** 约 60 分钟

## 问题

通用的多智能体系统产生通用的输出。三个程序员在一个群聊中写出三种版本同样平庸的代码。你可以增加更多的智能体，增加更多的轮次，但仍然达不到质量的阈值。

解决方法不是增加更多的智能体——而是使用*不同的*智能体。分配不同的角色。给予评审者规划者没有的工具。给予验证者一个客观的测试套件。现在系统有了内部的分歧与基于事实的纠正，而不仅仅是并行的猜测。

## 概念

### 四个规范角色

**规划者（Planner）。** 阅读目标，生成步骤列表或规格说明。工具：知识检索、文档。输出：结构化计划。

**执行者（Executor）。** 一次读取一个计划步骤，生成产物。工具：实际的工作工具（代码编译器、Shell、API 客户端）。输出：产物。

**评审者（Critic）。** 对照规划者的意图审查执行者的输出。工具：对产物的只读访问、静态分析。输出：接受/拒绝及理由。

**验证者（Verifier）。** 读取产物并运行确定性检查。工具：测试运行器、类型检查器、模式验证器。输出：通过/失败及证据。

评审者是主观的、有观点的，通常基于 LLM。验证者是客观的、确定性的，通常基于代码。它们不是同一个角色。

### MetaGPT 的 SOP 模式

MetaGPT（arXiv:2308.00352）将软件工程 SOP 编码为角色提示：

- **产品经理**编写 PRD。
- **架构师**产生系统设计。
- **项目经理**拆分任务。
- **工程师**实现。
- **QA 工程师**运行测试。

每个角色都有严格的输入/输出模式。角色提示说明了该角色*是什么*以及它*必须产出什么*。`Code = SOP(Team)` 的公式——确定性 SOP 将一组 LLM 转变为可预测的流水线。

### ChatDev 的沟通去幻觉化

ChatDev 增加了一个关键动作：当执行者需要一个未在计划中提供的具体细节时，它会在继续之前明确地向设计师询问。这防止了 LLM 典型的那种似是而非地编造细节的失败。

实现方式：角色提示中包含"当你需要未获得的具体信息时，在产生输出之前，按名称向相关角色询问。"

### 为什么验证者最重要

Cemri 等人（MAST）追踪了 1642 个多智能体执行失败。其中 21.3% 是验证缺口——系统交付了未经任何人检查的答案。其余 79% 通常可以追溯到"有一个检查，但它静默地失败了或从未运行过。"验证是承重角色。

普华永道报告（CrewAI 部署，2025 年）称，添加结构化验证循环后，准确率从 10% 提升到 70%。一个角色带来了 7 倍的提升。

### 评审者与验证者的区别

- 评审者是审查产物质量的 LLM。主观的。可能被看似合理的文字所欺骗。
- 验证者是在产物上运行的确定性程序。客观的。给出带有证据的通过/失败结论。

两者都要使用。评审者捕捉验证者无法表达的风格问题。验证者捕捉评审者由于只在运行时显现而无法看到的错误。

### 反模式

系统中的每个角色都是 LLM，并且每个角色的输出都是"看起来不错"。这是经典的 MAST 失败模式。至少添加一个验证者，其通过/失败由代码决定，而不是由 LLM 决定。

### 框架映射

- **CrewAI**——`Agent(role, goal, backstory)` 是教科书式的专业化接口。
- **LangGraph**——节点可以具有专门的提示；边强制执行流水线。
- **AutoGen**——GroupChat 中具有专业角色的 ConversableAgent，使用简短名称。
- **OpenAI Agents SDK**——角色专业化智能体之间的交接工具。

## 构建它

`code/main.py` 实现了一个构建简单 Python 函数的 4 角色流水线：

- **规划者**产生规格说明。
- **执行者**生成代码字符串。
- **评审者**（LLM 模拟）标记明显的问题。
- **验证者**在沙箱（`exec`）中针对测试用例运行生成的代码。

演示运行两次：一次执行者生成正确代码（评审者和验证者均通过），一次执行者生成不符合规格的代码（评审者因代码看起来合理而漏过了错误，验证者因测试失败而捕获了错误）。

运行：

```
python3 code/main.py
```

## 使用它

`outputs/skill-role-designer.md` 接收一个任务并生成角色名册（3-5 个角色）、每个角色的输入/输出模式以及验证者检查。在将智能体接入框架之前使用此工具。

## 交付清单

- **至少一个确定性验证者。** 永远不要全是 LLM。
- **每个角色有明确的 I/O 模式。** 规划者返回规格说明，而不是散文；执行者读取该模式。
- **沟通去幻觉化。** 执行者必须在信息缺失时询问规划者；永远不要编造。
- **评审者/验证者顺序。** 先运行评审者（廉价，捕捉设计问题），再运行验证者（慢，捕捉错误）。
- **循环预算。** 最多 2 轮评审者-执行者修订循环，之后升级到人工。

## 练习

1. 运行 `code/main.py` 并观察验证者如何捕获评审者漏掉的错误。添加一个静态分析检查（统计 `return` 出现次数）作为额外的验证者。它捕获了运行时测试漏掉的什么？
2. 添加第 5 个角色："需求分析师"，将用户愿望转化为规划者可用的规格说明。哪些沟通去幻觉化请求应该流向它？
3. 阅读 MetaGPT 第 3 节（"Agents"）。列出 MetaGPT 5 个角色中每个角色的输入/输出模式。
4. 阅读 ChatDev 的聊天链图（arXiv:2307.07924 图 3）。指出沟通去幻觉化在哪里打破了一个否则会无限循环的循环。
5. 普华永道 7 倍的准确率提升来自验证循环。假设三种添加验证者不会有所帮助的任务——即确定性检查正确性不可能或成本过高的情况。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------|---------|
| 角色专业化 (Role specialization) | "不同智能体，不同工作" | 为规划者/执行者/评审者/验证者角色调优的独特系统提示。 |
| SOP 模式 (SOP pattern) | "编码的标准操作程序" | MetaGPT 的框架：每个角色严格的 I/O 模式将团队转化为流水线。 |
| 沟通去幻觉化 (Communicative dehallucination) | "先问再编" | ChatDev 模式：执行者在细节缺失时询问规划者，而不是编造一个。 |
| 评审者 (Critic) | "LLM 审查者" | 主观的、有观点的审查者。捕捉风格问题。可能被看似合理的文字欺骗。 |
| 验证者 (Verifier) | "确定性检查" | 基于代码的通过/失败。测试运行器、类型检查器、模式验证器。不能被欺骗。 |
| 验证缺口 (Verification gap) | "没人检查" | 占 MAST 失败的 21.3%。答案在未经本可捕获错误的检查的情况下被交付。 |
| 修订循环 (Revision loop) | "评审者退回" | 评审者拒绝触发执行者根据反馈重新运行。需要有预算。 |
| 全 LLM 反模式 (All-LLM anti-pattern) | "看起来不错" | 每个角色都是 LLM，没有确定性检查。经典的 MAST 失败。 |
