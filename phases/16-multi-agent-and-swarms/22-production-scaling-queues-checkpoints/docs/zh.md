# 生产级扩展 —— 队列、检查点、持久性

> 将多智能体系统扩展到数千个并发运行需要**持久化执行**。LangGraph 的运行时在每次超级步骤后以 `thread_id` 为键写入检查点（默认使用 Postgres）；工作进程崩溃后释放租约，另一个工作进程接管。智能体可以无限期休眠等待人类输入。**MegaAgent**（arXiv:2408.09955）运行了一个每个智能体的生产者-消费者队列，具有三种状态（Idle / Processing / Response）和两层协调（组内聊天 + 组间管理员聊天）。**协程/异步** 在 LLM 流式传输方面优于每作业一线程：线程 99% 的时间在等待 token 时空闲，协程在 I/O 上合作式让出。反观点：Ashpreet Bedi 的"扩展智能体软件"主张 **FastAPI + Postgres + 无其他**，直到负载证明其他选择是必要的 —— 简单架构能走得更远。本课构建一个持久化检查点日志、一个带状态转换的每个智能体工作队列、一个异步与线程的对比演示，并夯实"从简单开始"的实用规则。

**类型：** 学习 + 构建
**语言：** Python（标准库、`asyncio`、`sqlite3`）
**前置条件：** Phase 16 · 09（并行群体网络），Phase 16 · 13（共享内存）
**时间：** ~75 分钟

## 问题

一个原型多智能体系统在单台笔记本电脑上通过内存事件循环运行三个智能体。你将其迁移到生产环境：

- 智能体有时运行数小时（长时间研究、人在回路中等待）。
- 工作进程崩溃。重启丢失状态。
- 峰值负载是平均值的 10 倍；你需要水平扩展。
- 用户按智能体运行付费；你需要计费的恰好一次语义。

内存事件循环无法做到这些。你需要一个底层的持久化执行层。2026 年的标准选项包括：

1. 带检查点的工作流引擎（Temporal、LangGraph 运行时）。
2. 带状态存储的消息队列（Postgres + SQS/RabbitMQ）。
3. Actor 模型框架（MegaAgent 的每个智能体生产者-消费者模式）。
4. 手工编写的 FastAPI + Postgres（Bedi 的主张）。

本课构建每个选项的微型版本。

## 概念

### 持久化执行 —— 模式

持久化执行引擎在每个"步骤"（LangGraph 术语中的超级步骤）后持久化完整的程序状态。崩溃时：

```
工作进程在步骤中崩溃
  -> 租约超时
  -> 另一个工作进程拾取 thread_id
  -> 从最后一个检查点恢复
  -> 无重复副作用
```

这需要满足以下条件：

- **可序列化状态。** 所有智能体状态必须可持久化。带有活跃数据库连接的函数闭包无法存活。
- **确定性恢复。** 给定相同的状态和相同的输入，智能体产生相同的动作（或将 LLM 调用委托给外部确定性预言机）。
- **幂等副作用。** 外部调用（工具调用、支付）必须是幂等的，或使用去重键。

LangGraph 在每个超级步骤后写入检查点；Temporal 在每个活动后写入；Restate 使用事件溯源日志。三者都实现相同的模式。

### LangGraph 的运行时

每个智能体有一个 `thread_id`；状态是一个类型化字典；每个超级步骤向检查点表写入一行。恢复时，运行时从最后一个检查点回放，而非从头开始。智能体可以 `interrupt()` 等待人类输入；运行时持久化并释放工作进程。输入到达时，任何工作进程都可以恢复。

这是 2026 年 4 月的参考生产设计。

### MegaAgent 的每个智能体队列

arXiv:2408.09955 描述了一个规模实验：一个集群中的数千个并发智能体。架构：

```
智能体 i:
  状态 ∈ {Idle, Processing, Response}
  in_queue   <- 发送给智能体 i 的消息
  out_queue  -> 回复 + 副作用

协调器:
  组内聊天  (同一组中的智能体)
  组间管理员聊天  (高层级路由)
```

两层协调使得组内对话密集进行，同时组间保持稀疏 —— 在数千个智能体中将成本保持在线性水平所使用的模式。

### 异步 vs 每作业一线程

LLM 调用是 I/O 密集型的。等待下一个 token 的线程 99% 的时间处于空闲状态。每个线程约消耗 1MB RAM；在 10,000 个并发调用时，仅堆栈就需要 10GB。

协程（Python `asyncio`、Go goroutines、Rust `tokio`）在 I/O 上合作式让出。同样的 10,000 个调用可以舒适地放入进程内。在 LLM 智能体规模下，异步不是优化 —— 它是架构本身。

例外：CPU 密集型的后处理（嵌入、分词器技巧）仍然需要线程或进程。将你的 I/O 层与 CPU 层分离。

### Bedi 的反观点

"扩展智能体软件"（Ashpreet Bedi，2026 年）认为大多数团队在测量负载之前就过度工程化了。务实的默认配置：

- FastAPI + Postgres。
- 每个智能体运行是一行；状态使用乐观并发原地更新。
- 通过 `pg_notify` 或简单的 Celery 工作进程实现后台作业。
- 应用代码中的重试策略。

对于约 100 个以下并发智能体运行可管理任务的情况，这通常就是你所需要的。当你测量到它失败时再进行升级。

规则：当你遇到简单架构无法解决的具体问题时，才采用持久化执行框架。过早采用会消耗时间于无法带来回报的仪式性工作。

### 恰好一次语义

对于付费的智能体运行，你需要"恰好一次有效"（至少一次投递 + 幂等消费者）。工程手段：

- **每次运行的去重键。** 将其包含在每个副作用调用中。
- **发件箱模式。** 副作用先写入一个表，然后一个单独的进程执行它们。两个步骤都是幂等的。
- **补偿事务。** 当副作用成功但其跟踪写入失败时，安排补偿。

这些是数据库工程模式，而非 LLM 专用。LLM 的额外代价仅在于 LLM 调用速度慢；其他一切是标准的分布式系统。

### 彩虹部署

Anthropic 的多智能体研究系统使用"彩虹部署"：智能体运行时的多个版本同时运行，这样长时间运行的智能体不必在每次代码部署时被终止。在一部分流量上金丝雀测试新版本；当旧版本的智能体完成时退役。

这对于长时间运行的有状态系统是标准做法；2026 年的改进在于智能体可能存活数小时，因此部署周期必须适应。

### 标准生产检查清单

- 持久化状态（检查点、快照或发件箱 + 可回放日志）。
- 幂等副作用。
- LLM 调用的异步 I/O 层。
- 带去重的至少一次投递。
- 有状态工作负载的彩虹/金丝雀部署。
- 可观测性：每个智能体追踪、超级步骤审计、重试计数。

## 构建它

`code/main.py` 实现：

- `CheckpointStore` —— 基于 SQLite 的检查点日志，以 thread-id 为键。每个超级步骤追加一行。
- `run_with_checkpoint(agent, thread_id)` —— 模拟运行中途崩溃；第二个工作进程从最后一个检查点恢复。
- `AgentQueue` —— 每个智能体的 Idle / Processing / Response 状态机，带一个小型工作队列。
- `demo_async_vs_threads()` —— 通过 asyncio 和线程运行 500 个并发的模拟"LLM 调用"；报告挂钟时间和峰值内存（近似）。

运行：

```
python3 code/main.py
```

预期输出：模拟崩溃后检查点恢复成功；异步版本在 < 1 秒内处理 500 个并发调用；线程版本花费数秒钟，每个并发单元使用的内存高出数个数量级。

## 交付它

标准生产加固：

- **从简单开始（Bedi 法则）。** FastAPI + Postgres，直到你测量到它失败。
- **先检测一切，再优化。** 每次运行的延迟直方图、每步时间、重试计数、失败分类。
- **副作用的发件箱模式。** 特别是支付和外部 API 调用。
- **彩虹部署。** 绝不在部署期间杀死正在进行的智能体运行。
- **采用持久化执行引擎（Temporal / LangGraph / Restate）的条件：** 遇到具体问题时 —— 数小时的人在回路中等待、跨区域协调、复杂的重试/补偿策略。
- **I/O 层使用异步。** 线程仅用于 CPU 密集型的后处理。

## 练习

1. 运行 `code/main.py`。确认检查点恢复工作；测量异步与线程的并发差异。
2. 实现一个**发件箱**表：每个工具调用先写入发件箱，然后一个单独的协程/任务执行。通过运行工具调用两次来验证幂等性。
3. 模拟一个**彩虹部署**：两个并发的运行时版本；将新 thread_id 的一半路由到每个版本；确认旧版本上正在进行的线程未被中断。
4. 阅读 LangGraph 的运行时文档（见下方链接）。确定运行时中哪些功能在手工编写的 FastAPI + Postgres 版本中最难复现。这是采用的理由，还是可以推迟？
5. 阅读 MegaAgent（arXiv:2408.09955）第 3 节。两层协调（组内 + 组间管理员聊天）是明确的。草拟如何将其映射到具有两个队列族的消息队列。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|---|---|---|
| Durable execution | "持久化程序状态" | 引擎在每个超级步骤后写入状态；崩溃恢复是确定性的。 |
| Super-step | "事务边界" | 检查点之间的工作单元。LangGraph 术语。 |
| thread_id | "智能体运行标识符" | 绑定检查点和恢复逻辑的键。 |
| Idempotency | "可安全重试" | 重复副作用产生与一次尝试相同的结果。 |
| Outbox pattern | "解耦副作用" | 将意图写入表；一个独立的执行器执行并标记完成。 |
| At-least-once delivery | "可能的重复" | 消息队列语义；去重键使消费者有效恰好一次。 |
| Rainbow deploy | "重叠版本" | 长时间运行工作负载期间多个运行时版本并发。 |
| Async fiber | "合作式让出" | 用户态并发；与线程相比，对于 I/O 密集型负载成本较低。 |
| Checkpoint | "状态快照" | 在超级步骤边界处序列化的状态；用于恢复的键。 |
