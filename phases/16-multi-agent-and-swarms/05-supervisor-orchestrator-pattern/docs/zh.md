# 监督者 / 编排器-工作者模式

> 一个主导智能体进行规划和委派；专门的工作者在独立的上下文中并行执行并报告结果。这是 Anthropic 研究系统（Claude Opus 4 为主导，Sonnet 4 为子智能体）背后的模式，在内部研究评估中比单智能体 Opus 4 高出 +90.2%。Anthropic 的工程文章报告说 BrowseComp 上 80% 的方差仅由 token 使用量解释——多智能体获胜很大程度上是因为每个子智能体都获得了一个全新的上下文窗口。本课程从原语构建监督者模式，并涵盖 2026 年来自生产部署的工程经验。

**类型：** 学习 + 构建
**语言：** Python（标准库，`threading`）
**前置条件：** 第16阶段 · 04（原语模型）
**时间：** ~75分钟

## 问题

研究是单智能体系统失败的典型任务。你问"2023 年到 2026 年间，多智能体系统发生了什么变化？"一个单智能体顺序阅读五篇论文，用论文文本填满一半上下文，然后必须一起推理所有论文。当它读到第五篇时已经忘记了第一篇。它无法并行化。

监督者模式解决了这个问题：一个主导智能体规划搜索，将每个子问题委派给一个工作者，然后进行综合。每个工作者针对一个狭窄问题获得自己 200k 的 token 窗口。主导者从不见原始论文——只看到工作者摘要。

Anthropic 的生产研究系统报告在内部研究评估中比单个 Opus 4 高出 +90.2%。同一篇文章指出 BrowseComp 上 80% 的方差仅由 *token 使用量* 解释。每个子智能体获得全新的上下文是主要机制。

## 概念

### 模式

```
                 ┌──────────────┐
                 │   主导者      │  规划、分解、
                 │  (Opus 4)    │  综合
                 └──┬────┬───┬──┘
                    │    │   │
            ┌───────┘    │   └───────┐
            ▼            ▼           ▼
      ┌─────────┐  ┌─────────┐  ┌─────────┐
      │ 工作者1  │  │ 工作者2  │  │ 工作者3  │
      │(Sonnet)  │  │(Sonnet)  │  │(Sonnet)  │
      └─────────┘  └─────────┘  └─────────┘
        全新         全新         全新
        上下文       上下文       上下文
```

主导者从不见原始材料。工作者在主导者综合之前从未看到彼此的工作。每个箭头都是一个带有狭窄产物的交接。

### 为什么它胜出

三个机制：

1. **每个子智能体有全新的上下文。** 探索"FIPA-ACL 传承"的工作者不会携带主导者规划时花费的 40k tokens。它为一个问题获得一个 200k 窗口。
2. **通过提示词实现专门化。** 主导者的提示词是"分解和综合"，而不是"研究"。每个工作者的提示词都很狭窄："找出 X 中发生了什么变化。"专注的提示词产生专注的输出。
3. **并行性。** 工作者并发运行。挂钟时间大约是 `max(worker_times) + plan + synthesis`，而不是 `sum(worker_times)`。

### 工程经验（Anthropic 2025）

Anthropic 的文章列出了一些在 2026 年仍然相关的生产经验：

- **根据查询复杂度调整投入。** 简单查询：一个智能体，3-10 次工具调用。复杂查询：10+ 个智能体。主导者必须估算这一点，而不是调用者。
- **先宽后窄。** 先分解为宽泛的子问题，然后如果答案值得深入，为每个子问题生成更多工作者。
- **彩虹部署。** 智能体是长期运行且有状态的。传统的蓝绿部署不起作用。Anthropic 使用彩虹部署：新版本逐步推出，同时旧版本排空。
- **token 使用量占主导。** 多智能体大约是单智能体的 15× token。只有在任务价值证明成本合理时才运行它。

### LangGraph 的转向

LangGraph 最初发布了一个 `langgraph-supervisor` 库，带有高级的 `create_supervisor` 辅助函数。2025 年，LangChain 将推荐改为直接通过工具调用来实现监督者模式，因为工具调用让开发者对*监督者看到什么*（上下文工程）有更多控制。该库仍然可用；文档现在推荐工具调用形式。

### 故障模式

- **主导者幻觉计划。** 如果主导者生成的子问题没有分解真正的问题，工作者会对错误的目标进行精确研究。
- **工作者过度探索。** 没有明确的边界范围，工作者会漂移超出分配的子问题并污染综合步骤。
- **综合冲突。** 两个工作者返回矛盾的事实。主导者必须要么重新询问（增加一轮），要么明确标注分歧。静默选择一边是最糟糕的失败：用户永远不知道发生了分歧。

### 监督者不合适的情况

- **顺序任务。** 如果第 2 步确实需要第 1 步的输出，并行性毫无意义。使用流水线（CrewAI Sequential、LangGraph 线性图）。
- **简单查询。** 单智能体处理得更快更便宜。在生成工作者之前使用主导者的"调整投入"检查。
- **严格确定性。** 监督者使用 LLM 选择的委派。当审计/回放比适应性更重要时，静态图更好。

## 动手构建

`code/main.py` 使用 `threading` 实现了一个具有三个并行工作者的监督者。主导者将查询分解为子问题，工作者在每个子问题上并发运行，主导者进行综合。没有真正的 LLM——工作者被脚本化以模拟获取和总结。

关键结构：

- `Lead.plan(query)` 将查询拆分为 3 个子问题。
- `Worker.run(sub_q)` 返回一个模拟摘要（在生产中可以是任何使用工具的智能体）。
- `Lead.run(query)` 在线程中启动工作者，等待，然后综合。

运行：

```
python3 code/main.py
```

输出显示计划、带有开始/结束时间戳的并行工作者跟踪，以及最终的综合。你可以看到挂钟时间的优势：三个 0.3 秒的工作者在约 0.35 秒内运行完毕，而不是 0.9 秒。

## 使用它

`outputs/skill-supervisor-designer.md` 接受用户查询并产生一个监督者模式设计：主导者系统提示词、工作者角色、子问题分解规则和综合模板。在构建新的研究型智能体系统之前使用它。

## 投入生产

部署监督者模式前的检查清单：

- **模型配对。** 主导者使用推理层模型（Opus 类、`o3` 类）。工作者使用更快、更便宜的模型（Sonnet、`o4-mini`）。
- **工作者超时。** 任何超过 2× 中位数运行时的工作者被杀死；主导者要么用更窄的范围重新生成，要么在没有它的情况下继续。
- **每个工作者的 token 上限。** 硬限制（比如说预期综合输入的 10×）防止失控的工作者超支预算。
- **可观测性。** 跟踪主导者的计划、每个工作者的工具调用和综合。这是任何事后调试的基础。
- **彩虹部署。** 有状态的长期运行智能体需要逐步版本过渡，而不是热切换。

## 练习

1. 运行 `code/main.py`，然后修改主导者使其生成 5 个工作者而不是 3 个。观察挂钟时间效果。在此演示中，工作者数量达到多少时生成开销超过并行节省？
2. 实现工作者超时：杀死任何运行超过 0.5 秒的工作者，并让主导者综合剩余结果。你需要什么可观测性来知道工作者已被切断？
3. 在主导者的综合中添加一个冲突检测步骤：如果两个工作者返回矛盾答案，主导者标注分歧而不是选择一个。如何在不调用 LLM 的情况下检测矛盾？
4. 阅读 Anthropic 研究系统工程文章。列出此玩具演示需要采用才能在生产中运行的三项实践。
5. 比较 LangGraph 的 `create_supervisor`（旧版）与新的工具调用推荐。哪个让你对监督者看到什么有更好的控制？为什么 Anthropic 明确只将子答案（而不是原始工作者上下文）传递到综合中？

## 关键术语

| 术语 | 人们所说的 | 实际含义 |
|------|-----------|---------|
| 监督者（Supervisor） | "主导智能体" | 一个规划、委派和综合的编排智能体。自己不做事。 |
| 工作者（Worker） | "子智能体" | 由监督者调用的专注智能体，具有狭窄范围和自己的上下文窗口。 |
| 编排器-工作者（Orchestrator-worker） | "监督者模式" | 同一事物，不同名称。2026 年的文献两者都用。 |
| 全新上下文（Fresh context） | "干净窗口" | 工作者的上下文从其系统提示词和分配的问题开始，而不是主导者的历史。 |
| 彩虹部署（Rainbow deployment） | "逐步推出" | 长期运行的有状态智能体需要版本化的排空和替换，而不是蓝绿部署。 |
| token 主导（Token dominance） | "上下文是变量" | 根据 Anthropic，80% 的研究评估方差来自使用的总 token，而不是模型选择。 |
| 调整投入（Scale effort） | "将智能体数量与复杂度匹配" | 主导者估算查询难度，相应地生成 1 个或 10+ 个工作者。 |
| 综合冲突（Synthesis conflict） | "工作者意见不合" | 两个工作者返回矛盾事实；主导者必须表面化分歧，而不是静默选择一个。 |
