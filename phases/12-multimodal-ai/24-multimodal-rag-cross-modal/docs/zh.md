# 多模态 RAG 与跨模态检索

> 视觉原生文档 RAG 只是其中的一部分。生产级多模态 RAG 的范围更广——跨文本、图像、音频和视频进行检索，适用于旅行规划（"找个安静且光线充足的素食早午餐"）、医疗分诊（"这张照片加上这些笔记对应什么伤势"）、电商（"和我这张自拍相似的衣服，我的尺码"）和现场服务（"根据引擎声音和零件照片诊断问题"）等工作流程。2025 年的三篇综述——Abootorabi 等人、Mei 等人、Zhao 等人——将子问题系统化：跨模态检索、检索融合、生成接地、多模态评估。本课程阅读这些综述并设计一个生产级流程。

**类型：** 构建
**语言：** Python（标准库，带融合 + 接地生成器的跨模态检索器）
**前置知识：** 阶段 12 · 23（ColPali），阶段 11（RAG 基础）
**时间：** 约 180 分钟

## 学习目标

- 设计跨模态检索：文本 → 图像、图像 → 文本、音频 → 视频等。
- 比较三种融合策略：分数融合、基于注意力的融合、MoE 融合。
- 解释生成接地：当来源混合多种模态时，"引用你的来源"是什么样子。
- 说出 2025 年三篇经典多模态 RAG 综述及其子问题分类法。

## 问题

单模态 RAG 是一个已解决的模式：嵌入查询，嵌入块，检索，塞入 LLM。多模态 RAG 需要：

1. 多个检索头（每种模态需要在兼容的空间中嵌入）。
2. 跨模态检索结果的融合。
3. 跨模态引用来源的生成接地。
4. 覆盖跨模态信号的评估指标。

2025 年的综述都得出了相同的分类法。

## 概念

### 跨模态检索

给定模态 A 的查询，检索模态 B 的文档。三种模式：

1. 共享嵌入空间。CLIP 和 CLAP 在共享空间中产生文本 + 图像/文本 + 音频嵌入。跨模态的余弦相似度直接可用。限于 CLIP 训练过的配对。

2. 每模态编码器 + 翻译。文本编码器 + 图像编码器 + 一个在空间之间映射的小型翻译模块。Gupta 等人的 Sen2Sen 及其他 2024 年的设计。灵活但增加了复杂性。

3. VLM 作为编码器。使用 VLM 的隐藏状态作为检索表示。VLM 支持的任何模态都可工作。质量更高，成本更高。

选择：CLIP / SigLIP 2 用于文本 + 图像；CLAP 用于文本 + 音频；VLM 隐藏状态用于前沿质量的跨模态检索。

### 融合策略

你检索到了 10 个结果：5 张图像、3 段文本、2 个音频片段。如何合并它们？

分数融合（最便宜）。每种模态有各自的检索器，各自返回分数。在模态内归一化分数然后求和。简单，通常有效。

基于注意力的融合。拼接所有检索到的项目，让一个小型注意力网络为它们加权。需要训练。

MoE 融合。门控网络将查询路由到特定模态的专家。不同的查询类型路由不同——视觉问题对图像权重更高。

生产默认：分数融合，对查询的主导模态略加偏置。如果 A/B 测试在你的领域显示出明显优势，则升级为 MoE。

### 生成接地

LLM 应该引用哪个检索到的项目驱动了每个声明。对于多模态：

- 文本来源：标准引用 `[1]`。
- 图像来源：`[img 3]` 加简短说明。
- 音频：`[audio 2 at 0:34]`。

使用接地感知数据训练生成器：训练目标中的每个声明都标记有来源索引。推理时，模型自然发出引用。

### 2025 年综述

Abootorabi 等人 (arXiv:2502.08826, "Ask in Any Modality")：多模态 RAG 的分类法。涵盖检索、融合、生成。覆盖面最广。

Mei 等人 (arXiv:2504.08748, "A Survey of Multimodal RAG")：专注于子任务基准和失败模式。对评估设计有用。

Zhao 等人 (arXiv:2503.18016)：以视觉为重点的综述。在 ColPali 系列工作方面很强。

阅读这三篇，你可以了解截至 2025 年春季的最新技术水平。大多数子问题仍然开放。

### MuRAG — 奠基性论文

MuRAG（Chen 等人，2022）是第一个多模态 RAG。从多模态知识库检索图像 + 文本，生成答案。在 VLM 浪潮之前显示了可行性。现代系统（REACT、VisRAG、M3DocRAG）建立在其基础上。

### 生产级旅行规划器示例

查询："找个安静且光线充足的素食早午餐。"

流程：

1. 分解查询。"安静" → 音频/评论关键词；"素食早午餐" → 菜单项；"自然光" → 图像特征。
2. 按模态检索：
   - 对评论进行文本检索："素食早午餐，安静氛围。"
   - 对餐厅照片进行图像检索："自然光，通风。"
   - 对环境声音片段进行音频检索："低分贝，无音乐。"
3. 融合分数。每家餐厅有一个综合分数。
4. 排名前 k 的餐厅 → VLM 生成器，加上所有证据 → 带引用的答案。

这远远超出了文本 RAG。每种模态都增加了文本单独无法捕获的信号。

### 智能体多模态 RAG

多跳：如果第一次检索未能返回高置信度答案，LLM 重新表述并再次检索。阶段 14 中的智能体 RAG 模式在这里适用。示例：

- 检索初始前 10 名 → LLM 发现"太吵了，过滤 <40 dB" → 重新检索。
- 检索图像 → LLM 发现其中一张有菜单 → 检索菜单文本 → 答案。

增加了复杂性，但能处理一次性检索无法处理的查询。

### 评估

跨模态评估仍然不成熟。常见的代理指标：

- 每种模态的 Recall@k。
- 融合后的 top-k 准确率。
- 人工评判的端到端满意度。
- 任务特定（完成的预订、完成的购买）。

没有标准基准涵盖所有模态。大多数论文在领域特定任务上进行评估。

## 使用

`code/main.py`：

- 三个模拟检索器（文本、图像、音频），在共享的餐厅语料库上操作。
- 分数融合，以可配置的权重组合模态分数。
- 一个生成器存根，发出带引用的最终答案。
- 一个简单的智能体循环，如果置信度低则重新表述查询。

## 练习

1. 提出一个医疗分诊多模态 RAG：查询 = 伤势照片 + 文本症状。什么模态从什么知识库检索？

2. 分数融合是简单的加权求和。它有什么 MoE 融合可以避免的失败模式？

3. 阅读 Abootorabi 等人的分类法（第 3 节）。三个经典子问题是什么？它们如何映射到你选择的产品？

4. 为旅行规划器多模态 RAG 设计评估规范。什么指标涵盖图像召回率、音频召回率和综合正确性？

5. 智能体多跳 RAG 每次往返有延迟代价。在什么查询难度下，准确率提升才值得付出延迟代价？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 跨模态检索（Cross-modal retrieval） | "用一种模态查询，检索另一种模态" | 文本查询检索图像；图像查询检索文本；需要共享空间或翻译器 |
| 分数融合（Score fusion） | "合并分数" | 每种模态检索分数的加权和；最简单的融合 |
| MoE 融合（MoE fusion） | "模态路由专家" | 门控网络根据查询选择信任哪种模态的分数 |
| 接地生成（Grounded generation） | "引用来源" | 答案中的每个声明都标记有来源索引 |
| MuRAG | "第一个多模态 RAG" | 确立了多模态 RAG 模式的 2022 年论文 |
| 智能体多跳（Agentic multi-hop） | "重新表述并重试" | 当首次通过置信度低时，LLM 重新查询检索器 |
