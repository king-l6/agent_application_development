# Omni 模型：Qwen2.5-Omni 与思考者-说话者分离

> GPT-4o 在 2024 年 5 月的产品演示具有颠覆性，不是因为底层模型，而是因为产品形态——一个语音界面，你说话，模型看到摄像头所见，并在 250ms 内回复。开放生态系统在 2024 年下半年和 2025 年竞相追赶这一产品形态。Qwen2.5-Omni（2025 年 3 月）是开放设计的参考：一个思考者（大型文本生成 transformer）加一个说话者（并行语音生成 transformer），通过流式语音 token 连接。Mini-Omni 简化了它，Moshi 匹配了其延迟，GLM-4-Voice 将其扩展到中文。本课程解读思考者-说话者架构以及使流式实时对话工作的延迟预算。

**类型：** 构建
**语言：** Python（标准库，流式流水线延迟模拟器 + VAD 循环）
**前置知识：** 阶段 12 · 19（音频-LLM），阶段 12 · 16（任意到任意）
**时间：** ~180 分钟

## 学习目标

- 将推理流水线拆分为思考者（文本推理）和说话者（语音合成），并解释为什么并行流式工作。
- 逐组件计算对话交互的首次音频字节输出时间（TTFAB）预算。
- 描述 TMRoPE 在思考者内跨视觉、音频和文本的时间对齐位置编码。
- 说出三种实时对话模式：半双工、轮替、全双工。

## 问题

一个实时语音助手必须快速完成很多事情：

1. 听到用户。实时语音分词、语音活动检测（VAD），以判断他们何时说完。
2. 可选地看到。以 2-4 FPS 的摄像头输入，与音频一起流式输入到思考者。
3. 思考。根据对话历史组成回复。
4. 说话。合成音频 token，解码为波形，流式传输到用户的扬声器。

每一步都会增加延迟。对话感要求总往返时间 < 500ms——低于这个阈值，用户就不再注意到延迟。GPT-4o 声称约 250ms。Moshi 约 160ms。Qwen2.5-Omni 约 350-500ms。

每个组件都必须流式工作。没有任何部分可以是"先批处理再解码"。

## 概念

### 思考者和说话者

Qwen2.5-Omni 的分解：

- 思考者：一个 7B-80B 的文本生成 transformer。消费交错的文本 + 图像 + 音频 token。输出代表要说什么的文本 token。
- 说话者：一个较小的语音生成 transformer（200M-1B）。消费思考者的文本输出 token 加上最近的语音上下文 token。输出离散的语音 token（残差 VQ 索引）。
- 语音解码器：一个流式波形解码器（SNAC、MoVQGAN 系列），实时将语音 token 转换为音频样本。

这种分离很重要。思考者必须大才能进行良好的推理。说话者可以小，因为它的任务是局部的——将文本转换为语音 token。更大的说话者并不会更有表现力；只会更慢。

并行运行两者：

1. 思考者发出文本 token t_i。
2. 说话者消费 t_i（通过流式）并发出语音 token s_i, s_{i+1}, ..., s_{i+k}。
3. 语音解码器在 token 到达时消费它们并发出音频样本。
4. 到思考者到达文本 token t_{i+3} 时，说话者已经流式输出了 t_0..t_{i+2} 的音频。

### TMRoPE——时间对齐的多模态位置

思考者需要整合图像帧（以比如 4 FPS 到达）、音频帧（以 50 帧/秒到达）和来自对话历史的文本。朴素的序列顺序（所有图像，然后所有音频，然后文本）会丢失时间对齐。

TMRoPE 为每个 token 分配绝对时间戳。t=2.3s 的视觉 token。t=2.32s 的音频 token。来自用户的"停止"的文本 token 在 t=2.35s。RoPE 按时间戳旋转注意力；模型将它们视为时间上并发的。

这是"他一边挥手一边说你好"能够工作的基础设施——模型在同一概念时刻看到视频帧和音频。

### 流式语音合成

语音 token 必须流式输出。Mini-Omni（Xie & Wu，2024）引入了"语言模型可以在流式思考的同时听到和说话"：思考者输出 token 和说话者输出 token 在同一序列中交错。说话者在思考者提交下一个文本 token 后立即触发。没有批处理边界。

Moshi（Defossez 等人，2024 年 10 月）是最快的开放实现。在单张 A100 上 TTFAB 为 160ms。架构：一个单一的 7B transformer，在交替位置上发出文本和语音 token，带有一个"内心独白"，将思考流与说话流分开。这实际上是思考者 + 说话者融合到一个模型中，通过精心设计的训练实现。

### VAD 和轮替

语音活动检测运行在输入侧。两种模式：

- 半双工：用户说话，模型听。模型说话，用户听。通过 VAD 静音检测清晰交接（约 200ms）。
- 全双工：双方可以同时说话。模型可以进行副声道回应（"嗯哼"）或打断。难得多。Moshi 支持这个。

Qwen2.5-Omni 默认支持半双工，通过静音阈值进行轮替。全双工需要应用层处理。

### Qwen3-Omni（2025 年 11 月）

继任者。Qwen3-80B 思考者，更大的说话者，改进的 TMRoPE-v2。延迟接近 GPT-4o 的 250ms。开放权重。在 OmniBench 上的基准测试与 Gemini 2.0 Live 相当。

### 生产延迟预算

对于一个典型的流式交互：

- 麦克风 -> 音频 token：40-80ms。
- 预填充（提示词 + 历史）：7B 为 100-200ms，70B 更多。
- 第一个思考者文本 token：40ms。
- 说话者处理第一个文本 token：20ms。
- 第一个语音 token 提交：40ms。
- 残差 VQ 解码：30ms。
- 语音波形解码：50-80ms。

总 TTFAB：7B 为 320-510ms，70B 为 600-900ms。前沿质量通常需要 70B+；因此前沿延迟差距存在。

### Token 速率数学

在 16kHz 语音、50 Hz 基础语音 token 下，每秒输出需要 50 个语音 token。说话者必须保持 >=50 tok/s 才能跟上。在 H100 上，以典型的 LLM 吞吐量 30-80 tok/s 计算，一个小的（200-300M）说话者足够快；一个 7B 说话者会跟不上。

这就是为什么存在小型专用说话者模型，而不是"直接用主模型"。

## 使用它

`code/main.py`：

- 使用模拟 token 发射率模拟一个思考者-说话者流水线。
- 为可配置的模型大小和麦克风采样率计算 TTFAB。
- 演示使用 VAD 静音阈值的半双工轮替。

## 交付它

本课程产出 `outputs/skill-omni-streaming-budget.md`。给定一个实时语音产品的目标 TTFAB 和功能集（视觉输入、双语、全双工），它在 Qwen2.5-Omni、Qwen3-Omni、Moshi 或 Mini-Omni 中选择，并确定思考者/说话者的大小。

## 练习

1. 你的目标 TTFAB 是 300ms。在一个 7B 思考者和 300M 说话者上，写出每个组件的延迟。

2. Qwen2.5-Omni 使用 TMRoPE。描述对于一个用户从 t=1s 开始说话、摄像头在 t=1.2s 捕捉到一个手势的提示词，模型看到了什么。

3. 全双工支持要求模型在监听时发出音频。提出一个教导这一行为的训练数据格式。

4. 阅读 Moshi 论文第 4 节。描述"内心独白"分离以及它为什么避免了思考者-说话者分离。

5. 计算吞吐量预算：说话者必须多快地发出 token 才能跟上以 50 个基础层 token/秒的 16kHz 语音？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| 思考者 | "推理大脑" | 大型文本生成 transformer，产生要说的内容 |
| 说话者 | "语音生成嘴巴" | 小型 transformer，从思考者的文本产生离散语音 token |
| TTFAB | "延迟预算" | 首次音频字节输出时间：从用户语音结束到首个音频样本输出 |
| TMRoPE | "时间对齐 RoPE" | 使用跨视觉、音频、文本的绝对时间戳的位置编码 |
| 半双工 | "轮替" | 用户和模型交替；VAD 静音检测用户说完 |
| 全双工 | "同时" | 模型可以同时说话和听；能够副声道回应 |
| 内心独白 | "Moshi 分离" | 单一模型设计，思考流和说话流交错 |
