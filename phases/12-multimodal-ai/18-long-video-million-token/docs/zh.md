# 百万 token 上下文的长视频理解

> 一个 1 小时的 4K 视频，24 FPS，经过块化和嵌入，产生约 6000 万个 token。一个 2 小时的播客片段转录后有 30,000 个 token。一部完整的蓝光故事片，即使经过激进的池化压缩，也有数十万个 token。Google 的 Gemini 1.5（2024 年 3 月）以 1000 万 token 的上下文开启了这一时代，在一小时长的视频上实现了可靠的海捞针召回。LWM（Liu 等人，2024 年 2 月）展示了环形注意力的扩展路径。LongVILA 和 Video-XL 进一步扩展了摄入能力。VideoAgent 用智能体检索替换了原始上下文。每种方法在计算、召回和工程复杂性方面都有不同的权衡。本课程将它们并列解读。

**类型：** 构建
**语言：** Python（标准库，海捞针模拟器 + 智能体检索路由器）
**前置知识：** 阶段 12 · 17（视频时间 token）
**时间：** ~180 分钟

## 学习目标

- 计算长视频在不同 FPS 和池化下的总视觉 token 数量。
- 解释三种扩展路径：暴力上下文（Gemini 1.5）、环形注意力（LWM）、token 压缩（LongVILA / Video-XL）。
- 在准确率和延迟方面比较原始上下文视频 VLM 与智能体检索视频 VLM（VideoAgent）。
- 为一个 30 分钟的视频设计一个海捞针测试，并测量在特定分钟处的召回率。

## 问题

Qwen2.5-VL 大小的块在 384 原生分辨率下，一帧约 729 个 token。3x3 池化后为每帧 81 个 token。一个以 1 FPS 采样的 30 分钟片段 = 1800 帧 = 145,800 个 token。对 2025 年的开放 VLM 来说可行，但已很紧张。以 2 FPS 计算，291,600 个 token——只有最大的上下文才能容纳。

一部以 1 FPS 采样的 2 小时电影是 583k 个 token。超出了大多数 2026 年开放模型的能力；需要 Gemini 2.5 Pro 或更激进的池化。

出现了三种扩展路径。

## 概念

### 路径 1：暴力上下文（Gemini 1.5、Claude Opus）

用硬件解决问题。将上下文扩展到数百万 token，在一次前向传播中处理所有内容。

Gemini 1.5 Pro 以 100 万 token 推出；Gemini 1.5 Ultra 增加到 1000 万；2026 年的 Gemini 2.5 Pro 能可靠地处理数小时的视频。论文（arXiv:2403.05530）记录了在约 950 万 token 范围内 99.7% 的海捞针召回率。

工程实现：具有内存层级（局部 + 全局 + 稀疏）的自定义注意力实现，加上用于长上下文效率的 MoE 专家路由。未完全公开细节。非开源。

### 路径 2：环形注意力（LWM、LongVILA）

环形注意力将长序列分布在设备间的"环"上，每个设备持有一个块。跨完整序列的注意力通过每个设备将其块发送给环中的下一个设备、计算部分注意力并聚合来实现。

LWM（Liu 等人，2024）以这种方式训练了一个 100 万 token 上下文的模型。训练计算随上下文线性扩展，而非二次——注意力的二次开销在环的设备间摊销。

LongVILA (arXiv:2408.10188) 将该模式适配到了 VLM。1400 帧视频，每帧 192 个 token = 268k 上下文，使用环形注意力在 8 路并行下训练。

### 路径 3：Token 压缩（Video-XL、LongVA）

比暴力上下文更便宜：在 LLM 看到序列之前激进压缩。

Video-XL (arXiv:2409.14485) 使用一个视觉摘要 token：每个 N 帧的片段产生一个单一的"摘要"token，该 token 关注这 N 帧。在推理时，LLM 每个片段看到一个摘要 token，大幅缩小上下文。

LongVA 通过"长上下文迁移"技术将 LLM 上下文从 200k 扩展到 200 万。在长上下文文本上训练，通过共享表示迁移到长上下文视频。

Token 压缩在特定时间戳的召回率与可扩展性之间进行权衡。模型大致知道发生了什么，但有时会错过确切的帧。

### 路径 4：智能体检索（VideoAgent）

不将完整视频输入给 LLM。相反，将视频视为数据库，并使用 LLM 来查询它。

VideoAgent (arXiv:2403.10517)：

1. LLM 读取问题。
2. LLM 向检索工具请求相关片段（"显示有猫的片段"）。
3. 工具返回匹配的片段时间戳。
4. LLM 通过 VLM 读取这些片段。
5. LLM 组成答案或提出后续查询。

这是应用于长视频的 LLM-as-agent 模式。推理更便宜（只有相关片段被编码），工程更困难（检索质量成为瓶颈）。

### 海捞针基准测试

标准的长上下文测试：在视频中随机位置插入一个独特的视觉或文本标记，然后提出需要回忆起它的查询。

指标：跨视频长度和标记位置的 Recall@k。

Gemini 2.5 Pro 在长达 90 分钟的视频中得分 >99% 的召回率。开放的 72B 模型（Qwen2.5-VL-72B、InternVL3-78B）在 30 分钟内得分约 85-90%，超过 60 分钟后性能下降。

VideoAgent 可以在 2 小时以上的视频中匹配或击败原始上下文模型，因为如果检索工具足够好，它就能找到"针"。

### 选择哪条路径

对于 15 分钟的片段，要求前沿准确率：开放 72B + 原生上下文通常可行。选择 Qwen2.5-VL-72B。

对于 30 分钟到 1 小时的内容：开放模型使用 LongVILA 或 Video-XL；闭源模型使用 Gemini 2.5 Pro。质量门槛决定——前沿性能倾向闭源。

对于 2 小时以上的内容：使用 VideoAgent 或类似的检索模式。或者，总结成更小的块并输入分层摘要。

### 2026 年生产模式

在实践中，生产级的长视频流水线是混合的：

1. 对整个视频运行动态 FPS 采样 + 激进池化（获得一个 10 万 token 的全局表示）。
2. 传递给一个 72B VLM 以生成全局摘要。
3. 如果用户提出详细问题，使用摘要作为索引运行智能体检索。

这结合了用于全局理解的暴力上下文和用于局部细节的检索。

2. 设计一个海捞针测试：你在第几分钟注入标记，确切的查询格式是什么？

3. 比较暴力上下文的 Qwen2.5-VL-72B（80k 上下文）与 VideoAgent（Claude 3.5 + 检索）在一小时视频上的表现。哪个在召回率上胜出？哪个在延迟上胜出？

4. 环形注意力的内存成本在序列长度上线性增长，在设备数量上也线性增长。解释为什么，以及如果跳过环旋转阶段会发生什么失败。

5. 阅读 Gemini 1.5 第 5 节关于海捞针的内容。论文发现了关于 100 万 vs 1000 万 token 边界处的召回率什么结论？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| 暴力上下文 | "更多 token 就完了" | 将 LLM 上下文扩展到数百万 token；一次处理所有内容 |
| 环形注意力 | "LWM 风格并行" | 分布式注意力模式，每个设备持有一个块并进行旋转 |
| Token 压缩 | "摘要 token" | 通过学习得到的压缩器在 LLM 之前减少每片段 token |
| 海捞针 | "NIH 测试" | 在随机位置插入一个独特标记，在测试时要求模型回忆它 |
| 智能体检索 | "LLM 作为查询规划器" | LLM 向检索工具请求相关片段，通过 VLM 读取它们，组成答案 |
| VideoAgent | "视频检索模式" | 规范的智能体检索设计：问题 -> 工具 -> 片段 -> 答案 |
