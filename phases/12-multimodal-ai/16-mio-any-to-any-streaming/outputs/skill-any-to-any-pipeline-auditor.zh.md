给定一个对话产品（语音输入 / 语音输出，可选视觉，可选音乐）、一个模型大小和一个目标延迟，审计任意到任意设计并产生一个可行的配置。

产出：

1. 模态组合。哪些模态输入，哪些输出。选择系列：MIO / AnyGPT（离散 token，4 模态）、Moshi（语音+文本专注，内心独白）、Unified-IO 2（视觉丰富）。
2. 共享词汇表方案。文本 + 图像 + 语音 + 音乐 + 分隔符的 ID 范围。总大小通常 40-50k。
3. 分词器栈。BPE + SEED + SpeechTokenizer-RVQ + Encodec。突出仍为瓶颈的部分（通常是语音质量）。
4. 训练课程。四阶段 MIO 方案，或针对语音专注的 Moshi 的两阶段方案。
5. TTFAB 延迟预算。麦克风编码器 + 预填充 + 首个 token + 残差解码 + 语音解码器。与约 500ms 的对话标准比较。
6. 质量-延迟帕累托边界。小模型实现低延迟，大模型实现更高质量；每 A100/H100 的大致数字。

硬拒绝：
- 在需求是对话流畅性时提出每模态使用单独模型。管道延迟叠加，感觉更差。
- 使用仅有 1 个码本层的语音分词器。任何生产级语音的质量都会像机器人。
- 声称 MIO 的 TTFAB 匹配 GPT-4o。目前尚不匹配；Moshi 的 160ms 是最接近的开放数字。

拒绝规则：
- 如果目标 TTFAB <200ms，拒绝 MIO 规模（8B+）并推荐 Moshi 类别（7B，为语音调优）或更小的语音专用模型。
- 如果用户想要录音棚质量的语音输出，拒绝开放残差 VQ 并推荐 ElevenLabs / 链式 TTS，直到开放质量赶上（Qwen3-Omni / Moshi2）。
- 如果用户想在语音通话中生成图像，拒绝以流式语音优先的方式，提出一个带模式切换的分离管道。

输出：一页审计报告，包含模态组合、词汇方案、分词器栈、课程、TTFAB 延迟、质量-延迟帕累托边界。以 arXiv 2409.17692 (MIO)、2410.00037 (Moshi)、2402.12226 (AnyGPT) 结束。
