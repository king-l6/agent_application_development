# Janus-Pro：统一多模态模型的解耦编码器

> 统一多模态模型面临一个不可避免的张力。理解任务需要语义特征——SigLIP 或 DINOv2 输出的向量富含概念级别的信息。生成任务需要利于重构的编码——能够清晰组合回像素的 VQ 令牌。这两个目标在单个编码器中是不兼容的。Janus（DeepSeek，2024 年 10 月）和 Janus-Pro（DeepSeek，2025 年 1 月）认为解决方法就是停止试图合并：解耦两个编码器。在任务之间共享 Transformer 主体，但将理解任务路由到 SigLIP，将生成任务路由到 VQ 分词器。在 7B 规模下，Janus-Pro 在 GenEval 上击败了 DALL-E 3，同时在 MMMU 上与 LLaVA 持平。本课程将解读为什么两个编码器在单个编码器失败的地方能够成功。

**类型：** 构建
**语言：** Python（标准库，双编码器路由 + 共享主体信号）
**前置要求：** 阶段 12 · 13（Transfusion），阶段 12 · 14（Show-o）
**时间：** ~120 分钟

## 学习目标

- 解释为什么单一共享编码器会损害理解或生成质量。
- 描述 Janus-Pro 的路由机制：输入侧的 SigLIP 特征用于理解，VQ 令牌在输入和输出端用于生成。
- 追踪使 Janus-Pro 比 Janus 更成功的数据混合规模化策略。
- 比较解耦（Janus-Pro）、耦合连续（Transfusion）和耦合离散（Show-o）架构。

## 问题

统一模型在理解和生成任务之间共享 Transformer 主体。先前的尝试（Chameleon、Show-o、Transfusion）都使用一个视觉分词器来处理两个方向。分词器是一种妥协：

- **为重构（生成）优化**：VQ-VAE 捕捉精细的像素细节，但产生的令牌语义连贯性较弱。
- **为语义（理解）优化**：SigLIP 嵌入将"猫"图像靠近"猫"令牌，但不允许良好的重构。

Show-o 和 Transfusion 为此付出了代价，在一个方向上有明显的质量损失。Janus-Pro 问道：当任务有不同的需求时，为什么还要要求使用一个分词器？

## 概念

### 解耦视觉编码

Janus-Pro 的架构分离了两个编码器：

- **理解路径**。输入图像 → SigLIP-SO400m → 2 层 MLP → Transformer 主体。
- **生成路径**。输入图像（如果以现有图像为条件）→ VQ 分词器 → 令牌 ID → Transformer 主体。
- **输出生成**。Transformer 预测的图像令牌 → VQ 解码器 → 像素。

Transformer 主体是共享的。主体上游和下游的所有内容都是任务特定的。

输入通过提示格式进行区分：`<understand>` 标签路由到 SigLIP；`<generate>` 标签路由到 VQ。或者路由由任务隐式决定。

### 为什么这有效

理解损失获得 SigLIP 特征，这些特征经过 CLIP 风格的预训练已经调整为语义相似性。模型的感知基准测试优于 Show-o / Transfusion，因为输入特征更适合任务。

生成损失获得 VQ 令牌，这些令牌经过分词器调整为重构。图像质量优于 Show-o，因为 VQ 编码可以干净地组合回像素。

共享的 Transformer 主体看到两种输入分布（SigLIP 和 VQ），并学会处理两者。其主张是：足够的数据 + 足够的参数，主体能够吸收切换。

### 数据规模化——Janus vs Janus-Pro

Janus（原始版，arXiv 2410.13848）引入了解耦思想，但规模较小（1.3B 参数，有限的数据）。Janus-Pro（arXiv 2501.17811）进行了规模化：

- 7B 参数（vs 1.3B）。
- 第一阶段（对齐）9000 万图像-文本对，从 7200 万提升。
- 第二阶段（统一）7200 万，从 2600 万提升。
- 第三阶段增加了 20 万图像生成指令样本。

结果：Janus-Pro-7B 在 MMMU 上与 LLaVA 持平（60.3 vs ~58），在 GenEval 上击败 DALL-E 3（0.80 vs 0.67）。一个开放模型，在统一频谱的两侧都具有竞争力。

### JanusFlow——修正流变体

JanusFlow（arXiv 2411.07975）将 VQ 生成路径替换为修正流生成路径（连续）。分割变为：用于理解的 SigLIP + 用于生成的修正流。质量上限进一步提升。架构仍然是解耦编码器 + 共享主体。

### 共享主体的职责

Transformer 主体处理统一的序列，但面对两种输入分布。它的职责是：

- **对于理解**：消费 SigLIP 特征 + 文本令牌 → 自回归地生成文本。
- **对于生成**：消费文本令牌 +（可选的图像 VQ 令牌）→ 自回归地生成图像 VQ 令牌。

主体没有每个块的模态特定权重。它是你在 Qwen 或 Llama 内部能找到的那种文本风格 Transformer，再加上两个输入适配器。

有趣的是，这意味着 Janus-Pro 的主体可以从预训练的 LLM 初始化。Janus-Pro 确实从 DeepSeek-MoE-7B 初始化。这个选择很重要：LLM 贡献了从零开始的统一模型难以达到的推理能力。

### 与 InternVL-U 的比较

InternVL-U（第 12.10 课）是 2026 年的后续工作。它结合了：

- 原生多模态预训练（InternVL3 主干）。
- 解耦编码器路由（输入 SigLIP，输出 VQ + 扩散头部）。
- 统一的理解 + 生成 + 编辑。

InternVL-U 将 Janus-Pro 的架构选择纳入一个更大的框架。解耦编码器思想现在已成为大规模统一模型的默认方案。

### 局限性

解耦编码器增加了架构复杂度。需要训练两个分词器，维护两个输入路径，有两套故障模式。对于不需要生成能力的产品，Janus-Pro 是过度设计的——选择 LLaVA 系列的理解模型。

对于不需要理解能力的产品，Janus-Pro 是过度合格的——选择 Stable Diffusion 3 / Flux 模型。

对于两者都需要产品，Janus-Pro 现在是参考性的开放架构。

## 使用它

`code/main.py` 模拟了 Janus-Pro 的路由：

- 两个模拟编码器：SigLIP 类（产生 256 维语义向量）和 VQ 类（产生整数编码）。
- 一个提示路由器，根据任务标签选择编码器。
- 一个共享主体（替身），处理无论来自哪个编码器的令牌序列。
- 一个从第一阶段（对齐）到第三阶段（指令微调）的加权采样调度切换。

打印 3 个示例的路由路径：图像问答、T2I、图像编辑。

## 练习

1. Janus-Pro-7B 在 GenEval 上击败了 DALL-E 3。解释为什么一个 7B 的开放模型能在生成上匹配前沿的专有模型，但在理解上却不能。

2. 实现一个路由器函数：给定提示文本，分类为 `understand` 或 `generate`。如何处理像"描述然后画出来"这样的模糊提示？

3. JanusFlow 将 VQ 路径替换为修正流。现在 Transformer 主体输出什么？损失有什么变化？

4. 提出 Janus-Pro 架构可以通过增加一个解耦编码器来处理的第四个任务。例如：图像分割（DINO 风格）、深度估计（MiDaS 风格）。

5. 阅读 Janus-Pro 第 4.2 节关于数据规模化的内容。哪个数据阶段对 T2I 质量的提升贡献最大（相比 Janus）？

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|------|-------------|---------|
| 解耦编码 | "两个视觉编码器" | 每个方向有单独的分词器或编码器：理解用语义的，生成用重构的 |
| 共享主体 | "一个 Transformer" | 单个 Transformer 处理任一编码器的输出；没有模态特定权重 |
| SigLIP 用于理解 | "语义特征" | CLIP 家族的视觉塔，提供丰富的概念特征但重构能力差 |
| VQ 用于生成 | "重构编码" | 向量量化令牌，能够干净地解码回像素 |
| JanusFlow | "修正流变体" | 使用连续流匹配生成头替代 VQ 的 Janus-Pro |
| 路由标签 | "任务标签" | 提示标记（`<understand>` / `<generate>`），选择输入编码器 |
