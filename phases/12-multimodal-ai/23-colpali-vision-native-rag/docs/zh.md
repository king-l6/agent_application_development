# ColPali 与视觉原生文档 RAG

> 传统 RAG 将 PDF 解析为文本，分块，嵌入块，存储向量。每一步都会丢失信号：OCR 丢失图表数据，分块破坏表格行，文本嵌入忽略图形。ColPali（Faysse 等人，2024 年 7 月）提出了一个更简单的问题：为什么要提取文本？直接从 PaliGemma 嵌入页面图像，使用 ColBERT 风格的后期交互（late interaction）进行检索，保留文档携带的所有布局、图形、字体和格式信号。已发表的基准测试：在视觉丰富的文档上，端到端准确率比文本 RAG 高 20-40%。ColQwen2、ColSmol 和 VisRAG 扩展了这一模式。本课程阅读视觉原生 RAG 论文，并构建一个微型 ColPali 风格索引器。

**类型：** 构建
**语言：** Python（标准库，多向量索引器 + MaxSim 评分器）
**前置知识：** 阶段 11（LLM 工程 — RAG 基础），阶段 12 · 05（LLaVA）
**时间：** 约 180 分钟

## 学习目标

- 解释双编码器检索（每文档一个向量）与后期交互检索（每文档多个向量）之间的区别。
- 描述 ColBERT 的 MaxSim 操作以及 ColPali 如何将其从文本令牌推广到图像块。
- 构建一个微型 ColPali 风格索引器：页面 → 块嵌入 → 查询词嵌入上的 MaxSim → 前 k 页面。
- 在发票/财务报告用例上比较 ColPali + Qwen2.5-VL 生成器与文本 RAG + GPT-4。

## 问题

PDF 上的文本 RAG 丢弃了文档的大部分内容。财务报告的第三季度收入增长通常在图表中；医疗报告的发现位于带注释的图像中；法律合同的签名块是布局事实，而非文本事实。

文本 RAG 流程：

1. PDF → 通过 OCR / pdftotext 转为文本。
2. 文本 → 300-500 令牌的块。
3. 块 → 双编码器嵌入（一个向量）。
4. 用户查询 → 嵌入 → 余弦相似度 → 前 k 块。
5. 块 + 查询 → LLM。

五个有损步骤。图表未被捕获。表格在块间断裂。多列布局被压平。图形注释消失。

ColPali 的修复：跳过 OCR，直接嵌入页面图像。使用 ColBERT 风格的后期交互进行检索，使模型能够在查询时关注细粒度块。

## 概念

### ColBERT（2020）

ColBERT（Khattab & Zaharia, arXiv:2004.12832）是一种文本检索方法。它不是为每篇文档生成一个向量，而是为每个令牌生成一个向量。在查询时：

- 查询令牌获得自己的嵌入（N_q 个向量）。
- 文档令牌获得嵌入（N_d 个向量，通常缓存）。
- 分数 = 对所有查询令牌求和，取与所有文档令牌的余弦相似度的最大值：Σ_i max_j cos(q_i, d_j)。

这就是 MaxSim 操作。每个查询令牌"选择"其最佳匹配的文档令牌。最终分数是总和。

优点：召回率高，处理词级语义。缺点：每篇文档 N_d 个向量，存储成本高。

### ColPali

ColPali（Faysse 等人，arXiv:2407.01449）将 ColBERT 模式应用于图像。

- 每个页面由 PaliGemma（ViT + 语言）编码为块嵌入：每页 N_p 个向量。
- 每个用户查询（文本）被编码为查询令牌嵌入：N_q 个向量。
- 分数 = Σ_i max_j cos(q_i, p_j)，即查询文本令牌与页面图像块上的 MaxSim。
- 按总分检索前 k 页。

在文档摄入时：用 PaliGemma 嵌入每个页面，存储所有块嵌入。在查询时：嵌入查询令牌，对所有存储的页面嵌入计算 MaxSim，返回前 k 页。

优点：在视觉丰富的文档上，端到端比文本 RAG 好 20-40%。每个块向量捕获局部布局和内容。

缺点：N_p 个块 × 4 字节浮点数 × D 维向量每页 = 存储增长迅速。通过 PQ / OPQ 量化缓解。

### ColQwen2 与 ColSmol

ColQwen2（illuin-tech，2024-2025）将 PaliGemma 替换为 Qwen2-VL。更好的基础编码器，更好的检索。

ColSmol 是用于本地/边缘场景的小规模变体。约 1B 参数的 ColSmol 检索器可在消费级 GPU 上运行。

### VisRAG

VisRAG（Yu 等人，arXiv:2410.10594）是一种不同的变体：不是对块进行 MaxSim，而是使用 VLM 将每个页面池化为单个向量，然后进行双编码器检索。更快的索引 + 更小的存储，但召回率较低。

质量与成本的权衡：ColPali 用于质量，VisRAG 用于规模。

### M3DocRAG

M3DocRAG（Cho 等人，arXiv:2411.04952）将多模态检索扩展到多页多文档推理。跨文档检索页面，为 VLM 组合多页上下文。

### ViDoRe — 基准测试

ColPali 的配套基准测试：视觉文档检索评估（Visual Document Retrieval Evaluation）。任务包括财务报告、科学论文、行政文档、医疗记录、手册。指标：nDCG@5。

ColPali-v1 在 ViDoRe 上得分约 80% nDCG@5；同一文档上的文本 RAG 得分约 50-60%。

### 端到端 RAG 流程

对于视觉原生 RAG：

1. 摄入：PDF → 页面图像 → PaliGemma 编码 → 存储所有块嵌入。
2. 查询：用户文本 → 查询令牌嵌入 → 对索引页面进行 MaxSim → 前 k 页面。
3. 生成：前 k 页面图像 + 查询 → VLM（Qwen2.5-VL 或 Claude）→ 答案。

全程无 OCR。图形、图表、字体、布局全部流入答案。

### 存储计算

一份 50 页的财务报告，每页 729 个块，128 维嵌入：

- ColPali：50 * 729 * 128 * 4 字节 = 约 18 MB 原始，PQ 后约 4 MB。
- 文本 RAG：50 个块 * 768 维 * 4 字节 = 约 150 kB。

ColPali 每份文档的存储量大约是文本 RAG 的 30 倍。大规模下，OPQ / PQ 可将其降至约 5-10 倍，通常是可接受的。

### 文本 RAG 仍然胜出的场景

- 纯文本文档，无布局信号（维基百科文章、聊天记录）。文本 RAG 更简单，存储成本更低。
- 数百万页的归档，存储主导成本。
- 严格的监管要求，需要可提取的 OCR 文本以及检索。

对于 2026 年的其他一切——财务报告、科学论文、法律合同、医疗记录、UX 文档——视觉原生 RAG 胜出。

## 使用

`code/main.py`：

- 玩具块编码器：将"页面"（小特征向量网格）映射到块嵌入数组。
- MaxSim 评分器：计算查询令牌嵌入集与页面块集之间的 ColBERT 风格分数。
- 索引 5 个玩具页面，运行 3 个查询，返回前 k 页及分数。

本课程产出 `outputs/skill-vision-rag-designer.md`。给定一个文档 RAG 项目，选择 ColPali / ColQwen2 / VisRAG / 文本 RAG 并估算存储大小。

## 练习

1. 一份 200 页的年度报告，每页 729 个块，128 维嵌入，4 字节浮点数。计算原始存储和 PQ 压缩后（8 倍）的存储。

2. MaxSim 是 Σ_i max_j cos(q_i, p_j)。这个总和捕获了简单均值相似度无法捕获的什么信息？

3. ColPali 将页面索引为块集。如果我们改为在词级别进行索引（如 ColBERT 所做），会有什么变化？权衡是什么？

4. 为一个 100 万页的语料库设计端到端流程，每个查询的延迟预算为 500ms。选择 ColQwen2 / VisRAG 并论证。

5. 阅读 M3DocRAG（arXiv:2411.04952）。描述多页注意力模式以及它如何不同于单页 ColPali 检索。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 后期交互（Late interaction） | "ColBERT 风格" | 使用每令牌或每块嵌入 + MaxSim 进行检索，而非单个文档向量 |
| MaxSim | "块上的最大值" | 对每个查询令牌，选择相似度最高的文档令牌；跨查询求和 |
| 双编码器（Bi-encoder） | "单向量" | 每篇文档一个向量；更快但丢失了粒度 |
| 多向量（Multi-vector） | "每文档多向量" | 每篇文档/页面存储 N_p 个向量；存储成本增加但召回率提高 |
| 块嵌入（Patch embedding） | "页面特征" | VLM 编码器为每个图像块生成的向量，按页面缓存 |
| ViDoRe | "视觉文档基准" | ColPali 的视觉文档检索基准套件 |
| PQ 量化（PQ quantization） | "乘积量化" | 压缩技术，保持向量相似度同时将存储缩小约 8 倍 |
