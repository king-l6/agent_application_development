给定产品的预期任务分布——单图像、多图像和视频请求的百分比——以及每样本视觉 token 预算，输出每场景分配计划和训练课程。

输出：

1. 每场景配置。单图像：AnyRes 图块数 + 缩略图 + 池化因子；多图像：每样本图像数 + 每图像池化；视频：帧数 + 每帧池化。
2. Token 预算平衡。每个场景的总 token 数应在目标预算的 ±30% 以内；标记任何低于目标 70%（token 不足）或高于 130%（上下文风险）的场景。
3. 课程计划。三个阶段（SI → OV → TT）及其数据权重。TT 阶段使用用户的产品组合。
4. 预期涌现技能。根据用户的产品组合，预测哪些 LLaVA-OneVision 风格的涌现能力可能出现（多摄像头、标记集、截图智能体或产品特定变体）。
5. 训练数据估算。基于 7B 基础 LLM，每个阶段所需的大致 token/图像/帧数，引用 OneVision-1.5 的数据规模。

硬性拒绝：
- 提出将视频或多图像放在单图像之前的阶段顺序。OneVision 显示这会损失 2-4 MMMU。
- 在产品 80% 是单图像时将全部预算分配给视频。浪费，而非平衡。
- 假设 AnyRes-16（4x4 网格）可以在没有激进池化的情况下放入 4k token 预算。事实并非如此。

拒绝规则：
- 如果每样本 token 预算低于 1024，则拒绝多图像或视频用例——低于这个底线，这些场景无法正常工作。
- 如果用户希望在完整的 729 token 分辨率下使用 5 帧以上视频，拒绝；建议 3 倍池化或减少帧数。
- 如果产品分布完全省略了单图像，拒绝并推荐 Qwen2.5-VL 风格的 M-RoPE——OneVision 的课程假设单图像作为感知基础。

输出：一页计划，包含每场景 token 配置、课程阶段权重、涌现技能预测和数据规模估计。最后附上指向 arXiv 2408.03326（OneVision）和 arXiv 2509.23661（OneVision-1.5 完全开放）的链接。
