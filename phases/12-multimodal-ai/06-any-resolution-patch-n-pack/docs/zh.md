# 任意分辨率视觉：Patch-n'-Pack 与 NaFlex

> 真实世界的图像并非 224x224 的正方形。收据是 9:16，图表是 16:9，医学扫描可能是 4096x4096，手机截图是 9:19.5。2024 年之前的 VLM 做法——将所有图像缩放到固定正方形——丢弃了让 OCR、文档理解和高分辨率场景解析得以工作的信号。NaViT（Google，2023）展示了可以将可变分辨率的 patch 打包到单个 transformer 批次中，并使用块对角掩码。Qwen2-VL 的 M-RoPE（2024）完全抛弃了绝对位置表。LLaVA-NeXT 的 AnyRes 将高分辨率图像平铺成基础图加子图像。SigLIP 2 的 NaFlex 变体（2025）现在是开放 VLM 的默认编码器，用单个检查点即可服务所有宽高比。本节课从头实现 patch-n'-pack。

**类型：** 构建
**语言：** Python（标准库，patch 打包器 + 块对角掩码）
**前置知识：** 第 12 阶段 · 01（ViT patch），第 12 阶段 · 05（LLaVA）
**时间：** ~120 分钟

## 学习目标

- 将一批可变分辨率图像的 patch 打包成一个序列，并构建块对角注意力掩码。
- 为给定任务在 AnyRes 平铺（LLaVA-NeXT）、NaFlex（SigLIP 2）和 M-RoPE（Qwen2-VL）之间做出选择。
- 在不对图像进行缩放的情况下，计算 OCR、图表和摄影的 token 预算。
- 指出正方形缩放的三种失败模式：文字挤压、内容裁剪、填充 token 浪费。

## 问题

Transformer 期望输入是一个序列。一个批次是等长序列的堆叠。如果你的图像是 224x224，每次你会得到 196 个 patch token，不需要填充，任务完成。在 224 上训练，在 224 上推理，再也不需要考虑分辨率的问题。

但现实世界不会配合。文档是竖屏的（8.5x11 英寸，大约 2:3）。图表截图是横屏的（16:9）。收据高而窄（1:3）。医学影像通常是 2048x2048 或更大。移动设备截图是 1170x2532（0.46:1）。

2024 年之前的三种方案及其各自的失败原因：

1. 缩放到固定正方形（224x224 或 336x336）。挤压会扭曲文字和人脸。下采样会破坏图表标签和 OCR 内容。这是 LLaVA-1.5 之前的标准做法。
2. 裁剪到固定宽高比。你会丢弃大部分图像，而且选择裁剪位置本身就是个视觉问题。
3. 填充到最长边。解决了扭曲问题，但将 50% 以上的 token 浪费在填充上，对于竖屏图像尤甚。所有填充 token 都会产生二次注意力开销。

2024-2025 年的答案：让 transformer 以图像的原始分辨率消耗 patch，并找到将异构批次打包到单个序列中而不浪费计算的方法。

## 概念

### NaViT 与 patch-n'-pack

NaViT（Dehghani 等人，2023）是证明这种方法可以大规模工作的论文。其思路是机械性的：

1. 对于批次中的每张图像，以选定的 patch 大小（比如 14）计算其原生 patch 网格。
2. 将每张图像的 patch 展平为其自身的可变长度序列。
3. 将所有图像的 patch 连接成该批次的一个长序列。
4. 构建一个块对角注意力掩码，使得图像 A 的 patch 只能在图像 A 内部相互注意。
5. 携带每个 patch 的位置信息（2D RoPE 或分数位置嵌入）。

一个包含三张图像（336x336（576 token）、224x224（256 token）和 448x336（768 token））的批次，会变成一个 1600 token 的序列，带有一个 1600x1600 的块对角掩码。没有填充。没有计算浪费。Transformer 可以处理任意宽高比。

NaViT 还引入了训练期间的分数 patch 丢弃——在整批中随机丢弃 50% 的 patch——既能正则化又能加速训练。SigLIP 2 继承了这一做法。

### AnyRes（LLaVA-NeXT）

LLaVA-NeXT 的 AnyRes 是更实用的替代方案。给定一张高分辨率图像和一个固定编码器（336 分辨率的 CLIP 或 SigLIP），将图像平铺：

1. 从预定义的网格布局集合——(1x1)、(1x2)、(2x1)、(1x3)、(3x1)、(2x2) 等——中选出最符合图像宽高比的布局。
2. 将整张图像平铺到该网格中；每个图块成为 336x336 的裁剪。
3. 同时生成一张缩略图：将整张图像缩放到 336x336 作为全局上下文 token。
4. 将所有图块通过冻结的 336 编码器进行编码。连接图块 token 和缩略图 token。

对于 672x672 的图像，使用 2x2 网格加缩略图：4 * 576 + 576 = 2880 个视觉 token。计算量大但有效——LLM 既能看到局部细节，也能看到全局上下文。

当编码器被冻结且只支持一种分辨率时，AnyRes 是首选方案。对于大图像，token 数量会爆炸（1344x1344 的图像使用 4x4 网格会产生 9216 + 576 ≈ 9800 个 token，几乎填满 8k LLM 上下文）。

### M-RoPE（Qwen2-VL）

Qwen2-VL 引入了多模态旋转位置编码。与 NaViT 的分数位置或 AnyRes 的图块加缩略图不同，每个 patch 携带一个 3D 位置（时间、高度、宽度）。query/key 旋转可以处理任意的 H、W 和时间长度。

M-RoPE 开箱即用地支持原生动态分辨率，无需重新训练。推理时，你输入任意 HxW 图像，patch 嵌入器生成 H/14 x W/14 个 token，每个 token 获得其 (t=0, r=行, c=列) 位置，RoPE 以正确的频率旋转注意力，完成。Qwen2.5-VL 和 Qwen3-VL 继续沿用这一方法。InternVL3 的 V2PE 是相同的思路，但每个模态使用可变编码。

与 AnyRes 不同，M-RoPE 的 token 数为 O(H x W / P^2) 的原始分辨率——没有乘性的图块开销。与 NaViT 不同，它仍然期望每次前向传播只有一张图像。跨分辨率批处理仍需要在上面叠加 patch-n'-pack。

### NaFlex（SigLIP 2）

NaFlex 是 SigLIP 2 检查点的原生灵活模式。单个模型在推理时支持多种序列长度（256、729、1024 token）。在内部，它在训练期间使用 NaViT 风格的 patch-n'-pack 以及每个 patch 的绝对分数位置。卖点：一个检查点，根据任务在推理时选择 token 预算。

对于语义任务（分类、检索），使用 256 token。对于 OCR 或图表理解，使用 1024 token。无需重新训练。

### 打包掩码

块对角掩码是大多数实现容易出错的地方。对于一个覆盖图像 `i=0..B-1`（长度分别为 `n_i`）的总长度为 `N_total` 的打包序列，形状为 `(N_total, N_total)` 的掩码 `M` 在两个索引落在同一图像的块内时为 1，否则为 0。你可以根据累积长度列表来构建它：

```
offsets = [0, n_0, n_0+n_1, ..., N_total]
M[i, j] = 1 当且仅当存在 b 使得 offsets[b] <= i < offsets[b+1] 且 offsets[b] <= j < offsets[b+1]
```

在 PyTorch 中，这可以用 `torch.block_diag` 或显式的 gather 操作一行完成。FlashAttention 的可变长度路径（`cu_seqlens`）完全跳过掩码，直接使用累积长度张量在序列内部进行注意力计算——对于典型批次，比稠密掩码快约 10 倍。

### Token 预算

根据任务选择策略：

- OCR / 文档：1024-4096 token。SigLIP 2 NaFlex 使用 1024，或 AnyRes 3x3 加缩略图。
- 图表和 UI：729-1024 token，使用 384-448 原生分辨率。Qwen2.5-VL 动态分辨率加最大像素上限。
- 自然照片：256-576 token 就足够了。下游 LLM 能看到足够的信息。在内容密度高的地方投入 token。
- 视频：空间池化后每帧 64-128 token，2-8 FPS。第 12.17 课涵盖此内容。

2026 年的生产规则：为每个任务选择一个最大像素上限，以原生宽高比编码到该上限，打包批次，跳过填充。Qwen2.5-VL 提供了 `min_pixels` 和 `max_pixels` 参数来实现这一调节。

## 使用

`code/main.py` 使用整数像素坐标为异构图像批次实现了 patch-n'-pack。它：

- 接收一个 (H, W) 图像尺寸列表。
- 以 patch 大小 14 计算每张图像的 patch 序列长度。
- 将它们打包成总长度为 `sum(n_i)` 的一个序列。
- 构建块对角注意力掩码（为清晰起见使用稠密形式）。
- 比较打包成本与正方形缩放和 AnyRes 平铺的成本。
- 为混合批次（收据、图表、截图、照片）打印 token 预算表。

运行它。得出的数字解释了为什么每款 2026 年的开放 VLM 都使用 patch-n'-pack。

## 练习

1. 一张收据尺寸为 600x1500（1:2.5）。patch 大小为 14 时，原生分辨率 token 是多少？缩放到 336 正方形后是多少？在实践中哪种会损失更多 OCR 精度？

2. 为一批四张图像（长度分别为 256、576、729、1024）构建块对角掩码。验证注意力矩阵为 2585x2585，并且恰好有 `256^2 + 576^2 + 729^2 + 1024^2` 个非零条目。

3. 对于一张 1792x896 的图像（patch 14），比较：(a) 缩放到 336 正方形后编码，(b) AnyRes 2x1 加缩略图，(c) 原生分辨率 M-RoPE。哪种使用 token 最少？哪种保留细节最多？

4. 实现分数 patch 丢弃：给定一个打包序列，均匀随机丢弃 50% 的 token，并相应更新块对角掩码。测量掩码稀疏度的变化。

5. 阅读 Qwen2-VL 论文（arXiv:2409.12191）第 3.2 节。用两句话描述 `min_pixels` 和 `max_pixels` 控制什么，以及为什么两个边界都很重要。

## 关键术语

| 术语 | 人们常说的 | 实际含义 |
|------|-----------|---------|
| Patch-n'-pack | "NaViT 风格打包" | 将来自不同图像的可变长度 patch 序列连接到一个批次维度中 |
| 块对角掩码 | "打包掩码" | 注意力掩码，将每张图像的 patch 限制为仅关注自身，而不是包中的邻居 |
| AnyRes | "LLaVA-NeXT 平铺" | 将高分辨率图像分割成固定大小图块网格加全局缩略图；用固定编码器编码每个图块 |
| NaFlex | "SigLIP 2 原生灵活" | 单个 SigLIP 2 检查点在推理时无需重新训练即可服务 256/729/1024 token 预算 |
| M-RoPE | "多模态 RoPE" | 3D 旋转位置编码（时间、行、列），无需位置表即可处理任意 H、W、T |
| cu_seqlens | "FlashAttention 打包" | FlashAttention 可变长度路径使用的累积长度张量，替代稠密块对角掩码 |
| min_pixels / max_pixels | "分辨率边界" | Qwen2.5-VL 的每请求调节参数，用于限制极小或极大输入的 token 数量 |
| 视觉 token 预算 | "每张图像多少 token" | 每张图像发出的 patch token 的粗略计数；决定了 LLM 的提示预算和注意力开销 |
