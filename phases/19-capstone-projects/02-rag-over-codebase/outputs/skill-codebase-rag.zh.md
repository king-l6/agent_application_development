给定至少 10 个仓库、总计至少 200 万行代码，构建一个摄入管道、一个混合索引和一个引用强制的查询 Agent，能够回答跨仓库问题并带有可验证的文件:行锚点。

构建计划：

1. 使用 tree-sitter 解析每个文件。在函数和类节点边界处分块。存储 `{repo, path, start_line, end_line, symbol, body}`。
2. 使用 Claude Haiku 4.5 或 Gemini 2.5 Flash 为每个块生成摘要，使用带提示缓存的系统提示。将一句话摘要与块一起存储。
3. 索引到三个结构中：Qdrant（稠密，Voyage-code-3 或 nomic-embed-code）、Tantivy（带字段权重的 BM25）和 kuzu（符号图边，表示导入、调用、继承）。
4. 构建一个包含三个节点的 LangGraph 查询 Agent：检索（稠密并行 BM25）、重排序（Cohere rerank-3 或 bge-reranker-v2-gemma-2b）、综合（Claude Sonnet 4.7，带提示缓存和文件:行引用要求）。
5. 后置过滤：拒绝任何没有可验证 `(repo/path:start-end)` 锚点的声明；重新询问或丢弃。
6. 接入 git 推送 webhook，计算符号级差异，只重新嵌入变更的块。目标：50 文件提交在 60s 内可搜索，基于 200 万行代码群。
7. 使用 100 个问题的保留集进行评估。报告 MRR@10、nDCG@10、引用忠实度和延迟百分位。
8. 运行每周漂移任务，重新执行评估并在 MRR@10 下降 > 5% 时发出警报。

评估标准：

| 权重 | 标准 | 衡量方式 |
|:-:|---|---|
| 25 | 检索质量 | 100 问题保留集上的 MRR@10 和 nDCG@10 |
| 20 | 引用忠实度 | 具有可验证文件:行锚点的答案声明比例 |
| 20 | 延迟和规模 | 在索引语料库大小上 p95 查询延迟 @ 10k QPS |
| 20 | 增量索引正确性 | 从 git 推送到可搜索的时间（50 文件提交） |
| 15 | 用户体验和答案格式 | 引用可点击性、代码片段预览、追问能力 |

硬性拒绝：

- 使用固定大小 token 分块而不是 AST 感知分块。会在生成代码密集的语料库上产生有毒结果。
- 仅余弦检索，没有 BM25 或重排序。已知在精确符号名查询上会失败。
- 没有强制文件:行引用的答案。
- 每次 git 推送时全语料库重新嵌入；必须是增量式的。

拒绝规则：

- 拒绝在没有读取其许可证的情况下索引仓库。有些禁止在第三方向量存储中嵌入。
- 拒绝回答声称引用索引从未见过的文件的查询；返回前始终验证锚点。
- 拒绝在 p95 以上 4s 时提供答案；改为返回部分结果和后续处理句柄。

输出：一个包含摄入管道、LangGraph 查询 Agent、100 问题标注评估集、Langfuse 仪表盘链接以及一份报告的仓库。报告命名你修复的三个检索失败模式（生成代码中毒、长尾符号召回、跨仓库符号解析）以及修复每个问题的确切变更。
