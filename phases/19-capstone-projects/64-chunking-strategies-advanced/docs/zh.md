# 分块策略对比

> 分块决定了你的检索器能召回的极限。边界选错了，任何嵌入模型、任何重排序器、任何 LLM 都无法在下游修复损害。

**类型：** 构建
**语言：** Python
**前置知识：** 阶段 11 课程 04（嵌入）、06（RAG）、07（高级 RAG）；阶段 19 轨道 B 基础（课程 20-29）
**时间：** ~90 分钟

## 学习目标
- 从头实现五种分块策略：固定窗口、句子、递归分割、语义聚类和结构化 Markdown 标题。
- 在带有黄金标注答案片段的夹具语料库上测量 recall@k，并解释为什么一种策略在散文上获胜而不同的策略在技术文档上获胜。
- 读取分块长度分布，识别每种策略注入的失败模式：孤儿句、符号中间切断、仅标题块、语义漂移。
- 通过检查三个属性为新语料库选择默认策略而无需运行基准测试：文档类型、平均段落长度、以及格式是否携带显式结构。

## 问题

每个 RAG 管道都从将源文档切割成足够小的片段（以便嵌入模型能容纳）且足够大（以便每个片段携带一个自包含的想法）开始。切割位置的选择不是一个超参数。它是检索器能返回的一切的上限。

一个询问"预算中止阈值是什么样"的查询只有在包含中止阈值的块可触及的情况下才能成功。如果固定窗口分割器将阈值值与周围上下文切分开，嵌入会移动到不同的聚类，BM25 分数下降，重排序器看到噪声，LLM 生成的答案就是错误的。2024 年的论文"LongRAG: Enhancing Retrieval-Augmented Generation with Long-context LLMs"测量出纯粹因分块选择导致的检索召回率波动高达 35 个百分点。2025 年关于上下文块标题的后续工作缩小了这一差距但并未消除它。

本课程并排构建五种策略，在带有黄金标注答案片段的夹具语料库上运行它们，让你自己阅读召回数字。

### 固定窗口

暴力基线。每 N 个字符切割一次。可选重叠，使得在位置 N 处被切割的句子完整地出现在从位置 N-overlap 开始的块中。快速、确定性、边界处糟糕。将其用作对照而非默认。

### 句子

用正则表达式或简单的状态机在句子边界处分割。将一个或多个句子打包到一个块中，不超过目标字符预算。不会在词中间切断。但仍会在段落中间和章节中间切断。许多早期 RAG 管道的默认选择，对于没有其他结构的散文来说是一个合理的选择。

### 递归分割

2023 年时代库推广的层次策略。首先尝试在最强的分隔符处分割（双换行、段落），回退到下一个（单换行），然后到句子，最后到字符。当块符合预算时递归终止。对于结构不一致的文档很有效，因为它按区域自适应。

### 语义聚类

嵌入每个句子。将共享主题质心的连续句子聚类。当与质心的运行相似度低于阈值时切割。边界反映语义而非字符。构建较慢且依赖于嵌入模型，但对于在段落内部切换主题的文档具有鲁棒性。

### 结构化 Markdown 标题

对于携带显式结构的文档（Markdown、reStructuredText、RFC 风格编号章节），在标题边界处切割。每个块成为标题加上其下直到下一个同级或上级标题的所有内容。每个主题的块最小，但仅当语料库结构良好时可用。

### recall@k 如何衡量边界选择

一个黄金标注的查询携带源文档中答案片段的确切字符偏移。分块后，你询问：检索器返回的前 k 个块中是否有任何块与黄金片段重叠？如果是，该查询的 recall@k 为 1；否则为 0。在查询集上取平均。对每种策略运行相同的评估，差异显示哪种边界策略在你拥有的语料库上胜出。

## 构建它

`code/main.py` 实现了：

- `fixed_window(text, size, overlap)` - 基线。
- `sentence_chunks(text, target)` - 简单句子打包器。
- `recursive_split(text, separators, target)` - 层次递归。
- `semantic_chunks(text, similarity_threshold)` - 基于确定性模拟嵌入的质心聚类。
- `structural_markdown(text)` - 标题感知分割器。
- `mock_embed(text, dim)` - 基于哈希的嵌入，使循环可离线运行。
- `DenseIndex` - 与阶段 19 轨道 B 混合检索课程中使用的相同形状。
- `eval_recall(strategy, corpus, queries, k)` - 比较循环。
- 一个 `main()`，在夹具语料库上运行每种策略并打印 recall@k 表格。

运行它：

```bash
python3 code/main.py
```

输出是一个小表格，每行一种策略，每列一个 k。句子在结构化夹具上表现差。结构化 Markdown 在 Markdown 夹具上获胜。递归在混合夹具上表现不错，因为递归会自适应。语义聚类在没有任何有用结构线索的散文夹具上获胜。

## 表格不会隐藏的失败模式

**孤儿句。** 句子打包产生缺少主题句的块。嵌入然后指向错误的聚类。

**符号中间切断。** 在代码或 YAML 内部使用固定窗口会将标识符切成两半。两半嵌入为噪声。

**仅标题块。** 结构化 Markdown 输出只包含 `## 标题` 的块。过滤掉它们或附加下一个块的第一段。

**语义漂移。** 当语料库主题一致时，语义聚类切割不足。一个 5000 字符的块将许多具体答案打包到一个模糊的嵌入中。将语义与硬字符上限结合使用。

**过时嵌入。** 语义聚类使用嵌入模型。如果你改变了模型，你也改变了块。将分块模型与检索模型分开固定，或一起重建索引。

## 无需运行基准测试即可选择默认策略

三个属性决定新语料库的默认分块器。

| 属性 | 值 | 默认 |
|----------|-------|---------|
| 文档类型 | 无结构的散文 | 递归分割，目标 800 |
| 文档类型 | Markdown / RFC / API 文档 | 结构化 Markdown |
| 文档类型 | 代码 | AST 感知（范围外；见阶段 19 课程 02） |
| 段落长度 | 长，单一主题 | 句子，目标 500 |
| 段落长度 | 短，混合主题 | 语义，阈值 0.6 |

有疑问时选择递归分割。它是最强的单策略基线。

## 使用它

生产模式：

- 在发布新管道之前运行评估；不要信任你的库默认使用的策略。
- 每当你更改嵌入模型或语料库组成时重新运行评估；胜出者依赖于语料库。
- 在每个块的元数据中持久化策略名称，以便后续可以归因回归。

## 投入生产

课程 69 中轨道 F 的端到端 RAG 系统使用此处选择的分块器作为其第一阶段。课程 68 中的评估框架从与本课程 `eval_recall` 相同的形状读取 recall@k。选择在你的语料库上获胜的策略并将其向前传递。

## 练习

1. 添加第六种策略：使用 `tiktoken` 的 token 窗口替代字符计数。在相同的夹具上与固定窗口比较。

2. 向散文夹具中注入 30% 的代码块。重新运行表格。解释为什么除结构化 Markdown 外的每种策略都会丢失召回率。

3. 将确定性嵌入替换为你项目真实提供商的嵌入。测量语义聚类的召回率差异。报告策略之间的差距是扩大还是缩小。

4. 为每个块添加一个 `summary` 字段：一句质心描述。将摘要附加到块体后重新运行评估。测量召回率提升。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|-----------------|------------------------|
| Recall@k | "我们拿到了正确的块吗？" | 前 k 个块中任一与黄金答案片段重叠的查询比例 |
| 块重叠（Chunk overlap） | "滑动窗口" | 在下一个块中重新包含前一个块的最后 N 个字符 |
| 结构分割器（Structural splitter） | "标题感知块" | 在 H1/H2/H3 边界处切割；标题文本是块的一部分 |
| 语义分块器（Semantic chunker） | "主题感知块" | 嵌入句子，按质心相似度聚类，在漂移处切割 |
| 质心漂移（Centroid drift） | "主题切换" | 运行均值与下一个句子之间的余弦相似度低于阈值 |
