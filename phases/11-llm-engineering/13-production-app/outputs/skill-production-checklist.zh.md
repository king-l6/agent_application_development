# 生产级 LLM 检查清单

在发布 LLM 应用时，按顺序检查此清单。每个部分都有带具体阈值的通过/失败标准。

## 1. 安全（发布阻塞）

这里的每一项必须在任何部署之前通过。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| API 密钥在环境变量中 | 代码库中零硬编码密钥 | `grep -r "sk-" --include="*.py"` 无返回 |
| 输入护栏已激活 | 提示注入模式被阻止 | 发送"忽略所有之前的指令"——返回阻止响应 |
| PII 编辑 | SSN、信用卡、邮件模式被捕获 | 发送"我的 SSN 是 123-45-6789"——PII 在 LLM 调用前被编辑 |
| 输出过滤 | 危险内容被阻止 | 模型无法返回 `DROP TABLE`、`rm -rf`、`exec()` 模式 |
| 速率限制 | 每用户请求上限已执行 | 同一用户在 10 秒内 100 个请求——最后 50+ 个被拒绝 |
| 所有端点有认证 | 无未经认证的 LLM 访问 | 不带 token 的 `curl /v1/chat` 返回 401 |
| CORS 受限 | 仅允许生产域名 | `Origin: evil.com` 请求被拒绝 |
| 最大输入 token | 超过限制的请求被拒绝 | 发送 50K token 输入——返回 413 或截断 |

## 2. 可靠性（第一周生存）

这些防止你的第一个值班事故。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 带退避的重试 | 对 5xx 重试 3 次，指数延迟 | 中途杀死 LLM 模拟——重试在日志中可见 |
| 备用模型链 | 链中有 2+ 个模型 | 主要模型不可用——响应仍从备用返回 |
| 请求超时 | 所有外部调用最长 30s | 慢速 LLM 模拟（60s）——请求在 30s 超时 |
| 优雅降级 | 缓存/RAG 失败不崩溃服务 | 停止缓存——请求仍成功（更慢，更贵） |
| 健康检查端点 | 返回依赖状态 | `GET /health` 返回 `{"status": "healthy", "cache": ..., "llm": ...}` |
| 流式传输有效 | 首个 token 在 500ms 内 | 测量的首个 token 时间，持续 < 500ms |
| 错误消息安全 | 内部错误从不泄露给用户 | 强制 500——用户看到通用错误，非堆栈追踪 |

## 3. 成本控制（第一个月经济）

这些防止 $50K 的意外账单。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 每次请求成本已追踪 | 每个请求记录 token 数 + USD 成本 | 请求日志有 `input_tokens`、`output_tokens`、`cost_usd` 字段 |
| 语义缓存已激活 | 重复模式命中率 > 20% | 1000 次测试请求后缓存统计显示命中率 |
| 缓存 TTL 已配置 | 条目过期（默认：1 小时） | 条目插入——TTL 后不再返回 |
| 每用户成本追踪 | 按 user_id 聚合成本 | 仪表板/API 显示按成本排名前 10 的用户 |
| 成本警报 | 在每日预算 80% 时告警 | 设置 $10 每日预算，发送 $8.50 的请求——警报触发 |
| 按成本模型路由 | 低复杂度查询使用更便宜模型 | 简单问题路由到 gpt-4o-mini，复杂路由到 gpt-4o |
| 最大输出 token 已设置 | 每个模板的响应上限 | max_output_tokens=512 的模板——响应绝不超限 |

**成本估算公式：**
```
月度 LLM 成本 = DAU x 每用户查询数 x 30 x (1 - 缓存命中率) x (平均输入 token x 输入价格 + 平均输出 token x 输出价格) / 1,000,000
```

**按规模的基准阈值：**

| DAU | 目标成本/请求 | 月度预算 |
|-----|-------------------|----------------|
| 1K | < $0.005 | < $750 |
| 10K | < $0.003 | < $4,500 |
| 100K | < $0.001 | < $15,000 |

## 4. 可观测性（生产调试）

你无法修复你看不到的东西。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 结构化 JSON 日志 | 每个请求产生一行 JSON 日志 | 日志包含：request_id, user_id, model, tokens, latency_ms, cost |
| 请求追踪 | 端到端追踪，含组件计时 | 单个请求显示：guardrail (5ms) + cache (2ms) + llm (3200ms) + eval (1ms) |
| 延迟追踪 | P50, P95, P99 已测量 | 1000 个请求后：P50 < 2s, P99 < 10s |
| 错误率监控 | 错误已计数和分类 | 仪表板显示：0.5% API errors, 0.1% guardrail blocks, 0.01% timeouts |
| 缓存指标 | 命中率、未命中率、条目数可见 | `GET /v1/cache/stats` 返回当前数字 |
| A/B 测试指标 | 每个变体的质量指标已记录 | 每个请求记录 prompt_template + version 用于比较 |
| 评估日志 | 每个请求记录质量信号 | 响应长度、延迟、模型、模板版本已存储用于离线分析 |

## 5. 提示管理

提示就是代码。像对待代码一样对待它们。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 版本化模板 | 每个模板有名称 + 版本字符串 | 模板更改创建新版本，旧版本保留 |
| A/B 测试支持 | 按确定性用户哈希分流 | 相同用户在实验内始终看到相同变体 |
| 回滚能力 | 在 < 1 分钟内恢复到以前版本 | 更改实验配置——流量立即切换 |
| 模板验证 | 变量在渲染前已验证 | 模板中缺少变量引发清晰错误，而非 KeyError |
| 系统提示分离 | 系统和用户消息在单独字段中 | 系统提示未拼接到用户消息中 |

## 6. 扩展就绪

发布时不需要。在 10 倍规模时需要。

| 检查 | 通过标准 | 如何验证 |
|-------|--------------|---------------|
| 异步 LLM 调用 | API 调用无线程阻塞 | 50 个并发请求——服务器 CPU 保持 < 30% |
| 连接池 | HTTP 连接重用 | 网络追踪显示与 LLM 提供商的持久连接 |
| 水平扩展 | 无状态服务器设计 | 负载均衡器后 2 个实例——所有请求成功 |
| 队列支持 | 非实时任务进入队列 | 摘要请求返回 job_id，结果可通过轮询获得 |
| 负载测试 | 100 并发用户，错误率 < 5% | `wrk` 或 `locust` 测试在目标并发下通过 |

## 新项目的实现顺序

1. **第 1 天：** API 服务器 + 提示模板 + 带重试的单 LLM 调用
2. **第 2 天：** 输入护栏 + 输出护栏 + 错误处理
3. **第 3 天：** 语义缓存 + 每次请求成本追踪
4. **第 4 天：** 流式传输 (SSE) + 健康检查端点
5. **第 5 天：** 结构化日志 + 请求追踪 + 评估日志
6. **第 2 周：** A/B 测试 + 提示版本控制 + 回滚
7. **第 3 周：** 备用模型链 + 优雅降级
8. **第 4 周：** 负载测试 + 异步优化 + 水平扩展

## 快速诊断

如果生产中出现问题，按此顺序检查：

1. **用户抱怨错误？** 检查健康端点，然后检查日志中的错误率，然后检查 LLM 提供商状态页面
2. **响应慢？** 检查 P99 延迟，然后检查缓存命中率，然后检查追踪中的 LLM 响应时间
3. **成本飙升？** 检查每次请求成本趋势，然后检查缓存命中率，然后检查按成本排名前 10 的用户，然后检查增加 token 数量的提示模板更改
4. **质量下降？** 检查是否部署了新提示版本，检查 RAG 检索准确率是否变化，检查模型提供商是否更改了默认模型版本
5. **安全事件？** 检查护栏阻止率（突然下降 = 护栏被禁用），检查请求日志中的异常模式，立即轮换 API 密钥
