# 高级 RAG 模式

基础 RAG：嵌入查询 -> 向量搜索 -> top-k -> 生成。
高级 RAG：嵌入查询 + BM25 -> 融合排名 -> 重排序 -> top-k -> 生成。

```
query -> [vector search (top-50)] -+-> RRF fusion -> reranker (top-5) -> prompt -> LLM
                                   |
query -> [BM25 search (top-50)]  --+
```

## 何时从基础 RAG 升级

- 检索质量低于70% Recall@5
- 用户报告错误或不相关的答案
- 语料库增长超过100K 块
- 查询使用与文档不同的词汇
- 多跳问题持续失败

## 实施检查清单

1. 在向量索引旁边添加 BM25 索引
2. 并行运行两种搜索（各 top-50）
3. 使用倒数排名融合合并（k=60）
4. 用交叉编码器重排序 top 候选
5. 取 top-5 用于最终提示
6. 在测试集上添加忠实度评估

## 技术选择指南

- **混合搜索**：始终在生产中使用。查询时无额外成本。
- **重排序**：当 Recall@50 良好但 Recall@5 较差时使用。增加50-200ms 延迟。
- **HyDE**：当查询模糊或使用与文档不同的词汇时使用。增加一次 LLM 调用。
- **父子块**：当小块缺乏上下文但大块稀释相关性时使用。
- **元数据过滤**：当语料库有明确类别（日期、来源类型、部门）时使用。
- **查询分解**：对需要来自多个文档信息的多跳问题使用。

## 常见错误

- 用不同的块集运行 BM25 和向量搜索（它们必须搜索相同的语料库）
- 重排序使用的候选池太小（top-10 太小；使用 top-50）
- 对每个查询添加 HyDE（仅当词汇不匹配是瓶颈时才有帮助）
- 不评估变更（在每次技术前后测量 Recall@k）
- 在测量失败位置之前过度设计流水线

## 评估工作流程

1. 创建50+ 个具有已知答案块的测试问题
2. 测量每种检索方法的 Recall@5 和 Recall@10
3. 对于检索成功的查询，测量生成答案的忠实度
4. 随着语料库增长，每周跟踪指标
5. 在添加更多技术之前调查个别失败
