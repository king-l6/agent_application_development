# LLM 成本优化模式

在构建一个需要控制成本的 LLM 应用时，应用此决策框架。

## 何时优化

**在以下情况下立即优化：**
- 每月 LLM 支出超过 $500 或基础设施预算的 10%
- 消费者产品的每次查询成本高于 $0.01
- 你的系统提示超过 1,000 token 且随每个请求发送
- 超过 30% 的查询是重复或近似重复的
- 你正从 100 个日活用户扩展到 10,000+ 个

**在以下情况下暂不优化：**
- 你只有不到 100 个 DAU，且仍在验证产品市场匹配
- 每月支出低于 $100 且增长缓慢
- 你仍在迭代提示设计（缓存会锁定提示）

## 缓存策略选择

### 精确缓存

**在以下情况下使用：** temperature=0，相同提示重复出现，需要确定性输出。

```python
key = sha256(json.dumps({"model": m, "messages": msgs, "temp": 0}))
```

- 实现时间：30 分钟
- 命中率：大多数应用 10-25%，FAQ 机器人 40-60%
- 延迟：<1ms（字典查找）
- 风险：如果底层数据发生变化，缓存的响应会过时

**在以下情况下跳过：** temperature > 0，每个查询都唯一，需要实时数据。

### 语义缓存

**在以下情况下使用：** 用户用不同词语问相同问题，FAQ 密集型产品，客户支持。

- 实现时间：2-4 小时（嵌入 + 相似度 + 存储）
- 命中率：在精确缓存基础上增加 15-35%
- 延迟：10-50ms（嵌入 + ANN 搜索）
- 风险：误报（为相似但不同的问题返回错误的缓存答案）

**阈值指南：**
- 0.98+：非常保守，几乎没有误报，但命中率较低
- 0.95：对事实性问答是良好的平衡
- 0.90：激进，命中率更高但存在错误答案的风险
- 0.85：仅适用于低风险应用（建议、自动补全）

**在以下情况下跳过：** 每个查询都有独特的上下文（代码生成），响应必须反映最新数据，查询空间无界。

### 提供商提示缓存

**在以下情况下使用：** 系统提示 > 1,024 token（OpenAI）或特定模型的最小值，相同前缀重复发送。

| 提供商 | 操作 | 节省 |
|----------|--------|---------|
| Anthropic | 在系统消息中添加 `cache_control: {"type": "ephemeral"}` | 缓存前缀 90%（在 25% 写入溢价后） |
| OpenAI | 什么都不做（自动） | 缓存前缀 50% |
| Google | 使用带显式 TTL 的 Context Caching API | 缓存上下文约 75% |

**在以下情况下跳过：** 系统提示每次请求都变化，提示低于最小长度。

## 模型路由规则

### 基于关键字（简单、快速）

```
简单：  <= 5 词 或 匹配 FAQ 关键字 -> gpt-4o-mini ($0.15/$0.60)
中等：  一般查询、摘要        -> claude-sonnet ($3/$15)
复杂：  "分析"、"比较"、"调试"     -> gpt-4o ($2.50/$10)
```

- 实现时间：1 小时
- 准确率：70-80%
- 节省：模型成本 40-60%

### 基于嵌入（更准确）

为每个类别嵌入 50-100 个带标签的查询。通过最近邻分类新查询。

- 实现时间：4-8 小时
- 准确率：85-92%
- 节省：模型成本 50-70%
- 额外成本：用于分类嵌入的 ~$0.02/100 万 token（可忽略）

### 基于 ML（生产级）

在历史查询/模型对上训练一个小型分类器（逻辑回归或小型 BERT）。

- 实现时间：1-2 周
- 准确率：90-95%
- 节省：模型成本 60-75%
- 需要：来自生产流量的带标签训练数据

## 速率限制配置

### 按层级的令牌桶参数

| 层级 | 桶大小 | 补充速率 | 最大 RPM | 每日上限 |
|------|-------------|-------------|---------|-----------|
| 免费 | 50K tokens | 500/秒 | 10 | 50K |
| Pro | 500K tokens | 5K/秒 | 60 | 500K |
| 企业 | 5M tokens | 50K/秒 | 300 | 5M |

### 实现检查清单

1. 将桶存储在 Redis 中（而非内存中）用于多实例应用
2. 使用原子操作（MULTI/EXEC）防止竞态条件
3. 在拒绝响应中返回 `Retry-After` 头部
4. 将拒绝的请求作为指标追踪（>5% 拒绝 = 层级限制太严格）
5. 实现优雅降级：先拒绝昂贵的模型请求，保留廉价模型访问

## 预算控制

### 三阈值断路器

| 阈值 | 操作 | 可逆 |
|-----------|--------|------------|
| 月度预算的 70% | 记录警告，通过 Slack/PagerDuty 提醒团队 | 是（自动） |
| 月度预算的 85% | 将所有流量路由到最便宜的模型 | 是（自动，下一个计费周期） |
| 月度预算的 95% | 仅提供缓存的响应，拒绝新的 LLM 调用 | 是（手动重置或下一周期） |

### 每用户成本追踪

追踪每个用户的累积成本。标记超过中位数 10 倍的用户。常见原因：
- 合理的重度用户（升级他们的层级）
- 提示注入循环（发送自动化请求的机器人）
- 低效集成（客户端在每次错误时重试）

## 成本追踪字段

记录每次 API 调用时包含这些字段：

```json
{
  "timestamp": "2026-04-02T10:30:00Z",
  "model": "gpt-4o",
  "input_tokens": 1523,
  "output_tokens": 487,
  "cached_input_tokens": 1024,
  "latency_ms": 1847,
  "cost_usd": 0.006142,
  "user_id": "user_abc123",
  "cache_status": "partial_hit",
  "request_category": "customer_support",
  "complexity_class": "medium",
  "routed_from": "gpt-4o"
}
```

### 需要在仪表板上展示的关键指标

- **每次查询成本**（P50, P95, P99）—— 按模型、按功能、按用户层级
- **缓存命中率** —— 精确 vs 语义，随时间变化趋势
- **模型分布** —— 每个模型的流量百分比，每个模型的成本
- **预算消耗率** —— 当前支出 vs 按当前速率预测的月度支出
- **拒绝率** —— 被速率限制的请求百分比，按层级

## 常见错误

| 错误 | 为什么有害 | 修复 |
|---------|-------------|-----|
| 在 temperature > 0 时缓存 | 非确定性输出，陈旧缓存失去多样性 | 仅缓存 temp=0 的调用，或接受缓存响应失去随机性 |
| 语义缓存阈值太低 | 为表面相似的查询返回错误答案 | 从 0.95 开始，仅在测量误报率后降低 |
| 没有缓存失效 | 当底层数据变化时响应变陈旧 | 设置 TTL（动态数据 1 小时，静态数据 24 小时），在数据更新时失效 |
| 将所有流量路由到最便宜的模型 | 质量下降，用户注意到 | 按复杂度路由，按层级衡量质量，设置最低质量标准 |
| 没有每用户限制 | 一个滥用用户烧光整个预算 | 始终实现每用户配额，即使设置得宽松 |
| 忽略输出 token | 输出每 token 比输入贵 2-5 倍 | 适当设置 max_tokens，使用停止序列，压缩输出 |
| 在提示稳定之前缓存 | 缓存中填充了旧提示的响应 | 仅在提示定稿后启用缓存，提示变更时清除缓存 |
