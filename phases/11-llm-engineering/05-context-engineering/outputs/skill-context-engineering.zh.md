# 上下文工程

在构建大语言模型应用时，应用此框架设计上下文组装流水线。

## 核心原则

1. **上下文是稀缺的。** 128K 窗口听起来很大，但很快就会被填满。显式预算每个组件。
2. **注意力是不均匀的。** 模型对开头和结尾的注意力更高。把关键信息放在那里。中间是死区。
3. **动态优于静态。** 不同的查询需要不同的上下文。每次查询组装，而不是在启动时组装一次。
4. **少即是多。** 一个精心策划的 10K 上下文优于随意倾倒的 100K 上下文。信噪比比总信息量更重要。
5. **衡量一切。** 你不能优化你无法衡量的东西。每次请求统计每个组件的词元数。

## 上下文预算指南

| 组件 | 典型范围 | 优先级 | 压缩策略 |
|------|---------|--------|---------|
| 系统提示词 | 200-1,000 词元 | 固定，高 | 写紧致，消除冗余 |
| 工具定义 | 500-3,000 词元 | 动态，中 | 按查询意图修剪 |
| 检索上下文 | 1,000-5,000 词元 | 动态，高 | 重排序 + 阈值 + 去重 |
| 对话历史 | 500-5,000 词元 | 动态，中 | 总结旧轮次 |
| 少样本示例 | 500-2,000 词元 | 动态，高 | 按任务相似性选择 |
| 用户查询 | 50-500 词元 | 固定，最高 | 不适用 |
| 生成预留 | 2,000-8,000 词元 | 固定 | 根据预期输出长度调整 |

## 何时使用每种记忆类型

**短期（对话历史）：** 当前会话。通过总结管理。压缩超过 5-10 轮对话的旧轮次。逐字保留最近 3-4 轮。

**长期（事实数据库）：** 跨会话持久化的偏好和项目事实。在会话开始时检索。示例："用户偏好 Python"、"项目使用 PostgreSQL"、"团队遵循主干开发"。存储在 CLAUDE.md、数据库或结构化记忆系统中。

**情景（过去交互）：** 与当前任务相关的特定过去对话。作为嵌入存储，通过相似性检索。"上周我们调试了类似的认证问题"就是情景记忆。

## 工具选择策略

不要在每次请求中包含所有工具。这会浪费词元并混淆模型。

1. 分类查询意图（代码、邮件、日历、研究、数据）
2. 将意图映射到工具类别
3. 只包含匹配的工具
4. 如果意图模糊，包含前 2 个类别的工具
5. 始终包含一个"通用"工具（如网络搜索）作为后备

预期节省：对意图清晰的查询，节省 60-80% 的工具定义词元。

## 检索最佳实践

- **检索后重排序。** 向量相似性是一个粗略的过滤器。重排序器（交叉编码器或基于 LLM 的）显著提高精度。
- **设置相关性阈值。** 不要包含余弦相似度低于 0.3 的块。它们增加噪声。
- **去重。** 如果两个块共享 80%+ 的内容，只保留分数更高的那个。
- **应用中间丢失排序。** 将最相关的块放在最前和最后。
- **限制检索总词元数。** 3-5 个高度相关的块胜过 15 个平庸的块。

## 历史管理

- 逐字保留最近 3-4 轮（模型需要近期上下文）
- 将更旧的轮次总结为摘要（"我们讨论过 X，决定 Y，卡在 Z"）
- 丢弃不添加信息的系统生成轮次（没有面向用户内容的工具调用）
- 当历史超过可用预算的 30% 时触发压缩

## 红旗信号

- 系统提示词超过 2,000 词元：可能包含应该是动态的信息
- 每次请求都包含所有工具：实现基于意图的选择
- 检索没有相关性过滤：你在向窗口倾倒噪声
- 历史无限制增长：没有实现总结
- 没有生成预留：模型截断其回复
- 相同信息出现在 3 个地方（系统提示词、检索文档、历史）：去重
- 上下文利用率超过 60%：你留给模型"思考"的空间太少
