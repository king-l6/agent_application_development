# 微调决策指南

在微调之前，按顺序尝试这些：

```
1. 提示工程（分钟，$0）
2. 提示中的 Few-shot 示例（分钟，$0）
3. 用于知识检索的 RAG（天，$10-100/月）
4. 使用 LoRA/QLoRA 微调（天，$5-50/次实验）
5. 全量微调（周，$100-10,000/次运行）
```

只有在前一步明显不够时才进入下一步。

## 何时微调

- 模型需要提示工程无法实现的一致输出风格或格式
- 你正在蒸馏一个更大的模型（从8B 模型获得 GPT-4 质量）
- 延迟很重要，few-shot 示例增加了太多 token
- 你需要模型可靠地遵循复杂的推理模式
- 你有1,000+ 个期望输入-输出行为的高质量示例

## 何时不微调

- 模型已经用正确的提示做到了你想要的
- 你需要模型了解事实（使用 RAG 代替）
- 你少于500个训练示例（很可能过拟合）
- 任务频繁变化（重新训练成本高）
- 你需要审计哪些数据影响了特定输出（微调是一个黑箱）

## 方法选择

| GPU 显存 | 7B 模型 | 13B 模型 | 70B 模型 |
|----------|----------|-----------|-----------|
| 16GB (T4) | QLoRA | 不可行 | 不可行 |
| 24GB (3090/4090) | QLoRA 或 LoRA | QLoRA | 不可行 |
| 40GB (A100) | LoRA 或 Full | QLoRA 或 LoRA | QLoRA |
| 80GB (A100/H100) | Full | LoRA 或 Full | QLoRA 或 LoRA |

## LoRA 配置检查清单

1. 从 r=16, alpha=32 开始（大多数任务的安全默认值）
2. 首先目标 q_proj 和 v_proj（最小可行 LoRA）
3. QLoRA 使用学习率 2e-4，LoRA fp16 使用 5e-5
4. 设置 lora_dropout=0.05
5. 训练 1-3 轮（更多可能导致过拟合）
6. 每100步在保留集上评估
7. 保存检查点，根据评估损失选择最佳的一个

## 常见错误

- 训练太多轮次（在小数据集上第2-3轮后过拟合）
- 使用与全量微调相同的学习率（LoRA 需要更高的学习率）
- 忘记设置填充 token（Llama 模型会导致 NaN 损失）
- 没有冻结基础模型（违背了 LoRA 的目的）
- 只在训练数据上评估（始终保留10-20%用于评估）
- 跳过提示工程基线（微调一个提示工程已经解决的问题）

## 质量验证

训练后，在200+ 个保留示例上比较：
1. 带最佳提示的基础模型（基线）
2. 带 LoRA 适配器的基础模型（你的微调模型）
3. 带相同提示的 GPT-4 或 Claude（天花板）

如果 LoRA 模型没有击败有提示的基线，那么需要改进的是你的训练数据或配置，而不是更多的算力。

## 适配器管理

- 为多任务服务保持适配器分开（每次请求切换适配器）
- 为单任务部署将适配器合并到基础权重中
- 在 Hugging Face Hub 上存储适配器（10-100MB，易于版本控制和共享）
- 在部署前测试合并模型输出是否与未合并输出匹配
- 使用 TIES-Merging 或 DARE 将多个适配器合并为一个

## 调试训练

如果损失没有下降：
1. 检查学习率（对 LoRA 来说太低，尝试 2e-4）
2. 验证 LoRA 层是否实际接收到了梯度
3. 确认基础模型权重已冻结
4. 检查数据格式（分词器必须匹配模型预期的格式）

如果损失下降但评估质量差：
1. 训练数据质量问题（垃圾进，垃圾出）
2. 过拟合（减少轮次，增加 dropout，添加更多数据）
3. 目标模块错误（为复杂任务添加 MLP 层）
4. 秩太低（尝试 r=32 或 r=64）
