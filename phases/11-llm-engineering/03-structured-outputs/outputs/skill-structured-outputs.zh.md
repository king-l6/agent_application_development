# 结构化输出策略

在构建需要结构化数据的大语言模型应用时，应用此决策框架。

## 何时使用每种方法

**基于提示（"返回 JSON"）：** 仅限原型设计。可接受用于内部工具，偶尔解析失败是可容忍的。添加 try/except 和重试。绝不要在生产流水线中使用。

**JSON 模式（API 标志）：** 你需要保证有效的 JSON，但 Schema 简单或灵活。当你在应用端验证数据结构时使用。可用：OpenAI、Anthropic（通过工具使用）、Google。

**Schema 模式（约束解码）：** 生产系统，每个输出必须匹配特定的 Schema。零解析失败。零 Schema 违规。默认情况下对任何生产提取或分类任务使用此方法。可用：OpenAI 结构化输出、Outlines、Guidance。

**函数调用 / 工具使用：** 模型需要选择调用哪个函数，而不仅仅是填充参数。你有多个 Schema，模型选择合适的那个。在与现有工具/函数基础设施集成时也使用此方法。

**Instructor 库：** 你想要带自动重试的 Pydantic 验证，支持任何提供者。Python 项目的最佳开发者体验。包装 OpenAI、Anthropic、Google 和开源模型。

## 提供者特定指导

**OpenAI：** 使用 `response_format` 和 `json_schema` 类型。约束解码是内置的。Pydantic 模型直接工作。最可靠的结构化输出实现。

**Anthropic：** 使用工具使用实现结构化输出。定义一个带有期望 Schema 的单一工具。模型返回匹配 Schema 的工具调用参数。可靠但需要工具使用 API 模式。

**开源模型（vLLM、Ollama）：** 使用 Outlines 或 Guidance 进行约束解码。这些库将 JSON Schema 编译为在生成过程中屏蔽无效词元的有限状态机。需要本地运行推理。

## Schema 设计指南

1. 尽可能保持 Schema 扁平。超过 2 层的嵌套对象会增加提取错误。
2. 对分类字段使用枚举。不要依赖模型创造正确的字符串。
3. 将模糊字段设为必需并显式支持 null，而非设为可选。迫使模型做出决策。
4. 为 Schema 属性添加描述。模型会将这些描述作为指令阅读。
5. 除非必要，避免联合类型（oneOf/anyOf）。它们增加解码复杂度。
6. 对数字设置最小值/最大值。捕获幻觉出来的极端值。
7. 对数组使用 minItems/maxItems，防止空或无界输出。

## 常见失败模式及修复

- **模型将 JSON 包裹在 Markdown 围栏中**：从基于提示的模式切换到 JSON 模式或 Schema 模式
- **Schema 有效但事实错误**：在提取后添加 LLM 作为评判的验证步骤
- **枚举值不一致**：切换到约束解码或添加后处理归一化
- **缺少可选字段**：将其设为必需，或在应用代码中添加默认值
- **提取速度非常慢**：约束解码增加 5-15% 延迟，如果对延迟敏感，降低 Schema 复杂度
- **包含多种项的大型数组**：分块输入并逐块提取，然后合并结果

## 可靠性阶梯

| 方法 | 解析成功率 | Schema 匹配率 | 设置工作量 |
|------|---------|-------------|----------|
| 基于提示 | ~90% | ~80% | 1 分钟 |
| JSON 模式 | 100% | ~90% | 5 分钟 |
| Schema 模式 | 100% | ~99% | 15 分钟 |
| 约束解码 | 100% | 100% | 30 分钟 |
| Instructor + 重试 | 100% | ~99.5% | 10 分钟 |
