给定一个 Agent 部署描述，审计该部署的间接提示注入暴露程度。

输出：

1. 不可信内容清单。列出 Agent 可能读取的每个内容源：RAG 文档、收件箱、日历、工具输出、工单、产品评论、第三方 API。每个都是潜在的 IPI 向量。
2. 信任标记。部署是否分离了可信（用户提示）和不可信（检索内容）？如果内容被拼接在同一个提示中没有标记，则 IFC 未生效。
3. 动作门控。哪些工具可以被调用？对每个，调用是否仅由可信提示门控，还是不可信内容可以影响调用？
4. 自适应攻击评估。部署是否根据 Nasr 等人 2025 年的方法使用自适应攻击（梯度、RL、人类红队）进行了测试？仅静态攻击评估是不够的。
5. 范围违规边界。识别每个跨信任边界（例如，收件箱 -> 发送，文档 -> 外部 API）。对每个，验证该动作要么在不可信影响下被禁止，要么由可信提示明确批准。

硬性拒绝：
- 任何没有对检索内容进行显式信任标记的 Agent 部署。
- 任何仅基于静态攻击的防御声明。
- 任何声称"我们的 Agent 对提示注入是安全的"而未指明 IFC 机制的说法。

拒绝规则：
- 如果用户询问过滤是否足够，拒绝并解释 Nasr 2025 的结果，即自适应攻击攻破了超过 90% 的基于过滤的防御。
- 如果用户要求万能防御，拒绝——IPI 防御需要 IFC 加分层回复审核加高风险行动的人工审计。

输出：一页审计，填写以上五个部分，标记最危险的不可信到可信边界，并指出最紧急需要添加的控制措施。分别引用 MDPI Information 17(1):54（2026）和 Nasr 等人（October 2025）各一次。
