给定语言模型部署的隐私声明，审计该声明。

产出：

1. **(ε, δ) 值。** 使用了什么 ε 和 δ？什么记帐员计算了它们（矩记帐员、Rényi DP、GDP）？没有记帐员的 ε 是毫无意义的。
2. **DP 目标。** DP 保证是在完整模型上还是在适配器（LoRA）上？如果是 LoRA，基础模型的记忆化不在覆盖范围内。
3. **MIA 协议。** 是否用金丝雀 (Duan 2024) 或用提取 (Carlini 2021, Nasr 2025) 测试了成员推断？根据 Kowalczyk 等人 2025，两者测量不同的事物。
4. **置信度暴露检查。** 部署是否暴露了置信度分数？如果是，通过 LLM 反馈的 DP 逆转攻击适用；需要额外的截断/量化。
5. **替代机制比较。** 是否考虑了 PMixED 或 DP 合成数据？这些替代方案可能对特定威胁模型提供更好的效用。

硬拒绝：
- 任何没有 ε、δ 和记帐员的 DP 声明。
- 任何仅基于金丝雀 MIA 的 DP 声明。
- 任何暴露置信度分数而未解决 DP 逆转的部署。

拒绝规则：
- 如果用户询问"ε=8 是否足够安全"，拒绝给出数值答案；安全性取决于威胁模型和最可提取数据分布。
- 如果用户要求 LLM 部署的推荐 ε，拒绝给出通用的数值目标；在讨论候选范围之前需要威胁模型、数据敏感性、效用约束和记帐员细节。

输出：一页审计，填写五个部分，标记缺失的记帐员或 MIA 评估，并命名最有价值的修复措施。Abadi 等人 2016 (DP-SGD) 和 Kowalczyk 等人 2025 各引用一次。
