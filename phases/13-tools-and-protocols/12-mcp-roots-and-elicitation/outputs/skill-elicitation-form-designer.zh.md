给定一个可能需要调用中途用户输入的工具，设计引导模式和消息。

生成内容：

1. 触发条件。说明导致工具调用 `elicitation/create` 的确切输入或歧义情况。
2. 消息模板。宿主向用户显示的一句话。清晰、具体、无行话。
3. 模式。包含类型化属性的扁平 JSON Schema，以及 `enum` 列表（用于消歧）或 `boolean`（用于确认）。不要嵌套。
4. 分支处理。将 `accept` / `decline` / `cancel` 映射到工具行为。
5. 速率限制规则。限制每次工具调用的引导次数；绝不要在循环内引导。

硬性拒绝：
- 任何嵌套对象的模式。引导 v1 是扁平的。
- 任何用于填充 LLM 本可以以文本形式询问的缺失参数的引导。
- 任何高频引导（每次工具调用超过一次）。

拒绝规则：
- 如果工具是只读且低风险的，拒绝使用引导，直接返回结果。
- 如果工具是破坏性的且宿主支持 `destructiveHint` 注解，建议使用注解并让客户端本地处理确认。
- 如果需要 OAuth 登录，推荐 URL 模式引导并标记 SEP-1036 漂移风险。

输出：一页设计文档，包含触发条件、消息模板、模式、分支处理、速率限制规则，以及关于表单模式还是 URL 模式更合适的说明。
