给定一个要运行的 MCP 服务器配置，生成一个客户端测试工具，用于启动每个服务器、与每个握手、将它们的工具列表合并为一个命名空间，并将每个调用路由到所属服务器。

生成内容：

1. 服务器配置解析器。将 `名称 -> {命令, 参数, 环境变量}` 建立映射。验证命令存在于路径中。
2. 启动计划。使用带有 stdin/stdout/stderr 管道的 subprocess.Popen，设置 `bufsize=1`、文本模式。每个服务器一个后台读取线程。
3. 握手管线。对于每个会话：发送 `initialize`，等待响应，持久化能力，发送 `notifications/initialized`。
4. 命名空间合并。选择冲突策略：`冲突时添加前缀`（默认）、`冲突时拒绝` 或 `静默覆盖`（禁止）。启动时打印合并后的工具列表。
5. 路由函数。`client.call(规范名称, 参数)` 查找所属会话并向其 stdin 写入 `tools/call` 消息。通过待处理请求表中的 future 等待匹配 ID 的响应。

硬性拒绝：
- 任何不在各自独立进程中启动每个服务器的测试工具。进程内多路复用破坏了隔离模型。
- 任何以 `静默覆盖` 作为默认冲突策略的测试工具。存在安全风险。
- 任何在 stdout 读取上阻塞主线程的测试工具。通知将停滞。

拒绝规则：
- 如果服务器的命令不受信任（不在锁定白名单中），拒绝启动并将问题引导至阶段 13 · 15 进行安全检查。
- 如果用户配置了超过 10 个服务器且无正当理由，发出警告并建议使用网关（阶段 13 · 17）。
- 如果被要求在此处理 OAuth，拒绝并将问题引导至阶段 13 · 16。

输出：一个完整的客户端测试工具 Python 文件（约 150 行），包含 Session、合并逻辑、路由和遍历每个已配置服务器的主循环。最后用一行总结冲突策略和合并后的工具数量。
