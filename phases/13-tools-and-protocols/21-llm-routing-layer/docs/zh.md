# LLM 路由层 — LiteLLM、OpenRouter、Portkey

> 供应商锁定成本高昂。不同的工具调用工作负载适合不同的模型。路由网关提供了一个统一的 API 表面、重试、故障转移、成本追踪和安全护栏。2026 年三种主流架构：LiteLLM（开源自托管）、OpenRouter（托管 SaaS）、Portkey（生产级，2026 年 3 月开源）。本课程说明决策标准并构建一个标准库路由网关。

**类型：** 学习
**语言：** Python（标准库，路由 + 故障转移 + 成本追踪器）
**前置知识：** 阶段 13 · 02（函数调用），阶段 13 · 17（网关）
**时间：** 约 45 分钟

## 学习目标

- 区分自托管、托管和生产级路由选项。
- 实现一个故障转移链，按定义的优先级顺序在供应商故障时重试。
- 跨供应商追踪每次请求的成本和令牌使用情况。
- 根据给定的生产约束决策选择 LiteLLM、OpenRouter 或 Portkey。

## 问题

需要供应商路由的场景：

1. **成本。** Claude Sonnet 的成本是 Haiku 的 3 倍。对于分类任务，Haiku 就足够了；对于综合任务，Sonnet 值得。按请求路由。

2. **故障转移。** OpenAI 出现故障。每个请求都失败。你希望自动回退到 Anthropic，而无需重新部署。

3. **延迟。** 实时聊天 UI 需要快速首次令牌时间。批量摘要器不需要。按延迟 SLA 路由。

4. **合规。** 欧盟用户必须留在欧盟区域。按区域路由。

5. **实验。** 对同一工作负载进行两种模型的 A/B 测试。按测试桶路由。

为每个集成手动编码所有这些是重复性的工作。路由网关提供一个 OpenAI 兼容的 API 并处理其余部分。

## 概念

### OpenAI 兼容的代理形状

每个人都使用 OpenAI 形状。路由网关暴露 `/v1/chat/completions`，接受 OpenAI 模式，并在内部代理到 Anthropic / Gemini / Cohere / Ollama / 任何其他服务。客户端无需关心。

### 模型别名

你的代码使用 `our_smart_model` 而非 `claude-3-5-sonnet-20251022`。网关将别名映射到真实模型。当 Anthropic 发布 Claude 4 时，你在服务端更改别名；你的代码无需改动。

### 故障转移链

```
primary: openai/gpt-4o
on 5xx: anthropic/claude-3-5-sonnet
on 5xx: google/gemini-1.5-pro
on 5xx: refuse
```

网关在配置中定义此链。重试计入预算，这样故障转移级联不会使成本爆炸。

### 语义缓存

相同或几乎相同的提示命中的是缓存而不是供应商。在重复的智能体循环上，节省可达 30% 到 60%。键基于嵌入；几乎相同的提示共享一个缓存槽。

### 安全护栏

网关级别：

- **PII 重写。** 在发送提示之前基于正则表达式或 ML 的过滤。
- **策略违规。** 拒绝包含禁止内容的提示。
- **输出过滤器。** 清理完成内容以防泄露。

Portkey 和 Kong 都提供了自带有主见的安全护栏。LiteLLM 将其保持为可选项。

### 每密钥速率限制

一个 API 密钥 = 一个团队。每密钥预算防止一个团队消耗共享配额。大多数网关支持此功能。

### 自托管与托管的权衡

| 因素 | LiteLLM（自托管） | OpenRouter（托管） | Portkey（生产级） |
|--------|-------------------|-------------------|-------------------|
| 代码 | 开源，Python | 托管 SaaS | 开源（2026 年 3 月）+ 托管 |
| 设置 | 部署代理 | 注册 | 任选 |
| 供应商 | 100+ | 300+ | 100+ |
| 计费 | 你自己的密钥 | OpenRouter 积分 | 你自己的密钥 |
| 可观测性 | OpenTelemetry | 仪表板 | 完整 OTel + PII 重写 |
| 最适合 | 需要完全控制的团队 | 快速原型开发 | 需要合规的生产环境 |

当你有 SRE 团队并希望数据主权时，LiteLLM 胜出。当你希望单一订阅且无基础设施时，OpenRouter 胜出。当你需要开箱即用的安全护栏和合规性时，Portkey 胜出。

### 成本追踪

每个请求携带 `provider`、`model`、`input_tokens`、`output_tokens`。乘以每模型每令牌价格（从网关维护的定价表中获取）。按用户/团队/项目聚合。

### MCP 加路由

网关可以路由 LLM 调用和 MCP 采样请求。当采样请求的 modelPreferences 偏好特定模型时，网关转换为正确的后端。这是阶段 13 · 17（MCP 网关）与本课程的路由网关有时合并为一个服务的地方。

### 路由策略

- **静态优先级。** 列表中的第一个；出错时回退。
- **负载均衡。** 轮询或加权。
- **成本感知。** 选择满足延迟/质量的最便宜模型。
- **延迟感知。** 选择过去 N 分钟内最快的模型。
- **任务感知。** 提示分类器将编程路由到一个模型，将摘要路由到另一个模型。

## 使用

`code/main.py` 在约 150 行代码中实现了路由网关：接受 OpenAI 形状的请求，转换为每个供应商的存根，运行优先级故障转移链，追踪每次请求的成本，并在输入上应用 PII 重写检查。用三种场景运行：正常请求、主供应商故障触发故障转移、PII 泄露被重写捕获。

需要关注的内容：

- `ROUTES` 字典：别名 -> 按优先级排序的具体供应商列表。
- 故障转移循环在 5xx 上重试。
- 成本追踪器将令牌使用量乘以每模型费率。
- PII 重写器在转发前清除 SSN 形状的模式。

## 练习

1. 运行 `code/main.py`。触发故障场景；确认故障转移落在第二个供应商上，成本正确归属。

2. 添加语义缓存：提示的 SHA256 作为查找键；缓存命中立即返回。测量重复调用的成本节省。

3. 添加提示分类器，将"代码 ..."提示路由到偏向智能的别名，"摘要 ..."提示路由到偏向速度的别名。

4. 设计团队预算：每个团队有月度支出上限；一旦达到上限，网关拒绝请求。选择强制粒度（每请求或窗口化）。

5. 并排阅读 LiteLLM、OpenRouter 和 Portkey 文档。指出每个提供的但其他两个没有的功能。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|----------|----------|
| 路由网关（Routing gateway） | "LLM 代理" | 众多供应商前面的统一 API 层 |
| OpenAI 兼容（OpenAI-compatible） | "说 OpenAI 模式的语言" | 接受 `/v1/chat/completions` 形状，转换为任何后端 |
| 模型别名（Model alias） | "our_smart_model" | 代码中的名称，网关将其映射到具体模型 |
| 故障转移链（Fallback chain） | "重试列表" | 失败时尝试的有序供应商列表 |
| 语义缓存（Semantic caching） | "提示嵌入缓存" | 键为提示的嵌入；几乎重复共享缓存命中 |
| 安全护栏（Guardrails） | "输入/输出过滤器" | 重写 PII，拒绝策略违规 |
| 每密钥速率限制（Per-key rate limit） | "团队预算" | 范围限定到 API 密钥的配额 |
| 成本追踪（Cost tracking） | "每请求支出" | 聚合令牌使用量 x 每模型价格 |
| LiteLLM | "开源代理" | 可自托管的 OSS 路由网关 |
| OpenRouter | "托管 SaaS" | 基于积分的计费托管网关 |
| Portkey | "生产级选项" | 开源 + 托管，内置安全护栏 |
