# 工具接口——为什么 Agent 需要结构化 I/O

> 语言模型产生 token。程序执行动作。这两者之间的鸿沟就是工具接口：一份契约，让模型可以请求一个动作，让宿主执行它。2026 年的每一个技术栈——OpenAI、Anthropic 和 Gemini 上的函数调用；MCP 的 `tools/call`；A2A 的任务片段——都是同一个四步循环的不同编码。这节课命名了这个循环，并展示了运行它所需的最小机制。

**类型：** 学习
**语言：** Python（标准库，无需 LLM）
**前置知识：** 第 11 阶段（LLM 补全 API）
**时间：** 约 45 分钟

## 学习目标

- 解释为什么一个只能生成文本的 LLM 无法自行对现实世界执行动作。
- 绘制四步工具调用循环（描述 -> 决策 -> 执行 -> 观察）并说明每一步由谁负责。
- 将工具描述写作三个部分：名称、JSON Schema 输入和确定性执行器函数。
- 区分纯工具和副作用工具，并说明这种区分对安全性的重要性。

## 问题

LLM 会为下一个 token 生成一个概率分布。这就是它全部的输出能力。如果你问一个聊天模型"班加罗尔现在天气如何"，它可以写出一个看似合理的句子，但它无法实际调用天气 API。那个句子可能只是巧合正确，或者已经是三天前的信息。

弥合这个差距就是工具接口的目的。宿主程序——你的 agent 运行时、Claude Desktop、ChatGPT、Cursor 或一个自定义脚本——向模型公布一个可调用工具的列表。当模型决定需要执行某个动作时，它会发出一个结构化的载荷，指明要调用的工具及其参数。宿主解析该载荷，真正执行工具，并将结果反馈给模型。这个循环持续进行，直到模型决定不再需要更多调用为止。

这份契约的第一个版本于 2023 年 6 月作为 OpenAI 的 "functions" 参数发布。Anthropic 在 Claude 2.1 中紧随其后推出了 `tool_use` 块。Gemini 在几个月后添加了 `functionDeclarations`。现在每个提供商都暴露了相同的形态：输入一个 JSON Schema 类型化的工具列表，输出一个 JSON 载荷的工具调用。模型上下文协议（2024 年 11 月）将这份契约泛化，使得一个工具注册表可以为所有模型服务。A2A（2026 年 4 月，v1.0）则将此同一原语扩展到了 agent 对 agent 的委派。

四步循环是所有这一切背后的不变模式。第 13 阶段的其余内容都是对它展开阐述。

## 概念

### 第一步：描述

宿主为每个工具声明三个字段。

- **名称。** 一个稳定的、机器可读的标识符。`get_weather`，而不是"天气那个东西"。
- **描述。** 一段自然语言的简要说明。"当用户询问某个城市的当前天气状况时使用。不要用于历史数据。"
- **输入模式。** 一个 JSON Schema 对象（draft 2020-12），描述该工具的参数。

模型接收这个列表。现代提供商会使用特定于提供商的模板将这些声明序列化到系统提示中，因此你作为调用者只需要处理结构化形式。

### 第二步：决策

根据用户的消息和可用的工具，模型从三种行为中选择一种。

1. **直接回答**文本。不调用工具。
2. **调用一个或多个工具。** 发出结构化的调用对象。在 `parallel_tool_calls: true`（OpenAI 和 Gemini 默认开启，Anthropic 可选开启）的情况下，模型可以在一次轮次中发出多个调用。
3. **拒绝。** 严格模式的结构化输出可以产生一个类型化的 `refusal` 块，而不是一个调用。

工具调用载荷有三个稳定的字段：调用 `id`、工具 `name` 和一个 JSON `arguments` 对象。`id` 的存在是为了让宿主能够将后续的结果与特定的调用关联起来，这在并行调用无序返回时非常重要。

### 第三步：执行

宿主接收调用，根据声明的模式验证参数，然后运行执行器。无效参数意味着模型虚构了一个字段或使用了错误的类型——这是较弱模型上非常常见的失败模式。生产环境的宿主在遇到无效参数时会做三件事之一：快速失败并将错误暴露给模型，使用受限解析器修复 JSON，或者将验证错误信息包含在提示中重新调用模型。

执行器本身是普通代码。Python、TypeScript、Shell 命令、数据库查询。它产生一个结果，通常是一个字符串，但也可以是任意 JSON 值或结构化的内容块（MCP 中的文本、图片或资源引用）。结果必须是可序列化的。

### 第四步：观察

宿主将工具结果附加到对话中（作为一个带有匹配 `id` 的 `tool` 角色消息），然后重新调用模型。模型现在在上下文中拥有了工具输出，可以生成最终答案或请求更多调用。这个过程持续进行，直到模型停止发出调用或宿主达到安全迭代次数上限。

### 信任分离

工具根据安全性可以分为两种类型。

- **纯工具。** 只读、确定性、无副作用。`get_weather`、`search_docs`、`get_current_time`。可以安全地投机性调用。
- **后果性工具。** 改变状态、花费金钱、接触用户数据。`send_email`、`delete_file`、`execute_trade`。必须设置关卡。

Meta 2026 年针对 agent 安全的"二元法则"规定，单次轮次最多只能同时包含以下三种中的两种：不可信输入、敏感数据、后果性动作。工具接口就是你执行这一规则的地方——通过拒绝调用、要求用户确认或提升作用域来实现。详见第 13 阶段第 15 课（完整的安全章节）和第 14 阶段第 09 课（agent 级别的权限策略）。

### 循环所在的位置

| 上下文 | 谁描述 | 谁决策 | 谁执行 |
|---------|---------------|-------------|--------------|
| 单轮函数调用（OpenAI/Anthropic/Gemini） | 应用开发者 | LLM | 应用开发者 |
| MCP | MCP 服务器 | LLM 通过 MCP 客户端 | MCP 服务器 |
| A2A | Agent Card 发布者 | 调用方 agent | 被调用 agent |
| 网页浏览器（函数调用 agent） | 浏览器扩展 / WebMCP | LLM | 浏览器运行时 |

无论在何处，都是同样的四步。列名会变；结构不会。

### 为什么不直接让模型输出 JSON？

"让模型以 JSON 格式回复"是函数调用出现之前的模式。在前沿模型上，它大约有 5% 到 15% 的失败率，在更小的模型上则高得多。失败模式包括缺少大括号、尾随逗号、虚构字段和类型错误。你需要一个 JSON 修复流程、一次重试或一个受限解码器来处理这些问题。

原生函数调用更好，原因有三。首先，提供商端到端地训练模型使用确切的调用格式，因此在严格模式下有效 JSON 率可以提升到 98% 到 99%。其次，调用载荷位于其自己的协议槽中，而不是在自由文本中——因此工具调用永远不会泄露到用户可见的回复中。第三，提供商通过受限解码（OpenAI 的严格模式、Anthropic 的 `tool_use`、Gemini 的 `responseSchema`）来强制模式合规。输出保证可以通过验证。

第 13 阶段第 02 课并列讲解三种提供商的 API。第 13 阶段第 04 课深入讲解结构化输出。

### 断路器

循环在模型停止发出调用或宿主达到最大轮次计数时终止。生产环境的宿主将此值设置在 5 到 20 轮次之间。超过这个范围，你几乎肯定处于一个模型无法退出的循环中。Claude Code 默认值为 20；OpenAI Assistants 为 10；Cursor 的 agent 模式为 25。

另一种选择——无限循环——每隔六个月就会以"agent 一夜间消耗了 400 美元 API 调用费用"的事后分析报告形式出现。不要在没有限制的情况下上线。

## 使用

`code/main.py` 在没有 LLM 的情况下运行四步循环。一个假的"决策器"函数通过对用户消息进行模式匹配来模拟模型；执行器、模式验证器和观察步骤框架都是真实的。运行它来看看完整的请求/响应编排过程及可打印的中间状态，然后在后续课程中用任何真实的提供商替换这个假的决策器。

关注点：

- 工具注册表为每个工具保存三个字段：名称、描述、模式和执行器引用。
- 验证器是一个仅使用标准库编写的最小 JSON Schema 子集（类型、required、enum、min/max）。第 13 阶段第 04 课提供了一个更完整的版本。
- 循环将迭代次数限制在五次。生产环境中的 agent 确实需要这种断路器。

## 交付

本课生成 `outputs/skill-tool-interface-reviewer.md`。给定一个草拟的工具定义（名称 + 描述 + 模式 + 执行器概要），该技能会审计其循环适应性：名称是否机器稳定、描述是否是完整的使用说明、模式是否正确使用了 JSON Schema 2020-12、以及纯工具与后果性工具的区分是否明确。

## 练习

1. 在 `code/main.py` 中添加第四个名为 `get_stock_price(ticker)` 的工具。将其描述写为"当用户询问某个股票代码的当前价格时使用。不要用于历史价格或市场摘要。"运行框架并确认假的决策器会将提及股票代码的查询路由到新工具。

2. 破坏模式验证器。传入一个 `arguments` 对象缺少必填字段的调用，确认宿主在执行前拒绝了它。然后传入一个带有额外未知字段的调用。决定：宿主应该拒绝还是忽略？用安全性论证来证明你的选择。

3. 将框架中的每个工具分类为纯工具或后果性工具。为需要的注册表条目添加一个 `consequential: true` 标志，并修改循环，使其在每次选择后果性工具时打印一条"将向用户确认"的信息。这就是每个生产环境宿主所需的确认关卡的基本形态。

4. 在纸上绘制四步循环，并为你最喜欢的客户端（Claude Desktop、Cursor、ChatGPT 或自定义栈）填写上面的提供商列表对照表。与第 13 阶段第 06 课中 MCP 特定的变体进行交叉对照。

5. 从头到尾阅读 OpenAI 的函数调用指南。找出一个存在于请求中但不在此处展示的四步循环中的字段。解释它增加了什么，以及为什么它是方便的而非必要的。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|------|----------------|------------------------|
| 工具 | "模型可以调用的东西" | 名称 + JSON Schema 类型化输入 + 执行器函数的三元组 |
| 函数调用 | "原生工具使用" | 提供商级别的 API 支持，用于发出结构化工具调用而非散文 |
| 工具调用 | "模型执行动作的请求" | 模型发出的包含 `id`、`name`、`arguments` 的 JSON 载荷 |
| 工具结果 | "工具返回的内容" | 执行器的输出，包装在带有匹配 id 的 `tool` 角色消息中 |
| 并行工具调用 | "一次多个调用" | 一次模型轮次中的多个调用对象，独立且可通过 id 排序 |
| 严格模式 | "保证 JSON" | 受限解码，强制模型输出验证声明的模式 |
| 纯工具 | "只读工具" | 无副作用；可以安全地重复执行 |
| 后果性工具 | "动作工具" | 改变外部状态；需要关卡、审计或用户确认 |
| 四步循环 | "工具调用周期" | 描述 -> 决策 -> 执行 -> 观察 |
| 宿主 | "Agent 运行时" | 持有工具注册表、调用模型并运行执行器的程序 |
