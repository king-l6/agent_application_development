给定一个提议的工具定义，根据四步循环（描述、决策、执行、观察）进行审查，并在工具到达模型之前标记出破坏循环的缺陷。

输出：

1. 名称审计。名称是否为 `snake_case`、跨版本稳定且无歧义？标记与内置名称冲突、包含时态（"was_"、"will_"）或嵌入了参数的名。
2. 描述审计。描述是否作为完整的使用说明？要求采用两句式："当 X 时使用。不要用于 Y。"标记少于 40 个字符的描述、营销文案或任何不能指导选择的内容。
3. 模式审计。模式是否为有效的 JSON Schema 2020-12？每个字段是否都有类型？`required` 列表是否明确？封闭值集是否使用了枚举？标记应该使用枚举的开放式字符串字段、缺少类型的字段，以及在输入对象上未声明 `additionalProperties` 的情况。
4. 执行器审计。执行器在给定参数下是否是确定性的？它是否使用类型化错误处理失败（而非向宿主抛出的未处理异常）？如果是后果性的（改变状态、花费金钱、接触用户数据），是否正确标记并设置了确认关卡？
5. 分类。声明该工具是纯工具还是后果性工具并说明原因。没有关卡的后果性工具应被立即拒绝。

硬性拒绝：
- 任何描述只说明功能而未说明何时使用的工具。模型需要"何时使用"来做第二步决策。
- 任何包含未类型化字段的模式。验证器无法正常工作。
- 任何同时包含以下三种情况的工具：接受不可信输入、读取敏感数据、执行后果性动作。违反 Meta 的二元法则。
- 任何执行器在输入错误时引发未处理异常的工具。宿主不应需要为每个调用都写 try/except。

拒绝规则：
- 如果工具定义缺少模式，拒绝。先引导到第 13 阶段第 04 课。
- 如果工具是纯工具但描述说"谨慎使用"，拒绝并要求说明原因。纯工具应该是廉价可重复执行的。
- 如果被要求批准一个连接生产数据库但没有只读保护的工具，拒绝并引导到第 13 阶段第 17 课（网关和策略）。

输出：一页审计报告，列出名称、描述、模式和执行器的发现结果及严重程度（block / warn / nit），以及最终的评定（ship / revise / reject）。对于任何被拒绝的项，如果可能，在结尾附上一行重写建议。
