给定一个拟议的 MCP 服务器的能力（以纯英文或工具列表草稿形式），将每项分类为工具、资源或提示，并附上一句话理由。

生成内容：

1. 每项能力的分类。对每个项目，返回 `{名称, 原语: 工具 | 资源 | 提示, 理由}`。
2. 资源 URI 方案。如果有能力变为资源，提出一个 URI 方案（`notes://`、`gh://`、`db://`）和一个模板模式。
3. 提示参数骨架。如果有能力变为提示，提出参数列表和必需/可选标志。
4. 订阅候选。标记那些经常变化且能从 `resources/subscribe` 中获益的资源。
5. 反模式标记。指出旧设计中将读取包装为工具的情况（例如 `notes_read(id)`），而此时使用资源会更合适。

硬性拒绝：
- 任何被分类为"既是工具又是资源"而未作拆分的能力。选择其一或搭建一对组合。
- 任何未标识必需参数的提示。在斜杠命令 UI 中呈现需要参数模式。
- 任何不可寻址的资源 URI 方案（自由格式字符串，不是 URI）。

拒绝规则：
- 如果所有能力都落入工具类别，请询问服务器是否有可能作为资源的只读数据。
- 如果没有能力适合作为提示，这没问题；提示是可选的。不要编造它们。
- 如果服务器的领域更适合 A2A（Agent 到 Agent 协作，不透明状态），拒绝并重定向到阶段 13 · 19。

输出：一页决策报告，包含分类表、URI 方案建议、提示参数骨架和订阅标志。最后指出此服务器中最具影响力的工具 -> 资源转换。
