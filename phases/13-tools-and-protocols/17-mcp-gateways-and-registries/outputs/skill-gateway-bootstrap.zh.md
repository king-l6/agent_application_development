给定一个企业 MCP 计划（用户、后端、合规约束），生成网关配置规范。

产出：

1. **后端列表**。每个后端标注其注册表（官方 / Glama / 自定义）、规范名称（反向 DNS）、钉选描述哈希。
2. **用户列表**。每个用户标注角色和允许的工具集。
3. **RBAC 矩阵**。每用户 x 后端-工具一行，标注允许/拒绝。
4. **速率限制**。每用户突发和持续限制；昂贵工具的每工具限制。
5. **审计计划**。日志目标（文件、OpenTelemetry、SIEM）、保留期限、捕获的字段。

硬性拒绝：
- 任何不在官方注册表中且未经管理员明确批准的后端。
- 任何允许所有用户使用所有工具的 RBAC 规则。权限爆炸。
- 任何没有不可变存储的审计计划。合规失败。

拒绝规则：
- 如果开发者数量超过 100 人且没有定义任何角色，拒绝启动并要求至少三个角色。
- 如果计划没有指定 OAuth 2.1 身份提供者，拒绝并推荐先采用 Keycloak 或 Auth0。
- 如果任何后端使用 stdio，拒绝通过 HTTP 网关代理它；stdio 服务器在本地按开发者运行。

输出：一页的配置文档，包含后端列表、用户列表、RBAC 矩阵、速率限制和审计计划。以团队应该首先实施的单一策略规则结尾。
