# Claude Code作为自主Agent：权限模式与自动模式

> Claude Code暴露七种权限模式。"plan"在每个动作前询问，"default"仅对有风险的动作询问，"acceptEdits"自动批准文件写入但仍确认shell执行，"bypassPermissions"批准一切。自动模式（2026年3月24日）将逐动作批准替换为两阶段并行安全分类器：一个单token快速检查运行在每个动作上；被标记的动作触发思维链深度审查。动作预算通过 `max_turns` 和 `max_budget_usd` 执行。自动模式以研究预览版发布——Anthropic已明确声明分类器本身是不够的。

**类型：** 学习
**语言：** Python（标准库，两阶段分类器模拟器）
**前置知识：** 第15阶段第01课（长时域agent），第15阶段第09课（编码agent全景）
**时间：** 约45分钟

## 问题

你机器上的自主编码agent是一个独特的安全类别。攻击面包括agent能触及的一切——文件系统、网络、凭证、剪贴板、任何浏览器标签页、任何打开的终端。Bruce Schneier等人已公开指出：计算机使用agent不是聊天机器人的"功能更新"，而是一种具有新型风险画像的新型工具。

Claude Code的权限系统是Anthropic的答案。不是单一的"自主/非自主"开关，而是有七种模式横跨一个能力阶梯：plan → default → acceptEdits → … → bypassPermissions。每种模式在速度和逐动作审查之间做出不同的权衡。自动模式（2026年3月）增加了一个两阶段分类器，对于分类器判断为安全的动作，将批准移出用户的关键路径，同时为分类器标记的动作保留审查层。

工程问题：这个系统能捕获什么，遗漏什么，以及给定的任务实际上需要哪种模式？

## 概念

### 七种权限模式

| 模式 | 行为 | 何时使用 |
|---|---|---|
| `plan` | Agent提出计划；用户批准整个计划；每个动作在执行前都被审查 | 不熟悉的任务；生产相关代码；首次在仓库上使用agent |
| `default` | Agent运行动作；对任何"有风险的"动作（shell执行、破坏性操作、网络调用）提示用户 | 大多数交互式编码会话 |
| `acceptEdits` | 文件写入自动批准；shell执行和网络调用仍需提示 | 跨多个文件的重构操作 |
| `acceptExec` | Shell命令在策展的允许列表内自动批准；写入自动批准 | 紧内循环，每个shell命令都是 `npm test` 或类似操作 |
| `autoMode` | 两阶段安全分类器；标记的动作升级审查 | 在受限工作区中的长时域无人值守运行 |
| `yolo` | 跳过大多数提示；仍运行工具允许/拒绝列表 | 临时沙箱、CI作业、研究脚本 |
| `bypassPermissions` | 批准一切 | 记录为"仅在愿意丢弃的临时容器内使用" |

（以上名称与公开的Claude Code文档一致；"yolo"是宽松中间地带的非正式简写。）

### 自动模式，一页说明

自动模式（2026年3月24日发布）是第一个将逐动作批准委托给模型的权限模式。结构：

1. **第一阶段——单token分类器。** 运行在每个提议的动作上。廉价。返回"安全"或"标记"。设计为与主agent循环并行运行，使已批准的动作不会在队列中等待。
2. **第二阶段——辅助策略/安全审查。** 仅在标记的动作上运行。对动作、会话当前状态和声明任务进行集中的策略和安全评估。若置信度低则升级到用户人工介入。

预算控制与分类器并存：

- `max_turns` — 会话中的总迭代次数。
- `max_budget_usd` — 终止会话的美元上限。
- 每个工具的动作计数限制（不超过N次 `WebFetch` 调用等）。

### 系统能捕获什么

- 对工具输入的直接提示注入，其中注入的指令映射到已知的有风险动作形态。
- 重复性的工具循环——分类器可以看到第N+1个动作与第N个动作几乎相同，连续五次。
- 在原本仅文件编辑的会话中明显超出范围的shell命令。

### 系统可能遗漏什么

- **微妙的提示注入**，在调节行为的同时不产生任何被标记的动作。间接提示注入不是一个可以完全修补的漏洞（OpenAI预备性负责人，2025年，关于浏览器agent——见第11课）。
- **语义级不当行为。** 每个单独的动作看起来安全，但组合起来的轨迹是有害的。分类器判断动作；它不会重新推导用户的意图。
- **通过合法渠道的外泄。** 将数据写入你拥有的文件，然后 `git push` 到公共仓库，是一系列允许的动作，其组合才是问题。

### 研究预览框架

Anthropic以研究预览形式发布了自动模式。文档明确说明分类器是一个层，而非解决方案：用户应结合预算、允许列表、隔离工作区和轨迹审计（第12-16课）来使用自动模式。预览框架也反映了已记录的评估vs部署差距（第1课）——通过离线评估的分类器在真实会话中可能表现不同，因为用户的上下文是模糊的。

### 这个阶梯在你工作流中的位置

- 不熟悉的任务：从 `plan` 开始。阅读计划比回滚糟糕的运行成本更低。
- 已知重构：`acceptEdits` 节省大量确认点击。
- 无人值守后台运行：`autoMode` 仅在你已测量过爆炸半径的工作区内使用（无凭证、无生产挂载、未经选择无法出网）。
- 临时容器：`yolo` / `bypassPermissions` 可以接受，当且仅当容器及其凭证是可丢弃的。

## 使用

`code/main.py` 模拟两阶段分类器。第一阶段是对提议动作的廉价关键字规则；第二阶段是较慢的多规则审查器。驱动代码输入一个短的合成轨迹（安全动作、一个提示注入尝试、一个重复循环），展示分类器在何处捕获、何处遗漏。

## 交付

`outputs/skill-permission-mode-picker.md` 将任务描述匹配到正确的权限模式、预算上限和所需的隔离。

## 练习

1. 运行 `code/main.py`。哪种合成动作类型从未被第一阶段标记但总被第二阶段捕获？哪种两者都未捕获？

2. 扩展第一阶段规则集以捕获特定的已知不良形态（例如 `curl $ATTACKER/exfil`）。在良性动作样本上测量假阳性率。

3. 阅读Anthropic的"agent循环工作原理"文档。列出agent在 `default` 模式下默认接触的每个外部状态。在无人值守运行 `autoMode` 之前，你需要单独限制哪些？

4. 设计一个24小时无人值守运行预算：`max_turns`、`max_budget_usd`、每个工具上限、允许列表。为每个数值提供理由。

5. 描述一个轨迹，其中每个单独的动作都被第一阶段和第二阶段批准，但组合起来的行为是不对齐的。（第14课涵盖了终止开关和金丝雀令牌如何解决此问题。）

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 权限模式 | "agent能做多少" | 控制逐动作批准的七个命名策略之一 |
| plan模式 | "做任何事前都询问" | Agent写计划；用户执行前批准 |
| acceptEdits | "让它写文件" | 文件写入自动批准；shell执行仍提示 |
| autoMode | "自动批准" | 两阶段安全分类器；标记的动作升级 |
| bypassPermissions | "全YOLO" | 批准一切；适用于临时容器 |
| 第一阶段分类器 | "快速token检查" | 对提议动作的单token规则；并行运行 |
| 第二阶段分类器 | "深度审查" | 对标记动作的思维链推理 |
| 研究预览 | "非GA" | Anthropic用于其失败模式仍在被映射的功能的框架 |
