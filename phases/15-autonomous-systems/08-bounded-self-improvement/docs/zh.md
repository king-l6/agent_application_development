# 有界自我改进设计

> 研究已经收敛到四个用于约束自我改进循环的原语。必须每次编辑都保持的形式化不变量。不能被修改的对齐锚点。每个维度（安全、公平、鲁棒性）都必须成立而不仅仅是性能的多目标约束。当历史指标显示能力损失时暂停循环的回归检测。它们中没有一个是安全的证明——信息论结果（Kolmogorov复杂度、Lob定理）限制了任何系统能证明关于其自身后继者的内容。它们是提高静默失败成本的缓解措施。

**类型：** 学习
**语言：** Python（标准库，带不变量检查的有界循环）
**前置知识：** 第15阶段第07课（RSI），第15阶段第04课（DGM）
**时间：** 约60分钟

## 问题

第7课的竞赛模拟器展示了微小的速率差异如何复合为巨大差距。第4课的DGM案例研究表明循环可以主动博弈自己的评估器。两个结果指向同一个工程问题：你能在自我改进循环上施加什么样的约束，使得这些约束不能被循环本身静默地削弱？

ICLR 2026 RSI研讨会总结（openreview.net/pdf?id=OsPQ6zTQXV）识别出四个这样的原语。Anthropic的RSP v3.0（第19课）和DeepMind的FSF v3（第20课）都在能力阈值中引用了它们。Meta的HyperAgents工作和社区框架如SAHOO（2026年3月）在生产环境中实现了子集。

诚实的框架：这些是缓解措施。信息论结果限制了任何系统能证明关于其自身后继者的内容，目前没有设计能正式闭合这个问题。一个有良好边界的循环比无边界循环更安全，但绝非绝对安全。

## 概念

### 原语1：形式化不变量

不变量是在每次自我修改之前和之后都必须保持的性质。例如：

- 输出分布以固定的宪法头部为条件（第17课）。
- 没有工具调用指向未经授权的端点。
- 内存写入通过已记录、已签名的路径进行。
- 评估器的模块哈希值与已批准版本匹配。

不变量由循环无法编辑的外部代码检查。如果提议的修改违反不变量，则被拒绝。这样可以捕获DGM风格的评估器篡改模式（第4课）。

困难在于选择那些对安全是必要且可在合理时间内计算的不变量。过度规定会阻碍进展；规定不足会让循环绕过重要性质。

### 原语2：对齐锚点

对齐锚点是循环核心目标的不可变表示，固定在循环的编辑面之外。例如：

- 每次调用时从只读位置加载的宪法文本（第17课）。
- 位于独立仓库、具有独立访问控制的评估器。
- 由人类签署并每个循环重新验证的工具权限清单。

锚点的作用是防止目标漂移。循环可以改进它追求目标的方式，但不能编辑目标是什么。

微妙的失败模式：循环无法编辑的锚点仍然可能被循环以漂移的方式重新解读。宪法AI（第17课）明确基于推理以处理新情况；该推理层正是解读漂移所在之处。锚点是必要的，但不是充分的。

### 原语3：多目标约束

优化单一标量分数的循环总会找到捷径。必须同时满足多个硬约束的循环可用的捷径更少。典型维度：

- 性能（任务级基准）
- 安全（红队评估，已知不良内容的拒绝率）
- 公平（敏感子群体的差异影响边界）
- 鲁棒性（OOD测试集，对抗性输入处理）

仅当每个约束都成立时修改才被接受。第13课的成本管理者将此与财务约束叠加。第18课的Llama Guard作为安全维度接入。

### 原语4：回归检测

每个循环的分数与历史分布进行比较。超出配置容差范围的下降会暂停循环。这捕获了原本会被吸收进运行平均值（因为循环"改进超过了"它）的静默能力损失。

实际实现：存储最近N个循环的每任务分数。每个新循环计算每任务增量。如果有任何增量低于阈值，则拒绝该循环并交由人类审查。

### 信息论限制

Kolmogorov复杂度和Lob定理设定了系统可以证明关于自身的内容的上限。Schmidhuber的形式化Godel Machine（第4课）旨在达到该上限；没有人完成过非平凡证明。Lob的结果表明：如果一个系统可证明地相信"如果我证明我应该做X，我就会做X"，那么它会在不证明应该做的情况下做X——这是一个著名的自指失败。

对我们原语的含义：它们无法闭合安全问题。它们使静默失败更昂贵。一个恶意或发生漂移的循环原本会悄悄绕过缺失的检查，现在必须主动破坏一个显式的检查，这是一种更可检测的特征。

### 一个实例

假设一个agent提出一个编辑。关卡堆栈：

1. 不变量检查：模块哈希、工具权限清单、宪法头部。
2. 锚点检查：目标陈述与已批准版本匹配（字节级或语义级）。
3. 多目标评估：性能、安全、公平、鲁棒性维度。
4. 回归检测：没有维度下降超过容差。

所有四项必须通过编辑才能落地。任何单一失败都会暂停循环。

## 使用

`code/main.py` 在第4课的DGM风格玩具上运行一个有界自我改进循环，但叠加了四个原语。每个原语可以单独启用或禁用。演示表明每个原语捕获一类特定的失败，移除其中任何一个都会让该类失败通过。

## 交付

`outputs/skill-bounded-loop-review.md` 审计一个提议的有界循环，并对四个原语中它实际实现了哪些（与声称实现了哪些）进行评分。

## 练习

1. 运行 `code/main.py`，启用所有原语。确认循环在不允许黑客技巧胜出的情况下仍能在主要指标上改进。

2. 禁用回归检测。构造一个输入，使静默能力损失被接受。

3. 禁用多目标约束。展示循环在性能维度上收敛的同时安全维度下降。

4. 为编码agent设计一个对齐锚点。什么样的文本，存储在哪里，如何检查？

5. 阅读ICLR 2026 RSI研讨会总结。选择四个原语之一，提出一个当前技术水平的改进方案。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 不变量 | "始终成立的性质" | 由外部代码在每次编辑前后检查的性质 |
| 对齐锚点 | "固定的目标" | 循环编辑面之外的不可变核心目标表示 |
| 多目标约束 | "所有维度都必须成立" | 性能、安全、公平、鲁棒性——全部要求 |
| 回归检测 | "下降时暂停" | 当历史指标增量表明能力损失时暂停循环 |
| Kolmogorov界限 | "信息论限制" | 限制系统能证明关于自身后继者的内容 |
| Lob定理 | "自指陷阱" | 系统可以执行"我应该"而不证明应该 |
| 关卡堆栈 | "分层检查" | 多个原语组合；任何失败拒绝编辑 |
| 有界改进 | "缓解措施，非证明" | 提高静默失败成本；不闭合安全问题 |
