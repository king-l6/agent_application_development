给定一个提议的长时间运行agent部署（Temporal + OpenAI Agents SDK、带PostgreSQL检查点的LangGraph、Microsoft Agent Framework、Claude Code Routines、Cloudflare Durable Objects或内部等效方案），根据持久执行模式审计设计。

生成：

1. **活动清单。** 列出每个活动（LLM调用、工具调用、HTTP请求、文件写入）。对于每个，确认其被包装为带有重试策略、超时和幂等键的活动。活动封装外的原始LLM调用是可靠性漏洞。
2. **工作流确定性。** 识别工作流代码中的每个非确定性读取（挂钟、随机数、外部状态）。每个必须注册为副作用活动，以便重放返回相同的值。隐藏的非确定性是重放漂移的最常见原因。
3. **检查点后端。** 命名后端（PostgreSQL、SQLite、Redis、Durable Objects）。确认它在部署后存活。SQLite仅用于开发。Redis需要AOF或快照配置。Cloudflare Durable Objects是透明的，但需要唯一键纪律。
4. **人类输入状态。** 确认为人工介入暂停是工作流的一等状态，而非轮询循环。工作流应阻塞在外部信号（审批队列、webhook、`interrupt()` 原语）上，该信号在审批到达时精确恢复。
5. **恢复时的人工介入策略。** 对于崩溃后的任何恢复，说明在执行为下一个活动之前是否需要新的人工介入。没有此策略，持久执行加上崩溃前已批准的审批可能会在上下文发生变化时重新触发已批准的动作。对于长时域至关重要。

硬性拒绝：
- Agent SDK使用中LLM调用未包装为活动。
- 不能在部署后存活的检查点后端。
- 嵌入挂钟或随机数但没有活动包装的工作流。
- 建模为轮询循环而非信号的人类输入。
- 没有恢复时人工介入策略的长时域运行（超过一小时）。
- 在持久性之上没有叠加预算终止开关（第13课）的运行。

拒绝规则：
- 如果用户提议的持久工作流没有对副作用活动的显式幂等性，拒绝并需要幂等键。否则重试将双重执行。
- 如果用户不能展示重放测试（运行工作流、中途崩溃、重放、断言无双重副作用），拒绝并要求该测试通过后才能生产。
- 如果用户提议24小时无人值守运行且没有人工介入检查点，拒绝。35分钟退化（第12课说明）使得即使持久性正确，这也是一个可靠性问题。

输出格式：

返回设计审查备忘录，包含：
- **活动表格**（活动、重试策略、超时、幂等键）
- **确定性审计**（非确定性读取及其处理方式）
- **检查点后端**（名称、部署后存活 是/否、重放测试状态）
- **人工介入状态形态**（一等状态/轮询/缺失）
- **恢复时人工介入策略**（显式，带理由）
- **就绪度**（生产/预演/仅研究）
