# 长时间运行的后台Agent：持久执行

> 生产级长时域agent不在 `while True` 中运行。每个LLM调用都成为一个带有检查点、重试和重放的活动。Temporal的OpenAI Agents SDK集成于2026年3月正式发布（GA）。Claude Code Routines（Anthropic）运行定时调度的Claude Code调用，无需持久的本地进程。会话在人类输入时暂停，在部署后存活，并从由 `thread_id` 索引的最新检查点恢复。新的工效学背后是一个古老模式——工作流编排——加上一个新的输入：LLM调用作为非确定性的、必须在恢复时确定性重放的活动。

**类型：** 学习
**语言：** Python（标准库，最小持久执行状态机）
**前置知识：** 第15阶段第10课（权限模式），第15阶段第01课（长时域agent）
**时间：** 约60分钟

## 问题

考虑一个运行四小时的agent。它调用三个工具，两次提示用户，并进行四十次LLM调用。一半时，运行它的主机重启。会发生什么？

- 在朴素的 `while True` 循环中：一切丢失。运行从头开始重启。三个工具调用（带有真实副作用）再次执行。用户再次被提示他们已经批准过的事情。四十次LLM调用被重新计费。
- 使用持久执行：运行从最近的检查点恢复。已经完成的活动不重新执行；其结果从持久日志中重放。用户无需重新批准他们已经批准过的事情。已经完成的LLM调用不重新计费。

这与工作流引擎已经提供十年之久的模式相同（Temporal、Cadence、Uber的Cherami）。新增的是LLM调用现在成为一种活动——非确定性的、昂贵的、带有副作用——并且它们干净地适配这个模式。

本课的主题：长时域可靠性衰减（METR观察到"35分钟退化"——成功率大致随任务时长二次方下降）。持久执行使得运行可以超出可靠性概况支持的长度，这是一种新的失败方式：设计正确时安全，设计错误时不安全。

## 概念

### 活动、工作流和重放

- **工作流**：确定性编排代码。定义了活动序列、分支和等待。必须是确定性的，以便可以从事件日志中重放而不会出现意外的发散。
- **活动**：一个非确定性的、可能失败的单元工作。LLM调用、工具调用、文件写入、HTTP请求。每个活动都记录其输入和（完成后）输出。
- **事件日志**：持久后端存储。每个活动开始、完成、失败、重试以及每个工作流决策都被记录。
- **重放**：恢复时，工作流代码从头开始重新运行；每个已完成的活动返回其记录的结果而不重新执行。仅尚未完成的活动实际运行。

这与React针对虚拟DOM的重新渲染，或Git从提交重建工作树是相同的形态。编排器中的确定性正是使持久性廉价的原因。

### 为什么LLM调用适配该模式

LLM调用：

- 非确定性（温度>0；即使温度0也会随模型版本漂移）。
- 昂贵（金钱和延迟）。
- 可能失败（速率限制、超时）。
- 带有副作用（如果它们调用工具）。

这正是活动概况。将每个LLM调用包装为活动，你就能获得带有指数退避的重试、跨重启的检查点以及用于调试的可重放轨迹。

### 由 `thread_id` 索引的检查点

LangGraph、Microsoft Agent Framework、Cloudflare Durable Objects和Claude Code Routines都收敛到相同的API形态：一个 `thread_id`（或等效标识符）标识会话；每个状态转换持久化到后端（PostgreSQL默认，SQLite用于开发，Redis用于缓存）；恢复读取最新的检查点。

后端选择很重要：

- **PostgreSQL**：持久化、可查询、在部署后存活。LangGraph的默认选择。
- **SQLite**：仅本地开发；跨主机丢失数据。
- **Redis**：快速但短暂，除非配置了AOF/快照。
- **Cloudflare Durable Objects**：透明分布；由唯一键限定范围；存活数小时到数周。

### 人类输入作为一等状态

提议-然后-提交（第15课）需要一个持久的"等待人类"状态。工作流暂停，外部队列持有待处理的请求，批准后从该确切点恢复。没有持久性这是尽力而为；有了持久性，隔夜的批准在早上到达时工作流继续执行。

### 35分钟退化

METR观察到每个被测量的agent类别在连续运行约35分钟后显示出可靠性衰减。任务时长翻倍大致使失败率翻两番。持久执行不能修复这个问题；它让你运行超过可靠性概况支持的时间。安全的模式是将持久性与以下两者结合：在重新进入时需要新的人工介入的检查点，以及无论挂钟时间如何都设定总计算量上限的预算终止开关（第13课）。

### 何时持久执行是错误答案

- 运行时间短于几分钟且无人类输入。开销大于收益。
- 严格的只读信息检索。
- 正确性要求端到端在单个上下文窗口内的任务（某些推理任务；某些一次性生成）。

## 使用

`code/main.py` 在标准库Python中实现了一个最小的持久执行引擎。它支持：

- `@activity` 装饰器，将输入和输出记录到JSON事件日志中。
- 一个编排活动序列的工作流函数。
- 一个 `run_or_replay(workflow, event_log)` 函数，重放已完成的活动而不重新执行它们。

驱动代码模拟一个三活动工作流，中途崩溃，并展示（a）朴素重试重新执行所有内容 vs（b）重放仅运行缺失的活动。

## 关键术语

| 术语 | 人们怎么说 | 实际含义 |
|---|---|---|
| 工作流 | "agent的脚本" | 确定性编排代码；可从事件日志重放 |
| 活动 | "一个步骤" | 非确定性单元（LLM调用、工具调用）；前后记录 |
| 事件日志 | "后端存储" | 每个状态转换的持久记录 |
| 重放 | "恢复" | 重新运行工作流；已完成的活动返回记录结果而不重新执行 |
| 检查点 | "保存点" | 由thread_id索引的持久化状态；恢复时取最新 |
| thread_id | "会话键" | 限定持久状态范围的标识符 |
| 35分钟退化 | "可靠性衰减" | METR：成功率大致随时域二次方下降 |
| 非确定性 | "重放中的漂移" | 挂钟、随机数、LLM输出；必须注册为副作用 |
