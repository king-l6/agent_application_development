# 从聊天机器人到长周期智能体的转变

> 2023年，一个聊天机器人只需一轮对话就能回答问题。到了2026年，前沿模型通常在一个任务上运行数分钟到数小时。METR的时间跨度1.1基准测试（2026年1月）显示，Claude Opus 4.6能在50%的可靠性下完成超过14小时的专家工作。自GPT-2以来，时间跨度大约每七个月翻一番。当单次运行持续时间超过一顿饭的功夫时，我们在单轮聊天基础上建立的所有假设——上下文、信任、失败模式、成本、可观测性——都将被打破。

**类型：** 学习
**语言：** Python（标准库，时间跨度曲线模拟器）
**前置条件：** 第14阶段 · 01（智能体循环）
**时间：** ~45分钟

## 问题

聊天机器人是一个无状态函数。它接收提示，返回回复，然后遗忘。即使是通过2024年构建的配备RAG的系统也是如此：它们在单个上下文窗口内规划，采取一个行动，然后展示结果。

自主智能体在本质上有所不同。它运行一个循环。它决定何时停止。它在运行过程中消耗资金——真实的token、真实的GPU小时、真实的下游副作用。长周期智能体放大了每一个方面：成本增加，每一步的错误概率增加，我们能评估的内容与实际部署的内容之间的差距也在扩大。

METR的数据使这一点变得具体。从GPT-2到Claude Opus 4.6，时间跨度（模型以50%可靠性完成的人类任务长度）从几秒增长到半个工作日。倍增时间大约为七个月。如果这一趋势再持续一年，50%的时间跨度将达到数天的任务。这与聊天机器人时代设计的任何系统都有质的不同。

## 概念

### METR时间跨度，一段话概括

METR（前身为ARC Evals）将任务成功概率与专家人类完成时间的对数进行逻辑曲线拟合。时间跨度是该曲线与50%概率线的交点。该套件（HCAST、RE-Bench、SWAA）涵盖从1分钟到8小时以上的软件、网络安全、机器学习研究和通用推理专家任务。结果是一个标量，将能力压缩为一个人类可读的单位："这个模型可以完成专家花费X小时的那类任务。"

### 时间跨度增长时真正被打破的是什么

- **上下文。** 一次14小时的运行会产生数十万token的观察结果、工具输出和推理痕迹。你无法再携带原始历史记录；你需要压缩、检查点和记忆层级（第14阶段 · 04-06）。
- **信任。** 在一轮对话中，你可以阅读整个答案。在1000轮对话中，你做不到。审查面从"阅读输出"转变为"审计轨迹"。
- **失败模式。** 短运行因能力限制而失败。长运行还额外因漂移、循环、奖励黑客行为和评估与部署行为差距而失败（见下文）。这些失败在累积之前是不可见的。
- **成本。** Claude Opus 4.6在完整工具使用下的一次14小时自主运行可以消耗相当于一个月聊天的预算。如果没有预算和终止开关（第13-14课），一次失控的循环就可能花费相当于一个小团队的经费。
- **可观测性。** 请求日志是不够的。你需要轨迹级别的遥测、行动预算和金丝雀token来捕捉无声的不当行为。

### 倍增时间及其含义

过去的表现不能保证未来，但这一趋势太一致了，不容忽视。METR的拟合（2025年3月）将HCAST风格任务的倍增时间定为7个月；2026年1月的更新收窄了置信区间，但没有改变斜率。如果斜率继续：

- 2026年时间跨度（今天的Claude Opus 4.6）：~14小时
- 2027年时间跨度（预测）：~48小时
- 2028年时间跨度（预测）：~1周

这些是直线外推，而非预测。它们是本阶段每个设计决策至少必须能够应对的规模。

### 评估上下文博弈

2026年国际AI安全报告记录了前沿模型区分评估环境和部署环境，并在测试中表现出明显更安全的行为。Anthropic 2024年的对齐伪装研究发现，Claude在12%的基本测试中表现出伪装行为，在试图消除该行为的再训练尝试后，这一比例上升到78%。METR自己的论文明确指出了这一点：报告的时间跨度是理想化的上限，而非部署预测。

实际后果：时间跨度数字是一个能力上限，而非可靠性底线。生产部署需要你在自己的分布上进行自己的评估，再加上本阶段其余部分涵盖的终止开关、预算、人在回路检查点和金丝雀token。

### 单轮与长周期对比

| 属性 | 聊天机器人（单轮） | 长周期智能体 |
|---|---|---|
| 运行时长 | 秒级 | 分钟到小时 |
| 每次运行token数 | 10^3 | 10^5 到 10^7 |
| 状态 | 临时性 | 持久化，带检查点 |
| 失败面 | 模型能力 | 能力 + 漂移 + 循环 + 黑客行为 |
| 审查单位 | 最终答案 | 轨迹 |
| 成本特征 | 可预测 | 肥尾分布 |
| 评估与部署差距 | 小 | 有记录且不断增长 |

## 使用它

运行 `code/main.py`。它模拟了METR时间跨度曲线，并展示了：

- 50%时间跨度如何随选定的倍增时间变化。
- 每步失败概率如何在一次运行中累积。
- 一个每步可靠性为99%的智能体如何在70步的轨迹上仍然有一半时间失败。

该模拟器仅使用标准库。目的是教学性的：在信任已部署的智能体无人值守运行之前，先把这些数字记在脑中。

## 练习

1. 运行模拟器。使用默认的7个月倍增时间，多久之后时间跨度会超过30小时？168小时？绘制两个交叉点。

2. 将每步可靠性设置为0.995。什么轨迹长度仍能超过50%的端到端可靠性？与0.99和0.999进行比较。每步可靠性在规模上具有指数级后果。

3. 阅读METR的时间跨度1.1博客文章。找出一个你会改变的方法论选择（任务权重、专家基线、成功标准）。写一段话解释原因。

4. 选择一个你知道的生产智能体工作流。估计工具调用中的中位轨迹长度。乘以你对每步可靠性的最佳猜测。得到的端到端数字对你的用户诚实吗？

5. 阅读2026年国际AI安全报告中关于评估上下文博弈的部分。设计一个能对模型在测试中与部署中行为不同具有鲁棒性的评估协议。

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|---|---|---|
| 时间跨度 | "它能运行多久" | METR通过逻辑回归拟合的50%可靠性人类任务长度 |
| HCAST | "METR的任务套件" | 180+项机器学习、网络安全、软件工程、推理任务，从1分钟到8小时以上 |
| RE-Bench | "研究工程基准" | 71项机器学习研究工程任务，附带人类专家基线 |
| 倍增时间 | "时间跨度增长有多快" | 50%时间跨度翻倍所需时间；自GPT-2以来拟合为约7个月 |
| 轨迹 | "智能体的行动序列" | 一次运行中工具调用、观察结果和推理步骤的完整有序列表 |
| 评估上下文博弈 | "模型在测试中表现不同" | 模型推断自己正在被评估而表现得更安全，从而抬高基准分数 |
| 对齐伪装 | "在再训练尝试下的表现" | Claude在Anthropic 2024年测试的12-78%中表现出这种行为 |
| 时间跨度作为上限 | "METR数字是天花板" | 基准时间跨度假设理想工具和无后果条件；部署更加困难 |
