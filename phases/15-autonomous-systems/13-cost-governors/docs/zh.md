# Action Budgets, Iteration Caps, and Cost Governors (行动预算、迭代上限与成本控制)

> 一家中等规模的电商代理月 LLM 成本从 1,200 美元跃升至 4,800 美元，原因仅仅是团队启用了"订单追踪"技能。这不是定价漏洞，而是一个代理找到了新的循环并持续在其中消费。微软的 Agent Governance Toolkit（2026 年 4 月 2 日）将此防御措施编撰成文：每次请求的 `max_tokens`、每任务的 Token 和美元预算、每日/每月上限、迭代上限、分层模型路由、提示缓存、上下文窗口、高成本操作的人工确认检查点、预算超支的终止开关。Anthropic 的 Claude Code Agent SDK 以不同名称提供了相同的原语。财务速度限制（如 10 分钟内超过 50 美元则切断访问）比月度上限更快地捕获循环。

**类型：** 理论学习
**语言：** Python（标准库，分层成本控制模拟器）
**前置要求：** 第 15 阶段 · 10（权限模式），第 15 阶段 · 12（持久化执行）
**预计时间：** ~60 分钟

## 问题

自主代理每执行一轮都会消耗真实资金。聊天机器人的糟糕输出是一次糟糕回复；而代理的糟糕循环则是一张账单。业界对此失效模式的记录术语是"钱包拒绝"——代理持续推理、持续调用工具、持续计费，而没有任何机制阻止它，因为从未设计过。

解决方案不是单一数字，而是不同时间尺度和粒度的限制堆栈：每次请求、每任务、每小时、每天、每月。设计良好的堆栈能在几分钟内捕获失控循环，几小时内捕获缓慢泄漏，一天内捕获不良发布。当代理需要长期自主运行时，同一堆栈才能始终保持预算。

这是一个工程课程：数学很简单，而团队常常在纪律性上失败。以下限制列表全部出现在微软 Agent Governance Toolkit 或 Anthropic Claude Code Agent SDK 文档中。

## 概念

### 成本控制堆栈

1. **每次请求的 `max_tokens`**。简单明了。防止任何一次调用产生无上限的补全内容。
2. **每任务 Token 预算**。在整个运行过程中，不超过 N 个 Token。到达上限时硬性停止。
3. **每任务美元预算**。与 Token 相同，但以货币计。Claude Code 中的 `max_budget_usd`。
4. **每个工具调用上限**。不超过 N 次 `WebFetch` 调用、N 次 `shell_exec` 调用等。
5. **迭代上限（`max_turns`）**。代理循环的总迭代次数；防止无限推理循环。
6. **每分钟/每小时/每天/每月上限**。滚动窗口。在不同时间尺度捕获泄漏。
7. **财务速度限制**。例如，"如果 10 分钟内支出超过 50 美元，切断访问。"在月度上限触发之前捕获基于循环的消耗。
8. **分层模型路由**。默认使用较小模型；仅当分类器判断任务需要时升级到较大模型。
9. **提示缓存**。系统提示和稳定上下文存储在提供商缓存中；重新发送的 Token 成本接近于零。
10. **上下文窗口管理**。压缩/摘要以将活动上下文保持在阈值以下；直接降低 Token 成本。
11. **高成本操作的人工确认检查点**。在执行已知高成本的操作（长时间工具调用、大文件下载、昂贵的模型升级）之前，需要人工确认。
12. **预算超支的终止开关**。任何上限触发时会话中止。记录上限触发情况；需要单独的重新启用路径。

### 为什么需要堆栈而非单一上限

单一的月度上限只能在钱包耗尽后捕获失控代理。单一的每次请求上限无法在会话级别捕获任何问题。不同的失效模式需要不同的时间尺度：

- **失控循环**（代理陷入 5 秒重试循环）：由速度限制捕获。
- **缓慢泄漏**（代理每个任务执行约 2 倍预期工作）：由每日上限捕获。
- **不良发布**（新版本使用 5 倍 Token）：由每周/每月上限捕获。
- **合法激增**（真实需求，而非故障）：由小时/日上限捕获，并附带清晰日志。

### Claude Code 的预算功能

Claude Code Agent SDK 公开了以下功能（来自公开文档）：

- `max_turns` —— 迭代上限。
- `max_budget_usd` —— 美元上限；超支时会话中止。
- `allowed_tools` / `disallowed_tools` —— 工具允许列表和拒绝列表。
- 工具使用前的钩子点，用于自定义成本核算。

结合权限模式阶梯（第 10 课）。没有 `max_budget_usd` 的 `autoMode` 会话就是无监管的自主运行。Anthropic 明确将自动模式视为需要预算控制；分类器与成本是正交的。

### 欧盟 AI 法案，OWASP 代理十大风险

微软的 Agent Governance Toolkit 涵盖了 OWASP 代理十大风险和欧盟 AI 法案第 14 条（人类监督）的要求。在欧盟的生产环境中，日志记录和上限执行是不可选的。

### 观察到的 1,200 美元 → 4,800 美元案例

微软文档中的真实案例：一个电商代理在添加新工具后月度成本翻了三倍。该工具允许代理在每个会话期间轮询订单状态。没有循环检测。没有每工具上限。没有周环比增长告警。解决方案是每工具上限加上每日增长告警。这是一个模板：每个新工具表面都是一个新的潜在循环；每个新工具都需要自己的上限和自己的告警。

## 使用

`code/main.py` 模拟了代理在有无分层成本控制堆栈两种情况下的运行。模拟代理在若干轮后进入轮询循环；分层堆栈在速度窗口内捕获它，而单一的月度上限要等到数天后才会触发。

## 练习

1. 运行 `code/main.py`。确认在轮询循环轨迹中，速度限制在迭代上限之前触发。然后禁用速度限制，测量代理在迭代上限捕获之前"花费"了多少。

2. 为浏览器代理（第 11 课）设计一套每工具上限。哪个工具需要最严格的上限？哪个工具可以无限制运行而不产生风险？

3. 阅读微软 Agent Governance Toolkit 文档。列出该工具包命名的每种上限类型。将每种映射到一种失效模式（失控循环、缓慢泄漏、不良发布、激增）。

4. 为实际任务（如"在一个仓库中分类 50 个 issue"）估算一次无人值守的夜间运行价格。将 `max_budget_usd` 设置为你点估计值的 2 倍。证明 2 倍的合理性。

5. Claude Code 的 `max_budget_usd` 在会话聚合成本上触发。设计一个你将在外部强制执行的补充速度限制。什么触发切断，重新启用是什么样的？

## 关键术语

| 术语 | 人们所说的 | 实际含义 |
|---|---|---|
| Denial of Wallet | "失控账单" | 代理循环产生支出，没有上限阻止它 |
| max_tokens | "每次请求上限" | 单次补全大小的上限 |
| max_turns | "迭代上限" | 会话中代理循环迭代次数的上限 |
| max_budget_usd | "美元终止开关" | 会话成本上限；超支时中止 |
| Velocity limit | "速率上限" | 短时间窗口内的支出限制（如 50 美元/10 分钟） |
| Tiered routing | "小模型优先" | 默认使用廉价模型；仅在分类器判断需要时升级 |
| Prompt caching | "缓存的系统提示" | 提供商端缓存将重新发送的 Token 成本降至接近零 |
| HITL checkpoint | "人类批准门" | 高成本操作前需要人工确认 |
