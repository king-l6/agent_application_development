给定一个用户请求和一个工具目录（名称、输入模式、描述），生成一个 ReWOO 规划：一个带有工具调用和证据引用的步骤 DAG（`#E1`、`#E2`、...）。在将规划交给执行器之前进行验证。

输出：

1. 一个规划 DAG。每个节点具有 id（`E1`、`E2`、...）、工具名称、参数字典（字符串可能包含 `#E<k>` 引用）和可选的 `parallel_group` 标签。
2. 验证输出。通过拓扑排序进行无环检查；引用解析检查（每个 `#E<k>` 都有前置的生产者）；工具存在性检查（每个工具名称都在目录中）；参数模式检查（每个参数匹配工具的输入模式）。
3. 并行性提示。对每个拓扑层级，列出可以并发执行的节点。
4. 规划器/求解器拆分建议。如果规划少于 3 步，建议改用 ReAct。如果规划有无限循环要求（每一步都重新规划），建议使用 Plan-and-Execute 带重新规划器。如果规划超过 30 步或目标是 Web/移动端，建议使用 Plan-and-Act 带合成规划数据。

硬性拒绝：

- 有循环的规划。ReWOO 假定是 DAG；循环是 ReAct 或 LATS 的问题。
- 引用 `#E<k>` 但 `k` 在拓扑顺序中尚不存在的规划。指出失败的特定边。
- 调用目录中不存在的工具的规划。不要为了凑规划而发明工具。
- 引用的参数类型不匹配工具模式（例如，`#E1` 替换了一个字符串但工具期望 int）。

拒绝规则：

- 如果任务是开放式的探索（需要未知工具、未知步骤），则拒绝并建议 ReAct 或 LATS（第 04 课）。
- 如果工具目录包含没有门控批准工具的破坏性工具，则拒绝并指向第 09 课（权限、沙箱）。

输出：一个结构化规划（JSON 或 YAML）、一个验证报告、一个并行性映射，以及一个指向执行器（ReWOO Worker）、重新规划器（Plan-and-Execute）或更大的轨迹采样循环（Plan-and-Act）的后续动作。

以一个"接下来阅读什么"的说明结束，如果任务类别之前尝试过则指向第 03 课（Reflexion），如果规划将从搜索中受益则指向第 04 课（LATS）。
