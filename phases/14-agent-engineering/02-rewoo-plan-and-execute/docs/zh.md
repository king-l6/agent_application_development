# ReWOO 与 Plan-and-Execute：解耦规划

> ReAct 将思考和行动交错在一个流中。ReWOO 将它们分离：前期的整体规划，然后执行。减少 5 倍的令牌数，在 HotpotQA 上提高 +4% 的准确率，并且你可以将规划器蒸馏到 7B 模型中。Plan-and-Execute 将其泛化；Plan-and-Act 将其扩展到网络导航。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 01（Agent 循环）
**时间：** ~60 分钟

## 学习目标

- 解释为什么 ReWOO 的规划器/工作器/求解器三部分拆分相比 ReAct 的交错循环节省令牌并提高鲁棒性。
- 实现一个规划 DAG、一个依赖排序的执行器和一个组合工作器输出的求解器 —— 全部使用标准库。
- 使用 2026 年的"五种工作流模式"框架（Anthropic），判断任务应该以"先规划后执行"的方式运行还是交错式的 ReAct。
- 认识到在哪些情况下需要 Plan-and-Act 的合成规划数据来处理长周期 Web 或移动端任务。

## 问题

ReAct 的交错思考-行动-观察循环简单灵活，但每次工具调用都必须携带完整的先前上下文 —— 包括每一个先前的思考。令牌使用量随深度呈二次方增长。更糟糕的是：当工具在循环中间失败时，模型必须从错误观察中重新推导整个规划。

ReWOO（Xu et al.，arXiv:2305.18323，2023 年 5 月）注意到了这一点并押注了一个方案：一次性规划整个任务，并行获取证据，最后组合答案。一次 LLM 调用来规划，N 次工具调用获取证据（可以并行），一次 LLM 调用来求解。代价是灵活性降低（规划是静态的），但令牌效率显著提高，失败模式也更加清晰。

## 概念

### 三个角色

```
Planner:  user_question -> [plan_dag]
Workers:  [plan_dag]     -> [evidence]        (tool calls, possibly parallel)
Solver:   user_question, plan_dag, evidence -> final_answer
```

规划器产生一个 DAG。每个节点命名一个工具、其参数以及它依赖的先前节点（引用如 `#E1`、`#E2`）。工作器按拓扑顺序执行节点。求解器将所有内容拼接在一起。

### 为什么令牌数减少 5 倍

ReAct 的提示长度随步数线性增长。在第 10 步时，提示包含思考 1 加上行动 1 加上观察 1 加上思考 2 加上行动 2 加上观察 2，依此类推。每个中间步骤还冗余包含原始提示。

ReWOO 支付一个大的规划器提示、N 个小的工人提示（每个只有工具调用，没有链式内容）和一个求解器提示。在 HotpotQA 上，论文测量到约 5 倍的令牌减少，同时准确率绝对提高了 +4%。

### 为什么更鲁棒

如果工作器 3 在 ReAct 中失败，循环必须在执行过程中推理出错误。在 ReWOO 中，工作器 3 返回一个错误字符串；求解器在上下文中看到它以及原始规划，可以优雅地降级。失败定位是基于节点的，而不是基于步骤的。

### 规划器蒸馏

论文的第二个发现：因为规划器不看到观察结果，你可以从 175B 教师模型的规划器输出中微调一个 7B 模型。小模型处理规划；大模型在推理时不需要使用。这现在已成为标准做法 —— 许多 2026 年的生产 agent 使用小规划器和大执行器，反之亦然。

### Plan-and-Execute（LangChain，2023）

LangChain 团队 2023 年 8 月的博文将 ReWOO 泛化为一个模式名称：Plan-and-Execute。前期规划器发出一个步骤列表，执行器运行每个步骤，可选的重新规划器可以在观察结果后修改规划。这比 ReWOO 更接近 ReAct（重新规划器将观察结果带回规划中），但保留了令牌节省的优势。

### Plan-and-Act（Erdogan et al.，arXiv:2503.09572，ICML 2025）

Plan-and-Act 将该模式扩展到长周期 Web 和移动 agent。关键贡献是合成规划数据：一个带标签的轨迹生成器产生训练数据，其中规划是显式的。用于微调规划器模型，使其在 WebArena 类任务中超过 30–50 步时仍然有效，而单个 ReAct 轨迹在此情况下会失去连贯性。

### 何时选择哪种

| 模式 | 何时使用 |
|---------|---------|
| ReAct | 短任务，未知环境，需要响应式异常处理 |
| ReWOO | 结构化的已知工具任务，对令牌敏感，可并行的证据获取 |
| Plan-and-Execute | 类似 ReWOO 但需要在部分执行后重新规划 |
| Plan-and-Act | 长周期（>30 步），Web/移动/计算机使用 |
| Tree of Thoughts | 值得为搜索付出代价时（第 04 课） |

Anthropic 2024 年 12 月的指导原则：从最简单的开始。如果任务只需一次工具调用加一个摘要，不要构建 ReWOO。如果任务是 40 步的研究任务，不要只使用 ReAct。

## 构建

`code/main.py` 实现了一个玩具 ReWOO：

- `Planner` —— 一个脚本化的策略，从提示中发出规划 DAG。
- `Worker` —— 通过注册表调度每个节点的工具调用。
- `Solver` —— 脚本化的组合，读取证据并产生最终答案。
- 依赖解析 —— 像 `#E1` 这样的引用被替换为早期工作器的输出。

演示回答"巴黎（法国首都）的人口是多少（四舍五入到百万）？"使用两步规划：（1）查找首都，（2）查找人口，然后求解。

运行：

```
python3 code/main.py
```

轨迹首先显示完整规划，然后是工作器结果，最后是求解器组合。将令牌计数（我们打印粗略字符数）与 ReAct 风格的交错运行进行比较 —— ReWOO 在这种结构化任务上胜出。

## 使用

LangGraph 将 Plan-and-Execute 作为配方提供（`create_react_agent` 用于 ReAct，自定义图用于 plan-execute）。CrewAI 的 Flows 直接编码了该模式：你提前定义任务，Flow DAG 执行它们。Plan-and-Act 的合成数据方法仍主要处于研究阶段；运行时模式（显式规划 DAG）通过 LangGraph 和 CrewAI Flows 进入生产。

## 交付

`outputs/skill-rewoo-planner.md` 根据用户请求和工具目录生成 ReWOO 规划 DAG。它在将规划交给执行器之前验证规划（无环，每个引用已解析，每个工具都存在）。

## 练习

1. 将独立规划节点的工作器执行并行化。在一个有 2 个并行组的 6 节点 DAG 上，它能带来什么好处？
2. 添加一个重新规划器节点，在任何工作器返回错误时触发。使 ReWOO 变为 Plan-and-Execute 的最小改动是什么？
3. 将 `Planner` 替换为小模型（7B 类），保持 `Solver` 使用前沿模型。比较端到端质量 —— 拆分在什么地方失败？
4. 阅读 ReWOO 论文中关于规划器蒸馏的第 4 节。概念上重现 175B -> 7B 的结果：你需要什么训练数据，如何评分规划质量？
5. 将玩具移植到 Plan-and-Act 的轨迹形式：规划是一个序列，而不是 DAG。哪些权衡发生了变化？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| ReWOO | "无观察的推理" | 先规划，然后并行获取证据，然后求解 —— 规划提示中没有观察结果 |
| Plan-and-Execute | "LangChain 的规划-执行模式" | ReWOO 加上执行后可选的重新规划器节点 |
| Plan-and-Act | "规模化规划-执行" | 显式的规划器/执行器拆分，使用合成规划训练数据进行长周期任务 |
| 证据引用 | "#E1, #E2, ..." | 规划节点占位符，在调度时替换为先前的工人输出 |
| 规划器蒸馏 | "小规划器，大执行器" | 在大教师模型的规划轨迹上微调小模型 |
| 令牌效率 | "更少的往返次数" | 在 HotpotQA 上相比 ReAct 减少 5 倍令牌 |
| DAG 执行器 | "拓扑调度器" | 按依赖顺序运行规划节点；在每个层级并行 |
