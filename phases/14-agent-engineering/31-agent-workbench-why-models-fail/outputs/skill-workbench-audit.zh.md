给定一个仓库路径以及将在其中运行的 Agent 产品，审计七个工作台面并生成一份就绪报告。

七个面：

1. 指令：Agent 首先读取的根文件（例如 `AGENTS.md`），简短，指向更深层的规则。
2. 状态：持久的、机器可读的文件，记录任务、触及的文件、阻塞项、下一步操作。
3. 范围：每个任务一份合同，列出允许的文件、禁止的文件、验收标准、回滚计划。
4. 反馈：捕获命令、标准输出、标准错误、退出码，并将结果反馈回循环的运行器。
5. 验证：运行测试、lint、类型检查、冒烟运行并确认验收标准的门。
6. 审查：使用不同角色的第二次检查，构建者不能给自己的作业评分。
7. 交接：总结改变了什么、为什么改变、还剩下什么以及下一个最佳行为的工件。

产出：

- 每个面的评分：0 缺失、1 部分存在、2 健康。将每个评分与你观察到的文件或过程关联。
- 按杠杆排序的三个优先级：哪个缺失的面如果首先添加，能消除最多的失败模式。
- 一份机器可读的 `workbench_audit.json` 报告以及一份人类可读的 `workbench_audit.md` 摘要。
- 最弱面的起始补丁：将评分从 0 提升到 1 的最小文件更改。

硬性拒绝：

- 没有文件路径或过程引用的"健康"评分。没有证据的审计会腐化。
- 单一的合并"Agent 配置"面。合并面在任务失败时隐藏了哪个面出了问题。
- 因为测试慢而跳过验证。如果验证不在工作台上，构建者就会给自己的作业评分。

拒绝规则：

- 如果仓库根本没有测试命令，拒绝验证评分并将其显示为阻塞性发现。
- 如果仓库没有版本控制历史，拒绝交接评分并将其显示为阻塞性发现。
- 如果 Agent 产品以 root 身份运行或拥有不受限制的文件访问权限，在沙箱或写入列表定义之前拒绝范围评分。

输出结构：

```
workbench-audit/
├── workbench_audit.json
├── workbench_audit.md
├── patches/
│   └── <最弱面>.patch
└── README.md
```

- 第 32 课了解最小仓库布局。
- 第 33 课深入了解指令面。
- 第 38 课了解验证门。
