# 使用 HTN 与进化搜索进行规划

> 符号规划处理那些计划可以被证明正确的情况。进化代码搜索处理那些适应度函数可被机器检查的情况。ChatHTN（2025）和 AlphaEvolve（2025）展示了各自在与 LLM 配对时能够解锁什么能力。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 02（ReWOO 与计划-执行）
**时间：** 约 75 分钟

## 学习目标

- 解释层次化任务网络：任务、方法、操作符、前置条件、效果。
- 描述 ChatHTN 的混合循环——符号搜索配合 LLM 回退分解。
- 解释 AlphaEvolve 的进化循环，以及为什么它只能在具有编程式评估器的情况下工作。
- 使用标准库实现一个玩具版 HTN 规划器和一个玩具版进化搜索。

## 问题

ReWOO（第 02 课）、计划-执行和 ReAct 覆盖了大多数 Agent 规划。但仍有两种情况它们处理得不够好：

1. **需要可证明正确性的计划。** 调度、航线规划、合规工作流——计划必须在构造上是正确的。一个流畅的 LLM 计划偶尔会出现幻觉步骤是不可接受的。
2. **具有可机器检查的适应度函数的优化。** 矩阵乘法、调度启发式、编译器优化——目标不是"一个正确的计划"而是"最佳的计划"。

HTN 规划和 AlphaEvolve 解决了这两个不同的问题。两者都将 LLM 作为放大器，而非替代品。

## 概念

### 层次化任务网络

HTN 的构成：

- **任务**——复合型（需要分解）和原始型（可直接执行）。
- **方法**——将复合任务分解为子任务的方式，带有前置条件。
- **操作符**——带有前置条件和效果的原始动作。
- **状态**——一组事实。

规划：给定一个目标任务和一个初始状态，找到一个按顺序满足前置条件的原始操作符分解方案。

HTN 比 LLM 更古老，至今仍是可证明正确计划的参考方案。

### ChatHTN（Gopalakrishnan 等人，2025）

ChatHTN（arXiv:2505.11814）将符号 HTN 与 LLM 查询交错进行：

1. 尝试用现有方法分解当前复合任务。
2. 如果没有方法适用，询问 LLM："在状态 `s` 中，你会如何分解任务 `T`？"
3. 将 LLM 响应翻译为候选子任务。
4. 对照操作符模式进行验证；拒绝无效的分解。
5. 递归。

论文的核心主张：生成的每个计划都是可证明正确的，因为 LLM 的建议只作为候选分解进入，从不直接修改计划。符号层拥有正确性；LLM 扩展方法库。

在线方法学习（OpenReview `gwYEDY9j2x`，2025 年后续研究）增加了一个学习器，通过回归泛化 LLM 产生的分解——将 LLM 查询频率降低高达 75%。

### AlphaEvolve（Novikov 等人，2025）

AlphaEvolve（arXiv:2506.13131，DeepMind，2025 年 6 月）是一个不同的生物：由 Gemini 2.0 Flash/Pro 集成协调的进化代码搜索。

循环：

1. 从一个种子程序和一个编程式评估器（返回适应度评分）开始。
2. LLM 集成提出变异。
3. 通过评估器运行变异。
4. 保留最佳结果；再次变异。

已发表的成果：

- 56 年来首次对 Strassen 算法在 4x4 复数矩阵乘法上的改进（48 次标量乘法）。
- 通过 Borg 调度启发式恢复了 0.7% 的 Google 计算资源。
- 在前沿工作负载上实现 32% 的 FlashAttention 加速。

硬性约束：适应度函数必须是可机器检查的。对散文答案进行进化搜索不会收敛。

### 何时使用哪种方法

| 问题类别 | 使用方法 | 原因 |
|---------------|-----|-----|
| 带有硬约束的调度 | HTN + ChatHTN | 可证明的正确性 |
| 编译器优化 | AlphaEvolve | 可机器检查的适应度 |
| 多步骤任务执行 | ReAct / ReWOO | LLM 在循环中，无形式化保证 |
| 带有测试的代码改进 | AlphaEvolve | 测试就是评估器 |
| 策略约束的自动化 | HTN | 前置条件编码策略 |

### 这种模式的陷阱

- **没有操作符的 HTN。** 没有前置条件/效果模式，正确性的主张就崩溃了。ChatHTN 的"LLM 建议分解"需要模式来拒绝无效动作。
- **没有真正评估器的 AlphaEvolve。** "问 LLM 代码是否更好"不是适应度函数。评估器必须是确定性的且快速的。
- **过度工程化。** 大多数 Agent 任务两者都不需要。先选择 ReAct 或 ReWOO。

## 动手构建

`code/main.py` 实现了两个玩具示例：

- 一个标准库 HTN 规划器，包含操作符、方法、前置条件、效果和一个在没有方法匹配复合任务时启用的 `LLMFallback`。"LLM"是一个脚本化的分解器，因此规划器可以离线运行。
- 一个标准库进化搜索，针对算术程序：演化出在测试集上使 `|f(x) - target|` 最小化的表达式。评估器是确定性的。

运行方式：

```
python3 code/main.py
```

跟踪输出显示 HTN 规划器分解复合任务（中间使用 LLM 回退），以及进化循环收敛到目标表达式。

## 使用场景

- **HTN 规划器**——`pyhop`、`SHOP3`，或为领域特定策略执行自行构建。
- **ChatHTN**——研究代码；该模式（符号 + LLM 回退）可干净地移植到任何 HTN 规划器。
- **AlphaEvolve**——DeepMind 论文；该模式（集成 + 评估器）是可复现的。OpenEvolve 和类似的开源分支正在出现。
- **Agent 框架**——目前还没有一个将 HTN 或 AlphaEvolve 作为一等公民提供。可将其构建为子 Agent 或后台工作器。

## 练习

1. 扩展 HTN 规划器以支持回溯：当操作符的后置条件在运行时失败时，回滚并尝试下一个方法。
2. 为 ChatHTN 添加 LLM 方法缓存：当 LLM 在状态模式 `P` 下分解任务 `T` 时，存储结果。下次调用时先检查方法库。
3. 将进化搜索的评估器替换为真实的测试套件。演化一个通过 20 个测试用例的排序函数；报告收敛所需的代数。
4. 阅读 AlphaEvolve 的评估器设计笔记。为你关心的领域设计一个评估器（SQL 查询优化、测试套件最小化、部署 YAML）。
5. 组合使用：用 HTN 将复合任务分解为子任务，然后在每个子任务的原始操作符上使用进化搜索。它在哪些场景下表现优异，哪些场景下过度设计？

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|------|----------------|------------------------|
| HTN | "层次化规划器" | 带有操作符、前置条件、效果的任务分解 |
| 方法 | "分解规则" | 将复合任务分解为子任务的方式 |
| 操作符 | "原始动作" | 带有前置条件和效果的具体步骤 |
| ChatHTN | "LLM + HTN" | 符号规划器在没有方法匹配时询问 LLM |
| AlphaEvolve | "进化代码搜索" | 集成 LLM 变异代码；确定性评估器选择 |
| 适应度函数 | "评估器" | 确定性的、可机器检查的输出评分 |
| 在线方法学习 | "缓存的 LLM 分解" | 存储 + 泛化 LLM 计划以降低查询成本 |
