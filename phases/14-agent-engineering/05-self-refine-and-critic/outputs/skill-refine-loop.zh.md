给定一个任务、一个迭代预算和可用的验证器（基于工具或仅自评估），输出一个评估器-优化器循环的提示和停止策略。

输出：

1. 生成器提示。用于第一个输出的确定性生产者。明确说明任务、输出格式和约束。
2. 评估器/验证器提示。如果工具可用（搜索、代码运行、测试、计算器、类型检查），指定如何调用它们以及如何产生结构化批评（JSON 格式：pass/fail、violations[]、suggested_fixes[]）。如果仅自评估可用，明确标记 Self-Refine 的走过场风险并使用结构不同的提示风格（例如对抗性的"至少找出一个缺陷"）。
3. 改进器提示。必须引用先前的输出和批评（历史记录）。声明"不重复先前迭代中标记的失败模式"是强制性的。
4. 停止策略。组合条件：验证器通过 或（自评估说没问题且迭代次数 >= 2）或 迭代次数 >= max_iterations。永远不要单条件。
5. 可观测性钩子。将每次迭代记录为 OpenTelemetry GenAI 跨度（evaluate、optimize）per 第 23 课，使整个改进轨迹是可审计的。

硬性拒绝：

- 生成器和批评者使用相同提示。走过场风险 —— 模型与自身达成一致。
- 没有迭代上限。无限改进循环会燃烧令牌；默认总是限制为 4。
- 请求自由形式散文反馈的验证器提示。仅限结构化 JSON —— pass/fail 加上逐项违规。
- 从改进器提示中丢弃历史。论文显示没有历史记录时质量崩溃。

拒绝规则：

- 如果任务没有验证器也无法构建一个，拒绝 CRITIC 并指出 Self-Refine 是可用但较弱的选项 —— 警告用户走过场风险。
- 如果 max_iterations >= 10，拒绝并建议重新架构任务。超过 3-4 次改进仍未收敛通常意味着生成器提示有误。
- 如果验证器调用破坏性工具（shell、git write），拒绝并要求沙箱边界（第 09 课）。

输出：一个包含所有提示、停止策略和工具列表的单一配置块，加上一个"接下来阅读什么"的说明，根据部署目标指向第 16 课（OpenAI Agents SDK 护栏）、第 12 课（Anthropic 评估器-优化器）或第 30 课（评估驱动的 agent 开发）。
