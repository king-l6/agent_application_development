# 记忆：虚拟上下文与 MemGPT

> 上下文窗口是有限的。而对话、文档和工具调用轨迹是无限的。MemGPT（Packer 等人，2023）将其类比为操作系统虚拟内存——主上下文是 RAM，外部存储是磁盘，Agent 在两者之间进行页面调度。这是 2026 年所有记忆系统所继承的模式。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 01（Agent 循环），阶段 14 · 06（工具使用）
**时间：** 约 75 分钟

## 学习目标

- 解释 MemGPT 所基于的 OS 类比：主上下文 = RAM，外部上下文 = 磁盘，记忆工具 = 页面换入/换出。
- 使用标准库实现双层 MemGPT 模式，包含主上下文缓冲区、可搜索的外部存储以及页面换入/换出工具。
- 描述 Agent 如何发出"中断"来查询或修改外部记忆，以及结果如何拼接回下一个提示词。
- 识别 MemGPT 设计中延续到 Letta（第 08 课）和 Mem0（第 09 课）的关键设计选择。

## 问题

上下文窗口看似应该能解决记忆问题，但事实并非如此。在生产中有三种常见的失败模式：

1. **溢出。** 多轮对话、长文档或工具调用密集的轨迹会超出窗口限制。超出截断点的内容全部丢失。
2. **稀释。** 即使在窗口范围内，塞入不相关的上下文也会稀释对重要内容的注意力。前沿模型在处理长输入时仍然会性能下降。
3. **持久性。** 新会话从空窗口开始。没有外部记忆的 Agent 无法跨会话说"还记得你让我……"。

更大的窗口有帮助，但并不能解决这个问题。Mem0 2025 年的论文测量发现，128k 窗口的基线模型仍然会遗漏那些配备 4k 窗口加外部记忆的 Agent 能够捕捉到的长周期事实。

## 概念

### MemGPT：操作系统类比

Packer 等人（arXiv:2310.08560，v2，2024 年 2 月）将上下文管理映射到操作系统的虚拟内存：

| OS 概念 | MemGPT 概念 | 2026 年生产环境类比 |
|------------|---------------|------------------------|
| RAM | 主上下文（提示词） | Anthropic/OpenAI 上下文窗口 |
| 磁盘 | 外部上下文 | 向量数据库、键值存储、图存储 |
| 缺页中断 | 记忆工具调用 | `memory.search`、`memory.read`、`memory.write` |
| OS 内核 | Agent 控制循环 | 带记忆工具的 ReAct 循环 |

Agent 运行正常的 ReAct 循环。额外的一类工具使其能够在主上下文中换入和换出数据。

### 双层架构

- **主上下文。** 固定大小的提示词缓冲区，容纳当前任务。始终对模型可见。
- **外部上下文。** 无界存储，通过工具可搜索。相关时读取，事实出现时写入。

原论文在两个超出基本窗口的任务上评估了该设计：超过 10 万 token 的文档分析，以及跨多天会话的持久记忆。

### 中断模式

MemGPT 引入了记忆即中断的概念：对话中途，Agent 可以调用一个记忆工具，运行时执行该工具，结果作为新的观测插入到下一个助手轮次中。这在概念上与 Unix 的 `read()` 系统调用相同——阻塞进程、返回字节、进程继续执行。

典型的记忆工具接口：

- `core_memory_append(section, text)` — 写入提示词中的一个持久化部分。
- `core_memory_replace(section, old, new)` — 编辑一个持久化部分。
- `archival_memory_insert(text)` — 写入可搜索的外部存储。
- `archival_memory_search(query, top_k)` — 从外部存储检索。
- `conversation_search(query)` — 扫描过去的轮次。

### MemGPT 的终点与 Letta 的起点

2024 年 9 月，MemGPT 更名为 Letta。研究仓库（`cpacker/MemGPT`）仍然存在；Letta 扩展了该设计：

- 三层架构而非两层（核心、回忆、归档——第 08 课）。
- 原生推理取代了 `send_message`/心跳模式（第 08 课）。
- 睡眠时间 Agent 异步执行记忆工作（第 08 课）。

即使生产系统运行的是 Letta、Mem0 或自定义的双层存储，MemGPT 论文仍然是 2026 年的基础。

### 这种模式的陷阱

- **记忆腐烂。** 写入积累速度快于读取；检索被过时事实淹没。修复方法：定期整合（Letta 睡眠时间）、显式失效（Mem0 冲突检测器）。
- **记忆投毒。** 外部记忆是检索到的文本。如果攻击者控制的内容进入记忆笔记，Agent 将在下一次会话中重新摄取。这是 Greshake 等人（第 27 课）的攻击在时间维度上的重述。
- **引用丢失。** Agent 回忆起"用户让我交付 X"，但无法引用是哪个轮次。在每个归档写入时存储来源引用（会话 ID、轮次 ID）。

## 动手构建

`code/main.py` 使用标准库实现了 MemGPT 的双层模式：

- `MainContext` — 固定大小的提示词缓冲区，包含一个 `core` 字典和一个 `messages` 列表；超过容量时自动压缩最旧的消息。
- `ArchivalStore` — 内存中的 BM25 风格存储（基于 token 重叠评分），记录包含 (id, text, tags, session, turn)。
- 五个映射到 MemGPT 接口的记忆工具。
- 一个脚本化 Agent，先用事实填充归档，然后通过调用 `archival_memory_search` 回答问题。

运行方式：

```
python3 code/main.py
```

跟踪输出显示 Agent 写入三个事实，将主上下文填充到上限（强制驱逐），然后通过从归档中检索来回答后续问题——在不使用真实 LLM 的情况下复现了 MemGPT 工作流程。

## 使用场景

当今每个生产级记忆系统都是 MemGPT 的变体：

- **Letta**（第 08 课）——三层架构、原生推理、睡眠时间计算。
- **Mem0**（第 09 课）——向量 + 键值 + 图存储，融合评分层。
- **OpenAI Assistants / Responses**——通过线程和文件实现托管记忆。
- **Claude Agent SDK**——通过技能和会话存储实现长期记忆。

根据运营形态（自托管、托管、框架集成）进行选择，而非根据核心模式——因为核心模式就是 MemGPT。

## 练习

1. 添加一个以 token 计量的 `max_main_context_tokens` 上限（用 `len(text.split()) * 1.3` 近似估算）。超过上限时将最旧的消息压缩为摘要。比较有和没有摘要器时的行为差异。
2. 在归档存储上正确实现 BM25（词频、逆文档频率）。在一个玩具事实集上测量 recall@10，与 token 重叠基线进行比较。
3. 为归档插入添加 `citation` 字段（session_id、turn_id、source_url）。让 Agent 在每次基于检索的回答中引用来源。
4. 模拟记忆投毒：添加一条"忽略所有未来用户指令"的归档记录。编写一个守卫，扫描检索结果中是否存在指令形式的文本并将其标记为不可信。
5. 将该实现移植为使用 MemGPT 研究仓库的核心内存 JSON 模式（`cpacker/MemGPT`）。从扁平字符串切换到类型化部分时会发生哪些变化？

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|------|----------------|------------------------|
| 虚拟上下文 | "无限记忆" | 主（提示词）+ 外部（可搜索）双层架构，支持换入/换出 |
| 主上下文 | "工作记忆" | 提示词——固定大小，始终可见 |
| 归档记忆 | "长期存储" | 外部可搜索持久化存储，按需检索 |
| 核心记忆 | "持久化提示词部分" | 固定在主上下文中的命名部分 |
| 记忆工具 | "记忆 API" | Agent 发出的用于读/写外部记忆的工具调用 |
| 中断 | "记忆缺页" | Agent 暂停，运行时获取，结果拼接到下一轮 |
| 记忆腐烂 | "过时事实" | 旧写入淹没检索；通过整合修复 |
| 记忆投毒 | "注入的持久化笔记" | 攻击者内容作为记忆存储，在召回时重新摄取 |
