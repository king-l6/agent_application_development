# 编排模式：监督器、群体、分层

> 四种编排模式在 2026 年框架中反复出现：监督器-工作者、群体/点对点、分层、辩论。Anthropic 的指导："关键在于为您的需求构建正确的系统。"从简单开始；仅在单一智能体加上五种工作流模式不足以解决问题时才添加拓扑。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 12（工作流模式），阶段 14 · 25（多智能体辩论）
**时间：** ~60 分钟

## 学习目标

- 说出四种反复出现的编排模式以及各自适用的情况。
- 描述 2026 年 LangChain 推荐：基于工具调用的监督 vs 监督器库。
- 解释 Anthropic 的"构建正确的系统"规则以及它如何约束拓扑选择。
- 在标准库中针对通用模拟 LLM 实现全部四种模式。

## 问题

团队在真正需要之前就急于采用"多智能体"。四种模式在不同框架中反复出现；一旦能命名它们，就可以选择正确的一种——或者完全跳过拓扑。

## 概念

### 监督器-工作者

- 中央路由 LLM 将任务分发给专业智能体。
- 决定：返回自身、交接给专业智能体、终止。
- 专业智能体之间不互相通信；所有路由都通过监督器。

框架：LangGraph `create_supervisor`、Anthropic 编排器-工作者、CrewAI 分层流程。

**2026 年 LangChain 推荐：** 通过直接工具调用执行监督，而不是使用 `create_supervisor`。提供更精细的上下文工程控制——您精确决定每个专业智能体看到什么。

### 群体/点对点

- 智能体通过共享的工具表面直接交接。
- 无中央路由器。
- 比监督器延迟更低（跳数更少）。
- 更难推理（没有单一控制点）。

框架：LangGraph 群体拓扑、OpenAI Agents SDK 交接（当所有智能体可以相互交接时）。

### 分层

- 监督器管理子监督器，子监督器管理工作者。
- 在 LangGraph 中实现为嵌套子图；在 CrewAI 中实现为嵌套 crew。
- 能扩展到大智能体群体，代价是操作复杂性。

何时需要：当单个监督器的上下文预算无法容纳所有专业智能体的描述时。

### 辩论

- 并行提议者 + 迭代交叉批评（第 25 课）。
- 不完全是编排——更像是验证——但在框架中作为拓扑选择出现。

### CrewAI Crew vs Flow

CrewAI 形式化了两种部署模式：

- **Flow**用于确定性事件驱动自动化（推荐的生产起点）。
- **Crew**用于自主基于角色的协作。

这与上述四种模式正交，但映射到拓扑：Flow 通常是监督器或分层；Crew 通常是带有 LLM 路由器的监督器。

### Anthropic 的指导

"在 LLM 领域，成功不在于构建最复杂的系统，而在于为您的需求构建正确的系统。"

决策顺序：

1. 单一智能体 + 工作流模式（第 12 课）——从这里开始。
2. 监督器-工作者——当有 2-4 个专业智能体时。
3. 群体——当延迟比推理清晰度更重要时。
4. 分层——仅当监督器上下文预算不足时。
5. 辩论——当准确率比成本更重要时。

### 此模式出错的地方

- **拓扑优先思维。** "我们需要多智能体"之前没有识别多智能体要解决什么问题。
- **群体中的来回交接。** A -> B -> A -> B。使用跳数计数器。
- **虚假分层。** 因为"企业级"而设三层；实际只有两个团队。精简。

## 构建

`code/main.py` 在标准库中针对模拟 LLM 实现了全部四种模式：

- `Supervisor`——中央路由器。
- `Swarm`——点对点直接交接。
- `Hierarchical`——监督器的监督器。
- `Debate`——并行提议者 + 批评。

每种模式处理相同的三种意图任务（退款/错误/销售）。追踪形状不同。

运行：

```
python3 code/main.py
```

输出：每种模式的追踪 + 操作计数。监督器最清晰；群体最短；分层最深；辩论最昂贵。

## 使用

- **LangGraph**用于监督器和分层（嵌套子图）。
- **OpenAI Agents SDK**用于作为工具的交接（监督器形态）。
- **CrewAI Flow**用于生产确定性流程。
- **自定义**用于辩论或当您想精确控制时。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| 监督器-工作者 | "路由器 + 专业智能体" | 中央 LLM 派发给专业智能体；它们不互相通信 |
| 群体 | "点对点" | 通过共享工具直接交接；无中央路由器 |
| 分层 | "监督器的监督器" | 针对大型群体的嵌套子图 |
| 辩论 | "提议者 + 批评" | 并行提议者，交叉批评（第 25 课） |
| 基于工具调用的监督 | "无需库的监督器" | 将监督器实现为直接工具调用以获得上下文控制 |
| Crew | "自主团队" | CrewAI 的基于角色的协作模式 |
| Flow | "确定性工作流" | CrewAI 的事件驱动生产模式 |
