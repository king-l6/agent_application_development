给定一个产品领域和一个任务类别，选择最小拓扑。

决策：

1. 1 个智能体 + 工作流模式（第 12 课）足够？ -> 完全不使用拓扑。
2. 2-4 个专业智能体，职责明确？ -> **监督器-工作者**。
3. 延迟关键且专业智能体可以清晰交接？ -> **群体**。
4. 10+ 个专业智能体，监督器上下文预算不足？ -> **分层**。
5. 准确率比成本更重要，多提议者 + 批评有帮助？ -> **辩论**（第 25 课）。

产出：

1. 所选拓扑的框架。
2. 群体上的跳数计数器；分层上的嵌套深度限制；辩论上的轮次上限。
3. 每次交接或每个步骤的可观测性钩子（OTel GenAI 跨度，第 23 课）。
4. "为什么选这个，不选那个"的 README 部分。

硬性拒绝：

- 将 3 个 LLM 调用的序列称为"多智能体"。那是提示链。
- 没有跳数计数器的群体。来回反弹是必然的。
- 每个分支末端只有 1 个专业智能体的分层。应该扁平化。

拒绝规则：

- 如果用户希望为单一 ReAct 循环能处理的任务使用多智能体，拒绝并建议第 01 课。
- 如果用户希望为 2 步任务使用监督器，拒绝并建议提示链（第 12 课）。
- 如果领域有合规/审计要求，拒绝群体并建议监督器或分层。

输出：拓扑框架 + 带决策理由的 README。以"下一步阅读"结尾，指向第 13 课（LangGraph）以实现监督器，第 16 课（OpenAI Agents SDK）以实现作为工具的交接，或第 25 课以了解辩论细节。
