# 智能体可观测性：Langfuse、Phoenix、Opik

> 2026年三大开源智能体可观测性平台占据主导地位。Langfuse（MIT）— 每月600万+安装量，追踪+提示管理+评估+会话回放。Arize Phoenix（Elastic 2.0）— 深度智能体特定评估、RAG相关性、OpenInference自动工具化。Comet Opik（Apache 2.0）— 自动提示优化、防护栏、LLM评判幻觉检测。

**类型：** 学习
**语言：** Python（标准库）
**前置知识：** 第14阶段·23（OTel GenAI）
**时间：** 约45分钟

## 学习目标

- 说出三大开源智能体可观测性平台及其许可证。
- 区分各自最强的领域：Langfuse（提示管理+会话）、Phoenix（RAG+自动工具化）、Opik（优化+防护栏）。
- 解释为什么到2026年89%的组织报告已经部署了智能体可观测性。
- 实现一个带有LLM评判评估的stdlib追踪到仪表板的管道。

## 问题

OTel GenAI（第23课）提供了模式。你仍然需要能够摄取跨度、运行评估、存储提示版本和发现回归的平台。三个竞争者各自强调生命周期的不同部分。

## 概念

### Langfuse（MIT）

- 每月600万+ SDK安装量，19k+ GitHub星标。
- 功能：追踪、带有版本控制+游乐场的提示管理、评估（LLM作为评判、用户反馈、自定义）、会话回放。
- 2025年6月：以前商业模块（LLM作为评判、注释队列、提示实验、游乐场）在MIT下开源。
- 最强领域：带有紧密提示管理循环的端到端可观测性。

### Arize Phoenix（Elastic License 2.0）

- 更深入的智能体特定评估：追踪聚类、异常检测、RAG检索相关性。
- 原生OpenInference自动工具化。
- 与托管的Arize AX配对用于生产。
- 没有提示版本管理 — 定位为与更广泛平台配合使用的漂移/行为回归工具。
- 最强领域：RAG相关性、行为漂移、异常检测。

### Comet Opik（Apache 2.0）

- 通过A/B实验进行自动提示优化。
- 防护栏（PII编辑、主题约束）。
- LLM评判幻觉检测。
- 来自Comet自身测量的基准：Opik日志+评估23.44秒 vs Langfuse 327.15秒（约14倍差距）——将厂商基准视为方向性参考。
- 最强领域：优化循环、自动实验、防护栏执行。

### 行业数据

根据Maxim（2026年实地分析）：89%的组织已部署智能体可观测性；质量问题是最主要的生产障碍（32%的受访者提及）。

### 选择指南

| 需求 | 选择 |
|------|------|
| 一体式带提示管理 | Langfuse |
| 深度RAG评估+漂移 | Phoenix |
| 自动优化+防护栏 | Opik |
| 开放许可，无ELv2 | Langfuse（MIT）或 Opik（Apache 2.0） |
| Datadog / New Relic集成 | 任意——它们都导出OTel |

### 这种模式失败的地方

- **没有评估策略。** 没有评估的追踪只是昂贵的日志记录。
- **自建LLM评判没有依据。** CRITIC模式（第05课）适用——评判需要外部工具进行事实验证。
- **提示版本未关联到追踪。** 当生产出现回归时，你无法二分定位到导致问题的提示。

## 动手构建

`code/main.py` 实现了一个stdlib追踪收集器+LLM评判评估器：

- 摄取GenAI格式的跨度。
- 按会话分组，标记失败运行（防护栏触发、低置信度评估）。
- 一个脚本化的LLM评判，根据评分标准对智能体响应打分。
- 类似仪表板的摘要：失败率、主要失败原因、评估分数分布。

运行它：

```
python3 code/main.py
```

输出：每个会话的评估分数和失败分类，匹配Langfuse/Phoenix/Opik会显示的内容。

## 使用建议

- **Langfuse** 自托管或云端；通过OTel或其SDK接入。
- **Arize Phoenix** 自托管；自动工具化OpenInference。
- **Comet Opik** 自托管或云端；自动优化循环。
- **Datadog LLM可观测性** 适用于已经运行Datadog的混合运维+ML团队。

## 练习

1. 将一周的OTel追踪导出到Langfuse云端（免费层级）。哪些会话失败了？为什么？
2. 为你的领域编写一个LLM评判评分标准（事实正确性、语气、范围遵守）。在50条追踪上测试。
3. 比较Langfuse提示版本管理与Phoenix的追踪聚类。哪个能更快告诉你什么出了问题？
4. 阅读Opik的防护栏文档。将一个PII编辑防护栏接入你的一个智能体运行。
5. 在你的语料库上基准测试这三个平台。忽略厂商发布的数字；衡量你自己的。

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| 追踪 | "跨度收集器" | 摄取OTel / SDK跨度；按会话索引 |
| 提示管理 | "提示CMS" | 版本化的提示，关联到追踪 |
| LLM作为评判 | "自动评估" | 单独的LLM根据评分标准对智能体输出评分 |
| 会话回放 | "追踪回放" | 逐步回放过去运行以进行调试 |
| RAG相关性 | "检索质量" | 检索到的上下文是否匹配查询 |
| 追踪聚类 | "行为分组" | 对相似运行进行聚类以检测漂移 |
| 防护栏执行 | "日志时策略" | 对记录内容的PII/毒性/范围检查 |
