给定一个任务领域，生成一个 agent 可以在 BFCL V4 各维度（代理型、多轮、真实、非真实、幻觉检测）上可靠使用的工具目录。

输出：

1. 工具定义。对每个工具：`name`（snake_case）、`description`（告诉模型何时使用以及何时不使用）、JSON Schema 输入，带有类型化属性、必填字段、适用的枚举、数值的最小值/最大值、每工具超时时间、每工具沙箱策略（文件系统范围、网络、内存上限）。
2. 描述质量检查。对每条描述运行"这是否告诉模型何时选择该工具而非其他工具？"如果两个工具有重叠的描述，拒绝并重写。
3. 并行分派规划。对每个现实任务，识别哪些工具调用是独立的（可并行化）以及哪些必须是顺序的。输出预期的分派图。
4. 验证策略。枚举检查、类型强制转换规则（例如"接受 int-as-string，拒绝 float-as-string"）、必填字段强制执行。每次失败返回结构化观察字符串，绝不抛到循环中。
5. 可观测性。每个工具发射一个带有属性 `gen_ai.tool.name`、`gen_ai.tool.call.id`、`gen_ai.tool.call.arguments`、`gen_ai.tool.call.result`（当内容策略要求时使用引用而非内联）的 OpenTelemetry GenAI `tool_call` 跨度。

硬性拒绝：

- 通用的 shell/命令执行工具。拒绝并拆分为具体动词（`git_status`、`fs_read`、`npm_test`）。
- 当参数有封闭值集合时缺少枚举。枚举验证是捕获漂移最便宜的方式。
- 两个不同的工具有相同的描述。模型无法可靠地在它们之间进行选择。
- `description` 只命名工具（"加两个数"）。要包括何时选择它而非其他选项。
- 没有超时时间。每个工具调用必须有上限。

拒绝规则：

- 如果单个 agent 的工具列表超过 30 个工具，拒绝并建议子 agent 委派（第 17 课）。
- 如果任何工具执行破坏性操作而没有确认门控，拒绝并指向第 09 课（权限、沙箱）。
- 如果任务是计算机使用（点击、输入、截图），拒绝并指向第 21 课 —— 那是另一种具有基于视觉动作的工具形态。

输出：一个准备好粘贴到 Anthropic / OpenAI / Gemini SDK 调用中的 JSON 工具目录、一个分派图图表、一个验证策略文档，以及注册表应通过的一个 BFCL 风格的小型评估。

以一个"接下来阅读什么"的说明结束：第 09 课（沙箱）、第 23 课（OTel GenAI 跨度）或第 30 课（评估驱动开发）。
