# 工具使用与函数调用

> Toolformer（Schick et al.，2023）开创了自我监督的工具标注。Berkeley Function Calling Leaderboard V4（Patil et al.，2025）设定了 2026 年的标准：40% 代理型、30% 多轮、10% 真实、10% 非真实、10% 幻觉检测。单轮调用已经解决。记忆、动态决策和长周期工具链尚未解决。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 01（Agent 循环），阶段 13 · 01（函数调用深入理解）
**时间：** ~60 分钟

## 学习目标

- 解释 Toolformer 的自我监督训练信号：仅当工具执行减少下一令牌损失时才保留工具标注。
- 说出 BFCL V4 的五个评估类别及其各自衡量的内容。
- 用标准库实现一个生产形态的工具注册表，包含模式验证、参数强制转换和执行沙箱。
- 诊断 2026 年的三个未解决问题：长周期工具链、动态决策和记忆。

## 问题

早期的工具使用问的是：模型能否预测正确的函数调用？现代的工具使用问的是：模型能否跨 40 步链接工具，带有记忆、不完全可观测性、从工具故障中恢复，且不幻觉出不存在的工具？

Toolformer 建立了基线：模型可以通过自我监督学习何时调用工具。BFCL V4 定义了 2026 年的评估目标。两者之间的差距正是生产 agent 所处的空间。

## 概念

### Toolformer（Schick et al.，NeurIPS 2023）

想法：让模型用自己的预训练语料库标注候选 API 调用。对每个候选，执行它。仅当包含工具结果能减少下一令牌的损失时才保留标注。在过滤后的语料库上微调。

覆盖的工具：计算器、问答系统、搜索引擎、翻译器、日历。自我监督信号纯粹是关于工具是否有助于预测文本 —— 没有人工标注。

规模结果：工具使用在规模上涌现。小模型因工具标注而受损；大模型从中获益。这就是为什么 2026 年的前沿模型内置了强大的工具使用能力，而大多数 7B 模型需要显式的工具使用微调才能可靠。

### Berkeley Function Calling Leaderboard V4（Patil et al.，ICML 2025）

BFCL 是 2026 年事实上的评估标准。V4 组成：

- **代理型（40%）** —— 完整的 agent 轨迹：记忆、多轮、动态决策。
- **多轮（30%）** —— 带工具链的交互式对话。
- **真实（10%）** —— 用户提交的真实提示（更难的数据分布）。
- **非真实（10%）** —— 合成测试用例。
- **幻觉检测（10%）** —— 检测何时不应调用工具。

V3 引入了基于状态的评估：在工具序列之后，检查 API 的实际状态（例如"文件是否已创建？"），而不是匹配工具调用的 AST。V4 增加了网络搜索、记忆和格式敏感性类别。

2026 年的关键发现：单轮函数调用几乎已经解决。失败集中在记忆（跨轮次携带上下文）、动态决策（基于先前结果选择工具）、长周期链（20+ 步后漂移）和幻觉检测（没有工具合适时拒绝调用）。

### 工具模式

每个提供商都有各自的模式。它们在细节上有所不同，但共享相同的形式：

```
name: string
description: string (what it does, when to use it)
input_schema: JSON Schema (properties, required, types, enums)
```

Anthropic 直接使用 `input_schema`。OpenAI 使用 `function.parameters`。两者都接受 JSON Schema。描述至关重要 —— 模型通过阅读描述来选择正确的工具。糟糕的工具描述是选错工具失败的首要原因。

### 参数验证

不要信任任何工具调用。验证：

1. **类型强制转换。** 模型可能返回字符串"5"而模式要求 int。如果无歧义则强制转换；否则拒绝。
2. **枚举验证。** 如果模式说 `status in {"open", "closed"}` 而模型输出 `"in_progress"`，用描述性错误信息拒绝。
3. **必填字段。** 缺少必填字段 -> 立即向模型返回错误观察，而不是崩溃。
4. **格式验证。** 日期、邮箱、URL —— 用具体的解析器验证，而不是正则表达式。

每次验证失败都应返回结构化观察结果，以便模型可以用正确格式重试。

### 并行工具调用

现代提供商支持一次助手轮次中的并行工具调用。循环：

1. 模型发出 3 个具有不同 `tool_use_id` 的工具调用。
2. 运行时执行它们（如果独立则并行）。
3. 每个结果作为由 `tool_use_id` 关联的 `tool_result` 块返回。

工程规则：将关联 ID 视为关键信息。交换它们会导致工具结果路由错误。

### 沙箱

工具执行是沙箱边界。详见第 09 课。简而言之：每个工具应指定读/写范围、网络访问、超时时间和内存上限。通用的 `run_shell(cmd)` 是一个危险信号；具体的 `git_status()` 更安全。

## 构建

`code/main.py` 实现了一个生产形态的工具注册表：

- JSON Schema 子集验证器（仅标准库）。
- 工具注册，包含描述、输入模式、超时和执行器。
- 参数强制转换和枚举验证。
- 带有相关 ID 的并行工具调度。
- 结构化字符串形式的错误观察。

运行：

```
python3 code/main.py
```

轨迹显示一个小型 agent 在一轮中调用三个工具，其中一个调用故意格式错误，被一个描述性错误拒绝，模型可以据此采取行动。

## 使用

每个提供商都有自己的工具模式 —— Anthropic、OpenAI、Gemini、Bedrock。如果需要多提供商支持，使用转换层（OpenAI Agents SDK、Vercel AI SDK、LangChain 工具适配器）。如果工具使用是产品的核心，BFCL 是参考基准 —— 在交付前针对你的 agent 运行它。

## 交付

`outputs/skill-tool-registry.md` 为给定任务领域生成工具目录、模式和注册表。包含描述质量检查（每个工具的描述是否告诉模型何时使用它？）。

## 练习

1. 添加一个"无操作"工具，让模型可以显式拒绝使用任何其他工具。在类似 BFCL 的幻觉测试上进行测量。
2. 实现 int-as-string 和 float-as-string 的参数强制转换。强制转换在什么情况下开始隐藏真正的 bug？
3. 添加每个工具的超时时间和一个熔断器（连续 3 次失败后拒绝该工具 60 秒）。这会如何改变模型的恢复方式？
4. 阅读 BFCL V4 描述。选择一个类别（例如"多轮"）并通过你的 agent 运行 10 个示例提示。报告通过率。
5. 将标准库验证器移植到 Pydantic 或 Zod。Pydantic/Zod 发现了什么玩具实现遗漏的问题？

## 关键术语

| 术语 | 人们说的 | 实际含义 |
|------|---------|---------|
| 函数调用 | "工具使用" | 带验证模式的结构化输出工具调用 |
| Toolformer | "自我监督工具标注" | Schick 2023 —— 保留结果能减少下一令牌损失的工具调用 |
| BFCL | "Berkeley Function Calling Leaderboard" | 2026 年基准：40% 代理型、30% 多轮、10% 真实、10% 非真实、10% 幻觉检测 |
| 工具模式 | "模型的函数签名" | 名称、描述、参数的 JSON Schema |
| tool_use_id | "关联 ID" | 将工具调用与其结果关联；对并行调度至关重要 |
| 幻觉检测 | "知道何时不调用" | V4 类别：没有工具合适时拒绝调用 |
| 参数强制转换 | "字符串到整数的修复" | 对可预测的模式不匹配进行窄修复；不明确时拒绝 |
| 沙箱 | "工具执行边界" | 每个工具的读/写范围、网络、超时、内存上限 |
