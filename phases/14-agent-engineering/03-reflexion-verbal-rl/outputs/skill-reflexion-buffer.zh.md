给定一个任务类别（重复类型的 agent 运行 —— 例如"重构一个函数"、"关闭一个支持工单"），维护一个情景记忆反思缓冲区。每条反思记录一个失败模式及用自然语言表述的纠正洞见。该缓冲区被预置到同一任务类别的下一次试验中。

输出：

1. 反思捕获。在一次试验以低于阈值的评估器分数结束后，发出一行形如"我未能做 X 因为 Y; 下次，做 Z。"的反思。丢弃对外部失败（网络、上游 500 错误）的反思，除非它们可重现。
2. TTL 和去重。反思在默认 N 次试验后过期（建议 10 次）。完全相同的重复项合并。近重复项（在小嵌入模型上余弦 > 0.9，或共享子串 >= 80%）只保留最近的。
3. 范围策略。三个范围：任务类别（按任务名称）、用户（同一用户跨任务）、agent（跨所有用户）。默认为任务类别。仅在反思涉及用户特定偏好时升级到用户范围；绝不自动升级到 agent 范围。
4. 压缩。当缓冲区超出预算时，运行睡眠时间压缩：聚类近重复项，摘要，合并。压缩在非热点路径上运行 —— 不要延迟主 agent 的响应。
5. 提示集成。发出一个标题为"我从先前试验中学到的"的单一区块，带项目符号列表。提示中最多 6 项；溢出项进入一个单独的摘要条目（"... 以及 4 条关于超时的旧反思"）。

硬性拒绝：

- 将反思写为"下次更小心。"这不是可执行的。用强制具体下一步指令的提示重新运行反射器。
- 基于挂钟时间而非试验次数使反思过期。TTL 应基于试验范围而非时间范围，以便离线回放运行。
- 存储引用密钥（API 密钥、令牌、PII）的反思。在提交到缓冲区之前，用特定的"包含密钥"类错误拒绝。

拒绝规则：

- 如果没有附加评估器，则拒绝并建议第 05 课（Self-Refine/CRITIC）—— 反思需要信号，而不是直觉。
- 如果任务类别是一次性的（从不重复），则拒绝；情景记忆对从不重复的任务毫无作用。

输出：一个结构化缓冲区文件（包含反思对象的 JSON：试验 id、任务类别、范围、文本、created_at、ttl_remaining），一个下一次试验的提示区块，以及一个列出即将过期条目的"过期反思"报告。

以一个"接下来阅读什么"的说明结束，如果缓冲区持续达到上限则指向第 06 课（上下文压缩），或将压缩移出热点路径则指向第 08 课（Letta 睡眠时间计算）。
