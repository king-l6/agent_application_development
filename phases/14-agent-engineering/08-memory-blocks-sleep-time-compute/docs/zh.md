# 记忆块与睡眠时间计算（Letta）

> MemGPT 在 2024 年演变为 Letta。2026 年的演进增加了两个理念：模型可以直接编辑的离散功能记忆块，以及在主 Agent 空闲时异步整合记忆的睡眠时间 Agent。这就是如何将记忆扩展到超越单次对话的方法。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 07（MemGPT）
**时间：** 约 75 分钟

## 学习目标

- 说出 Letta 使用的三个记忆层（核心、回忆、归档）及每层的作用。
- 解释记忆块模式：Human 块、Persona 块和用户自定义块作为一等类型化对象。
- 描述什么是睡眠时间计算，为什么它位于关键路径之外，以及为什么它可以运行比主 Agent 更强的模型。
- 实现一个脚本化的双 Agent 循环，其中主 Agent 服务响应，睡眠时间 Agent 在轮次之间整合记忆块。

## 问题

MemGPT（第 07 课）解决了虚拟内存的控制流程。但出现了三个生产环境问题：

1. **延迟。** 每次记忆操作都位于关键路径上。如果 Agent 在用户等待时需要进行修剪、摘要或协调，尾延迟会急剧增加。
2. **记忆腐烂。** 写入不断积累。矛盾的事实仍然存在。检索被过时内容淹没。
3. **结构缺失。** 扁平的归档存储无法表达"Human 块始终在提示词中；Persona 块始终在提示词中；Task 块每个会话不同"。

Letta（letta.com）是 2026 年的重写版本。记忆块使结构显式化；睡眠时间计算将整合移出关键路径。

## 概念

### 三层架构

| 层 | 范围 | 存储位置 | 写入方 |
|------|-------|----------------|------------|
| 核心 | 始终可见 | 主提示词内部 | Agent 工具调用 + 睡眠时间重写 |
| 回忆 | 对话历史 | 可检索 | 自动轮次记录 |
| 归档 | 任意事实 | 向量 + 键值 + 图 | Agent 工具调用 + 睡眠时间摄取 |

核心是 MemGPT 的核心。回忆是对话缓冲区及其被驱逐的尾部。归档是外部存储。这种拆分清理了 MemGPT 双层架构中的过载问题。

### 记忆块

记忆块是核心层中的类型化、持久化、可编辑部分。最初的 MemGPT 论文定义了两个：

- **Human 块**——关于用户的事实（姓名、角色、偏好、目标）。
- **Persona 块**——Agent 的自我认知（身份、语气、约束条件）。

Letta 将其泛化为任意用户自定义块：用于当前目标的 `Task` 块、用于代码库事实的 `Project` 块、用于硬约束的 `Safety` 块。每个块都有 `id`、`label`、`value`、`limit`（字符上限）、`description`（让模型知道何时编辑它）。

块通过工具接口进行编辑：

- `block_append(label, text)`
- `block_replace(label, old, new)`
- `block_read(label)`
- `block_summarize(label)`——压缩接近上限的块。

### 睡眠时间计算

2025 年 Letta 新增功能：在后台运行第二个 Agent，置于关键路径之外。睡眠时间 Agent 处理对话记录和代码库上下文，将 `learned_context` 写入共享块，并整合或失效归档记录。

随之而来的特性：

- **无延迟成本。** 主响应不需要等待记忆操作。
- **允许更强模型。** 睡眠时间 Agent 可以使用更昂贵、更慢的模型，因为它不受延迟约束。
- **自然的整合窗口。** 在用户不等待时进行去重、摘要、失效矛盾事实。

这种模式与人类的工作方式相符：你执行任务，你睡一觉，长期记忆在一夜之间沉淀。

### Letta V1 与原生推理

Letta V1（`letta_v1_agent`，2026）废弃了 `send_message`/心跳和内联 `Thought:` token，转而使用原生推理。Responses API（OpenAI）和带有扩展思考功能的 Messages API（Anthropic）在独立通道上发出推理，并在各轮次之间传递（生产环境中跨提供商加密）。控制循环仍然是 ReAct。思考轨迹是结构化的，而非提示词形状的。

### 这种模式的陷阱

- **块膨胀。** 无限的 `block_append` 会很快触及限制。在即将超过上限的写入之前接入一个块摘要器。
- **静默漂移。** 睡眠时间 Agent 重写了一个块，而主 Agent 从未察觉。对块进行版本控制并在跟踪中显示差异。
- **被投毒的整合。** 睡眠时间 Agent 将攻击者可访问的内容处理到核心中。第 27 课同样适用于睡眠时间接口。

## 动手构建

`code/main.py` 实现了：

- `Block` — id、label、value、limit、description。
- `BlockStore` — CRUD + `near_limit(label)` 辅助方法。
- 两个脚本化 Agent — `PrimaryAgent` 服务一轮对话，`SleepTimeAgent` 在轮次之间进行整合。
- 一个跟踪输出，展示了三轮对话的块写入，以及一次睡眠时间处理——对块进行摘要并使一个过时事实失效。

运行方式：

```
python3 code/main.py
```

日志显示了分工：主轮次快速执行并产生原始写入；睡眠处理进行压缩和清理。

## 使用场景

- **Letta**（letta.com）作为参考实现。自托管或使用托管云。
- **Claude Agent SDK 技能**作为块状知识——技能是一个命名的、带版本号的、可检索的指令块，Agent 按需加载。
- **自定义构建**适用于希望控制存储后端的团队。使用 Letta API 契约，以便后续可以迁移。

## 交付产物

`outputs/skill-memory-blocks.zh.md` 为任何运行时生成一个 Letta 风格的块系统，包含睡眠时间钩子、安全规则和引用线路。

## 练习

1. 添加一个 `block_summarize` 工具，当 `near_limit` 返回 true 时，用模型生成的摘要替换块的值。哪个触发阈值能在摘要调用次数和块溢出之间取得最佳平衡？
2. 在归档上实现睡眠时间去重：文本 token 重叠超过 90% 的两条记录合并为一条。仅在睡眠处理中执行，绝不放在关键路径上。
3. 对块进行版本控制。每次写入时记录旧值和差异。暴露 `block_history(label)`，以便运维人员可以调试"为什么 Agent 忘记了 X"。
4. 将睡眠时间 Agent 视为不可信写入者。当它触及 Persona 或 Safety 块时，需要第二个 Agent 审查后才能提交。
5. 将示例移植为使用 Letta API（`letta_v1_agent`）。块模式有哪些变化？原生推理如何改变跟踪输出的形状？

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|------|----------------|------------------------|
| 记忆块 | "可编辑的提示词部分" | 类型化、持久化、LLM 可编辑的核心记忆片段 |
