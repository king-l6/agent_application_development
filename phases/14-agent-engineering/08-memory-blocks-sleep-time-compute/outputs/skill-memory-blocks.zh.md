给定一个目标运行时、一个主模型和一个（可能更强的）睡眠时间模型，生成一个具有显式块类型和异步整合的三层记忆系统。

生成内容：

1. `Block` 类型，包含 `label`、`value`、`limit`、`description`、`version`、`history`。每次写入都会增加版本号并记录旧值。暴露 `near_limit(threshold=0.8)`。
2. 一个 `BlockStore`，至少包含三个默认块：`human`（关于用户的事实）、`persona`（Agent 自我认知）和 `task`（当前范围）。允许用户自定义块。
3. 一个 `Recall` 存储——按会话分页的轮次日志。自动写入每一轮。尾部在达到上限时驱逐但仍可检索。
4. 一个 `Archival` 存储——至少两个后端（向量、键值）。插入返回记录 id。矛盾时使记录失效而非删除。
5. 一个 `PrimaryAgent`，处理轮次并仅发出原始写入。关键路径上不进行摘要。
6. 一个 `SleepTimeAgent`，在轮次之间运行：对超过阈值的块进行摘要、使矛盾的归档记录失效、将 `learned_context` 写入共享块。

硬性拒绝：

- 在用户面对轮次期间同步执行的任何记忆操作，但直接查找除外。摘要、整合、失效属于睡眠时间处理。
- 在产生矛盾时删除归档记录。应使其失效以保持历史可审计。
- 未经审查步骤向 Persona 或 Safety 块写入数据。这些块全局性地塑造行为；静默写入会掩盖 bug。

拒绝规则：

- 如果运行时无法跨会话持久化块，拒绝发布被描述为具有"记忆"的产品。降低声明的等级。
- 如果睡眠时间 Agent 没有跟踪输出，拒绝。静默整合是调试的死区。
- 如果用户要求"不做失效，永远信任最新写入"，在历史性声明重要的领域（合规、医疗、法律）中拒绝。

输出：每个组件一个文件，外加一个 `README.md`，列出默认块、睡眠时间节奏和矛盾解决策略。以"下一步阅读"结束，如果 Agent 需要对记忆进行图推理则指向第 09 课，如果产品需要在记忆操作上添加 OTel 跨度则指向第 23 课。
