给定一个任务类别（预期持续时间、步骤数量、触发类型、延迟预算），选择运行时形态。

决策：

1. < 30 秒，用户等待 -> **请求-响应**。
2. 渐进式用户体验或语音 -> **流式**。
3. 数分钟到数小时，用户不等待 -> **基于队列**。
4. 响应外部事件 -> **事件驱动**。
5. 定期维护 -> **cron**。
6. 以上任何情况且重启成本高 -> 添加 **持久执行**。

产出：

1. 您技术栈中的形态框架。
2. 可观测性：OTel GenAI 跨度（第 23 课），接入后端（第 24 课）。
3. 对于队列：DLQ + 重试策略 + 队列深度指标。
4. 对于事件：显式的订阅者注册表 + 重放路径。
5. 对于 cron：锁文件或分布式锁以防止重叠运行。
6. 对于持久：检查点后端 + 恢复语义。

硬性拒绝：

- 为 5 分钟任务使用同步 HTTP。用户挂断；工作者堆积。
- 基于队列但没有 DLQ。失败的任务消失。
- 没有追踪导出的后台工作。失败在用户投诉之前不可见。
- "不需要持久状态，我们会重试。"长周期任务必须使用检查点。

拒绝规则：

- 如果产品有 SLA + 重放需求，拒绝群体拓扑 + 非持久运行时。
- 如果任务受合规约束，拒绝没有审计追踪的事件驱动。
- 如果用户想要 cron 但没有锁，拒绝。重叠的 cron 运行轻则重复工作，重则数据损坏。

输出：运行时框架 + 可观测性钩子 + 包含 SLA、重试策略、检查点选择的 README。以"下一步阅读"结尾，指向第 23 课（OTel）、第 24 课（可观测性）或第 17 课（托管的长运行 Managed Agents）。
