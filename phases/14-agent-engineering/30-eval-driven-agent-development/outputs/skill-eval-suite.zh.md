给定一个智能体产品，构建一个接入 CI 的三层评估套件。

产出：

1. **静态基准层**——至少一个相关基准（用于代码的 SWE-bench Verified、用于工具使用的 BFCL V4、用于网页的 WebArena、用于桌面的 OSWorld、用于通才的 GAIA）。始终报告 + 审核后的分数。
2. **自定义离线层**——至少一个在领域特定维度（事实准确性、语气、范围、拒绝质量）上评分的 LLM 评判者评分标准。至少一个基于执行的用例，探测智能体运行后的实际状态。至少一个带有黄金路径的基于轨迹的用例。
3. **在线评估层**——会话回放、护栏触发的告警、通过 OTel GenAI 跨度（第 23 课）的每步成本/延迟追踪。
4. **评估器-优化器运行器**——将智能体包裹在提议/评判/精炼中，带有轮次上限。
5. **CI 门控**——在 >=5% 与基线相比的回归时使构建失败。随时间追踪基线。
6. **用例映射**——阶段 14 课程中的每个护栏和每个学到的规则至少有一个用例。

硬性拒绝：

- 没有基线的评估套件。没有参照就无法检测回归。
- 在事实性任务上没有外部依据的 LLM 评判者。需要 CRITIC 模式（第 05 课）。
- 没有固定种子或快照状态的不稳定用例。误报会侵蚀团队对评估的信任。

拒绝规则：

- 如果用户想要"仅快乐路径"，拒绝。每个故障模式（第 26 课）都应该有一个用例。
- 如果用户想要"没有 CI 门控"，对于面向付费用户的产品拒绝。否则评估漂移不可见。
- 如果用户想要"全部使用 LLM 评判者"，在事实性和合规任务上拒绝。那里需要基于执行或程序化的评估者。

输出：`cases/benchmarks/`、`cases/custom/`、`cases/online/`、`runner.py`、`ci_gate.py`、`README.md`，解释评分标准、基线和阶段 14 映射表。以"下一步阅读"结尾，指向第 24 课（可观测性）、第 26 课（故障模式）或第 23 课（OTel）以了解基础设施。
