# Anthropic 的工作流模式：简约优于复杂

> Schluntz 和 Zhang（Anthropic，2024 年 12 月）区分了工作流（预定义路径）和 Agent（动态工具使用）。五种工作流模式覆盖了大多数场景。从直接 API 调用开始。只有当步骤无法预测时才添加 Agent。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 01（Agent 循环）
**时间：** 约 60 分钟

## 学习目标

- 说出 Anthropic 的五种工作流模式：提示词链、路由、并行化、编排器-工作者、评估器-优化器。
- 解释 Agent 与工作流的区别以及各自工程成本。
- 识别何时选择工作流优于 Agent（反之亦然）。
- 使用标准库针对脚本化 LLM 实现全部五种模式。

## 问题

团队为本来只需要单一函数调用的问题，就去使用多 Agent 框架。代价是真实的：框架增加了层，掩盖了提示词，隐藏了控制流，并引入了不必要的复杂性。Schluntz 和 Zhang 在 2024 年 12 月的文章是业界引用最多的逆向推动：从简单开始，只有在复杂性值得其成本时才增加。

## 概念

### 工作流 vs Agent

- **工作流。** 通过预定义代码路径编排的 LLM 和工具。工程师拥有图结构。
- **Agent。** LLM 动态地自行指挥其工具并采取自己的步骤。模型拥有图结构。

两者各有用武之地。工作流更便宜、更快、更易调试。Agent 解锁了开放式问题，但使失败模式更难推理。

### 增强型 LLM

所有五种模式的基础：一个具备三种能力的 LLM——搜索（检索）、工具（动作）、记忆（持久化）。任何 API 调用都可以使用这些能力。

### 五种模式

1. **提示词链。** 调用 1 的输出是调用 2 的输入。当任务有清晰的线性分解时使用。步骤之间可选编程式门控。

2. **路由。** 一个分类器 LLM 选择调用哪个下游 LLM 或工具。当分类不同的输入需要不同处理时使用（一级支持 vs 退款 vs bug 报告 vs 销售）。

3. **并行化。** 并发运行 N 个 LLM 调用，汇总结果。两种形态：分段（不同块）和投票（相同提示词，N 次运行，多数/综合）。

4. **编排器-工作者。** 编排器 LLM 动态决定运行哪些工作者（也是 LLM），并综合它们的输出。类似于 Agent 循环，但编排器不会无限循环。

5. **评估器-优化器。** 一个 LLM 提出答案，另一个 LLM 评估它。迭代直到评估器通过。这是 Self-Refine（第 05 课）的泛化形式。

### 工作流何时胜过 Agent

- **可预测的任务。** 如果你能列举出步骤，你就应该这么做。
- **成本受限的任务。** 工作流有有限的步骤计数；Agent 可能失控。
- **合规受限的任务。** 审计人员希望读取图结构，而不是从轨迹中推断。

### Agent 何时胜过工作流

- **开放式研究。** 当下一步取决于上一步返回的结果时。
- **可变长度的任务。** 几分钟到几小时的工作，步骤数未知。
- **新领域。** 当你还不知道正确的工作流时——先探索，后固化。

### 上下文工程的配套

"面向 AI Agent 的有效上下文工程"（Anthropic 2025）将相邻学科形式化：20 万 token 窗口是一个预算，而非一个容器。包含什么、何时压缩、何时让上下文增长。在阶段 14 关于上下文压缩的课程中有详细讨论（本课程重新编号前的第 06 课）。

## 动手构建

`code/main.py` 针对 `ScriptedLLM` 实现了全部五种工作流模式：

- `prompt_chain(input, steps)` — 顺序执行。
- `route(input, classifier, handlers)` — 分类 + 分发。
- `parallel_vote(prompt, n, aggregator)` — N 次运行，汇总。
- `orchestrator_workers(task, workers)` — 编排器选择工作者。
- `evaluator_optimizer(task, proposer, evaluator, max_iter)` — 循环直到通过。

运行方式：

```
python3 code/main.py
```

每种模式都会打印其跟踪信息。每种模式的代码行数约为 10-15 行；框架的成本以千行计。

## 使用场景

- 对大多数任务使用直接 API 调用。
- 仅当模式确实需要持久状态（LangGraph）、参与者模型并发（AutoGen v0.4）或角色模板（CrewAI）时才使用框架。
- 当你想要 Claude Code 的 harness 形状而又不想重新构建时，选择 Claude Agent SDK。

## 练习

1. 实现带有置信度阈值的路由。低于阈值 -> 升级给人工处理。对于一级支持用例，阈值应设在哪里？
2. 为 `parallel_vote` 添加超时。当一个调用挂起时会发生什么？如何在缺少投票的情况下进行汇总？
3. 将 `evaluator_optimizer` 转换为一个 bandit：跨迭代保留 top-2 输出，这样一个好的结果不会因为后续的不良结果而被覆盖。
4. 将提示词链与路由结合：一个路由器选择三条链之一。测量 token 成本与单一大型提示词替代方案的对比。
5. 选择你的一个生产功能。绘制工作流图。计算步骤数。Agent 在这里真的会更好吗？

## 关键术语

| 术语 | 人们说的意思 | 实际含义 |
|------|----------------|------------------------|
| 工作流 | "预定义流程" | 工程师拥有的 LLM 和工具调用图 |
| Agent | "自主 AI" | 模型拥有的图；动态工具指挥 |
| 增强型 LLM | "带工具的 LLM" | LLM + 搜索 + 工具 + 记忆；原子单元 |
| 提示词链 | "顺序调用" | 调用 N 的输出是调用 N+1 的输入 |
| 路由 | "分类器分发" | 选择哪个链/模型处理输入 |
| 并行化 | "扇出" | N 个并发调用；通过分段或投票汇总 |
| 编排器-工作者 | "调度者 Agent" | 编排器 LLM 动态选择专门 LLM |
| 评估器-优化器 | "提议者 + 评判者" | 迭代直到评估器通过；Self-Refine 的泛化 |
