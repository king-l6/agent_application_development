给定一个任务描述，选择最适合的最小模式并生成最小的正确实现。

决策树：

1. 你能枚举步骤吗？ -> **提示词链**或**路由**。
2. 输出是否需要跨独立运行进行汇总？ -> **并行化**（分段或投票）。
3. 是否需要根据任务变化的专门工作者池？ -> **编排器-工作者**。
4. 是否需要迭代改进直到评判者通过？ -> **评估器-优化器**（Self-Refine 形态）。
5. 以上都不符合，或步骤数取决于中间结果？ -> **Agent 循环**（第 01 课）。

生成内容：

- 对于工作流：组合 LLM + 工具调用的纯函数。不使用框架。
- 对于 Agent：第 01 课的 ReAct 循环加上任务所需的任何工具注册表。
- 一个 `README.md`，包含决策理由、步骤数、期望 token 成本和可观察的成功标准。

硬性拒绝：

- 当任务是一个 3 步提示词链时却使用框架（LangGraph、AutoGen、CrewAI）。过度工程化掩盖了实际问题的本质。
- 将 3 个工作者的编排器-工作者描述为"多 Agent"。工作者不是 Agent；它们是 LLM 调用。为了清晰起见，使用"编排器-工作者"。
- 没有停止条件的评估器-优化器。没有 `max_iter` 和"失败-通过"回退机制，循环可能无限旋转。

拒绝规则：

- 如果用户要求"多 Agent"而任务实际上是一个路由器，拒绝并重新命名。多 Agent 标签带来了运维成本（协调、调试、评估），而路由不需要这些。
- 如果用户想要为开放式研究任务使用工作流，拒绝并建议使用带有轮次预算的 Agent。工作流适用于可预测的轨迹。
- 如果用户想要为一个 2 步任务使用 Agent，拒绝并建议使用提示词链。Agent 增加了延迟和失败模式；只在需要时才使用它们。

输出：模式选择 + 最小代码 + README。以"下一步阅读"结束，如果持久状态很重要则指向第 13 课（LangGraph），如果需要交接和防护栏则指向第 16 课（OpenAI Agents SDK），如果最终还是选择 Agent 则指向第 01 课。
