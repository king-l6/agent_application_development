# 混合记忆：向量 + 图 + 键值存储（Mem0）

> Mem0（Chhikara 等人，2025）将记忆视为三个并行存储——向量用于语义相似性，键值存储用于快速事实查找，图存储用于实体关系推理。一个评分层在检索时融合三者。这是 2026 年外部记忆的生产标准。

**类型：** 构建
**语言：** Python（标准库）
**前置知识：** 阶段 14 · 07（MemGPT），阶段 14 · 08（Letta 块）
**时间：** 约 75 分钟

## 学习目标

- 解释为什么单一存储（仅向量、仅图、仅键值）不足以作为 Agent 记忆。
- 说出 Mem0 的三种并行存储及各自的优化目标。
- 描述 Mem0 的融合评分——相关性、重要性、时效性——以及为什么它是加权求和而非层级结构。
- 使用标准库实现一个玩具版三存储记忆系统，包含写入全部三个存储的 `add()` 和融合结果的 `search()`。

## 问题

单一存储对于以下三种查询类型中至少两种是错误的：

- **语义相似性**——"上周我们讨论了关于 Agent 漂移的哪些内容？"向量存储胜出；键值存储和图存储无法胜任。
- **事实查找**——"用户的电话号码是多少？"键值存储胜出；向量存储浪费资源，图存储过度设计。
- **关系推理**——"哪些客户共享同一个计费实体？"图存储胜出；向量存储和键值存储无法回答。

生产环境中的 Agent 在一次会话中会发出所有三种类型的查询。单一存储记忆总是对其中的两种是错误的。Mem0 的贡献在于将三者整合在一个统一的 `add`/`search` 接口后面，并带有一个融合它们的评分函数。

## 概念

### 三种并行存储

Mem0（arXiv:2504.19413，2025 年 4 月）在 `add(text, user_id, metadata)` 时的操作：

1. 从文本中提取候选事实（由 LLM 驱动的步骤）。
2. 将每个事实写入向量存储（嵌入向量）用于语义搜索。
3. 将每个事实写入键值存储，以 `(user_id, fact_type, entity)` 为键，用于 O(1) 查找。
4. 将每个事实写入图存储（Mem0g）作为类型化边，用于关系查询。

在 `search(query, user_id)` 时：

1. 向量存储通过嵌入向量的余弦相似度返回 top-k 结果。
2. 键值存储返回基于查询派生的 `(user_id, type, entity)` 键的直接命中结果。
3. 图存储返回从查询实体出发可达的子图。
4. 评分层融合三者。

### 融合评分

```
score = w_relevance * relevance(q, record)
      + w_importance * importance(record)
      + w_recency * recency(record)
```

- **相关性**——向量余弦相似度、键值精确匹配、图路径权重。
- **重要性**——在写入时标记或通过学习获得（某些事实更重要：姓名、ID、策略）。
- **时效性**——自上次写入或读取以来的指数衰减。

权重按产品调整。聊天 Agent 使用更高的 `w_recency`；合规性 Agent 使用更高的 `w_importance`；检索 Agent 使用更高的 `w_relevance`。

### Mem0g 与时间推理

Mem0g 增加了一个冲突检测器。当新事实与现有边矛盾时，现有边被标记为无效但不会被删除。时间查询（"用户三月份所在的城市是哪里？"）会遍历在特定时间有效的子图。

这是 Letta 的失效模式在合规级别上的泛化。

### 基准数据

Mem0 论文报告（2025 年）的数据：

- **LoCoMo**（长形式对话记忆）：91.6
- **LongMemEval**（长周期情景记忆）：93.4
- **BEAM 1M**（100 万 token 记忆基准）：64.1

对比基线（全上下文 128k LLM、扁平向量存储、扁平键值存储）都落后 10 个百分点以上。仅凭基准数据不足以决定选择——运营形态才是关键——但数据表明融合设计并非四舍五入的误差。

### 作用域分类

Mem0 按作用域划分记忆：

- **用户记忆**——跨会话持久化，以 `user_id` 为键。
- **会话记忆**——在线程内持久化。
- **Agent 记忆**——每个 Agent 实例的状态。

每次写入选择一个作用域。检索可以按每个作用域的权重跨作用域查询。不加思考地混合作用域会导致"助手告诉 Alice 关于 Bob 的项目"之类的事故。

### 这种模式的陷阱

- **嵌入漂移。** 向量结果在前一百次查询中看起来很好，但随着语料库的增长会退化。定期重新嵌入使用频率最高的记录。
- **键值模式蔓延。** `(user_id, type, entity)` 看起来很简洁，直到每个团队都添加自己的 `type`。每季度审计一次类型集合。
- **图爆炸。** 一个嘈杂的提取器每消息添加 50 条边。限制每次 `add` 调用的图写入数量；丢弃低置信度的边。

## 动手构建

`code/main.py` 使用标准库实现了三存储模式：

- `VectorStore` — 使用简单的 token 重叠相似度作为嵌入向量的替代。
- `KVStore` — 以 `(user_id, fact_type, entity)` 为键的字典。
- `GraphStore` — 类型化边（subject, relation, object, valid）。
- `Mem0` — 顶层外观类，提供 `add()`、`search()`、融合评分和感知作用域的检索。
- 一个多用户、多会话对话的工作跟踪示例。

运行方式：

```
python3 code/main.py
```

输出显示三条独立的召回路径以及融合后的 top-k 结果。调整 `main()` 顶部的评分权重，观察排序的变化。

## 使用场景

- **Mem0（Apache 2.0 许可）** — 生产就绪。使用 Postgres + Qdrant + Neo4j 自托管，或使用托管云服务。
- **Letta** — 三层核心/回忆/归档架构；可自带向量和图存储后端。
- **Zep** — 商业替代方案，提供时间知识图谱和事实提取功能。
- **自定义构建** — 当需要精确控制提取器（合规场景）或融合权重（时效性占主导的语音 Agent）时使用。

## 练习

1. 用真实的嵌入模型（sentence-transformers、Ollama、OpenAI 嵌入）替换玩具向量相似度。在合成长对话上测量 recall@10。经过 1000 次写入后排名的漂移情况如何？
2. 添加时间查询：`search(query, as_of=timestamp)`。仅返回在该时间或之前有效的记录。哪个存储需要做最多的工作？
3. 实现冲突检测器：如果传入的事实与图边矛盾，使旧边失效并记录两者。用"用户住在柏林" -> "用户住在里斯本"进行测试。
4. 将融合评分器扩展为包含 `user_feedback` 维度（对检索记录点赞）。如何防止作弊（Agent 只返回它已经喜欢的记录）？
5. 阅读 Mem0 文档（`docs.mem0.ai`）。将玩具实现移植为使用 `mem0` 客户端调用。在相同的 20 个测试查询上比较检索质量。
