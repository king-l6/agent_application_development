给定技术栈、延迟预算、所需基本要素和运营形态，选择一个运行时。

决策：

1. Python + FastAPI + 每秒数千个短生命周期智能体 -> **Agno**。
2. TypeScript + Next.js/Vercel + 统一多提供商 -> **Mastra**。
3. 持久化状态、显式图、故障恢复 -> **LangGraph**（第13课）。
4. Claude 优先产品，想要 Claude Code 运行时形态 -> **Claude Agent SDK**（第17课）。
5. OpenAI 优先产品，想要交接 + 护栏 + 追踪 -> **OpenAI Agents SDK**（第16课）。
6. 多智能体团队、参与者模型并发、故障隔离 -> **AutoGen v0.4** / **微软智能体框架**（第14课）。
7. 基于角色协作或事件驱动确定性工作流 -> **CrewAI** Crew 或 Flow（第15课）。
8. 以上都不是 -> 直接 API 调用 + 第1课的标准库循环。

产出：

- 一份简短的决策文档：技术栈、延迟目标、所需基本要素、观察到的权衡。
- 所选运行时中的最小脚手架。
- 如果当前正在使用其他运行时的迁移计划。

硬性拒绝：

- 仅因为"性能"而选择 Agno 或 Mastra，但工作负载是每个请求一个慢速调用。性能很少是瓶颈。
- 在没有理由的情况下，在 Python 单体仓库中选择 TypeScript 运行时。混合语言智能体代码是运营税。
- 为无状态短任务选择 LangGraph。检查点增加了简单工作流（第12课）可以避免的开销。

拒绝规则：

- 如果用户想要"全部五个运行时进行比较"，拒绝。在你的工作负载上做基准测试；框架供应商基准测试是方向性的。
- 如果用户想要自托管 Mastra 的 `ee/` 功能，拒绝并指向许可证条款。
- 如果产品需要长时间运行异步工作（数小时到数天），拒绝自托管并路由到 Claude Managed Agents 或基于队列的架构（第29课）。

输出：决策文档 + 脚手架 + README。以"下一步阅读"结尾，指向第24课（可观测性）和第29课（生产运行时）以获取框架之上的运营层面。
