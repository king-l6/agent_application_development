给定一个产品领域和一个追踪存储，为智能体故障模式生成检测器。

产出：

1. 每种模式的检测器：`hallucinated_action`、`scope_creep`、`cascading_errors`、`context_loss`、`tool_misuse`、`success_hallucination`。
2. 领域特定检测器（例如，对于开发工具："创建了 PR 但没有关联 issue"；对于营销工具："未经确认就对超过 5 个收件人发送邮件"）。
3. 将所有检测器应用于每个追踪并输出分布的标签器。
4. 基于阈值的告警：如果今天 >=5% 的追踪标记了某种模式，则发通知或创建工单。
5. 样本保留：对于每个被标记的追踪，保留输入 + 输出 + 状态快照以供运维人员审查。

硬性拒绝：

- 在生产中每个追踪需要 LLM 调用的检测器。使用基于模式的检测器；将 LLM 判断器保留给抽样审查。
- 仅基于崩溃进行标记。大多数故障产生看起来正常的输出。需要对内容 + 状态进行签名检查。
- 存储被标记的追踪而不进行 PII 脱敏。故障样本携带最糟糕的内容；存储前必须清洗。

拒绝规则：

- 如果用户想要"永久存储所有追踪"，出于成本和合规原因拒绝。按标签 + 速率采样。
- 如果产品没有"已知正常"的基线，拒绝漂移告警。漂移需要参照。
- 如果检测器没有版本管理，拒绝。检测器回归会在没有预警的情况下破坏您的信号。

输出：`detectors.py`、`tagger.py`、`alerts.py`、`retention.py`、`README.md`，解释阈值、保留策略、告警路由。以"下一步阅读"结尾，指向第 24 课（可观测性后端）或第 27 课（提示注入）以了解对抗性故障模式。
