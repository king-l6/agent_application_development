# LangGraph：有状态图与持久化执行

> LangGraph 是 2026 年底层有状态编排的参考实现。智能体即状态机；节点即函数；边即转移；状态不可变且在每一步后检查点持久化。从任何故障处精确恢复至中断点。

**类型：** 学习 + 构建
**语言：** Python（标准库）
**前置条件：** 第14阶段·01（智能体循环），第14阶段·12（工作流模式）
**时长：** ~75分钟

## 学习目标

- 描述 LangGraph 的核心模型：具有不可变状态、函数节点、条件边和步后检查点的状态机。
- 列举文档强调的四个能力：持久化执行、流式输出、人工介入、全面记忆。
- 解释 LangGraph 支持的三种编排拓扑：监督者、点对点（群集）、层次化（嵌套子图）。
- 使用标准库实现一个具有不可变状态、条件边和检查点/恢复循环的状态图。

## 问题

智能体和工作流共享一个问题：当一个40步的运行在第38步失败时，你希望从第38步恢复，而不是从头开始。二流的状态模型让操作人员在一个假设全新运行的库上拼凑重试逻辑。

LangGraph 的设计答案：状态是一等类型对象，变更显式化，检查点在每个节点后持久化。恢复就是一个 `load_state(session_id)` 调用。

## 概念

### 图

一个图由以下元素定义：

- **状态类型。** 每个节点读取和变更的类型化字典（或 Pydantic 模型）。
- **节点。** 纯函数 `(state) -> state_update`。返回值后更新合并到状态中。
- **边。** 节点间的条件或直接转移。
- **入口和出口。** `START` 和 `END` 哨兵节点标记边界。

示例：一个具有 `classify`、`refund`、`bug`、`sales`、`done` 节点的智能体——一个路由工作流以图的形式呈现。

### 持久化执行

每个节点返回后，运行时序列化状态并写入检查点（SQLite、Postgres、Redis、自定义）。如果在第N步失败，运行时可以调用 `resume(session_id)` 并从第N+1步以精确状态继续。

LangGraph 文档明确列举了关注此特性的生产用户：Klarna、Uber、J.P. Morgan。关键不在图的结构，而在图结构加上检查点使恢复变得廉价。

### 流式输出

每个节点可以产出部分输出。图向调用方流式发送每节点增量事件，使 UI 在图运行时实时更新。

### 人工介入

在节点之间检查和修改状态。实现方式：在关键节点前暂停，向人工展示状态，接受修改，恢复。检查点使这变得简单，因为状态已经序列化。

### 记忆

短期（同一运行内——状态中的对话历史）和长期（跨运行——通过检查点加独立的长期存储持久化）。LangGraph 通过工具与外部记忆系统（Mem0、自定义）集成。

### 三种拓扑

1. **监督者。** 中央路由 LLM 分派给专业子智能体。`create_supervisor()` 在 `langgraph-supervisor` 中（尽管 LangChain 团队在2026年建议通过工具调用直接实现以获得更好的上下文控制）。
2. **群集 / 点对点。** 智能体通过共享工具表面直接交接。无中央路由器。
3. **层次化。** 监督者管理子监督者，以嵌套子图实现。

### 这种模式的失败点

- **检查点太小。** 仅检查点对话轮次会使工具状态和记忆写入无法恢复。完整状态必须序列化。
- **非确定性节点。** 恢复假设节点输入产生相同的状态更新。随机种子、时钟时间、外部 API 必须被捕获。
- **条件边过度使用。** 每条边都是条件边的图是一个无法推理的状态机。优先使用带偶有分支的线性链。

## 构建

`code/main.py` 实现了一个标准库有状态图：

- `State` — 包含 `messages`、`step`、`route`、`output`、`human_approval` 的类型化字典。
- `Node` — 接受状态并返回更新字典的可调用对象。
- `StateGraph` — 节点 + 边 + 条件边 + 运行 + 恢复。
- `SQLiteCheckpointer`（内存模拟）— 每个节点后序列化状态；`load(session_id)` 恢复。
- 演示图：分类 -> 分支（退款 / 缺陷 / 销售） -> 人工门控 -> 发送。

运行：

```
python3 code/main.py
```

跟踪显示第一次运行在人工门控处失败，状态持久化，然后恢复产生最终输出。

## 关键术语

| 术语 | 人们的说法 | 实际含义 |
|------|-----------|---------|
| 状态图（State graph） | "智能体即状态机" | 类型化状态 + 节点 + 边 + 归约器 |
| 检查点（Checkpointer） | "持久化后端" | 每个节点后序列化状态；支持恢复 |
| 归约器（Reducer） | "状态合并器" | 将当前状态与节点更新合并的函数 |
| 条件边（Conditional edge） | "分支" | 由状态的函数选择的边 |
| 子图（Subgraph） | "嵌套图" | 作为另一个图中的节点使用的图 |
| 持久化执行（Durable execution） | "从故障恢复" | 在最后一个成功节点以精确状态重启 |
| 监督者（Supervisor） | "路由 LLM" | 为专业子智能体服务的中央分派器 |
| 群集（Swarm） | "P2P 智能体" | 智能体通过共享工具交接；无中央路由器 |
